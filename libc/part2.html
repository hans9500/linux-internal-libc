Part IIlibc 의 정체 — 세 가지 일

Part I 에서 user-space 와 kernel-space 의 경계를 보았다. 이제 그 경계 위에 서 있는 libc 자체가 무엇을 하는지 본다.

이 Part 의 핵심 주장은 — libc 는 syscall wrapper 의 모음이 아니다. Part I 의 1장에서 잠시 보았듯 libc 가 하는 일은 크게 세 가지로 나뉘는데, 그 중 syscall wrapper 는 전체의 1/3 도 되지 않는다. 나머지 2/3 가 — 추상화 층과 순수 user-space 함수 — 가 libc 의 진짜 가치를 만든다.

각 영역을 차례로 깊이 본다. master 의 실제 코드를 인용해 가며 — glibc 와 musl 의 차이도 비교한다.

9libc 의 3 영역 개관

libc 가 제공하는 함수들을 분류하면 다음 세 가지 영역으로 나뉜다.

┌─────────────────────────────────────────────────────────────────┐
│ libc 의 3 영역                                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  (a) syscall wrapper                                              │
│      kernel 의 syscall 을 C 함수로 감싼 얇은 함수                    │
│      한 syscall = 한 wrapper 가 일반적                              │
│      예: read, write, open, close, fork, ...                     │
│                                                                  │
│  (b) 추상화 층                                                     │
│      여러 syscall 의 조합 + 상태 관리 + buffer + caching            │
│      한 호출이 0~여러 번의 syscall 을 일으킬 수 있음                  │
│      예: malloc, free, printf, fopen, pthread_create, ...        │
│                                                                  │
│  (c) 순수 user-space 함수                                          │
│      kernel 을 한 번도 호출하지 않는 함수                            │
│      CPU 안에서 메모리 연산/산술만 수행                              │
│      예: strcpy, strlen, memcpy, sin, cos, qsort, ...            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

영역별 코드 라인 비율

glibc 의 실제 master 소스를 보면 — 디렉토리 구조 자체가 이 세 영역의 비율을 잘 보여준다.

[glibc master 의 주요 디렉토리 구조]

  glibc/
  ├── sysdeps/             ← syscall wrapper + 아키텍처별 코드
  │   └── unix/sysv/linux/  → Linux syscall wrapper 들
  │
  ├── stdio-common/        ← (b) 추상화 — stdio 의 일부
  ├── stdio/               ← (b) 추상화 — stdio
  ├── malloc/              ← (b) 추상화 — malloc
  ├── nptl/                ← (b) 추상화 — pthread
  ├── elf/                 ← (b) 추상화 — dynamic linker
  ├── resolv/              ← (b) 추상화 — DNS 해석
  ├── locale/              ← (b) 추상화 — locale
  │
  ├── string/              ← (c) 순수 — strcpy, memcpy, ...
  ├── math/                ← (c) 순수 — sin, cos, sqrt, ...
  ├── stdlib/              ← (c) 순수 — qsort, bsearch, atoi, ...
  ├── ctype/               ← (c) 순수 — isalpha, tolower, ...
  ├── time/                ← 혼합 — gettimeofday 는 (a), strftime 은 (c)
  └── wctype/, wcsmbs/     ← (c) 순수 — wide char 처리

실제로 라인 수를 세어 보면 (master 기준):

[glibc 코드 라인 비율 (대략)]

  syscall wrapper (sysdeps/unix/sysv/linux 등):     ~ 15-20%
  추상화 층 (malloc, stdio, nptl, elf, 등):          ~ 35-40%
  순수 user-space (string, math, stdlib, 등):       ~ 30-35%
  나머지 (locale, iconv, debug 등):                  ~ 10-15%

대부분 사람의 인상과 달리 — syscall wrapper 는 가장 작은 영역이다. libc 의 진짜 가치는 그 위에 쌓인 추상화 층과, kernel 과 무관한 순수 user-space 함수들에 있다.

영역 판별의 기준 — "syscall 횟수"

세 영역을 명료히 구분하는 기준은 — 한 호출이 일으키는 syscall 의 횟수다.

┌──────────────────────────────────────────────────────────────┐
│ 호출당 syscall 횟수로 본 영역 분류                              │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  영역 (a) syscall wrapper                                      │
│    호출 1번 = syscall 1번 (보통)                                 │
│    예외: 인자 변환이나 EINTR retry 가 있을 때 0~2 번             │
│                                                               │
│  영역 (b) 추상화 층                                              │
│    호출 1번 = syscall 0번 ~ 수십 번                              │
│    상황에 따라 다름                                              │
│    예: malloc — 보통 0 번, 가끔 1 번 (brk/mmap)                  │
│       printf — 보통 0 번, buffer full 시 1 번                  │
│       getaddrinfo — 5~20 번 (여러 syscall 조합)                 │
│       pthread_create — 1~5 번 (clone + 보조 syscall)            │
│                                                               │
│  영역 (c) 순수 user-space                                       │
│    호출 1번 = syscall 0번 (항상)                                 │
│                                                               │
└──────────────────────────────────────────────────────────────┘

이 분류가 흥미로운 이유는 — 응용 프로그램의 성능 분석에 직접 영향을 준다.

[성능 분석 시사점]

영역 (a) 함수가 핫스팟이면 → kernel 진입 비용이 큼 → 호출 줄이거나 batch
영역 (b) 함수가 핫스팟이면 → 분석 필요 (buffer 효과인가, 호출 횟수인가)
영역 (c) 함수가 핫스팟이면 → CPU 연산 자체가 비싼 것 → 알고리즘/SIMD 검토

strace 로 syscall 을 보면 영역 (a) 와 영역 (b) 의 효과는 보이지만, 영역 (c) 는 보이지 않는다. perf 같은 CPU 프로파일러로만 영역 (c) 를 본다.

함수가 어느 영역인지 빠르게 판별하는 법

man page 의 section 번호가 첫 단서다.

[man section 으로 보는 영역 판별]

man 2  →  System call (= kernel 함수)
          → 그 이름의 libc 함수는 보통 영역 (a) wrapper
          
man 3  →  Library function (= libc 의 user-space 함수)
          → 영역 (b) 또는 (c)
          → 더 세분화는 함수의 본체를 봐야 함
# [user-space - bash]
$ man -k read | grep "(2)"        # syscall 인 것들
read (2)              - read from a file descriptor
readv (2)             - read or write data into multiple buffers
...

$ man -k read | grep "(3)"        # 라이브러리 함수인 것들
fread (3)             - binary stream input/output
readdir (3)           - read a directory
...

fread 가 section 3 이라는 사실이 결정적이다 — 그건 kernel 함수가 아니라 glibc 의 추상화 함수 (read syscall 위에 buffer 를 얹은 것) 라는 뜻이다.

[man 2 vs man 3 의 의미]

  read   (man 2)  →  glibc 의 read() = kernel sys_read 의 wrapper
                     → 영역 (a)
                     → man 2 가 kernel ABI 문서
                     
  fread  (man 3)  →  glibc 의 fread() = stdio buffer 위의 함수
                     → 영역 (b)
                     → man 3 가 libc API 문서
                     → 내부에서 read() 호출 (또는 buffer 사용)

이 장의 결론

libc 는 단일한 무언가가 아니다. 세 영역의 함수들이 한 라이브러리 안에 모여 있을 뿐이다. 각 영역은:

  • 목적이 다르다 — wrapper, 추상화, 순수 연산
  • kernel 과의 관계가 다르다 — 1번, 0~여러 번, 0번
  • 성능 특성이 다르다 — kernel 진입 비용, buffer 효과, CPU 연산
  • 출처가 다르다 — kernel ABI, libc API, ISO C/POSIX 표준 함수

이 다음 4 개의 장에서 — 각 영역의 본체를 master 의 실제 코드를 보면서 깊이 분석한다. 다음 장은 영역 (a) syscall wrapper 의 실제다.

10syscall wrapper 의 실제 — glibc / musl 비교

이 장은 libc 의 영역 (a) — syscall wrapper — 의 본체를 본다. glibc 와 musl 두 구현체의 master 소스를 직접 인용하면서 — 그 디자인 철학의 차이도 함께 본다.

musl 의 write() — 가장 단순한 형태

먼저 musl 부터 본다. musl 의 철학은 "단순함과 표준 준수" 다. src/unistd/write.c 의 본체는 단 7 줄.

// [user-space - musl src/unistd/write.c master]
#include <unistd.h>
#include "syscall.h"

ssize_t write(int fd, const void *buf, size_t count)
{
    return syscall_cp(SYS_write, fd, buf, count);
}

이게 전부다. syscall_cp 매크로 한 줄로 syscall 호출하고 반환. cancellation point 지원이 _cp 접미사로 표시되어 있다.

syscall_cp 매크로는 — __syscall_cp 라는 어셈블리 함수를 호출하는 wrapper 다. 그 본체는 아키텍처별로 src/internal/<arch>/syscall_cp.s 에 있다 (x86_64):

; [user-space - musl src/thread/x86_64/syscall_cp.s master]
.global __syscall_cp_asm
.hidden __syscall_cp_asm
.type   __syscall_cp_asm,@function
__syscall_cp_asm:
__cp_begin:
    mov (%rdi), %eax        ; cancel flag 체크 (rdi = pointer to cancel state)
    test %eax, %eax
    jnz __cp_cancel         ; cancel 요청되었으면 cancel 처리
    mov %rdi, %r11          ; r11 에 cancel pointer 보관
    mov %rsi, %rax          ; syscall 번호 (2nd arg)
    mov %rdx, %rdi          ; 3rd arg → rdi (1st syscall arg)
    mov %rcx, %rsi          ; 4th arg → rsi (2nd syscall arg)
    mov %r8,  %rdx          ; 5th arg → rdx (3rd syscall arg)
    mov %r9,  %r10          ; 6th arg → r10 (4th syscall arg, rcx 우회)
    mov 8(%rsp), %r8        ; stack 의 7th arg → r8 (5th syscall arg)
    mov 16(%rsp), %r9       ; stack 의 8th arg → r9 (6th syscall arg)
    mov %r11, 8(%rsp)       ; r11 (cancel ptr) 을 stack 에 복원
    syscall                  ; ← user/kernel 전환
__cp_end:
    ret
__cp_cancel:
    jmp __cancel

핵심은 — 인자 register 재배치 + r10 변환 + syscall 명령어. Part I 의 5장에서 본 그 그대로다. cancel 처리가 추가되어 있을 뿐.

__syscall_cp_asm 의 인자 배치가 특이한데 — 첫 인자 (rdi) 가 syscall 번호가 아니라 cancel 상태 pointer 다. 두 번째 인자 (rsi) 가 syscall 번호. 이건 cancel handling 위한 디자인이다 — 시스템 호출 전에 thread 의 cancel flag 를 확인하고, 호출 중 cancel 이 발생하면 __cancel 로 점프.

cancel 처리가 필요 없는 syscall 은 더 단순한 __syscall 을 쓴다. musl 의 src/internal/x86_64/syscall.s:

; [user-space - musl src/internal/x86_64/syscall.s master]
.global __syscall
.hidden __syscall
.type __syscall,@function
__syscall:
    movq %rdi, %rax              ; syscall 번호
    movq %rsi, %rdi              ; arg1
    movq %rdx, %rsi              ; arg2
    movq %rcx, %rdx              ; arg3
    movq %r8,  %r10              ; arg4 (rcx → r10 으로!)
    movq %r9,  %r8               ; arg5
    movq 8(%rsp), %r9            ; arg6 (stack 에서)
    syscall                       ; ← kernel 진입
    ret                           ; 반환 (rax 그대로)

10 줄짜리 어셈블리. musl 의 디자인이 얼마나 단순한지 보여준다.

glibc 의 write() — 더 복잡한 구조

glibc 의 sysdeps/unix/sysv/linux/write.c 도 본체는 비슷하지만 매크로의 깊이가 훨씬 많다.

// [user-space - glibc sysdeps/unix/sysv/linux/write.c master]
#include <unistd.h>
#include <sysdep-cancel.h>

/* Write NBYTES of BUF to FD.  Return the number written, or -1.  */
ssize_t __libc_write (int fd, const void *buf, size_t nbytes)
{
    return SYSCALL_CANCEL (write, fd, buf, nbytes);
}
libc_hidden_def (__libc_write)

weak_alias (__libc_write, __write)
libc_hidden_weak (__write)
weak_alias (__libc_write, write)
libc_hidden_weak (write)

세 개의 심볼이 정의된다 — __libc_write, __write, write. 그러나 함수 본체는 하나 (__libc_write) 다. __write 와 write 는 weak_alias 매크로로 만들어진 같은 함수의 다른 이름이다 — 별개 함수가 아니라 단일 심볼의 여러 이름. 왜 이렇게 많은 별칭이 필요한가?

[glibc 의 다중 별칭 패턴]

  write              → application 이 부르는 표준 이름
                     → weak alias, 사용자가 redefine 가능
                     
  __write            → glibc 내부에서 부르는 이름
                     → 사용자가 write 를 redefine 해도 영향 안 받음
                     
  __libc_write       → 더 내부적인 이름
                     → 일부 시스템에서 cancel point 우회용

이 다중 별칭은 — glibc 의 ABI 안정성과 사용자 코드의 자유를 동시에 보장하기 위한 디자인이다. 사용자가 write 를 자기 함수로 정의해서 LD_PRELOAD 로 가로채도 — glibc 내부는 __write 를 쓰므로 영향이 없다.

SYSCALL_CANCEL 매크로의 전개를 따라가면 결국 INLINE_SYSCALL 또는 INTERNAL_SYSCALL 에 도달한다. master 의 sysdeps/unix/sysv/linux/sysdep.h:

// [user-space - glibc sysdep.h master 단순화]
#define INLINE_SYSCALL(name, nr, args...)                       \
    ({                                                          \
        unsigned long int resultvar = INTERNAL_SYSCALL (name, , nr, args); \
        if (__glibc_unlikely (INTERNAL_SYSCALL_ERROR_P (resultvar, ))) \
        {                                                       \
            __set_errno (INTERNAL_SYSCALL_ERRNO (resultvar, )); \
            resultvar = (unsigned long int) -1;                 \
        }                                                       \
        (long int) resultvar;                                   \
    })

이 매크로가 — Part I 5장의 "libc wrapper 가 하는 일" 다섯 단계를 정확히 수행한다.

[INLINE_SYSCALL 매크로의 실제 일]

1. INTERNAL_SYSCALL (name, ...) 
   → syscall 명령어 실행, 결과를 resultvar 에 저장

2. INTERNAL_SYSCALL_ERROR_P (resultvar, )
   → -4095 ~ -1 범위인지 검사

3. 에러면 __set_errno()
   → __errno_location() 의 위치에 -resultvar 저장
   
4. 에러면 resultvar = -1
   → 사용자에게 표준 에러 반환값 -1

5. (long int) resultvar 반환

아키텍처별 INTERNAL_SYSCALL — x86_64

INTERNAL_SYSCALL 의 아키텍처별 구현이 syscall 명령어를 호출하는 곳이다. master 의 sysdeps/unix/sysv/linux/x86_64/sysdep.h :

// [user-space - glibc x86_64 sysdep.h master 단순화]
#define INTERNAL_SYSCALL(name, err, nr, args...)                \
    internal_syscall##nr (SYS_ify (name), err, args)

#define internal_syscall3(number, err, arg1, arg2, arg3)        \
    ({                                                          \
        unsigned long int resultvar;                            \
        TYPEFY (arg3, __arg3) = ARGIFY (arg3);                  \
        TYPEFY (arg2, __arg2) = ARGIFY (arg2);                  \
        TYPEFY (arg1, __arg1) = ARGIFY (arg1);                  \
        register TYPEFY (arg3, _a3) asm ("rdx") = __arg3;       \
        register TYPEFY (arg2, _a2) asm ("rsi") = __arg2;       \
        register TYPEFY (arg1, _a1) asm ("rdi") = __arg1;       \
        asm volatile (                                          \
            "syscall\n\t"                                       \
            : "=a" (resultvar)                                  \
            : "0" (number), "r" (_a1), "r" (_a2), "r" (_a3)     \
            : "memory", REGISTERS_CLOBBERED_BY_SYSCALL);        \
        (long int) resultvar;                                   \
    })

핵심을 본다.

[internal_syscall3 의 분해]

input:
  number = SYS_write 의 번호 (= 1)
  arg1, arg2, arg3 = fd, buf, count

inline assembly:
  - rax 에 number 로드 (constraint "0" 가 rax 강제)
  - rdi 에 arg1 (constraint "r" 와 register 선언)
  - rsi 에 arg2
  - rdx 에 arg3
  - syscall 명령어
  - rax 의 값을 resultvar 로 (constraint "=a")
  - clobber: memory (메모리 변경 가능), rcx, r11 (syscall 이 destroy)

REGISTERS_CLOBBERED_BY_SYSCALL 매크로는 "cc", "r11", "cx" 를 포함한다. 이 점이 매우 중요 — Part I 5장에서 본 "syscall 명령어가 rcx, r11 을 destroy" 가 여기 명시되어 있다.

glibc 의 syscall() — 임의의 syscall 호출

특수한 wrapper 가 하나 더 있다 — syscall(2). 이건 임의의 syscall 번호를 받아서 호출하는 generic 함수다. master 의 sysdeps/unix/sysv/linux/x86_64/syscall.S:

; [user-space - glibc syscall.S master 인용]
; Usage: long syscall (syscall_number, arg1, arg2, arg3, arg4, arg5, arg6)
; We need to do some arg shifting, the syscall_number will be in rax.

ENTRY (syscall)
    movq %rdi, %rax     ; Syscall number -> rax
    movq %rsi, %rdi     ; shift arg1 - arg5
    movq %rdx, %rsi
    movq %rcx, %rdx
    movq %r8,  %r10     ; arg4: rcx → r10
    movq %r9,  %r8      ; arg5
    movq 8(%rsp), %r9   ; arg6 is on the stack
    syscall             ; Do the system call
    cmpq $-4095, %rax   ; Check %rax for error
    jae SYSCALL_ERROR_LABEL  ; Jump to error handler if error
    ret                 ; Return to caller
PSEUDO_END (syscall)

이 코드 한 블록에 Part I 5장 의 모든 핵심 사실이 압축되어 있다.

[syscall.S 의 모든 사실]

1. C 함수 호출 ABI 로 인자 도착:
   rdi = syscall 번호 (1st C arg)
   rsi, rdx, rcx, r8, r9 = arg1..arg5 (2nd~6th C arg)
   stack[8] = arg6 (7th C arg)

2. syscall ABI 로 재배치:
   rdi → rax (syscall 번호)
   rsi → rdi (arg1)
   rdx → rsi (arg2)
   rcx → rdx (arg3)
   r8  → r10 (arg4, rcx 우회)
   r9  → r8  (arg5)
   stack[8] → r9 (arg6)

3. syscall 명령어 실행 (CPU 가 권한 상승)

4. 결과가 rax 에 있음
   -4095 보다 작으면 ($-4095 와 비교해서 jae = jump if above or equal) 에러
   → SYSCALL_ERROR_LABEL 로 점프 (errno 설정 + -1 반환)
   
5. 정상이면 ret (rax 그대로 반환)

이 함수가 사용자가 직접 syscall(SYS_write, 1, "hi", 2) 같은 식으로 호출하는 generic wrapper 다. 개별 wrapper (read, write, ...) 가 없는 새로운 syscall (예: pidfd_open) 을 호출할 때 자주 쓰인다.

// [user-space - 사용자가 syscall(2) 를 직접 부르는 예]
#include <sys/syscall.h>
#include <unistd.h>

int main(void) {
    // pidfd_open 의 wrapper 가 없을 수 있음 (libc 가 오래된 경우)
    int pidfd = syscall(SYS_pidfd_open, 1234, 0);
    return 0;
}

이 자동 생성 시스템이 — glibc 에 수백 개의 wrapper 가 있어도 유지보수가 가능한 이유다. 새 syscall 이 kernel 에 추가되면 syscalls.list 에 한 줄만 추가하면 된다.

본질은 같지만 — glibc 는 매크로로 처리하고, musl 은 별도 함수로 처리한다. musl 의 방식이 더 직관적이지만, glibc 의 매크로 방식이 더 최적화 여지가 있다 (inline 됨).

직접 syscall 을 호출하는 코드 — wrapper 없이

가끔은 wrapper 없이 syscall 을 직접 호출하고 싶을 때가 있다. inline assembly 로 가능하다.

// [user-space - inline assembly 로 직접 syscall 호출]
#include <unistd.h>
#include <sys/syscall.h>

int main(void) {
    long ret;
    const char *msg = "hello, kernel\n";
    
    /* x86_64 inline assembly */
    asm volatile (
        "syscall"
        : "=a" (ret)                  /* output: rax */
        : "0" (SYS_write),            /* input: rax = SYS_write */
          "D" (1),                    /* rdi = fd */
          "S" (msg),                  /* rsi = buf */
          "d" (14)                    /* rdx = count */
        : "rcx", "r11", "memory"      /* clobbered */
    );
    
    return 0;
}

GCC 의 constraint 들의 의미:

[GCC inline assembly constraints — syscall 호출]

  "=a" → output 이 rax 에
  "0"  → input 이 첫 번째 output 과 같은 register (= rax)
  "D"  → input 이 rdi 에
  "S"  → input 이 rsi 에
  "d"  → input 이 rdx 에
  
  clobber:
  "rcx", "r11" → syscall 명령어가 망가뜨리는 register
  "memory"     → syscall 이 메모리를 변경할 수 있음
                  (컴파일러가 register 캐싱 무효화)

이 코드를 컴파일하고 실행하면 hello, kernel\n 이 출력된다. libc 의 write() 함수를 한 번도 거치지 않고 kernel 의 sys_write 가 호출된다. errno 설정이나 -1 반환 같은 처리는 없으니, 사용자가 직접 해야 한다.

이 장의 결론

syscall wrapper 의 본체는 — 결국 매우 단순하다.

[모든 syscall wrapper 의 공통 구조]

1. 인자를 정해진 register 에 배치
2. syscall 번호를 rax (또는 x8, a7 등) 에 로드
3. syscall (또는 svc, ecall) 명령어 실행
4. 결과 검사: 에러 범위 (-4095 ~ -1) 면 errno 설정 + -1 반환
5. 정상이면 결과 반환

glibc 는 이걸 매크로로 추상화해서 수백 개의 wrapper 를 자동 생성하고, musl 은 각 wrapper 를 명시적 .c 파일로 짧게 쓴다. 두 접근 다 같은 일을 한다.

다음 장에서는 영역 (b) 의 추상화 층 — 같은 syscall wrapper 들 위에 어떻게 정교한 추상화가 쌓이는지 — 를 본다. 시작은 stdio 의 buffering 메커니즘이다.

11추상화 층의 정체

이 장은 libc 의 영역 (b) — 추상화 층 — 을 본다. syscall wrapper 위에 어떻게 정교한 자료구조와 상태 관리가 쌓이는지, 그래서 한 함수 호출이 syscall 의 횟수와 어떻게 분리되는지를 본다.

추상화 층은 libc 의 가장 흥미로운 영역이다. 사용자 인터페이스와 syscall 사이의 거리가 멀수록 추상화 가치가 크다. 후속 Part 에서 깊이 다룰 주요 추상화 — malloc, stdio, pthread — 의 핵심 아이디어를 미리 본다.

추상화의 공통 패턴

영역 (b) 의 모든 함수는 — syscall wrapper 위에 다음 다섯 가지 중 하나 또는 여러 개의 layer 를 얹는다.

┌─────────────────────────────────────────────────────────────┐
│ 추상화 layer 의 다섯 종류                                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. State (상태 관리)                                          │
│     예: stdio 의 FILE*, malloc 의 arena                       │
│                                                              │
│  2. Buffer (사용자 공간 버퍼링)                                  │
│     예: stdio 의 buffer, getline 의 동적 buffer                │
│                                                              │
│  3. Caching (재사용 캐시)                                       │
│     예: malloc 의 tcache, dlopen 의 라이브러리 캐시              │
│                                                              │
│  4. Multi-syscall composition (여러 syscall 묶음)              │
│     예: getaddrinfo (여러 socket 조작), printf (여러 write)    │
│                                                              │
│  5. Lazy init (지연 초기화)                                     │
│     예: pthread 자료구조, locale, dynamic linker                │
│                                                              │
└─────────────────────────────────────────────────────────────┘

각 layer 가 — 사용자 호출과 syscall 사이의 직접 1:1 관계를 깬다. 호출 1번에 syscall 0번이 될 수도, 여러 번이 될 수도 있다. 이게 영역 (b) 의 정의다.

사례 1 — stdio 의 buffering

가장 친숙한 예. 사용자가 printf("a") 를 100 번 호출하면 syscall 은 몇 번 일어날까?

// [user-space - 일반적인 응용 코드]
for (int i = 0; i < 100; i++) {
    printf("a");
}
fflush(stdout);

직관: "100 번 printf 했으니 syscall 도 100 번?" — 틀렸다. 보통 1번 또는 0번이다.

[printf 의 실제 동작 — stdout 이 fully buffered 일 때]

  Call 1:  printf("a")
            → vfprintf 가 'a' 를 stdout 의 buffer 에 append
            → buffer: "a"
            → syscall 없음
            
  Call 2:  printf("a")
            → buffer: "aa"
            → syscall 없음
            
  Call 3~99: ...
            → buffer: "aaa...a" (99 글자)
            → syscall 없음
            
  Call 100: printf("a")
            → buffer: "aaa...a" (100 글자)
            → syscall 없음 (buffer 가 아직 안 참)
            
  fflush(stdout):
            → write(1, "aaa...a", 100)
            → syscall 1번

printf 호출 100 번이 write syscall 1 번으로 압축된다. buffer 가 그 변환을 수행한다.

vfprintf 의 master 구조

master 의 stdio-common/vfprintf-internal.c 가 vfprintf 의 본체를 가진다. 핵심 흐름을 단순화해서 보면:

_IO_BUFSIZ 가 buffer 크기 — 보통 8192 바이트. 즉 unbuffered stream 도 한 번의 vfprintf 호출 안에서는 stack 의 임시 buffer 에 누적했다가, 마지막에 한 번 write syscall 로 flush.

FILE 의 buffering mode

stdio buffer 의 동작은 stream 의 buffering mode 에 따라 다르다. POSIX 가 세 종류를 정의한다.

┌──────────────────┬──────────────────────────────────────────┐
│ Mode             │ 동작                                       │
├──────────────────┼──────────────────────────────────────────┤
│ Fully buffered   │ buffer 가 차야 flush                       │
│                  │ 또는 명시적 fflush() 호출 시               │
│                  │ → 기본: 파일 stream, stdout (터미널 아닐 때) │
│                  │                                            │
│ Line buffered    │ 줄바꿈 ('\n') 만나면 flush                  │
│                  │ 또는 buffer 가 차면 flush                   │
│                  │ → 기본: stdout (터미널일 때)                │
│                  │                                            │
│ Unbuffered       │ 매 호출마다 즉시 syscall                    │
│                  │ → 기본: stderr                              │
└──────────────────┴──────────────────────────────────────────┘

이 차이가 매우 흥미로운 결과를 만든다. 같은 프로그램이 터미널에서 실행되느냐 파일로 redirect 되느냐에 따라 stdout 의 buffering 이 달라진다.

// [user-space - 같은 코드, 다른 결과]
printf("hello");
sleep(10);
printf(", world\n");
[실행 환경별 동작]

터미널에서 실행:
  stdout = line buffered
  → "hello" 가 즉시 안 보임 (\n 없으니까)
  → 10초 후 "hello, world\n" 한 번에 출력
  
파일로 redirect ($ ./prog > out):
  stdout = fully buffered
  → "hello" 도 안 보임, "hello, world\n" 도 안 보임
  → 프로그램 종료 시 (또는 4KB+ 출력 시) 한 번에 flush
  
stderr 로 출력:
  stderr = unbuffered
  → "hello" 가 즉시 출력
  → 10초 sleep
  → ", world\n" 출력

같은 코드가 환경에 따라 다르게 동작하는 이유는 — stdio 의 buffer 가 환경을 감지해서 mode 를 자동 선택하기 때문이다.

사례 2 — malloc 의 brk/mmap 추상화

malloc 은 더 정교한 추상화의 좋은 예다. malloc(size) 가 호출되어도 — 대부분의 경우 syscall 이 발생하지 않는다.

[malloc 의 일반적 호출 패턴]

  malloc(100):
    → arena 의 free list 에서 100 바이트 chunk 찾음
    → 발견 → 그 chunk 반환 (syscall 없음)
    → 못 찾으면 → brk 또는 mmap 으로 OS 에서 더 큰 영역 받기 (syscall 1번)
                  → 그걸 잘라서 100 바이트 chunk 반환
                  
  free(p):
    → chunk 를 arena 의 free list 로 반환
    → 보통 syscall 없음 (메모리를 OS 에 즉시 안 돌려줌)
    → 가끔: 큰 영역이 모두 free 되면 brk(축소) 또는 munmap (syscall 1번)

이 디자인의 결과로 — 일반적 응용 프로그램이 malloc/free 를 수십만 번 호출해도 syscall 은 수십 번~수백 번 정도다. 비율이 1000:1 이상이 흔하다.

brk vs mmap 의 분기점

glibc 가 OS 에서 메모리를 받을 때 — brk 와 mmap 둘 중 하나를 선택한다. 기준은 요청 크기다.

// [user-space - glibc malloc/malloc.c master 의 분기 로직 단순화]
if (req_size >= mp_.mmap_threshold) {
    /* 큰 요청 → mmap (독립 영역) */
    return mmap_chunk(req_size);
} else {
    /* 작은 요청 → brk (heap 끝 확장) */
    return brk_chunk(req_size);
}

mmap_threshold 의 기본값은 128 KB (DEFAULT_MMAP_THRESHOLD = 128 * 1024). 이 값은 동적으로 조정된다 (free 된 mmap chunk 의 크기에 따라).

사례 3 — getaddrinfo 의 multi-syscall composition

getaddrinfo(3) 는 호스트 이름을 IP 주소로 변환하는 함수다. 한 호출 안에서 여러 syscall 이 일어난다.

// [user-space - 호스트네임 → IP 변환]
struct addrinfo hints = { .ai_family = AF_INET };
struct addrinfo *res;

getaddrinfo("example.com", "80", &hints, &res);

이 한 줄의 결과로 — 내부에서 다음 syscall 들이 일어난다.

[getaddrinfo 가 실제 호출하는 syscall 들 (단순화)]

1. /etc/nsswitch.conf 읽기
   open + read + close      (3 syscall)
   
2. /etc/resolv.conf 읽기 (DNS 서버 정보)
   open + read + close      (3 syscall)
   
3. /etc/hosts 읽기 (먼저 정적 매핑 확인)
   open + read + close      (3 syscall)
   
4. UDP 소켓 생성 + DNS 쿼리
   socket + connect + sendto + recvfrom + close   (5+ syscall)
   
5. (실패 시) 다른 DNS 서버 시도, TCP 폴백 등
   추가 syscall ...

Total: 보통 15~30 syscall

strace -e trace=network ./prog 로 보면 실제로 수십 개의 syscall 이 일어나는 것을 볼 수 있다.

이게 — 단일 함수 호출이 매우 많은 syscall 의 sequence 로 변환되는 추상화의 극단이다. 사용자는 getaddrinfo("example.com", ...) 한 줄로 끝나지만, 내부는 매우 복잡한 multi-step 프로토콜이다.

사례 4 — pthread_create 의 lazy init

pthread_create 는 multi-syscall composition 이면서 동시에 lazy init 의 좋은 예다.

// [user-space - 쓰레드 생성]
pthread_t t;
pthread_create(&t, NULL, worker, NULL);

이 한 줄이 내부에서 일으키는 syscall:

[pthread_create 의 syscall 시퀀스 - glibc master 단순화]

1. allocate_stack:
   mmap (새 thread 의 stack + TLS 영역 할당)     1 syscall
   
2. 새 thread 의 TLS 초기화:
   mprotect (guard page 설정)                    1 syscall
   
3. clone (실제 thread 생성):
   clone (CLONE_VM | CLONE_FS | CLONE_FILES | ...)  1 syscall
   ↑ 새 thread 의 시작점, stack pointer 지정
   
4. (필요 시) rseq 등록:
   rseq                                          (1 syscall)

Total: 보통 3~4 syscall

그리고 첫 호출 시에는 추가로 lazy 초기화가 일어난다.

[첫 pthread_create 시 추가 init]

- NPTL 자료구조 초기화
- atfork handler 준비
- libgcc_s.so 로드 (cancellation support)
- ...

→ 이건 first call 한 번만 발생

이 lazy init 패턴이 영역 (b) 의 또 다른 흔한 특징이다. 첫 호출이 무겁고 이후 호출은 가벼움. malloc 의 첫 호출도 비슷하다 — heap 영역 확보, tcache 초기화 등.

printf 100번 vs write 100번 — 실험

이 장을 마무리하면서, 추상화의 가치를 측정하는 실험을 보자. 같은 출력을 두 가지 방법으로:

// [user-space - 방법 1: printf 100번 (추상화 사용)]
#include <stdio.h>
int main(void) {
    for (int i = 0; i < 100; i++) {
        printf("a");
    }
    fflush(stdout);
    return 0;
}

// [user-space - 방법 2: write 100번 (추상화 우회)]
#include <unistd.h>
int main(void) {
    for (int i = 0; i < 100; i++) {
        write(1, "a", 1);
    }
    return 0;
}

strace -c 로 측정하면:

[strace 결과 — 방법 1 (printf)]

  % time    seconds  usecs/call   calls    syscall
  -------   --------  ----------  ------   --------
   ...
   100.00    ...         ...         1     write   ← 1번!
   ...

[strace 결과 — 방법 2 (write)]

  % time    seconds  usecs/call   calls    syscall
  -------   --------  ----------  ------   --------
   ...
   100.00    ...         ...       100     write   ← 100번!

같은 결과 ("a" 100 번) 를 출력하지만 — syscall 횟수가 100 배 차이. 각 syscall 이 약 100ns ~ 1μs 정도 걸리므로, 100 번의 차이가 누적되면 약 10μs ~ 100μs 의 latency 차이가 된다.

대용량 출력에서는 이 차이가 더 커진다. 백만 개 정수를 출력하는 경우 — printf 는 약 100~250 번의 write syscall (8KB buffer 가 차는 횟수) 로 끝나지만, putc 또는 write 를 직접 쓰면 백만 번이 된다.

이 장의 결론

추상화 층은 — libc 가 syscall wrapper 의 모음이 아니라는 사실을 증명하는 영역이다.

[영역 (b) 의 가치 = syscall 횟수의 변화량]

순진한 strlen — 한 글자씩 읽기

strlen 의 가장 단순한 구현은:

// [user-space - 순진한 strlen]
size_t strlen(const char *s) {
    size_t n = 0;
    while (*s != '\0') {
        s++;
        n++;
    }
    return n;
}

이게 동작하지만, 매우 느리다. 한 글자씩 읽으니 long string 에서는 비효율.

glibc 의 strlen — 워드 단위 트릭

master 의 string/strlen.c 는 훨씬 영리한 portable 구현을 가진다. 핵심은 — 한 번에 한 word (4 또는 8 바이트) 씩 읽고, 그 안에 NUL 바이트가 있는지를 산술 연산으로 검사.

// [user-space - glibc string/strlen.c master 단순화]
size_t strlen(const char *str)
{
    const char *char_ptr;
    const unsigned long int *longword_ptr;
    unsigned long int longword, himagic, lomagic;

    /* 1. align 안 맞은 처음 몇 바이트는 byte-by-byte */
    for (char_ptr = str;
         ((unsigned long int) char_ptr & (sizeof(longword) - 1)) != 0;
         ++char_ptr) {
        if (*char_ptr == '\0')
            return char_ptr - str;
    }

    longword_ptr = (unsigned long int *) char_ptr;
    
    /* magic numbers: 64-bit 일 때 */
    himagic = 0x80808080L;
    lomagic = 0x01010101L;
    if (sizeof(longword) > 4) {
        himagic = ((himagic << 16) << 16) | himagic;
        lomagic = ((lomagic << 16) << 16) | lomagic;
    }

    /* 2. word 단위로 한꺼번에 검사 */
    for (;;) {
        longword = *longword_ptr++;
        
        /* trick: longword 안에 0 바이트가 있는지 산술로 검사 */
        if (((longword - lomagic) & ~longword & himagic) != 0) {
            /* NUL 발견! 정확한 위치 찾기 */
            const char *cp = (const char *) (longword_ptr - 1);
            
            if (cp[0] == 0) return cp - str;
            if (cp[1] == 0) return cp - str + 1;
            if (cp[2] == 0) return cp - str + 2;
            if (cp[3] == 0) return cp - str + 3;
            if (sizeof(longword) > 4) {
                if (cp[4] == 0) return cp - str + 4;
                if (cp[5] == 0) return cp - str + 5;
                if (cp[6] == 0) return cp - str + 6;
                if (cp[7] == 0) return cp - str + 7;
            }
        }
    }
}

이 magic-number 트릭이 핵심이다.

[strlen 의 magic number 트릭 — bit 단위 NUL 검출]

  himagic = 0x80808080 80808080  (각 바이트의 MSB 만 set)
  lomagic = 0x01010101 01010101  (각 바이트의 LSB 만 set)
  
  검사 식: (longword - lomagic) & ~longword & himagic
  
  어떻게 동작하는가?
  
  - 한 바이트 b 가 0 이면:
      b - 0x01 = 0xff (under-flow!)
      ~b = 0xff
      0xff & 0xff & 0x80 = 0x80 (non-zero!)  ← NUL 검출
      
  - 한 바이트 b 가 0이 아니면 (예: b = 'a' = 0x61):
      b - 0x01 = 0x60
      ~b = 0x9e
      0x60 & 0x9e & 0x80 = 0x00              ← 정상

이 트릭으로 8 바이트를 한 번에 검사한다. 산술 명령어 몇 개로 — 8 개의 분기 명령어 (한 글자씩 비교) 를 대신한다. CPU 의 pipeline 활용도 훨씬 좋아져서, 일반적으로 4~8 배 빨라진다.

strcpy 의 비슷한 트릭

strcpy 도 비슷한 word-level 트릭을 쓴다. master 의 string/strcpy.c 는 사실 매우 단순하다 — 실제 본체는 __stpcpy 에 위임한다.

// [user-space - glibc string/strcpy.c master]
char *
STRCPY (char *dest, const char *src)
{
    __stpcpy (dest, src);
    return dest;
}

__stpcpy 가 진짜 본체다. master 의 string/stpcpy.c 를 보면 — 그것 역시 portable 버전은 단순하고, SIMD 가속된 빠른 버전들은 sysdeps/<arch>/multiarch/ 에 있다 (다음 13장에서 자세히 다룬다).

흥미로운 점은 — strcpy 가 strlen 한 후 memcpy 하는 단순한 조합이라는 통념이 master 에 없다는 점이다. master 의 stpcpy 는 직접 byte-by-byte (또는 word-by-word) 로 복사하면서 NUL 도 함께 찾는다. 별개 두 단계로 나누는 것보다 캐시 친화적.

수학 함수 — sin, cos, sqrt

<math.h> 의 함수들은 모두 영역 (c) 다. CPU 의 부동소수점 명령어 또는 알고리즘만으로 결과를 계산한다.

[수학 함수의 구현 전략]

  sqrt
    → CPU 의 SQRTSD/SQRTSS 명령어 (x86_64) 또는 직접 알고리즘
    → 단일 instruction 으로 끝날 수도 있음
    
  sin, cos, tan
    → Taylor 또는 Chebyshev 다항식 근사
    → CPU 의 FSIN, FCOS (x87) 또는 software 구현
    → glibc 는 보통 software (정확도 보장 위해)
    
  log, exp, pow
    → 다항식 근사 + bit manipulation
    → 매우 빠른 hand-tuned 구현
    
  ceil, floor, round
    → CPU 의 ROUND 명령어 또는 bit trick

glibc 의 sin 의 master 구현은 — IBM Accurate Mathematical Library (IBMlib) 이 원조다. master 의 sysdeps/ieee754/dbl-64/s_sin.c 에서 볼 수 있다. 수백 줄의 복잡한 코드인데 — 정확한 IEEE 754 결과를 보장하기 위해.

        /* |x| < 2, sin(x) = cos(x - pi/2) 변환 */
        ...
    }

}

크기에 따라 다른 알고리즘으로 분기. 작은 값은 단순한 근사로 끝내고, 큰 값은 복잡한 변환을 한다. 이게 영역 (c) 의 디자인 — kernel 진입 없이 CPU 와 메모리만으로 정확한 결과를 만든다.

qsort 의 본체

qsort 도 영역 (c) 다. master 의 stdlib/qsort.c 가 본체. 이름과 달리 glibc 의 qsort 는 quicksort 가 아니다 — mergesort 가 기본이고, 메모리 할당 실패 시 heapsort 로 fallback 한다.

(이 디자인은 흥미로운 역사를 가진다. 2023년 10월에 quicksort/heapsort 기반의 introsort 로 변경되었다가, 2024년 1월에 다시 mergesort 로 되돌려졌다. 이유는 — POSIX 와 C 표준이 stability 를 요구하지 않지만, 많은 프로그램이 mergesort 의 stability 동작에 의존하고 있어서 — "Hyrum's Law" 의 적용. master 는 stability 를 유지하기로 결정.)

// [user-space - glibc stdlib/qsort.c master 단순화]
void __qsort_r (void *const pbase, size_t total_elems, size_t size,
                __compar_d_fn_t cmp, void *arg)
{
    size_t total_size = total_elems * size;
    char *tmp = NULL;
    
    /* 적당한 크기면 mergesort */
    if (size <= MAX_THRESH || (tmp = malloc (total_size)) != NULL) {
        qsort_r_mergesort (pbase, total_elems, size, cmp, arg, tmp);
    } else {
        /* malloc 실패 시 heapsort 로 fallback */
        if (!qsort_r_malloc (pbase, total_elems, size, cmp, arg, total_size))
            heapsort_r (pbase, total_elems - 1, size, cmp, arg);
    }
}
libc_hidden_def (__qsort_r)
weak_alias (__qsort_r, qsort_r)

매우 흥미로운 점은 — glibc qsort 는 메모리를 추가 할당한다는 것. mergesort 가 추가 메모리를 요구하므로 malloc(total_size) 으로 보조 buffer 를 잡는다. 메모리 할당이 실패하면 — heapsort 로 fallback. heapsort 는 in-place 라 추가 메모리가 불필요.

그래서 — qsort 는 보통 영역 (c) 지만, 가끔 malloc 호출로 영역 (b) 가 되기도 한다. 매우 큰 배열을 정렬하면 — 그 크기만큼의 임시 메모리가 잡힌다.

이 fallback 디자인이 stability 의 미묘한 문제를 만든다. mergesort 는 stable 이지만 heapsort 는 unstable. 즉 — 일반적인 정렬은 stable 이지만, 시스템에 메모리가 부족하면 같은 입력의 같은 코드가 unstable 한 결과를 줄 수 있다. master 의 알려진 이슈.

ctype 함수 — 매우 빠른 lookup

isalpha, isdigit, tolower, toupper 등은 매우 빠른 lookup table 로 구현되어 있다.

// [user-space - glibc ctype 의 아이디어 단순화]

/* 256 바이트 table, 각 바이트가 문자의 속성 비트들 */
extern const unsigned short int **__ctype_b_loc(void);

#define _ISdigit  0x0800
#define _ISalpha  0x0400
#define _ISspace  0x2000
...

#define isdigit(c) ((__ctype_b_loc())[c] & _ISdigit)
#define isalpha(c) ((__ctype_b_loc())[c] & _ISalpha)

함수 호출도 아닌 매크로 표현이라 — 컴파일러가 inline 으로 펼치면 lookup table 한 번 + bit AND 한 번이 전부. 1~2 nanosecond 안에 끝난다.

__ctype_b_loc() 는 — locale 마다 다른 table 을 가리키는 함수다. 이 부분이 약간의 추상화 (locale-aware) 를 만든다.

locale-aware vs locale-free 의 구분

영역 (c) 안에서도 두 가지 변형이 있다.

┌─────────────────────────────────────────────────────────────┐
│ locale 의존 / 비의존 구분                                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│ Locale-free (어디서나 같은 결과):                                │
│   strlen, strcpy, strncpy                                    │
│   memcpy, memmove, memset                                    │
│   atoi, atol (단순한 변환)                                     │
│   sin, cos, sqrt                                             │
│                                                              │
│ Locale-aware (locale 에 따라 결과 다름):                        │
│   strcoll  ← strcmp 의 locale 인식 버전                       │
│   strxfrm                                                    │
│   tolower, toupper (특정 locale 에서)                          │
│   strtod, strtol (decimal separator 차이)                    │
│   strftime, strptime                                         │
│   isalpha (UTF-8 locale 에서)                                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘

locale-aware 함수는 — LC_* 환경 변수나 setlocale() 호출에 따라 동작이 달라진다. 예를 들어 독일 locale 에서는 "ä" 도 isalpha 로 true 가 반환된다.

// [user-space - locale 의 영향 데모]
#include <ctype.h>
#include <locale.h>
#include <stdio.h>

int main(void) {
    char c = 0xE4;   // 'ä' in Latin-1
    
    printf("Default: %d\n", isalpha((unsigned char) c));  // 0
    
    setlocale(LC_CTYPE, "de_DE.ISO-8859-1");
    printf("German: %d\n", isalpha((unsigned char) c));   // non-zero
    
    return 0;
}

이 변동성 때문에 — locale-aware 함수는 약간의 추상화 cost 가 있다. lookup table 의 indirection 한 번이 추가된다. 그래도 여전히 nanosecond 단위로 끝나서 영역 (c) 다.

영역 (c) 의 가치 — 표준화와 이식성

영역 (c) 의 함수들의 진짜 가치는 — 사실 속도가 아니다. 표준화된 API 라는 점이다.

[영역 (c) 함수의 가치]

  1. 표준화된 API
     ISO C 표준, POSIX 표준
     어느 libc 구현체에서도 같은 시그니처와 동작
     
  2. 이식성
     Linux, BSD, macOS, Windows (MSVC) 모두 동일
     코드를 컴파일러만 바꿔서 옮길 수 있음
     
  3. 검증된 정확도
     수십 년간 사용되며 코너 케이스가 모두 처리됨
     특히 수학 함수의 IEEE 754 준수
     
  4. 최적화의 집약
     SIMD, alignment, branch prediction 모두 활용
     사용자가 직접 짜는 것보다 훨씬 빠름
     
  5. 보안 검증
     buffer overflow, UB, race 등 코너 케이스가 검증됨

특히 4번이 다음 장의 주제다 — 같은 strlen, memcpy 라도 SIMD 로 가속된 버전이 훨씬 빠르다. glibc 의 모든 string 함수에 SIMD 가속이 적용되어 있다.

이 장의 결론

영역 (c) 의 함수는 — libc 의 가장 단순한 영역이지만, 가장 자주 쓰이는 영역이기도 하다. 모든 C 프로그램이 strlen, strcpy, memcpy 를 쓰고, 많은 프로그램이 sin, cos, qsort 를 쓴다.

[영역 (c) 의 본질]

  ✓ kernel 진입 없음 (syscall 0번)
  ✓ 결과가 결정적 (locale 빼고)
  ✓ ISO C / POSIX 표준에 정의됨
  ✓ 매우 빠름 (nanosecond 단위)
  ✓ 다른 영역의 빌딩 블록 (printf 안에서도 호출됨)

다음 장에서는 — 이 함수들이 어떻게 SIMD 명령어로 가속되어, glibc 가 어떻게 런타임에 가장 빠른 구현을 선택하는지를 본다. ifunc 메커니즘과 함께.

13SIMD 가속된 libc 함수

이 장은 영역 (c) 의 핵심 성능 최적화 — SIMD (Single Instruction Multiple Data) 가속과 ifunc (indirect function) 메커니즘을 본다. 같은 memcpy, strlen 라는 함수 이름이지만, 실행 시점에 CPU 가 지원하는 가장 빠른 구현이 선택되는 영리한 디자인이다.

SIMD 의 정체

SIMD 는 한 명령어로 여러 데이터를 동시에 처리하는 CPU 기능이다. x86 의 SSE/AVX, ARM 의 NEON 같은 명령어 셋이 SIMD 다.

[Scalar vs SIMD 비교 — 8 바이트 복사]

  Scalar (일반 명령어):
    mov  byte [dst+0], byte [src+0]
    mov  byte [dst+1], byte [src+1]
    mov  byte [dst+2], byte [src+2]
    mov  byte [dst+3], byte [src+3]
    mov  byte [dst+4], byte [src+4]
    mov  byte [dst+5], byte [src+5]
    mov  byte [dst+6], byte [src+6]
    mov  byte [dst+7], byte [src+7]
    → 8개 명령어

  SIMD (SSE2):
    movdqu xmm0, [src]      ; 128-bit (16 바이트) 한 번에 로드
    movdqu [dst], xmm0      ; 한 번에 저장
    → 2개 명령어로 16 바이트 처리

  AVX (256-bit):
    vmovdqu ymm0, [src]     ; 32 바이트 한 번에
    vmovdqu [dst], ymm0
    → 2개 명령어로 32 바이트 처리
  
  AVX-512 (512-bit):
    vmovdqu zmm0, [src]     ; 64 바이트 한 번에
    vmovdqu [dst], zmm0
    → 2개 명령어로 64 바이트 처리

memcpy, memset, strlen 같은 bulk 데이터 처리 함수가 SIMD 의 가장 큰 수혜자다. 한 cycle 에 16~64 바이트를 처리할 수 있게 된다.

SIMD 명령어 셋의 종류

x86 과 ARM 의 SIMD 명령어 셋을 정리하면:

┌──────────────┬─────────────────┬───────────────────────────┐
│ 아키텍처      │ SIMD 명령어 셋    │ 폭                          │
├──────────────┼─────────────────┼───────────────────────────┤
│ x86          │ MMX (옛)         │ 64-bit                    │
│              │ SSE             │ 128-bit                   │
│              │ SSE2 / SSE3 /   │ 128-bit                   │
│              │ SSSE3 / SSE4    │ 128-bit                   │
│              │ AVX             │ 256-bit                   │
│              │ AVX2            │ 256-bit (integer 확장)     │
│              │ AVX-512         │ 512-bit                   │
│              │                 │                           │
│ ARM (32)     │ NEON (Advanced  │ 128-bit                   │
│              │  SIMD)          │                           │
│              │                 │                           │
│ ARM64        │ NEON (필수)      │ 128-bit                   │
│              │ SVE             │ 128-bit ~ 2048-bit (가변)  │
│              │ SVE2            │ 가변                        │
└──────────────┴─────────────────┴───────────────────────────┘

문제는 — 모든 CPU 가 모든 명령어 셋을 지원하지는 않는다. 2010년 이전 x86 CPU 는 AVX 가 없다. 2020년 이전 일반 소비자 CPU 는 AVX-512 가 없다. ARM64 의 SVE 는 매우 최근 (서버급 CPU) 에만 있다.

binary 가 한 가지 명령어 셋만 사용하도록 컴파일된다면 — 옛 CPU 에서는 동작 안 하고, 새 CPU 의 빠른 명령어를 활용 못 한다. 이 문제를 어떻게 해결할까?

해결책 — ifunc (Indirect Function)

GNU 의 답은 ifunc 다. STT_GNU_IFUNC 라는 ELF 심볼 타입을 도입해서 — 한 함수 이름이 실행 시점에 여러 구현 중 하나로 분기되게 한다.

[ifunc 의 핵심 아이디어]

  binary 에 여러 구현이 다 들어 있음:
    memcpy_generic     (모든 CPU 에서 동작)
    memcpy_sse2        (SSE2 지원 CPU 만)
    memcpy_avx         (AVX 지원 CPU 만)
    memcpy_avx512      (AVX-512 지원 CPU 만)
    
  + resolver 함수 하나:
    memcpy_resolver()  → 현재 CPU 의 기능을 검사하고
                         가장 빠른 구현을 반환
    
  사용자가 memcpy() 호출하면:
    프로그램 시작 시 dynamic linker 가 resolver 호출
    PLT 에 선택된 구현의 주소가 채워짐
    이후 모든 호출이 그 구현을 직접 호출

이 메커니즘으로 — 같은 binary 가 모든 CPU 에서 최적 성능을 낸다. 옛 CPU 면 generic, 새 CPU 면 AVX-512. 사용자 코드는 변경 불필요.

ifunc 의 동작 예제

binutils 2.20.1 / glibc 2.11.1 이상에서 도입된 기능이다 (master 인용):

// [user-space - ifunc 사용 예]
#include <stddef.h>

/* 여러 구현 */
void foo_c(unsigned *data, size_t len)       { /* generic */ }
void foo_sse42(unsigned *data, size_t len)   { /* SSE4.2 */ }
void foo_avx2(unsigned *data, size_t len)    { /* AVX2 */ }

/* CPU 기능 확인 */
extern int cpu_has_sse42(void);
extern int cpu_has_avx2(void);

/* resolver */
static void *resolve_foo(void) {
    if (cpu_has_avx2())  return foo_avx2;
    if (cpu_has_sse42()) return foo_sse42;
    return foo_c;
}

/* foo 는 ifunc — resolver 가 실제 주소 결정 */
void foo(unsigned *data, size_t len)
    __attribute__((ifunc("resolve_foo")));

이 binary 가 link 되면, ELF 에 STT_GNU_IFUNC 타입의 foo 심볼이 들어간다. 실행 시점:

[ifunc 의 실행 흐름]

  1. dynamic linker (ld-linux.so) 가 binary 를 로드
  
  2. STT_GNU_IFUNC 심볼들을 찾음
  
  3. 각 심볼에 대해 resolver 함수 호출
       resolve_foo() 가 호출됨
       → CPU 검사
       → foo_avx2 의 주소 반환 (예)
  
  4. PLT (Procedure Linkage Table) 의 foo 항목을 그 주소로 패치
  
  5. 사용자 코드가 foo() 호출 시:
       PLT 거쳐서 foo_avx2 가 직접 호출
       resolver 는 다시 안 부름

이 모든 일이 프로그램 시작 시 한 번만 일어난다. 이후 함수 호출은 — 일반 함수 호출과 같은 비용. ifunc 의 overhead 가 거의 zero.

glibc 의 ifunc 활용

glibc 는 — 성능 critical 한 거의 모든 string/memory 함수에 ifunc 를 적용한다. master 의 sysdeps/x86_64/multiarch/ 디렉토리를 보면:

[glibc master sysdeps/x86_64/multiarch/ 의 일부]

  memcpy.c                       ← ifunc resolver (memmove.S 의 구현 재사용)
  memmove-sse2-unaligned-erms.S  ← SSE2 (memcpy 도 이걸 사용)
  memmove-ssse3.S                ← SSSE3
  memmove-avx-unaligned-erms.S   ← AVX
  memmove-avx512-no-vzeroupper.S ← AVX-512
  memmove-vec-unaligned-erms.S   ← 공통 구현 (다른 .S 들이 include)
  
  memset.c                       ← ifunc resolver
  memset-sse2-unaligned-erms.S   ← SSE2
  memset-avx2-unaligned-erms.S   ← AVX2
  memset-avx512-no-vzeroupper.S  ← AVX-512
  
  strlen.c                       ← ifunc resolver
  strlen-sse2.S
  strlen-avx2.S
  strlen-avx2-rtm.S
  strlen-evex.S                  ← AVX-512 EVEX 인코딩
  ...

흥미로운 사실 — memcpy 의 구현은 사실 memmove 와 공유된다. 두 함수의 차이는 — memmove 가 overlapping 영역도 안전하게 처리한다는 것뿐. master 는 memmove-*-unaligned-erms.S 들이 memcpy / memmove 모두에 쓰이도록 통합했다. 그래서 memcpy-*.S 라는 이름의 파일은 거의 없고, memmove-*.S 가 둘 다 담당.

각 함수가 4~7 가지의 구현을 갖는다. CPU 에 따라 다른 구현이 선택된다.

memcpy.c 의 resolver 본체 (master 단순화):

// [user-space - glibc sysdeps/x86_64/multiarch/memcpy.c master 단순화]
#define SYMBOL_NAME memcpy
#include "ifunc-memmove.h"

libc_ifunc_redirected(__redirect_memcpy, __new_memcpy,
                      IFUNC_SELECTOR());

static inline void *IFUNC_SELECTOR(void)
{
    const struct cpu_features *cpu_features = __get_cpu_features();
    
    /* AVX-512 지원? */
    if (CPU_FEATURE_USABLE_P(cpu_features, AVX512F)
        && !CPU_FEATURES_ARCH_P(cpu_features, Prefer_No_AVX512))
        return OPTIMIZE(avx512_no_vzeroupper);
    
    /* AVX 지원? */
    if (CPU_FEATURE_USABLE_P(cpu_features, AVX))
        return OPTIMIZE(avx_unaligned_erms);
    
    /* SSSE3? */
    if (CPU_FEATURE_USABLE_P(cpu_features, SSSE3))
        return OPTIMIZE(ssse3);
    
    /* fallback: SSE2 (모든 x86_64 가 지원) */
    return OPTIMIZE(sse2_unaligned_erms);
}

매우 정교한 분기. CPU_FEATURES_ARCH_P 는 CPU 가 명령어를 지원하더라도 그 모델에서 — 그 명령어가 실제로 빠른가를 본다. 일부 CPU (특히 옛 AMD) 는 AVX-512 가 켜져 있지만 느려서 안 쓰는 게 낫다.

CPU 기능 검사 — __get_cpu_features

__get_cpu_features 는 — 프로그램 시작 시 한 번 CPU 의 CPUID 명령어 (x86) 또는 AT_HWCAP (ARM) 를 읽어서 결과를 캐싱한 것이다.

// [user-space - glibc sysdeps/x86/cpu-features.h master 단순화]
struct cpu_features {
    struct cpuid_features cpuid[COMMON_CPUID_INDEX_MAX];
    unsigned int basic;
    unsigned int kind;
    unsigned int model;
    unsigned int family;
    
    /* preferred features (CPU 마다 더 빠른 것) */
    unsigned int preferred[PREFERRED_FEATURE_INDEX_MAX];
    
    ...
};

const struct cpu_features *__get_cpu_features(void);

x86 의 경우 — glibc 가 부팅 초기에 CPUID 명령어를 여러 번 호출해서 — feature flag 들을 모두 읽는다. SSE, SSE2, SSSE3, AVX, AVX2, AVX-512 등 수십 가지의 비트가 설정된다. 그 결과를 전역 변수에 저장하고 — 이후 ifunc resolver 들이 그것을 읽는다.

ARM64 는 비슷한 일을 AT_HWCAP auxiliary vector (Part I 8장 vDSO 부분 참조) 에서 한다.

실제 성능 차이

이 모든 노력이 — 실제로 얼마나 빠른가? 측정값 (대략):

[memcpy 의 구현별 성능 (1KB 복사 기준)]

  byte-by-byte loop:     ~ 1000 ns
  word-by-word loop:     ~ 130 ns
  SSE2 (16B at once):    ~ 50 ns
  AVX (32B):             ~ 30 ns
  AVX-512 (64B):         ~ 20 ns
  
  byte-by-byte 대비 AVX-512:  약 50배 빠름
[strlen 의 구현별 성능 (1KB 문자열)]

  byte-by-byte:          ~ 1000 ns
  word-magic-number:     ~ 130 ns
  SSE2 pcmpeqb + pmovmskb: ~ 30 ns
  AVX2 vpcmpeqb:         ~ 18 ns
  
  → SIMD 가 약 50~60배 빠름

이 차이는 — 응용 프로그램이 데이터 처리 위주이면 매우 큰 영향이다. JSON 파서, 텍스트 검색, 데이터베이스 — 모두 string 함수에 의존하고, 그 성능이 ifunc 덕분에 자동으로 최적화된다.

SIMD strlen 의 핵심 아이디어

SIMD strlen 이 어떻게 빠른지 잠깐 본다. SSE2 의 경우:

[SSE2 strlen 의 idea]

  1. 16 바이트 (XMM register 폭) 씩 로드
  
  2. pcmpeqb 명령어: 각 바이트를 0 과 비교
       16 바이트 → 16 개의 비교 결과 (각 바이트가 0xff 또는 0x00)
  
  3. pmovmskb 명령어: 각 바이트의 MSB 를 모아 16-bit 마스크 생성
       → 만약 mask = 0 이면 16 바이트 안에 NUL 없음
       → mask != 0 이면 그 안에 NUL 있음, bsf 로 위치 찾기
  
  4. NUL 없으면 다음 16 바이트 로드, 반복
; [user-space - SSE2 strlen 의 핵심 부분 어셈블리]
; rdi = string 시작 주소

    pxor    xmm0, xmm0          ; xmm0 = 16 개의 0 바이트

.loop:
    movdqa  xmm1, [rdi]         ; 16 바이트 로드 (aligned)
    pcmpeqb xmm1, xmm0          ; 각 바이트를 0 과 비교
    pmovmskb eax, xmm1          ; 16개 비교 결과를 16-bit 마스크로
    test    eax, eax            ; mask == 0?
    jnz     .found              ; non-zero 면 NUL 발견
    add     rdi, 16
    jmp     .loop

.found:
    bsf     eax, eax            ; 첫 set bit 위치 (= NUL 위치)
    add     rdi, rax            ; NUL 의 정확한 주소
    sub     rdi, start          ; 길이 계산
    ret

3개의 명령어 (movdqa, pcmpeqb, pmovmskb) 로 16 바이트의 NUL 검사를 끝낸다. byte-by-byte 의 16분의 1 사이클 안에 — pipeline 효율까지 고려하면 30배 가까이 빠르다.

ARM64 NEON 의 비슷한 접근

ARM64 NEON 도 같은 패턴이다. master 의 sysdeps/aarch64/strlen.S 에는:

[ARM64 NEON strlen 의 idea]

  1. ld1 명령어로 16 바이트 (Q register 폭) 로드
  
  2. cmeq 명령어: 각 바이트와 0 비교
       → 결과는 byte 별 0x00 또는 0xff
  
  3. shrn 명령어: 결과를 압축
       16 바이트 결과 → 8 바이트 mask
  
  4. fmov 로 mask 를 GPR 로 이동
  
  5. NUL 없으면 다음 16 바이트, 반복

x86 의 pmovmskb 같은 직접적인 mask 명령어가 ARM64 에는 없다 (SVE 까지 가야 있음). 그래서 NEON 의 strlen 은 약간 다른 트릭을 쓰지만 — 본질은 동일.

SIMD 의 약점 — 짧은 데이터의 overhead

SIMD 가 항상 빠른 건 아니다. 짧은 데이터 에서는 — SIMD 의 setup overhead 가 더 클 수 있다.

[strlen 의 길이별 성능 비교]

  string 길이     │ scalar 빠름  │ SIMD 빠름   │ 차이
  ────────────────┼─────────────┼─────────────┼──────
  5 bytes         │ ✓           │             │ scalar 가 약 2배 빠름
  20 bytes        │             │ ✓           │ SIMD 가 약 1.5배 빠름
  100 bytes       │             │ ✓           │ SIMD 가 약 5배 빠름
  1000 bytes      │             │ ✓           │ SIMD 가 약 30배 빠름
  10000 bytes     │             │ ✓           │ SIMD 가 약 50배 빠름

그래서 glibc 의 SIMD 함수는 — 시작 부분에 길이 검사를 한다. 길이가 매우 짧으면 byte-by-byte loop 로, 길면 SIMD 본체로.

[glibc 의 SIMD strlen 시작 부분 패턴]

  if (len < SHORT_THRESHOLD) {
      /* 짧으면 byte-by-byte 빠른 path */
      return short_strlen(s);
  }
  
  /* SIMD 본체 */
  ...

이런 hybrid 디자인이 — glibc 의 string 함수가 어떤 입력에서도 빠른 이유다.

이 장의 결론

SIMD 와 ifunc 의 조합은 — libc 가 사용자에게 보이지 않게 제공하는 마법이다.

[ifunc + SIMD 의 가치]

  사용자 코드:
    memcpy(dst, src, n);   ← 단순한 한 줄
    
  실제로는:
    프로그램 시작 시: resolver 가 CPU 감지, 최적 구현 선택
    호출 시:        SIMD 명령어로 한 cycle 에 16~64 바이트 처리
    
  결과:
    같은 binary 가 옛 CPU 와 새 CPU 모두에서 최적
    프로그래머는 plain C 만 쓰면 됨
    글로벌 응용 프로그램 성능이 자동으로 향상

이 디자인이 — 영역 (c) 의 정점이다. 함수 호출의 의미는 그대로 두고, 그 내부 구현을 — runtime 에 가장 빠른 것으로 dispatch.
