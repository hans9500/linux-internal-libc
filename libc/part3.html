Part III_start 부터 main 까지 — program startup 의 깊이

지금까지 본 모든 것은 — 프로그램이 이미 실행 중이라는 가정 아래의 이야기였다. read() 가 호출된다는 건 어떤 함수 안에 있다는 뜻이고, 그 함수는 누군가 부른 것이고, 결국 main 까지 거슬러 올라간다.

그런데 **main 은 누가 부르는가?**

C 표준은 답을 모른다. ISO C 의 5.1.2.2 절은 단지 "구현체가 main 을 호출한다" 고만 적는다. 그 구현체란 — Linux 의 경우 kernel + dynamic linker + glibc 의 협업이다.

이 Part 는 그 협업의 깊이를 본다. 사용자가 ./hello 를 입력한 그 순간부터 — main 의 첫 줄이 실행되기까지 — 무엇이 일어나는지.

14ELF binary 의 구조와 entry point

./hello 라는 파일은 그냥 0과 1의 나열이 아니다. ELF (Executable and Linkable Format) 라는 정밀한 구조를 가진 파일이다. kernel 이 이 구조를 읽어서 — 어느 부분을 어디에 메모리 매핑할지, 어디에서부터 실행을 시작할지를 결정한다.

ELF 의 전체 구조

ELF 파일은 다음 5 가지 영역으로 구성된다.

┌──────────────────────────────────────────────────────┐
│ ELF 파일의 구조                                          │
├──────────────────────────────────────────────────────┤
│                                                       │
│  1. ELF Header                                        │
│     - magic number (0x7F 'E' 'L' 'F')                │
│     - 32/64 bit, endianness, OSABI                    │
│     - file type (executable, shared, relocatable)     │
│     - architecture (x86_64, ARM64, ...)               │
│     - entry point address (e_entry)                  │
│     - program header table 위치                       │
│     - section header table 위치                       │
│                                                       │
│  2. Program Headers (= segments)                     │
│     kernel 이 로딩에 사용                              │
│     - LOAD: 메모리에 매핑할 영역                       │
│     - INTERP: dynamic linker 의 경로                  │
│     - DYNAMIC: 동적 링크 정보                          │
│     - PHDR: program header 자체에 대한 정보            │
│     - NOTE, TLS, GNU_STACK, ...                       │
│                                                       │
│  3. Sections (= 더 세분화된 영역)                       │
│     linker / debugger 가 사용                          │
│     - .text: 코드                                      │
│     - .data: 초기화된 데이터                            │
│     - .bss: 0 으로 초기화될 데이터                       │
│     - .rodata: 읽기 전용 데이터                         │
│     - .init_array, .fini_array: 생성자/소멸자 배열       │
│     - .got, .plt: 동적 링크 테이블                      │
│     - .symtab, .strtab: 심볼 정보                       │
│     - 등                                                │
│                                                       │
│  4. Section Header Table                              │
│     모든 section 의 메타데이터                          │
│                                                       │
│  5. (선택) String Tables 등                            │
│                                                       │
└──────────────────────────────────────────────────────┘

핵심을 강조해 둔다 — segment 와 section 은 다른 개념이다.

[segment vs section]

  segment (program header):
    - kernel 이 메모리에 매핑할 단위
    - 보통 4~8 개
    - 메모리 권한 (R/W/X) 단위
    - readelf -l 로 봄
    
  section:
    - linker 가 다루는 더 세분화된 단위
    - 보통 30~50 개
    - 각각 다른 용도 (.text, .data, .bss, ...)
    - readelf -S 로 봄
    
  관계:
    여러 section 이 하나의 segment 로 묶임
    예: .text + .rodata + .init_array → 하나의 R-X segment
        .data + .bss → 하나의 RW- segment

ELF Header 보기

readelf -h 로 ELF Header 를 본다.

핵심은 **Entry point address: 0x6ab0** 다. kernel 이 ELF 를 로드한 후 — 이 주소로 점프해서 실행을 시작한다.

이 주소는 — 보통 **_start 심볼의 주소** 다. 확인:

# [user-space - bash]
$ readelf -s /bin/ls | grep _start
    66: 0000000000006ab0    47 FUNC    GLOBAL DEFAULT   16 _start
    ↑ 같은 0x6ab0!

Type: DYN 의 의미 — PIE

Type: DYN (Position-Independent Executable file) 이 흥미롭다. 현대 Linux 의 실행 파일은 대부분 PIE (Position-Independent Executable) 다. 옛날엔 Type: EXEC 였다.

[EXEC vs DYN (PIE) 의 차이]

  EXEC (옛 방식):
    - 고정된 주소에 매핑 (예: 0x400000)
    - 매번 같은 주소에서 실행
    - ASLR 적용 안 됨
    - 보안 취약
    
  DYN/PIE (현재):
    - 매번 다른 주소에 매핑 (랜덤화)
    - ASLR (Address Space Layout Randomization) 적용
    - 공유 라이브러리 (.so) 와 같은 형식
    - 보안에 강함

PIE 의 entry point 는 상대 주소다. 0x6ab0 은 — 매핑된 base 로부터의 offset. kernel 이 base 를 결정한 후 base + 0x6ab0 으로 점프한다.

핵심 항목들을 본다:

[중요한 program headers]

  PHDR     : program header 자체 위치 (메타데이터)
  INTERP   : dynamic linker 의 경로
             /lib64/ld-linux-x86-64.so.2
             → ★ kernel 이 이 파일을 추가로 로드한다
             
  LOAD x4  : 메모리에 매핑할 4 개 segment
    R--    : 읽기 전용 (.note, .dynsym, .gnu.hash 등)
    R-E    : 읽기+실행 (.text - 코드!)
    R--    : 읽기 전용 (.rodata, .eh_frame 등)
    RW-    : 읽기+쓰기 (.data, .bss)
             
  DYNAMIC  : 동적 링크 정보 (어떤 .so 가 필요한지 등)
  NOTE     : GNU 빌드 ID, ABI tag 등
  GNU_STACK: stack 의 권한 (RW-, executable stack 방지)
  GNU_RELRO: relocation 후 read-only 로 만들 영역

INTERP 의 존재가 결정적이다. ELF 가 dynamic linker 의 경로를 자기 안에 명시한다 — /lib64/ld-linux-x86-64.so.2. kernel 이 ./hello 를 실행하라고 받아도 — 실제로는 그 dynamic linker 를 먼저 로드하고, dynamic linker 가 ./hello 를 처리한다.

static binary 의 경우 — INTERP 없음

static 으로 link 된 binary 는 INTERP 가 없다.

# [user-space - bash]
$ gcc -static hello.c -o hello_static
$ readelf -l hello_static | grep INTERP
# 아무것도 출력 안 됨

$ readelf -l hello_static | head -20
Elf file type is EXEC (Executable file)
Entry point 0x401b50
...

Type: EXEC (PIE 아님), INTERP 없음. 이 경우 kernel 이 직접 entry point 로 점프 — dynamic linker 우회. 자세한 차이는 22장에서.

  .text           : 프로그램 코드 (실행 가능)
  .data           : 초기화된 전역 변수

  .fini           : 옛 스타일 종료 함수

  .fini_array     : ★ 현대 fini 함수 포인터 배열
  .preinit_array  : init_array 전에 호출될 함수들

  .got            : 글로벌 오프셋 테이블 (Global Offset Table)
  .dynamic        : 동적 link 정보 (필요한 .so 등)
  .dynsym         : 동적 심볼 테이블
  .dynstr         : 동적 심볼 이름 문자열들

이 중 .init_array 가 — 19장에서 깊이 다룰 핵심 메커니즘이다. C++ 의 글로벌 객체 생성자, __attribute__((constructor)) 함수가 모두 여기 등록된다.

.bss 의 trick — 파일에 영역 없음

.bss 의 묘미는 — 파일에 영역이 없다는 것.

# [user-space - bash]
$ readelf -S /bin/ls | grep -E "\.(data|bss)"
  [24] .data             PROGBITS         0x21000     0x20000 0x270      WA
  [25] .bss              NOBITS           0x21280     0x20270 0x1278     WA
                          ↑               ↑          ↑       ↑
                          NOBITS!         시작 주소   파일에선   메모리에선
                                                     0 bytes    이 크기 차지

.bss 의 Type 이 NOBITS — 파일에 실제 데이터를 저장하지 않는다는 뜻. 0 으로 초기화될 거니까 — kernel 이 메모리에 매핑할 때 0 으로 채우면 된다. 디스크 공간 절약.

// [user-space - C code]
int x;          // ← .bss (초기값 없음 = 0 초기화)
int y = 0;      // ← .bss (명시적 0 초기화)
int z = 42;     // ← .data (non-zero 초기화)
char buf[1024]; // ← .bss (큰 배열)

const char *s = "hello";  // 변수 s 자체는 .data
                          // 문자열 "hello" 는 .rodata

큰 zero-filled 배열은 binary 크기를 키우지 않는다. 1 MB 짜리 static char buf[1024*1024] 도 — 디스크에서는 0 bytes, 메모리에서만 1 MB. 이 사실이 c 프로그래머에게 익숙해야 한다.

kernel 의 시각 — 무엇을 알아야 하는가?

kernel 이 ELF 를 로드할 때 필요한 모든 정보는 — ELF Header + Program Headers 에 있다.

[kernel 이 ELF 로딩에 사용하는 정보]

  ELF Header:
    e_machine    : 어느 아키텍처? (x86_64? ARM64?)
    e_type       : EXEC? DYN?
    e_entry      : 어디로 점프?
    e_phoff      : program header 위치
    e_phnum      : program header 개수
    
  Program Headers (각 PT_LOAD 마다):
    p_vaddr      : 가상 주소
    p_paddr      : (보통 무시)
    p_offset     : 파일 내 위치
    p_filesz     : 파일에서의 크기
    p_memsz      : 메모리에서의 크기 (>= filesz; bss 영역)
    p_flags      : 권한 (R/W/X)
    p_align      : 정렬 (보통 페이지 크기 4KB)
    
  PT_INTERP:
    어느 dynamic linker 를 쓸지의 경로
    
  PT_DYNAMIC:
    동적 링크 정보

p_memsz > p_filesz 인 경우 — 그 차이가 .bss 영역이다. kernel 이 그 부분을 0 으로 채운다.

다음 — kernel 이 무엇을 하는가

이 장에서는 ELF binary 의 구조를 봤다. 이건 _static_ 한 정보 — 디스크에 있는 파일의 형태다. 다음 장에서는 — kernel 이 이 파일을 받아 무엇을 하는지 를 본다. execve syscall 의 내부를 따라간다.

15kernel 의 execve 가 하는 일

./hello 를 실행했을 때 — kernel 안에서 일어나는 일을 따라간다. shell 이 execve("./hello", argv, envp) 를 호출하는 순간부터, user-space 의 첫 명령어가 실행되기까지.

shell 의 fork + execve 패턴

먼저 큰 그림. shell 이 명령어를 실행하는 패턴은 fork + execve 다.

[shell 의 일반적인 명령 실행 패턴]

  shell (예: bash):
    fork()              ← 자식 프로세스 생성
       │
       ├──── 부모 (계속 shell)
       │     wait(child)    ← 자식이 끝나기 기다림
       │     ...
       │
       └──── 자식 (새 프로세스)
             execve("./hello", argv, envp)
             ↑ 이 호출은 성공하면 반환 안 함
             ↑ 자식의 메모리가 전부 hello 로 교체됨

fork 는 자기 자신을 복사한 자식을 만든다. 그 자식에서 execve 가 호출되면 — 자식의 모든 메모리가 새 프로그램으로 교체된다. 자식의 PID 는 유지되지만, 그 안의 코드와 데이터는 완전히 새 ELF 로 바뀐다.

execve 가 성공하면 반환하지 않는다. 왜냐하면 — 반환할 메모리 (호출자의 stack) 자체가 새 프로그램으로 덮어쓰여졌으니까.

execve syscall 의 시작 — do_execveat_common

kernel 안의 execve 입구는 master 의 fs/exec.c 에 있다.

// [kernel-space - fs/exec.c master 단순화]
SYSCALL_DEFINE3(execve,
    const char __user *, filename,
    const char __user *const __user *, argv,
    const char __user *const __user *, envp)
{

}

static int do_execve(struct filename *filename,
                     const char __user *const __user *__argv,
                     const char __user *const __user *__envp)
{
    /* ... 인자 변환 ... */
    return do_execveat_common(AT_FDCWD, filename, argv, envp, 0);
}

do_execveat_common 이 진짜 본체. 매우 길지만 핵심 흐름:

// [kernel-space - fs/exec.c do_execveat_common 핵심 단순화]
static int do_execveat_common(int fd, struct filename *filename,
                              struct user_arg_ptr argv,
                              struct user_arg_ptr envp,
                              int flags)
{
    struct linux_binprm *bprm;
    int retval;
    
    /* 1. binprm 구조체 할당 — 실행 정보의 컨테이너 */
    bprm = alloc_bprm(fd, filename, flags);
    
    /* 2. argc, argv, envp 카운트 */
    retval = count(argv, MAX_ARG_STRINGS);
    bprm->argc = retval;
    
    /* 3. 새 stack 위한 임시 vma 준비 */
    retval = bprm_stack_limits(bprm);
    
    /* 4. 파일을 열고 첫 128 바이트 (BINPRM_BUF_SIZE) 읽기 */
    retval = prepare_binprm(bprm);
    
    /* 5. filename, argv, envp 을 새 stack 에 복사 */
    retval = copy_string_kernel(bprm->filename, bprm);
    retval = copy_strings_kernel(bprm->argc, argv, bprm);
    retval = copy_strings_kernel(bprm->envc, envp, bprm);
    
    /* 6. ★ 적합한 binary handler 찾아서 호출 */
    retval = exec_binprm(bprm);
    
    return retval;
}

핵심은 6번 — exec_binprm 가 binary handler 들을 순회하면서, 이 파일을 처리할 수 있는 handler 를 찾는다.

binary handler 들

Linux kernel 은 여러 형식의 binary 를 지원한다. 각 형식마다 별도의 handler 가 있다.

[Linux kernel 의 주요 binary handler 들]

  binfmt_elf.c         : ELF (가장 일반적)
  binfmt_elf_fdpic.c   : ELF FDPIC (no-MMU 시스템)
  binfmt_flat.c        : Flat binary (no-MMU)
  binfmt_script.c      : #! shebang scripts
  binfmt_misc.c        : 사용자 등록 가능한 handler
                         (예: Java .class, Wine 의 .exe)

#!/bin/bash 로 시작하는 스크립트는 — binfmt_script.c 가 처리한다. 그게 첫 줄을 읽어서 interpreter 의 경로를 찾고, 그 interpreter 를 execve 로 다시 실행. 그래서 shell script 가 동작.

ELF binary 는 binfmt_elf.c 가 처리. 이게 — 우리의 관심사.

load_elf_binary — ELF 의 본체 처리

master 의 fs/binfmt_elf.c 의 load_elf_binary 가 ELF 로딩의 본체다. 매우 긴 함수인데, 핵심 흐름:

// [kernel-space - fs/binfmt_elf.c load_elf_binary 핵심 단순화]
static int load_elf_binary(struct linux_binprm *bprm)
{
    struct file *interpreter = NULL;
    unsigned long load_bias = 0;
    int first_pt_load = 1;
    struct elfhdr *elf_ex = (struct elfhdr *)bprm->buf;
    struct elfhdr *interp_elf_ex = NULL;
    
    /* 1. ELF magic number 확인 */
    if (memcmp(elf_ex->e_ident, ELFMAG, SELFMAG) != 0)
        goto out;
    
    /* 2. ELF 타입 확인 (EXEC 또는 DYN 만) */
    if (elf_ex->e_type != ET_EXEC && elf_ex->e_type != ET_DYN)
        goto out;
    
    /* 3. Program Headers 로드 */
    elf_phdata = load_elf_phdrs(elf_ex, bprm->file);
    
    /* 4. INTERP segment 찾기 (dynamic linker 의 경로) */
    for (i = 0; i < elf_ex->e_phnum; i++, elf_ppnt++) {
        if (elf_ppnt->p_type == PT_INTERP) {
            /* dynamic linker 경로 읽기 */
            elf_interpreter = kmalloc(elf_ppnt->p_filesz, GFP_KERNEL);
            elf_read(bprm->file, elf_interpreter,
                     elf_ppnt->p_filesz, elf_ppnt->p_offset);
            /* dynamic linker 파일 열기 */
            interpreter = open_exec(elf_interpreter);
            break;
        }
    }
    
    /* 5. ★ 기존 프로세스의 메모리 모두 해제 */
    retval = begin_new_exec(bprm);
    /* ↑ 이 시점부터 돌아갈 수 없음 */
    
    /* 6. 각 LOAD segment 를 메모리에 mmap */
    for (i = 0; i < elf_ex->e_phnum; i++) {
        if (elf_ppnt->p_type != PT_LOAD) continue;
        
        /* 첫 LOAD 이고 ET_DYN 이면 ASLR base 결정 */
        if (first_pt_load && elf_ex->e_type == ET_DYN)
            load_bias = randomize_va_space();
        
        /* mmap: 파일을 메모리에 매핑 */
        elf_map(bprm->file, load_bias + elf_ppnt->p_vaddr,
                elf_ppnt, /* ... */);
        first_pt_load = 0;
    }
    
    /* 7. BSS 영역 0 으로 채우기 (memsz > filesz) */
    
    /* 8. dynamic linker (interpreter) 도 로드 */
    if (interpreter) {
        elf_entry = load_elf_interp(interp_elf_ex, interpreter, ...);
        /* elf_entry = ld.so 의 _start */
    } else {
        elf_entry = elf_ex->e_entry + load_bias;
        /* static binary: 프로그램의 _start 직접 */
    }
    
    /* 9. argc, argv, envp, auxv 를 새 stack 에 배치 */
    create_elf_tables(bprm, elf_ex, ...);
    
    /* 10. CPU 레지스터 초기화 + entry point 로 점프 */
    START_THREAD(elf_ex, regs, elf_entry, bprm->p);
    
    return 0;
}

매우 많은 일이 일어난다. 단계별로 더 자세히 본다.

단계 5: begin_new_exec — 돌아올 수 없는 다리

5번이 결정적이다. begin_new_exec 가 호출되면 — 기존 프로세스의 모든 메모리가 해제된다. 이후로는 실패해도 원래 프로세스로 돌아갈 수 없다.

[begin_new_exec 의 대표 작업]

  - 기존 mm_struct 해제
  - 새 mm_struct 할당
  - signal handlers 초기화
  - SUID/SGID 처리 (필요 시)
  - close-on-exec 인 fd 들 닫기

close-on-exec 가 흥미롭다 — O_CLOEXEC 플래그로 열린 fd 는 execve 시 자동으로 닫힌다. 보안을 위함이다.

단계 6: LOAD segment 매핑과 ASLR

각 PT_LOAD segment 가 mmap 으로 매핑된다. PIE binary 의 base 는 랜덤하게 결정된다 — ASLR.

[ASLR 의 적용]

  매번 실행마다 다른 base:
  
  실행 1: 0x555555554000 + (.text offset)
  실행 2: 0x555a3f2b1000 + (.text offset)
  실행 3: 0x55b6c1a0a000 + (.text offset)
  
  목적:
  - exploit 의 ROP gadget 주소 예측 불가
  - 같은 binary 도 매번 다른 주소에 매핑됨

단계 8: dynamic linker 도 로드

PT_INTERP 가 있는 경우 — kernel 이 dynamic linker 도 동시에 로드한다. 그리고 점프할 elf_entry 가 **프로그램의 _start 가 아니라 dynamic linker 의 _start** 가 된다.

[엔트리 결정 — dynamic 와 static 의 차이]

  Dynamic linked (PT_INTERP 있음):
    elf_entry = ld.so 의 _start
    → ld.so 가 먼저 실행
    → 라이브러리 로드, relocation 후
    → 프로그램의 _start 로 점프
    
  Static linked (PT_INTERP 없음):
    elf_entry = 프로그램의 _start
    → 직접 프로그램의 _start 실행

단계 10: 점프

마지막 — kernel 이 user-space 로 점프. master 의 arch/x86/kernel/process_64.c 의 start_thread:

// [kernel-space - arch/x86/kernel/process_64.c master 단순화]
void start_thread(struct pt_regs *regs, unsigned long new_ip,
                  unsigned long new_sp)
{
    /* registers 초기화 */
    regs->ip = new_ip;       /* RIP = elf_entry */
    regs->sp = new_sp;       /* RSP = 새 stack 의 top */
    regs->cs = __USER_CS;    /* user-space code segment */
    regs->ss = __USER_DS;    /* user-space data segment */
    /* 다른 GP register 들도 0 으로 (보안) */
}

이 함수가 반환되면 — kernel 의 syscall return 경로가 실행되고, 결국 **iret (또는 sysret) 으로 user-space 의 new_ip 로 점프**한다.

[kernel → user-space 의 첫 점프 시점의 register 상태]

  RIP = elf_entry              (보통 ld.so 의 _start)
  RSP = stack 의 top           (argc 를 가리킴)
  CS  = __USER_CS              (user-space 코드 세그먼트)
  SS  = __USER_DS              (user-space 데이터 세그먼트)
  나머지 GP register = 0       (보안 — kernel 정보 누출 방지)

여기서부터 — user-space 의 첫 명령어가 실행된다. dynamic linker 의 _start 가 실행되기 시작.

16_start 어셈블리 본체 — glibc start.S

_start 가 ELF 의 entry point 다. kernel 이 user-space 로 점프했을 때 처음 실행되는 사용자 영역 코드 — 이건 이전 장의 모든 흐름의 끝이자, 다음 단계 (__libc_start_main) 의 시작이다.

이 장은 glibc 의 _start 본체를 master 인용으로 깊이 본다.

[crt1.o 의 위치]

  /usr/lib/x86_64-linux-gnu/crt1.o     ← static binary 용
  /usr/lib/x86_64-linux-gnu/Scrt1.o    ← PIE 용 (현재 기본)
  /usr/lib/x86_64-linux-gnu/gcrt1.o    ← gprof 프로파일링 용

이 .o 파일들의 source 는 — glibc 의 sysdeps/<arch>/start.S 다.

glibc x86_64 의 start.S — master 본체

master 의 sysdeps/x86_64/start.S 의 _start 본체를 정확히 인용한다.

; [user-space - glibc sysdeps/x86_64/start.S master 인용]

#include <sysdep.h>

ENTRY (_start)
    /* Clearing frame pointer is insufficient, use CFI.  */
    cfi_undefined (rip)
    /* Clear the frame pointer.  The ABI suggests this be done, to mark
       the outermost frame obviously.  */
    xorl %ebp, %ebp

    /* Extract the arguments as encoded on the stack and set up
       the arguments for __libc_start_main (int (*main) (int, char **, char **),
                int argc, char *argv,
                void (*init) (void), void (*fini) (void),
                void (*rtld_fini) (void), void *stack_end).
       The arguments are passed via registers and on the stack:
    main:        %rdi
    argc:        %rsi
    argv:        %rdx
    init:        %rcx
    fini:        %r8
    rtld_fini:   %r9
    stack_end:   stack.    */

    mov %RDX_LP, %R9_LP    /* Address of the shared library termination
                              function.  */
#ifdef __ILP32__
    mov (%rsp), %esi       /* Simulate popping 4-byte argument count.  */
    add $4, %esp
#else
    popq %rsi              /* Pop the argument count.  */
#endif
    /* argv starts just at the current stack top.  */
    mov %RSP_LP, %RDX_LP
    /* Align the stack to a 16 byte boundary to follow the ABI.  */
    and  $~15, %RSP_LP

    /* Push garbage because we push 8 more bytes.  */
    pushq %rax

    /* Provide the highest stack address to the user code (for stacks
       which grow downward).  */
    pushq %rsp

#ifdef PIC
    mov main@GOTPCREL(%rip), %RDI_LP    /* Pass address of main to %rdi.  */
#else
    mov $main, %RDI_LP                   /* Pass address of main to %rdi.  */
#endif

    xorl %ecx, %ecx        /* Pass NULL for init.  */
    xorl %r8d, %r8d        /* Pass NULL for fini.  */

    /* Call the user's main function, and exit with its value.
       But let the libc call main.    */
    call __libc_start_main@PLT

    hlt    /* Crash if somehow `exit' does return.  */
END (_start)

이게 핵심. 30 줄 안 되는 어셈블리. 단계별로 파해친다.

단계 1: frame pointer 초기화

xorl %ebp, %ebp

%ebp (frame pointer) 를 0 으로. ABI 요구사항이다 — 가장 바깥 frame 임을 표시. debugger 가 stack 을 거슬러 올라갈 때 — %ebp = 0 을 만나면 멈춘다.

[왜 frame pointer 를 0 으로?]

  보통 함수 호출 시:
    push %ebp       (호출자의 ebp 저장)
    mov  %esp, %ebp (새 frame pointer)
    
  stack frame 들이 linked list 처럼 연결됨:
    [func3 ebp] → [func2 ebp] → [func1 ebp] → [_start ebp = 0]
                                                ↑
                                                debugger 가 여기서 멈춤
  
  ebp = 0 이 "최상위 frame" 의 marker

cfi_undefined (rip) 도 같은 맥락 — debugger 한테 "여기서부터는 RIP 가 undefined" 라고 알림. unwind 가 여기서 멈춰야 한다는 신호.

단계 2: stack 에서 인자 추출

mov %RDX_LP, %R9_LP    /* Address of the shared library termination function. */
popq %rsi              /* Pop the argument count.  */
mov %RSP_LP, %RDX_LP   /* argv starts just at the current stack top. */

%RDX_LP, %R9_LP, %RSP_LP 같은 매크로는 — 64비트면 %rdx/%r9/%rsp, x32 ABI 면 %edx/%r9d/%esp. 일반적인 x86_64 에서는 64비트 register.

핵심 동작:

[Step 2 동작]

  진입 시 register 상태:
    %rdx = ld.so 가 넘긴 rtld_fini 의 주소
    (kernel 이 직접 호출한 경우엔 0)
    
  진입 시 stack:
    [rsp+0]   = argc           (★ 다음에 pop 됨)
    [rsp+8]   = argv[0]
    [rsp+16]  = argv[1]
    ...
    [rsp+N]   = NULL (argv 종료)
    [rsp+N+8] = envp[0]
    ...

  1. mov %rdx, %r9    : rtld_fini → r9 (6th arg of __libc_start_main)
  2. popq %rsi        : argc → rsi (2nd arg), rsp += 8
  3. mov %rsp, %rdx   : 이제 rsp 는 argv[0] 을 가리킴, rdx = argv

popq 한 후 — %rsp 가 argv[0] 을 가리키게 된다. 그래서 %rdx = %rsp 로 — argv 의 주소 (= argv[0] 의 주소) 가 들어간다.

단계 3: stack 16 바이트 정렬

and  $~15, %RSP_LP    /* Align the stack to a 16 byte boundary */
pushq %rax            /* Push garbage because we push 8 more bytes. */
pushq %rsp            /* Provide the highest stack address */

x86_64 ABI 의 stack alignment 요구사항 — call 명령어 직전에 RSP 가 16 바이트 정렬되어야 함.

[stack alignment 의 trick]

  현재 rsp 가 어떤 값인지 모름 (argc 위 위치)
  
  1. and $~15, %rsp    : rsp 의 하위 4 비트 클리어
                          → 16 바이트 경계로 내림
                          
  2. pushq %rax        : 8 바이트 push (alignment 유지 위한 garbage)
  
  3. pushq %rsp        : stack 의 top 주소를 stack 에 push
                          → __libc_start_main 의 stack_end 인자
                          → "7번째 인자는 stack 위에서"
                          
  4. call ...          : 함수 호출 (rsp 16-aligned 상태)

7개 인자를 register 6개 + stack 1개로 전달하기 위해 — 이 ABI dance 가 필요하다.

단계 4: register 에 인자 배치 + 호출

mov main@GOTPCREL(%rip), %RDI_LP    /* Pass address of main */
xorl %ecx, %ecx                      /* init = NULL */
xorl %r8d, %r8d                      /* fini = NULL */
call __libc_start_main@PLT

x86_64 의 함수 호출 ABI 에 따라:

[__libc_start_main 호출 시 register 상태]

  %rdi = &main             (main 함수의 주소)
  %rsi = argc              (이전 단계에서 pop)
  %rdx = argv              (이전 단계에서 rsp 복사)
  %rcx = NULL (init)       (★ 옛날엔 __libc_csu_init, 지금은 NULL)
  %r8  = NULL (fini)       (★ 옛날엔 __libc_csu_fini, 지금은 NULL)
  %r9  = rtld_fini         (이전 단계에서 rdx → r9)
  stack[0] = stack_end     (이전 단계에서 push %rsp)

흥미로운 점이 **init 과 fini 인자에 NULL 이 들어간다**는 것. 옛 glibc 에서는 __libc_csu_init 와 __libc_csu_fini 가 link 되어 그 주소가 들어갔는데 — glibc 2.34 (2021.8) 부터 제거되었다. 자세한 내용은 18장에서 다룬다.

단계 5: hlt — 도달 불가 보호

call __libc_start_main@PLT
hlt

__libc_start_main 은 exit 를 호출해서 끝나므로 — 정상적으로는 _start 로 돌아오지 않는다. 만약 어쩌다 돌아오면 — hlt 명령어가 실행돼서 즉시 crash (user-space 에서는 SIGSEGV).

이게 "절대 일어나서는 안 될 상황" 에 대한 방어 코드다.

ARM64 의 start.S

x86_64 와 동일한 일을 ARM64 에서 어떻게 하는지 본다. master 의 sysdeps/aarch64/start.S.

; [user-space - glibc sysdeps/aarch64/start.S master 인용]

#include <sysdep.h>

ENTRY (_start)
    /* Create an initial frame with 0 LR and FP */
    cfi_undefined (x30)
    mov     x29, #0
    mov     x30, #0

    /* Setup rtld_fini in argument register */
    mov     x5, x0

    /* Load argc and a pointer to argv */
    ldr     PTR_REG (1), [sp, #0]
    add     x2, sp, #PTR_SIZE

    /* Setup stack limit in argument register */
    mov     x6, sp

    /* Set up the other arguments in registers */
    adrp    x0, :got:main
    ldr     PTR_REG (0), [x0, #:got_lo12:main]

    mov     x3, #0    /* nullify init */
    mov     x4, #0    /* nullify fini */

    /* __libc_start_main (main, argc, argv, init, fini, rtld_fini, stack_end) */
    bl      __libc_start_main

    /* should never get here....*/
    bl      abort
END (_start)

같은 일을 한다.

[ARM64 _start 의 인자 배치]

  x29, x30 = 0           (frame pointer, link register 클리어)
  x5 = x0 (들어올 때)      (rtld_fini, 진입 시 x0 에 있음)
  x1 = argc              (stack 의 첫 word)
  x2 = sp + 8 = argv
  x6 = sp                (stack_end)
  x0 = main 의 주소        (got 에서 로드)
  x3 = 0 (init = NULL)
  x4 = 0 (fini = NULL)
  
  bl __libc_start_main
  
  ARM64 의 함수 호출 ABI 는 x0~x7 이 첫 8개 인자.
  → 모든 인자가 register 에 (stack 사용 안 함)

x86_64 가 6개 register + 1개 stack 인 반면, ARM64 는 7개 모두 register. ARM64 의 ABI 가 더 register 친화적.

_start 의 본질 — ABI 변환

_start 가 하는 일을 한 줄로 요약하면 — kernel ABI 에서 C ABI 로의 변환이다.

[_start 의 본질]

  kernel ABI:
    stack 에 모든 게 있음 (argc, argv, envp, auxv)
    register 는 0 으로 초기화됨
    
  C ABI:
    register 에 인자 배치 (System V x86_64 ABI)
    stack 은 16 바이트 정렬
    
  _start = kernel ABI → C ABI 변환기
        + __libc_start_main 호출
        + 돌아오면 즉시 crash (도달 불가)

이 변환이 — 모든 C 프로그램에서 일어나는 첫 user-space 코드다. 매우 단순하지만 매우 중요한 코드.

static binary 의 경우

dynamic linked binary 의 경우 — _start 가 실행되기 전에 dynamic linker (ld.so) 가 먼저 실행된다. ld.so 의 _start 도 같은 위치에 있지만, 이름만 같고 다른 코드다 (glibc 의 elf/dl-init.c 등).

static linked binary 의 경우 — kernel 이 직접 프로그램의 _start 로 점프. dynamic linker 우회.

[dynamic vs static 의 _start 호출 흐름]

  Dynamic:
    kernel → ld.so 의 _start
            → ld.so 가 라이브러리 로드, relocation
            → ld.so 가 프로그램의 _start 호출
            → _start 가 __libc_start_main 호출
            → main
    
  Static:
    kernel → 프로그램의 _start 직접
            → _start 가 __libc_start_main 호출
            → main

static 의 경우 ld.so 가 없으니 — _start 진입 시 %rdx = 0 (rtld_fini 없음). 그래서 __libc_start_main 의 %r9 = 0 이 되고, 그 안에서 NULL 체크가 있다 (다음 장).

gcc 가 어떻게 _start 를 link 하는가

gcc hello.c -o hello 의 link 과정을 보면:

# [user-space - bash]
$ gcc -v hello.c -o hello 2>&1 | grep collect2 | head -1
# ↑ collect2 (linker wrapper) 의 호출 명령어를 봄

/usr/libexec/gcc/x86_64-linux-gnu/13/collect2 \
  -plugin ... \
  -m elf_x86_64 \
  --eh-frame-hdr \
  --dynamic-linker /lib64/ld-linux-x86-64.so.2 \
  -pie \
  /usr/lib/x86_64-linux-gnu/Scrt1.o \      ← ★ _start 가 여기
  /usr/lib/x86_64-linux-gnu/crti.o \       ← _init 의 시작 부분
  ... gcc 내부 라이브러리들 ...
  hello.o \                                ← 사용자 코드
  ... gcc 내부 라이브러리들 ...
  -lc \                                    ← libc
  /usr/lib/x86_64-linux-gnu/crtn.o \       ← _init 의 끝 부분
  ...

Scrt1.o, crti.o, crtn.o 가 자동으로 link 된다. 사용자가 그것들에 대해 알 필요 없지만 — 이게 main 이 호출되기까지의 모든 사슬을 만든다.

이 장의 결론

_start 의 본질:

✓ ELF 의 entry point
✓ glibc 의 sysdeps/<arch>/start.S 에 정의
✓ crt1.o (또는 Scrt1.o) 에 컴파일되어 모든 프로그램에 link
✓ kernel ABI → C ABI 변환
✓ __libc_start_main 호출
✓ 돌아오지 않음 (hlt/abort)

매우 작은 코드지만 — 모든 C 프로그램의 실행이 정확히 여기서 시작한다. 다음 장에서는 _start 가 호출한 __libc_start_main 의 깊이를 본다. 거기서 main 이 호출되기까지의 모든 준비가 일어난다.

17__libc_start_main 의 깊이

_start 가 호출한 __libc_start_main 은 — Program startup 의 진짜 본체다. 이 함수 안에서 — TLS, pthread, atexit, locale, stack canary, ifunc, init_array — 모든 초기화가 일어난 후, **main 이 호출**된다.

master 의 csu/libc-start.c 에 있고, 매우 길고 복잡한 함수 (수백 줄). 이 장은 그 구조를 단계별로 본다.

시그니처

먼저 시그니처. master 의 csu/libc-start.c:

// [user-space - glibc csu/libc-start.c master 시그니처]
STATIC int
LIBC_START_MAIN (int (*main) (int, char **, char ** MAIN_AUXVEC_DECL),
                 int argc,
                 char **argv,
                 __typeof (main) init,
                 void (*fini) (void),
                 void (*rtld_fini) (void),
                 void *stack_end)

7개 인자. 이전 장에서 본 _start 가 정확히 이 순서로 인자를 배치한다.

[__libc_start_main 의 7 인자]

  main      : 사용자의 main 함수
  argc      : argument count
  argv      : argument vector
  init      : 옛 __libc_csu_init (현재 NULL)
  fini      : 옛 __libc_csu_fini (현재 NULL)
  rtld_fini : dynamic linker 의 termination function
              (static binary 면 NULL)
  stack_end : 새 프로세스의 stack 최대 주소
              (alternative stack 설정 등에 사용)

MAIN_AUXVEC_DECL 매크로는 — 일부 아키텍처 (PowerPC 등) 에서 main 에 auxv 도 인자로 추가하기 위한 것. x86_64 에서는 빈 매크로.

전체 흐름 — master 본체 (단순화)

매우 단순화한 형태로 본체를 보면:

// [user-space - glibc csu/libc-start.c master 핵심 흐름 단순화]
STATIC int
LIBC_START_MAIN (int (*main)(int, char**, char**),
                 int argc, char **argv,
                 __typeof(main) init,
                 void (*fini)(void),
                 void (*rtld_fini)(void),
                 void *stack_end)
{
    /* environment 포인터 계산: argv 끝 +1 */
    char **ev = &argv[argc + 1];
    
    /* 1. 전역 변수 설정 */
    __environ = ev;
    __libc_stack_end = stack_end;
    
    /* 2. argv, argc 를 전역에 저장 (psinfo, getauxval 등에서 사용) */
    _dl_aux_init(...);    /* auxv 파싱 */
    
    /* 3. ★ 동적 링크 환경 초기화 (SHARED 빌드만) */
#ifndef SHARED
    /* static binary 인 경우: 직접 초기화 */
    __pthread_initialize_minimal();
    __libc_init_first(argc, argv, __environ);
#endif
    
    /* 4. rseq 시스템 호출 등록 (Restartable Sequences, Linux 4.18+) */
    __rseq_size = ...;
    
    /* 5. stack canary (스택 보호) 설정 */
    uintptr_t stack_chk_guard = _dl_setup_stack_chk_guard();
    __stack_chk_guard = stack_chk_guard;
    
    /* 6. pointer mangling 키 설정 */
    uintptr_t pointer_chk_guard = _dl_setup_pointer_guard();
    __pointer_chk_guard_local = pointer_chk_guard;
    
    /* 7. dynamic linker 의 종료 함수 등록 (atexit) */
    if (rtld_fini != NULL)
        __cxa_atexit((void (*)(void *))rtld_fini, NULL, NULL);
    
    /* 8. fini 종료 함수 등록 (옛 방식, 현재는 NULL) */
    if (fini != NULL)
        __cxa_atexit((void (*)(void *))fini, NULL, NULL);
    
    /* 9. ★ 메인 프로그램의 .init_array 호출 */
    /* SHARED 빌드: csu/libc-start.c 안의 별도 call_init 함수가 처리
       (의존 라이브러리들은 이전에 ld.so 의 _dl_init 가 처리해둠) */
#ifdef SHARED
    call_init(argc, argv, __environ);   /* 메인 프로그램의 init_array 호출 */
#else
    /* static 빌드: ld.so 가 없으니 모든 일을 직접 */
    _dl_relocate_static_pie();
    _init();    /* DT_INIT */
    /* __init_array_start / __init_array_end linker 심볼로 init_array 순회 */
    const size_t size = __init_array_end - __init_array_start;
    for (size_t i = 0; i < size; i++)
        (*__init_array_start[i])(argc, argv, __environ);
#endif

    /* 옛 방식: init 인자 호출 (현재 init == NULL 이므로 skip) */
    if (init != NULL)
        (*init)(argc, argv, __environ MAIN_AUXVEC_PARAM);
    
    /* 10. 새 thread 의 cleanup handler 등록 (취소 처리) */
    /* setjmp() with __libc_start_call_main 의 unwind 처리 */
    
    /* 11. ★ 진짜 main 호출 */
    int result = main(argc, argv, __environ MAIN_AUXVEC_PARAM);
    
    /* 12. main 이 반환되면 exit(result) */
    exit(result);
}

각 단계가 매우 깊다. 차례로 본다.

단계 1-2: 환경 설정

char **ev = &argv[argc + 1];
__environ = ev;
__libc_stack_end = stack_end;

이전 장에서 본 stack 구조에서 — argv 의 NULL terminator 직후가 envp 의 시작이다. 즉 argv[argc] = NULL, argv[argc + 1] = envp[0].

[envp 의 위치 계산]

  stack:
    argc
    argv[0]
    argv[1]
    ...
    argv[argc-1]
    NULL              ← argv[argc]
    envp[0]           ← argv[argc + 1] = &ev
    envp[1]
    ...
    NULL              ← envp 끝
    (auxv)

__environ 이 — 전역 변수다. 사용자 코드에서 extern char **environ 으로 접근 가능. getenv("PATH") 같은 함수가 이걸 읽는다.

단계 3: pthread 와 libc 초기화

#ifndef SHARED
__pthread_initialize_minimal();
__libc_init_first(argc, argv, __environ);
#endif

이 부분이 — dynamic linker 와의 분업의 핵심이다.

[SHARED vs static 의 초기화 분업]

  Dynamic linked (SHARED 빌드):
    - ld.so 가 __libc_start_main 호출 전에
      모든 초기화를 끝냄 (libc.so 로드, relocation 등)
    - __libc_start_main 안에서는 일부만 수행
    
  Static linked (!SHARED):
    - ld.so 가 없으니
      __libc_start_main 이 모든 일을 직접 함
    - __pthread_initialize_minimal, __libc_init_first 등

__pthread_initialize_minimal 이 — 첫 thread (main thread) 의 TLS 와 pthread 자료구조를 초기화한다.

[__pthread_initialize_minimal 의 일]

  - main thread 의 pthread struct (= struct pthread) 할당
  - TCB (Thread Control Block) 초기화
  - %fs register 설정 (TLS 접근용, Part I 7장 참조)
  - thread-local 변수들의 초기값 복사 (TLS template)
  - tid 가져오기 (set_tid_address syscall)
  - cleanup stack 초기화
  - rseq 등록 (Linux 4.18+)

단계 5: stack canary — 스택 보호

uintptr_t stack_chk_guard = _dl_setup_stack_chk_guard();
__stack_chk_guard = stack_chk_guard;

매우 흥미로운 부분. buffer overflow 공격 방어를 위한 magic number 설정.

[stack canary 의 동작]

  컴파일러가 함수 진입 시:
    canary = __stack_chk_guard       ← 전역 변수에서 읽음
    stack 의 buffer 바로 위에 저장
    
  함수 반환 시:
    저장된 canary == __stack_chk_guard 확인
    다르면 → __stack_chk_fail() 호출 → 즉시 abort
    
  공격자가 buffer overflow 로 return address 덮어쓰려면
  중간의 canary 를 통과해야 함 → 매우 어려움

__stack_chk_guard 의 값은 — AT_RANDOM (kernel 이 준 16 byte 랜덤) 에서 첫 8 바이트. 매번 실행마다 다른 값. 보통 마지막 바이트가 \0 으로 — strcpy 같은 string 함수가 canary 까지 덮지 못하도록.

이 메커니즘이 gcc -fstack-protector 의 본체다. 현대 Linux 의 모든 binary 에 적용.

단계 7-8: atexit 등록

if (rtld_fini != NULL)
    __cxa_atexit((void (*)(void *))rtld_fini, NULL, NULL);
if (fini != NULL)
    __cxa_atexit((void (*)(void *))fini, NULL, NULL);

__cxa_atexit 가 — 프로그램 종료 시 호출할 함수들의 큐에 등록한다. rtld_fini 는 dynamic linker 의 정리 함수.

[atexit 등록의 의미]

  main 이 끝나면 (또는 exit 호출되면):
    1. atexit 큐의 함수들 역순으로 호출
    2. .fini_array 의 함수들 역순으로 호출
    3. 등록된 destructor 들 호출
    4. _exit syscall 로 kernel 진입
    
  rtld_fini 등록의 효과:
    프로그램 종료 시 ld.so 가 정리 작업 수행
    (mmap 된 라이브러리들 해제 등)

이 메커니즘이 20장 (atexit 와 프로그램 종료) 에서 다룰 핵심.

단계 9: .init_array 호출

가장 흥미로운 부분이 9번 — **.init_array 의 함수들을 호출**. master 의 실제 분담은 미묘하다.

[.init_array 실행의 정확한 책임 분담]

  Dynamic linked (SHARED 빌드):
    
    a) 의존 라이브러리들의 .init_array
       → ld.so 의 _dl_init() 이 처리
       → __libc_start_main 호출 _전에_ 이미 끝남
       → 의존 그래프의 leaf 부터 root 순서
       → 메인 프로그램은 skip!
    
    b) 메인 프로그램의 .init_array
       → __libc_start_main 안의 별도 call_init() 함수가 처리
       → main 호출 _직전_
    
  Static linked (!SHARED 빌드):
    ld.so 가 없으니 __libc_start_main 이 모든 일 직접:
    - __init_array_start / __init_array_end (linker 심볼) 로 순회
    - _init() 도 직접 호출
[.init_array 의 의미]

  사용자가 작성:
    __attribute__((constructor)) 함수
    C++ 의 글로벌 객체 생성자
    
  컴파일러가 처리:
    그 함수들의 주소를 .init_array section 에 모음
    
  런타임:
    .init_array 의 함수들을 차례로 호출 (main 전에!)
    .init_array 끝나면 main 호출

자세한 내용은 19장에서.

단계 10-11: setjmp + main 호출

master 의 실제는 — __libc_start_call_main 이라는 함수가 별도로 있고, 거기에서 setjmp 와 main 호출이 일어난다. master 의 sysdeps/nptl/libc_start_call_main.h:

// [user-space - glibc sysdeps/nptl/libc_start_call_main.h master 단순화]
_Noreturn static void
__libc_start_call_main(int (*main) (int, char **, char **),
                       int argc, char **argv)
{
    int result;
    
    /* unwind 정보 설정 (longjmp/cancellation 처리용) */
    struct pthread *self = THREAD_SELF;
    struct __jmp_buf_tag unwind_buf;
    
    /* setjmp - 여기로 longjmp 로 돌아올 수 있음 */
    if (!__sigsetjmp(&unwind_buf, 0)) {
        /* 첫 진입 — main 호출 */
        self->cleanup_jmp_buf = &unwind_buf;
        
        /* ★ 진짜 main 호출 */
        result = main(argc, argv, __environ MAIN_AUXVEC_PARAM);
    } else {
        /* longjmp 로 돌아온 경우 — pthread_exit 등 */
        result = 0;
    }
    
    /* main 이 반환되면 exit */
    exit(result);
}

setjmp 가 흥미로운 부분이다 — pthread_exit(NULL) 이 main thread 에서 호출되면 — 일반적인 함수 반환이 아니라 longjmp 로 여기로 돌아온다. 그래야 stack unwind 가 안전.

  ✓ TLS / pthread 초기화 (main thread)

  ✓ exit(main 의 반환값)

사용자 코드 main 한 줄 위에는 — 이 모든 무대가 깔려 있다. 다음 장에서는 — 이 무대를 만드는 데 사용된 부품들 (crt1.o, crti.o, crtn.o) 의 구조와 그들이 어떻게 합쳐지는지 본다.

18crt1.o, crti.o, crtn.o 의 역할

지금까지의 장에서 — _start (crt1.o 안) 와 __libc_start_main (libc.so 안) 사이의 통로를 봤다. 하지만 그 사이에 한 가지가 더 있다 — **_init 과 _fini 함수, 그리고 그들을 만드는 crti.o + crtn.o** 의 쌍.

이 장은 그 .o 파일들이 어떻게 협력해서 — _init 와 _fini 라는 함수가 완성되는지를 본다. ELF 의 매우 영리한 메커니즘이다.

4 가지 .o 파일

C 프로그램이 link 될 때 — gcc 가 자동으로 합치는 .o 파일들이 있다.

[gcc 가 자동 link 하는 C runtime 부속 파일들]

  crt1.o     :  _start 의 정의
                (또는 PIE 용 Scrt1.o, profiling 용 gcrt1.o)
                
  crti.o     :  _init / _fini 함수의 "prologue" (시작 부분)
                
  crtbegin.o :  gcc 제공, .init/.fini 의 "중간 시작" 부분
                생성자 호출 코드 포함
                
  crtend.o   :  gcc 제공, .init/.fini 의 "중간 끝" 부분
                
  crtn.o     :  _init / _fini 함수의 "epilogue" (끝 부분)

흥미로운 점은 — **_init 와 _fini 함수는 단일 .o 파일에 정의되지 않는다**. 여러 파일의 조각들이 link 시점에 합쳐져서 한 함수가 된다. 이 메커니즘을 이해하려면 ELF section 의 동작을 봐야 한다.

핵심 아이디어 — .init / .fini section 조각

_init 와 _fini 는 — .init 와 .fini 라는 section 안에 들어가는 코드다. linker 는 — 같은 이름의 section 들을 모두 연결한다.

[linker 가 .init section 을 합치는 방식]

  crti.o 의 .init section (시작 부분):
    _init:
        subq $8, %rsp        ; stack alignment
        ; ★ 함수 본체가 끝나지 않음! ret 없음

  crtbegin.o 의 .init section (가운데):
        ; gcc 가 넣는 코드 (deregister_tm_clones 등)

  사용자 코드의 .init section (만약 있다면):
        ; 보통 비어 있음

  crtend.o 의 .init section (가운데 끝):
        ; gcc 가 넣는 코드

  crtn.o 의 .init section (끝 부분):
        addq $8, %rsp        ; stack 복원
        ret                  ; ★ 여기서 함수 끝

link 시점에 — 이 모든 조각이 순서대로 합쳐져서 **한 개의 _init 함수**가 된다.

[합쳐진 _init 의 모습]

  _init:
      subq $8, %rsp              ← crti.o
      
      ... deregister_tm_clones... ← crtbegin.o
      
      ... (사용자 코드 비어있음) ...
      
      ... register_tm_clones ...  ← crtend.o
      
      addq $8, %rsp              ← crtn.o
      ret                        ← crtn.o

이게 매우 영리한 디자인이다. 각 .o 파일이 자기 부분만 정의하고, linker 가 알아서 합친다. 새로운 라이브러리가 추가되어도 — 그 .o 가 .init section 에 코드를 넣으면 자동으로 호출됨.

glibc 의 crti.S — master 본체

master 의 sysdeps/x86_64/crti.S 를 본다.

; [user-space - glibc sysdeps/x86_64/crti.S master 인용]

#include <libc-symbols.h>
#include <sysdep.h>

#ifndef PREINIT_FUNCTION
# define PREINIT_FUNCTION __gmon_start__
#endif

#ifndef PREINIT_FUNCTION_WEAK
# define PREINIT_FUNCTION_WEAK 1
#endif

#if PREINIT_FUNCTION_WEAK
weak_extern (PREINIT_FUNCTION)
#else
.hidden PREINIT_FUNCTION
#endif

.section .init,"ax",@progbits
.p2align 2
.globl _init
.type _init, @function
_init:
    /* Maintain 16-byte stack alignment for called functions.  */
    subq $8, %rsp
#if PREINIT_FUNCTION_WEAK
    /* PREINIT_FUNCTION (보통 __gmon_start__) 가 link 되면 호출.
       link 안 되면 skip. */
    movq PREINIT_FUNCTION@GOTPCREL(%rip), %rax
    testq %rax, %rax
    je .Lno_weak_fn
    call *%rax
.Lno_weak_fn:
#else
    call PREINIT_FUNCTION
#endif
    /* ★ crtn.S 의 끝 부분이 여기 이어짐. ret 가 거기 있음. */

.section .fini,"ax",@progbits
.p2align 2
.globl _fini
.type _fini, @function
_fini:
    subq $8, %rsp
    /* ★ crtn.S 의 끝 부분이 여기 이어짐 */

PREINIT_FUNCTION (__gmon_start__) 가 — 보통 gprof 프로파일링용. 일반 binary 에서는 weak 이라 없으므로 skip.

crtn.S — _init 와 _fini 의 끝

master 의 sysdeps/x86_64/crtn.S:

; [user-space - glibc sysdeps/x86_64/crtn.S master 인용]

#include <sysdep.h>

.section .init,"ax",@progbits
    addq $8, %rsp
    ret

.section .fini,"ax",@progbits
    addq $8, %rsp
    ret

매우 단순. 단 4 줄. **.init 와 .fini section 모두에서 stack 복원 후 ret**. 이 코드가 — crti.S 의 prologue 와 합쳐져서 완전한 함수가 된다.

_init 가 누가 호출하는가?

_init 는 — dynamic linker (ld.so) 가 자동 호출한다. ELF 의 **DT_INIT** dynamic entry 가 — _init 의 주소를 가리킨다.

# [user-space - bash]
$ readelf -d ./hello | grep -E "(INIT|FINI)"
 0x000000000000000c (INIT)               0x1000      ← _init 의 주소
 0x000000000000000d (FINI)               0x119c      ← _fini 의 주소
 0x0000000000000019 (INIT_ARRAY)         0x3da8      ← 다음 장의 주제
 0x000000000000001b (INIT_ARRAYSZ)       8 (bytes)
 0x000000000000001a (FINI_ARRAY)         0x3db0
 0x000000000000001c (FINI_ARRAYSZ)       8 (bytes)
  • DT_INIT → 옛 방식의 초기화 함수 (_init)
  • DT_FINI → 옛 방식의 종료 함수 (_fini)
  • DT_INIT_ARRAY → 새 방식의 초기화 함수 배열
  • DT_FINI_ARRAY → 새 방식의 종료 함수 배열

_init 의 과거 — 옛 메커니즘

옛날엔 — 사용자가 .init section 에 자기 코드를 직접 추가할 수 있었다.

// [user-space - 옛 방식]
asm(".section .init\n"
    "    call my_initializer\n"
    ".section .text\n");

void my_initializer(void) {
    printf("called before main!\n");
}

이 방식이 동작은 했지만 — 어셈블리가 필요하고, 인자 전달이 어렵고, 호출 순서를 통제할 수 없었다. 그래서 — **현대에는 .init_array 가 표준**이다.

glibc 2.34 의 큰 변경 — __libc_csu_init 제거

이전 17장에서 잠깐 언급한 — glibc 2.34 (2021.8) 에서 완성된 큰 변경이 이 영역과 관련 있다. 사실 이 변경은 갑작스러운 게 아니라 — glibc 2.28 (2018, Florian Weimer 의 patch BZ#23323) 부터 점진적으로 진행되었다. ROP gadget 보안 이슈가 직접적 동기.

옛 방식 (glibc < 2.28):

[옛 방식 — __libc_csu_init 가 매개]

  _start:
    rdi = &main
    rcx = &__libc_csu_init    ← gcc 가 자동으로 추가하는 함수
    r8  = &__libc_csu_fini
    call __libc_start_main
    
  __libc_start_main:
    (*init)(...)              ← __libc_csu_init 호출
    
  __libc_csu_init:
    _init();                  ← .init section 호출
    .preinit_array 순회 호출
    .init_array 순회 호출
    
  → 그 다음 main 호출

중간 단계 (glibc 2.28 ~ 2.33):

csu/libc-start.c 의 __libc_start_main 이 — init == NULL 인 경우 main_map 을 직접 찾아서 처리하도록 개선. 옛 binary 와의 호환성 유지를 위해 __libc_csu_init 자체는 남아 있되, 새로 컴파일되는 binary 는 init/fini 인자로 NULL 을 넘기게 됨.

새 방식 (glibc 2.34+):

[새 방식 — ld.so + __libc_start_main 의 분담]

  ld.so 의 _dl_init:
    프로그램 로드 후, _start 호출 전:
    각 공유 라이브러리의 DT_INIT 와 DT_INIT_ARRAY 호출
    (libc.so, libpthread.so, 사용자 .so 들)
    ★ 메인 프로그램은 skip!
    
  _start (crt1.o):
    rdi = &main
    rcx = NULL                ← csu_init 안 넘김
    r8  = NULL                ← csu_fini 도 안 넘김
    call __libc_start_main
    
  __libc_start_main (csu/libc-start.c, SHARED 빌드):
    ...
    call_init(argc, argv, env)
      ↓
      메인 프로그램의 link_map 가져옴
      메인 프로그램의 DT_INIT 호출
      메인 프로그램의 DT_INIT_ARRAY 호출
    
    if (init != NULL)
        (*init)(...)          ← NULL 이므로 skip (옛 binary 만 의미)
    main(argc, argv, env)
  
  glibc 2.34 에서:
    csu/elf-init.c 파일이 완전 삭제됨
    → __libc_csu_init 와 __libc_csu_fini 가 더 이상 link 되지 않음

결과: __libc_csu_init 와 __libc_csu_fini 가 binary 에 더 이상 link 되지 않는다.

$ nm ./hello_new | grep csu_init
# 아무것도 없음 (binary 에 link 되지 않음)

# 옛 binary 에는 있을 수 있음:
$ nm ./hello_old | grep csu_init
0000000000xxxx T __libc_csu_init    ← glibc 2.27 이하로 빌드된 binary

이 변경이 — PIE binary 의 ROP 공격 표면을 줄이는 보안 개선이기도 했다. 논문 "return-to-csu: A New Method to Bypass 64-bit Linux ASLR" (Marco-Gisbert and Ripoll-Ripoll) 가 — __libc_csu_init 안의 ROP gadget 으로 ASLR 우회 가능함을 보였고, 이게 직접적 동기가 되었다.

static binary 의 경우

dynamic linked binary 는 — ld.so 가 .init_array 를 호출. static binary 는 — __libc_start_main 안에서 직접 호출한다.

// [user-space - glibc csu/libc-start.c master, static 빌드만의 분기]
#ifndef SHARED
    /* static binary: 직접 .init_array 호출 */
    _dl_relocate_static_pie();
    _init();                      /* DT_INIT */
    
    /* .init_array 순회 */
    const size_t size = __init_array_end - __init_array_start;
    for (size_t i = 0; i < size; i++)
        (*__init_array_start[i])(argc, argv, __environ);
#endif

__init_array_start 와 __init_array_end 는 — linker 가 자동으로 정의하는 심볼이다. .init_array section 의 시작과 끝 주소를 가리킨다.

4 .o 파일의 link 순서

gcc 가 link 할 때의 정확한 순서가 중요하다.

[gcc link 시 .o 파일들의 순서]

  Scrt1.o      ← _start
  crti.o       ← _init / _fini 의 prologue
  crtbeginS.o  ← gcc 의 init/fini 보조 코드 (시작)
  
  사용자 .o 들  ← 사용자의 main, init_array 등
  
  -lc 등 라이브러리들
  
  crtendS.o    ← gcc 의 init/fini 보조 코드 (끝)
  crtn.o       ← _init / _fini 의 epilogue

이 순서가 — .init section 안의 조각이 합쳐지는 순서다. crti.o 의 시작 + crtbeginS.o + (다른 .o 들 — 보통 없음) + crtendS.o + crtn.o 의 끝.

gcc 의 link 보기

-v 옵션으로 gcc 의 link 명령어를 직접 볼 수 있다.

이 장의 결론

crt1.o, crti.o, crtn.o 는 — C 프로그램의 보이지 않는 부속품이다.

[부속품의 역할 정리]

  crt1.o (Scrt1.o)  : _start 정의
                      kernel ABI → C ABI 변환
                      __libc_start_main 호출
                      
  crti.o             : _init 와 _fini 의 시작 부분
                       (prologue: stack alignment)
                       
  crtn.o             : _init 와 _fini 의 끝 부분
                       (epilogue: stack 복원 + ret)
                       
  crtbegin*.o (gcc)  : .init / .fini 의 중간 부분
                       (deregister_tm_clones 등)
                       
  crtend*.o (gcc)    : .init / .fini 의 중간 끝
                       (register_tm_clones 등)

이 부속품이 합쳐져서 — _init 와 _fini 라는 완전한 함수가 만들어진다. 그리고 ld.so 가 그 함수들을 호출해 — 사용자의 초기화 코드와 종료 코드를 실행한다.

다음 장에서는 — 현대적이고 더 강력한 메커니즘 — .init_array 와 __attribute__((constructor)) 를 본다. 그리고 C++ 의 글로벌 객체 생성자가 어떻게 처리되는지.

19전역 객체 초기화와 .init_array

C 와 C++ 에서 — main 이 실행되기 전에 자동으로 실행되는 코드가 있다. C++ 의 글로벌 객체 생성자, __attribute__((constructor)) 함수 — 이들이 어떻게 호출되는지의 핵심 메커니즘이 .init_array 다.

이 장은 그 메커니즘을 깊이 본다.

문제 정의 — main 전에 실행될 코드를 어떻게 등록?

C++ 예제로 시작.

// [user-space - C++]
#include <iostream>
#include <vector>

class Counter {
public:
    Counter() {
        std::cout << "Counter constructor called!\n";
        count = 0;
    }
    int count;
};

Counter global_counter;    // ← 글로벌 객체

int main() {
    std::cout << "main started\n";
    return 0;
}

이 코드를 실행하면 출력은:

Counter constructor called!
main started

Counter 의 생성자가 — main 이 시작되기 전에 호출된다. 어떻게?

C 로도 같은 일을 할 수 있다 (gcc 확장).

// [user-space - C with gcc extension]
#include <stdio.h>

__attribute__((constructor))
void my_init(void) {
    printf("my_init called!\n");
}

int main(void) {
    printf("main started\n");
    return 0;
}

출력은 똑같이:

my_init called!
main started

이게 — 어떻게 동작하는가?

답 — .init_array section

답은 — **컴파일러가 __attribute__((constructor)) 함수 (와 C++ 글로벌 객체 생성자) 의 주소를 .init_array section 에 모으고, 런타임에 ld.so 또는 __libc_start_main 이 그 함수들을 순회 호출**한다.

[전체 흐름]

  컴파일 시:
    gcc 가 my_init 의 주소를 .init_array section 에 등록
    
  link 시:
    linker 가 모든 .o 의 .init_array 들을 합침
    __init_array_start 와 __init_array_end 심볼 생성
    DT_INIT_ARRAY dynamic entry 생성
    
  런타임:
    ld.so (또는 static 의 경우 __libc_start_main) 가
    __init_array_start ~ __init_array_end 순회
    각 함수 포인터 호출
    
  main 실행

.init_array 의 실제 — readelf 로 보기

위의 C 코드를 컴파일하고 .init_array 를 본다.

$ gcc init_demo.c -o init_demo
$ readelf -d init_demo | grep INIT_ARRAY

.init_array 에는 두 개의 함수 포인터가 있다.

[.init_array 의 두 entry]

  Entry 0: frame_dummy    ← gcc 가 자동으로 추가
                          예외 처리 정보 등록
                          (crtbeginS.o 에서 옴)
  
  Entry 1: my_init        ← 사용자의 constructor

런타임에 — 이 두 함수가 순서대로 호출된다.

__attribute__((constructor)) 의 동작

컴파일러가 어떻게 처리하는지를 보면:

// [user-space - C source]
__attribute__((constructor))
void my_init(void) { ... }
; [user-space - gcc 가 생성하는 어셈블리]
.section .init_array,"aw"
.align 8
.quad my_init                ; my_init 의 주소를 .init_array 에 추가

.text
my_init:
    ...                      ; 함수 본체 (보통의 함수처럼)
    ret

매우 단순. 그냥 .init_array section 에 함수의 주소를 추가하는 게 전부.

우선순위 — constructor(N)

여러 constructor 가 있으면 — 호출 순서를 통제할 수 있다.

// [user-space - C]
#include <stdio.h>

__attribute__((constructor(101)))
void init_a(void) { printf("init_a (101)\n"); }

__attribute__((constructor(200)))
void init_b(void) { printf("init_b (200)\n"); }

__attribute__((constructor))      // 기본 priority
void init_c(void) { printf("init_c (default)\n"); }

int main(void) { return 0; }

출력:

init_a (101)         ← 가장 낮은 priority 먼저
init_b (200)
init_c (default)     ← priority 없는 건 마지막

priority 가 낮을수록 먼저. 1~100 은 gcc/libc 가 예약, 사용자는 101~ 사용 권장.

[priority 범위 규칙]

  1 ~ 100    : 시스템/라이브러리 예약
  101 ~ ...  : 사용자 코드용
  (없음)     : 모든 priority constructor 다음에

priority 가 있는 constructor 는 — .init_array.101, .init_array.200 처럼 별도 section 에 들어간다. linker 가 priority 순으로 정렬해서 합친다.

# [user-space - bash]
$ readelf -S init_demo | grep init_array
  [20] .init_array.101   INIT_ARRAY       ...
  [21] .init_array.200   INIT_ARRAY       ...
  [22] .init_array       INIT_ARRAY       ...      ← priority 없는 것

.preinit_array — 더 일찍 호출

.init_array 보다도 먼저 호출되는 메커니즘이 있다 — .preinit_array. 이건 보통 사용자가 직접 안 만들고, gcc 의 내부용.

[호출 순서]

  1. .preinit_array     ← 매우 일찍 (C++ vtable validation 등)
  2. _init               ← 옛 방식 (.init section)
  3. .init_array         ← 사용자 constructor 들 (priority 순)
  4. main                ← 사용자 코드!
  5. main 반환
  6. atexit 큐의 함수들 (역순)
  7. .fini_array         ← (priority 역순)
  8. _fini               ← 옛 방식 (.fini section)

ld.so 의 호출 코드

ld.so 가 .init_array 를 호출하는 본체. master 의 elf/dl-init.c:

// [user-space - glibc elf/dl-init.c master 단순화]
void
_dl_init (struct link_map *main_map, int argc, char **argv, char **env)
{
    /* preinit_array 처리 (메인 프로그램의 것만, 의존 라이브러리는 X) */
    ElfW(Dyn) *preinit_array = main_map->l_info[DT_PREINIT_ARRAY];
    if (preinit_array != NULL) {
        /* preinit_array 의 함수들 순회 호출 */
        ...
    }
    
    /* l_initfini[] 배열 거꾸로 순회 — 의존성 그래프의 leaf 부터 root */
    /* (이 배열은 _dl_sort_maps 가 의존성 순서로 미리 정렬해둠) */
    unsigned int i = main_map->l_searchlist.r_nlist;
    while (i-- > 0)
        call_init (main_map->l_initfini[i], argc, argv, env);
}

static void
call_init (struct link_map *l, int argc, char **argv, char **env)
{
    /* ★ 메인 프로그램 (executable) 은 _dl_init 에서 skip!
       메인 프로그램의 init_array 는 __libc_start_main 의 별도 call_init 가 처리 */
    if (l->l_type == lt_executable)
        return;
    
    /* 이미 init 호출됐으면 skip (순환 의존성 대비) */
    if (l->l_init_called) return;
    l->l_init_called = 1;
    
    /* DT_INIT (= _init) 호출 — 옛 방식 */
    if (l->l_info[DT_INIT] != NULL) {
        DL_CALL_DT_INIT(l, l->l_addr + l->l_info[DT_INIT]->d_un.d_ptr,
                        argc, argv, env);
    }
    
    /* DT_INIT_ARRAY 순회 호출 */
    ElfW(Dyn) *init_array = l->l_info[DT_INIT_ARRAY];
    if (init_array != NULL) {
        ElfW(Addr) *array = (ElfW(Addr)*)(l->l_addr + init_array->d_un.d_ptr);
        size_t size = l->l_info[DT_INIT_ARRAYSZ]->d_un.d_val / sizeof(ElfW(Addr));
        for (size_t i = 0; i < size; i++)
            ((dl_init_t)array[i])(argc, argv, env);
    }
}

매우 미묘한 부분 — **_dl_init 안의 call_init 가 메인 프로그램 (executable) 은 skip 한다**. 메인 프로그램의 .init_array 는 누가 호출하나? — __libc_start_main 안의 별도 call_init 함수가 호출 (다음 절). master 의 csu/libc-start.c 와 elf/dl-init.c 에 동명이인의 call_init 함수가 두 개 있다.

(역사적 이유: 옛날 glibc 에서는 ld.so 가 메인 프로그램을 skip 했고, __libc_start_main 이 __libc_csu_init 를 통해 그것을 처리했다. glibc 2.28 부터 __libc_csu_init 가 점진적으로 제거되었지만, 메인 프로그램 처리의 책임은 여전히 __libc_start_main 쪽에 남아 있다. glibc 2.34 에서 __libc_csu_init 가 완전 제거되었고, csu/libc-start.c 안의 call_init 함수가 그 자리를 차지.)

__libc_start_main 안의 call_init — 메인 프로그램용

master 의 csu/libc-start.c 의 SHARED 빌드:

// [user-space - glibc csu/libc-start.c master SHARED branch 인용]
#ifdef SHARED
/* Initialization for dynamic executables.  Find the main executable
   link map and run its init functions.  */
static void
call_init (int argc, char **argv, char **env)
{
    /* 메인 프로그램의 link_map 가져옴 */
    struct link_map *l = GL(dl_ns)[LM_ID_BASE]._ns_loaded;
    
    /* DT_PREINIT_ARRAY 는 여기서 처리하지 않음.
       _dl_init 에서 이미 처리됨. */
    
    /* 메인 프로그램의 DT_INIT 호출 */
    if (ELF_INITFINI && l->l_info[DT_INIT] != NULL)
        DL_CALL_DT_INIT(l, l->l_addr + l->l_info[DT_INIT]->d_un.d_ptr,
                        argc, argv, env);
    
    /* 메인 프로그램의 DT_INIT_ARRAY 호출 */
    ElfW(Dyn) *init_array = l->l_info[DT_INIT_ARRAY];
    if (init_array != NULL) {
        unsigned int jm = l->l_info[DT_INIT_ARRAYSZ]->d_un.d_val / sizeof(ElfW(Addr));
        ElfW(Addr) *addrs = (void *)(init_array->d_un.d_ptr + l->l_addr);
        for (unsigned int j = 0; j < jm; ++j)
            ((dl_init_t)addrs[j])(argc, argv, env);
    }
}
#endif

이 call_init 함수가 — __libc_start_main 안에서 호출되어, **메인 프로그램의 .init_array** 를 호출한다. ld.so 가 의존 라이브러리들을 모두 처리한 후 — 마지막에 메인 프로그램의 init_array 가 호출되어, 그 다음 main 으로 진행.

핵심 — 함수 포인터를 차례로 호출. 인자로 (argc, argv, env) 가 넘어가는 게 흥미롭다 (그래서 사용자의 constructor 도 이 인자를 받을 수 있음).

의존 라이브러리의 init_array 도 호출

ld.so 가 — 의존 라이브러리들의 .init_array 도 모두 호출한다. 호출 순서는 — depth-first post-order. 즉 의존성 그래프의 leaf 부터.

[의존 그래프와 init 호출 순서 예]

  hello (main_map)
    └── libstdc++.so
          └── libgcc_s.so
          └── libm.so
                └── libc.so

  init 호출 순서 (leaf 부터):
  
    1. libc.so 의 init_array
    2. libm.so 의 init_array
    3. libgcc_s.so 의 init_array
    4. libstdc++.so 의 init_array
    5. hello 의 init_array
    6. → main 호출

이 순서가 보장되기 때문에 — libstdc++ 의 글로벌 객체들 (std::cout 등) 이 hello 의 main 시작 전에 이미 초기화되어 있다. C++ 의 글로벌 객체 사용이 안전한 이유.

C++ 글로벌 객체 — 같은 메커니즘

C++ 글로벌 객체의 생성자도 — 같은 .init_array 메커니즘으로 호출된다. 컴파일러가 만든다.

// [user-space - C++]
#include <iostream>
Counter global_counter;

g++ 가 만드는 어셈블리 (단순화):

; [user-space - g++ 가 생성하는 어셈블리]
.section .init_array,"aw"
.quad _GLOBAL__sub_I_init_demo_main.cpp    ; 컴파일러 생성 함수

.text
_GLOBAL__sub_I_init_demo_main.cpp:
    ; 이 .cpp 파일의 모든 글로벌 객체 생성자 호출
    lea global_counter(%rip), %rdi
    call _ZN7CounterC2Ev                    ; Counter::Counter()
    
    ; atexit 으로 소멸자 등록
    lea __dso_handle(%rip), %rdx
    lea _ZN7CounterD2Ev(%rip), %rdi
    mov $global_counter, %rsi
    call __cxa_atexit
    
    ret

컴파일러가 합쳐진 함수를 자동 생성한다. 한 .cpp 파일 안의 모든 글로벌 객체의 생성자가 한 함수에 들어간다. 그 함수의 주소가 .init_array 에 추가.

동적 라이브러리 안의 constructor

.so 안의 constructor 는 — 그 라이브러리가 로드되는 시점에 호출된다.

[constructor 호출 시점]

  실행 파일 (main binary):
    main 시작 전, __libc_start_main 안의 call_init 에서
    (★ ld.so 의 _dl_init 이 아니라 __libc_start_main)
    
  link 시 의존 라이브러리:
    main 시작 전, ld.so 의 _dl_init 에서
    (의존성 그래프의 leaf 부터 root 순서)
    
  dlopen 으로 로드되는 라이브러리:
    dlopen() 호출 시점
    → dlopen 안에서 _dl_init 호출
    → 이 경우는 메인 프로그램이 아니므로 정상 처리

dlopen 한 .so 의 constructor 가 — dlopen 함수 반환 전에 모두 실행된다. 사용자 코드는 dlopen 후의 라이브러리가 "준비된" 상태를 볼 수 있다.

동적 라이브러리의 init_array — LD_PRELOAD 의 동작

LD_PRELOAD 의 동작도 — .init_array 의 호출 순서와 관련된다.

# [user-space - bash]
$ LD_PRELOAD=./mylib.so ./hello

ld.so 가 — mylib.so 를 다른 모든 의존 라이브러리보다 먼저 로드한다. 그래서 mylib.so 의 .init_array 가 — 다른 라이브러리들보다 나중에 호출된다 (의존성 그래프의 root 에 가까우니까).

[LD_PRELOAD 의 init 호출 순서]

  hello
    └── libc.so
    └── mylib.so (preloaded)
    
  호출 순서:
    1. libc.so 의 init_array
    2. mylib.so 의 init_array      ← LD_PRELOAD 라이브러리
    3. hello 의 init_array
    4. main

글로벌 객체의 위험 — 초기화 순서 문제

C++ 의 글로벌 객체에는 — "static initialization order fiasco" 라 불리는 유명한 문제가 있다.

// [user-space - C++ - 위험한 코드]
// file1.cpp:
extern int b;
int a = b * 2;   // ← b 가 초기화됐는지?

// file2.cpp:
int b = 5;

다른 .cpp 파일의 글로벌 객체끼리의 초기화 순서는 컴파일 단위 사이에서 정의되지 않는다. linker 가 어떤 순서로 .o 를 합치냐에 따라 다를 수 있다.

[초기화 순서 문제]

  같은 .cpp 안:
    선언 순서대로 초기화됨 (보장됨)
    
  다른 .cpp 간:
    링크 순서에 따라 다름
    → 위 예제에서 a 가 먼저 초기화되면 b = 0 (UB!)

해결책: Singleton pattern, Construct on First Use, etc.

// [user-space - C++ - 안전한 패턴]
int& get_b() {
    static int b = 5;   // 첫 호출 시 초기화
    return b;
}

int a = get_b() * 2;   // get_b() 가 b 의 초기화 보장

static initialization vs dynamic initialization

C++ 의 글로벌 초기화는 사실 두 단계로 나뉜다.

[C++ 글로벌 초기화의 두 단계]

  1. static initialization (컴파일 시점)
     - 상수 표현식으로 초기화 가능한 것
     - 0 으로 초기화 (zero initialization)
     - .data 또는 .bss 에 값이 미리 들어감
     - 런타임 비용 없음
     - 예: int x = 5;
     
  2. dynamic initialization (런타임)
     - 함수 호출이나 복잡한 표현식
     - .init_array 의 함수가 호출되어 수행
     - 위에서 본 메커니즘
     - 예: int x = compute_something();
     - 예: std::vector<int> v = {1, 2, 3};

constexpr 와 constinit (C++20) 가 — 모든 글로벌이 static initialization 으로 끝나도록 강제하는 기능. dynamic initialization 의 fiasco 를 회피.

atexit 와의 대칭

.init_array 의 함수들이 main 전에 호출되듯 — .fini_array 의 함수들은 main 후에 호출된다. 자세한 흐름은 다음 장 (20장 atexit) 에서.

[init_array 와 fini_array 의 대칭]

  init_array 호출 (priority 작은 것 먼저):
    constructor(101)
    constructor(200)
    constructor (priority 없음)
    
  main 실행
  
  fini_array 호출 (priority 큰 것 먼저, init 의 역순):
    destructor (priority 없음)
    destructor(200)
    destructor(101)

atexit() 로 등록한 함수는 — .fini_array 와 별개의 큐에 들어간다. 다음 장에서 모두 정리.

이 장의 결론

.init_array 는 — main 전 코드 실행의 핵심 메커니즘.

[.init_array 정리]

  ✓ __attribute__((constructor)) 함수의 주소가 들어감
  ✓ C++ 글로벌 객체 생성자도 같은 메커니즘
  ✓ priority 로 호출 순서 통제 가능 (101~)
  ✓ ld.so 가 의존 라이브러리들의 init_array 도 처리
  ✓ leaf 부터 root 순으로 호출 (depth-first post-order)
  ✓ glibc 2.34 부터 ld.so 가 프로그램의 init_array 도 직접 처리
  ✓ static binary 는 __libc_start_main 안에서 직접 호출

main 전 코드의 모든 비밀이 여기 있다. 사용자가 __attribute__((constructor)) 라고 적은 함수가 — ELF 의 .init_array 에 등록되어, ld.so 가 호출.

다음 장에서는 — 그 반대편을 본다. main 이 반환되거나 exit 가 호출되었을 때 — 어떻게 정리가 일어나는지. atexit, .fini_array, __cxa_atexit 의 협력.

20atexit 와 프로그램 종료

프로그램의 시작을 봤으니 — 이제 끝을 본다. main 이 반환되거나 exit 가 호출되면 — 어떤 정리 절차가 일어나는가? 이 장은 그 흐름의 깊이를 본다.

프로그램 종료의 4 가지 경로

먼저 정의. 프로그램이 종료될 수 있는 경로가 4 가지다.

┌──────────────────────────────────────────────────────────────┐
│ 프로그램 종료의 4 가지 경로                                       │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. main 의 return                                              │
│     → __libc_start_main 으로 돌아옴                              │
│     → exit(반환값) 호출                                          │
│     → "정상 종료" 의 전체 절차 수행                                │
│                                                               │
│  2. exit() 호출                                                  │
│     → atexit 핸들러 + .fini_array 순회                          │
│     → _exit syscall                                            │
│     → "정상 종료"                                                │
│                                                               │
│  3. _exit() 또는 _Exit() 직접 호출                                │
│     → atexit 등 우회                                             │
│     → 즉시 syscall                                              │
│     → "빠른 종료"                                                │
│                                                               │
│  4. abort() 또는 assert 실패                                      │
│     → SIGABRT 발생                                              │
│     → core dump (optional)                                     │
│     → 비정상 종료                                                │
│                                                               │
└──────────────────────────────────────────────────────────────┘

이 중 1번과 2번이 — atexit 핸들러와 destructor 가 실행되는 경로다. 3번은 그것들을 우회. 4번은 비정상.

exit() 의 본체

exit() 함수가 종료의 본체. master 의 stdlib/exit.c:

// [user-space - glibc stdlib/exit.c master 단순화]
void
exit(int status)
{
    __run_exit_handlers(status, &__exit_funcs, true, true);
}

void
__run_exit_handlers(int status, struct exit_function_list **listp,
                    bool run_list_atexit, bool run_dtors)
{
    /* atexit + __cxa_atexit 큐 처리 */
    __libc_lock_lock(__exit_funcs_lock);
    
    while (true) {
        struct exit_function_list *cur = *listp;
        if (cur == NULL) break;
        
        /* 각 entry 를 역순으로 처리 */
        while (cur->idx > 0) {
            cur->idx--;
            const struct exit_function *const f = &cur->fns[cur->idx];
            
            switch (f->flavor) {
                case ef_at:    /* atexit 으로 등록 */
                    (*f->func.at)();
                    break;
                case ef_on:    /* on_exit 으로 등록 */
                    (*f->func.on.fn)(status, f->func.on.arg);
                    break;
                case ef_cxa:   /* __cxa_atexit 으로 등록 */
                    (*f->func.cxa.fn)(f->func.cxa.arg);
                    break;
            }
        }
        
        *listp = cur->next;
    }
    
    __libc_lock_unlock(__exit_funcs_lock);
    
    if (run_list_atexit) {
        /* run_dtors == true 면 thread-local destructor 실행 */
        if (run_dtors)
            __call_tls_dtors();
    }
    
    /* 마지막: _exit syscall */
    _exit(status);
}

핵심 흐름:

[exit() 의 흐름]

  1. __exit_funcs 큐 (atexit, __cxa_atexit) 처리
     역순으로 (LIFO) — 마지막에 등록된 것부터 호출
     
  2. thread-local destructor 호출 (__call_tls_dtors)
     thread_local 변수의 destructor
     
  3. _exit syscall
     ─ kernel 진입 ─
     프로세스 종료

이전 장에서 ld.so 가 _dl_fini (= rtld_fini) 를 __cxa_atexit 로 등록한다고 했다. 그래서 — exit() 의 큐 처리 안에서 _dl_fini 가 호출되고, ld.so 가 — 모든 라이브러리의 .fini_array 와 _fini 를 처리한다.

atexit 의 등록

atexit(func) 는 — 함수 포인터를 큐에 추가한다. master 의 stdlib/cxa_atexit.c:

// [user-space - glibc stdlib/cxa_atexit.c master 단순화]
int
__cxa_atexit(void (*func)(void *), void *arg, void *d)
{
    return __internal_atexit(func, arg, d, &__exit_funcs);
}

int
atexit(void (*func)(void))
{
    return __cxa_atexit((void (*)(void *)) func, NULL, NULL);
}

atexit 는 — __cxa_atexit 의 wrapper. 인자 없이 등록하는 형태.

atexit 큐의 자료구조

__exit_funcs 가 atexit 큐다. master 의 stdlib/exit.h:

// [user-space - glibc stdlib/exit.h master 단순화]
enum {
    ef_free,
    ef_us,
    ef_on,        /* on_exit() 으로 등록 */
    ef_at,        /* atexit() 으로 등록 */
    ef_cxa        /* __cxa_atexit() 으로 등록 */
};

struct exit_function {
    long int flavor;
    union {
        void (*at)(void);                                    /* atexit */
        struct { void (*fn)(int, void *); void *arg; } on;   /* on_exit */
        struct {
            void (*fn)(void *);
            void *arg;
            void *dso_handle;
        } cxa;                                                /* __cxa_atexit */
    } func;
};

struct exit_function_list {
    struct exit_function_list *next;
    size_t idx;
    struct exit_function fns[32];     /* 32개 항목씩 묶음 */
};

extern struct exit_function_list *__exit_funcs;

각 atexit entry 가 — 함수 포인터 + 인자 + DSO handle 을 담는다. DSO handle 이 흥미로운 부분 — dlclose 로 라이브러리를 unload 할 때 사용된다.

__cxa_atexit — 라이브러리별 정리

C++ 의 destructor 등록은 — __cxa_atexit 를 쓴다. 일반 atexit 과의 차이는 — DSO (Dynamic Shared Object) handle 을 함께 등록한다는 점.

// [user-space - g++ 가 C++ 글로벌 객체 처리]
// 글로벌 객체의 destructor 등록
__cxa_atexit(destructor_function, &object, &__dso_handle);

__dso_handle 은 — 현재 라이브러리/실행파일을 식별하는 magic value. 이 정보가 — dlclose 시점에 활용된다.

[dlclose 와 __cxa_atexit 의 협력]

  dlclose(handle):
    1. handle 의 __dso_handle 값을 가져옴
    2. __exit_funcs 큐에서 그 __dso_handle 을 가진 entry 들 찾기
    3. 그 entry 들의 함수만 호출 (destructor)
    4. 큐에서 제거
    5. 라이브러리 메모리 해제

그래서 — dlopen 한 라이브러리의 글로벌 객체가 — dlclose 시점에 정확히 정리된다. 메모리 안전성.

.fini_array — 또 다른 정리 메커니즘

.init_array 의 대응물이 .fini_array 다. **__attribute__((destructor)) 함수**의 주소가 여기 들어간다.

// [user-space - C]
__attribute__((destructor))
void my_cleanup(void) {
    printf("my_cleanup called!\n");
}

int main(void) {
    printf("main started\n");
    return 0;
}

출력:

main started
my_cleanup called!

ELF 의 DT_FINI_ARRAY 가 — 이걸 ld.so 에게 알려준다.

# [user-space - bash]
$ readelf -d ./prog | grep FINI
 0x000000000000000d (FINI)               0x119c
 0x000000000000001a (FINI_ARRAY)         0x3db0
 0x000000000000001c (FINI_ARRAYSZ)       16 (bytes)

.fini_array 의 호출자

.fini_array 도 — __cxa_atexit 을 통해 호출된다. 정확히는:

[.fini_array 의 호출 메커니즘]

  ld.so 의 _dl_init 안에서:
    각 라이브러리의 .init_array 처리
    그리고 그 라이브러리의 _dl_fini 를 __cxa_atexit 으로 등록
    
  exit() 호출 시:
    __exit_funcs 큐 처리 (역순)
    그 안에서 _dl_fini 호출됨
    _dl_fini 가:
      각 라이브러리의 .fini_array 순회 호출 (역순)
      각 라이브러리의 _fini 호출

그래서 — 글로벌 객체 destructor 와 atexit 등록 함수가 — 하나의 큐에서 통합 처리된다. 순서는 등록 역순 (LIFO).

atexit 의 호출 순서 — LIFO

atexit 의 호출 순서는 LIFO. 마지막에 등록된 게 먼저 호출.

// [user-space - C 예제]
#include <stdio.h>
#include <stdlib.h>

void a(void) { printf("a\n"); }
void b(void) { printf("b\n"); }
void c(void) { printf("c\n"); }

int main(void) {
    atexit(a);
    atexit(b);
    atexit(c);
    printf("main exiting\n");
    return 0;
}

출력:

main exiting
c
b
a

LIFO 인 이유는 — 의존성 정리다. 나중에 등록된 게 먼저 등록된 것에 의존할 수 있으니, 그 의존성의 역순으로 정리.

main 의 return 의 미묘함

main 의 return 도 — exit(반환값) 과 같다. __libc_start_main 의 마지막에:

// [user-space - glibc __libc_start_main 의 끝]
result = main(argc, argv, __environ);
exit(result);

그래서 — return 0 와 exit(0) 는 동일한 결과. 둘 다 atexit 핸들러를 실행하고 _exit(0) 으로 끝.

_exit() 와 _Exit() — 정리 우회

_exit() (POSIX) 또는 _Exit() (C99) 는 — 모든 정리를 우회하고 즉시 종료.

// [user-space - glibc sysdeps/unix/sysv/linux/_exit.c master]
void
_exit(int status)
{
    while (1) {
        INLINE_SYSCALL_CALL(exit_group, status);
#ifdef ABORT_INSTRUCTION
        ABORT_INSTRUCTION;
#endif
    }
}

exit_group syscall 을 호출. **exit syscall 이 아니라 exit_group** 인 게 흥미롭다. exit 는 현재 thread 만 끝내고, exit_group 은 전체 process (모든 thread) 를 끝낸다.

[exit vs exit_group syscall]

  exit (syscall 60)        : 현재 thread 만 종료
                            다른 thread 들은 계속 실행
                            pthread_exit() 의 내부
                            
  exit_group (syscall 231) : 전체 프로세스 종료
                            모든 thread 종료
                            _exit(), _Exit(), main return 의 내부

atexit 의 위험 — fork 후

atexit 의 미묘한 위험이 있다 — fork 한 자식 프로세스도 등록된 atexit 함수들을 상속.

// [user-space - C - 위험한 패턴]
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

void cleanup(void) {
    printf("cleanup called! pid=%d\n", getpid());
}

int main(void) {
    atexit(cleanup);
    
    pid_t pid = fork();
    if (pid == 0) {
        printf("child exit\n");
        return 0;       /* 자식 종료 → cleanup 호출됨! */
    }
    
    printf("parent exit\n");
    return 0;           /* 부모도 cleanup 호출 */
}

출력:

parent exit
cleanup called! pid=12345     ← 부모
child exit
cleanup called! pid=12346     ← 자식도!

cleanup 이 두 번 호출된다. 자원 해제하는 함수면 — double-free 위험.

자식에서 atexit 안 부르고 싶으면 — _exit() 사용.

if (pid == 0) {
    /* child */
    _exit(0);   /* atexit 우회 */
}

또는 fork 후 자식에서 atexit 큐를 비우는 방법도 있지만 — 보통 _exit 가 깔끔.

exec 후의 atexit

fork 와 달리, **exec (execve) 후에는 atexit 큐가 새로 시작**한다. exec 가 모든 메모리를 새 프로그램으로 교체하니까 — 옛 atexit 큐도 사라짐.

[fork vs exec 의 atexit 상속]

  fork:  atexit 큐 복사됨 (자식도 호출 위험)
  exec:  atexit 큐 새로 시작 (메모리 다 교체됨)

signal 로 인한 종료

signal (SIGTERM, SIGINT, SIGSEGV 등) 로 프로세스가 죽으면 — 기본적으로 atexit 호출 안 됨. signal 의 default action 이 즉시 종료.

[signal 종료 시 atexit]

  SIGTERM (kill)     : default = 즉시 종료, atexit 안 호출
  SIGINT (Ctrl+C)    : default = 즉시 종료, atexit 안 호출
  SIGSEGV            : default = core dump + 종료, atexit 안 호출
  SIGKILL (kill -9)  : 어떻게도 잡을 수 없음, atexit 안 호출

atexit 호출되도록 하려면 — signal handler 안에서 명시적으로 exit() 호출.

// [user-space - C]
#include <signal.h>
#include <stdlib.h>

void handler(int sig) {
    exit(128 + sig);   /* atexit 호출됨 */
}

int main(void) {
    atexit(my_cleanup);
    signal(SIGTERM, handler);
    /* ... */
}

다만 — signal handler 안에서 exit 호출은 async-signal-safe 아님. 안전한 패턴 : flag 설정해서 main loop 에서 처리.

atexit 의 등록 한도

옛 C 표준은 atexit 의 최소 등록 한도를 32 개 로 요구했다. 현대 구현은 보통 무제한이다.

[atexit 등록 한도]

  C89/C90: ATEXIT_MAX >= 32
  POSIX:   ATEXIT_MAX >= 32
  glibc:   사실상 무제한 (메모리 한도)
  musl:    사실상 무제한

glibc 의 exit_function_list 가 — 32 개씩 노드로 연결된 linked list. 부족하면 새 노드 동적 할당.

__call_tls_dtors — thread-local 정리

thread_local (C11/C++11) 변수의 destructor 는 — __call_tls_dtors 가 처리.

// [user-space - C++ 예제]
class Logger {
public:
    Logger() { open_file(); }
    ~Logger() { close_file(); }
};

thread_local Logger tls_logger;   // thread 마다 별도

이 객체의 destructor 는 — thread 가 끝날 때 (pthread_exit 또는 main thread 의 exit) 호출. 일반적인 atexit 과는 별도 큐.

[exit 의 정리 순서 — 더 정확히]

  1. atexit / __cxa_atexit 큐 처리 (LIFO)
     → 그 안에서 _dl_fini 호출
     → 각 라이브러리의 .fini_array, _fini 호출
     
  2. __call_tls_dtors
     → thread_local 변수들의 destructor
     
  3. _exit / exit_group syscall

이 장의 결론

프로그램 종료의 흐름:

[종료 흐름 정리]

       │
       ▼

       │

21dynamic linker 와의 협조

지금까지 — _start 와 __libc_start_main 사이에서 일어나는 일을 봤다. 하지만 그 전에 — **kernel 이 점프한 첫 user-space 코드는 사실 dynamic linker (ld.so) 의 _start** 다 (15장 참고). 프로그램의 _start 가 호출되기 전에 — ld.so 가 매우 많은 일을 한다.

이 장은 dynamic linker 의 깊이를 본다.

ld.so 는 무엇인가

/lib64/ld-linux-x86-64.so.2 — 이게 dynamic linker 의 정체다. 이건 일반적인 .so 가 아니라 실행 가능한 ELF 파일.

ld.so 는 — 다른 ELF 의 INTERP 로 지정되어 호출되기도 하고, 직접 실행되기도 한다. 이 자체가 statically linked 라는 점에 주목 — ld.so 가 자신을 동적으로 link 할 수는 없으니까 (자기가 자기를 부르는 닭과 달걀 문제).

[glibc master 의 elf/ 디렉토리]

  elf/rtld.c               : dl_main — ld.so 의 본체
  elf/dl-init.c            : _dl_init — init_array 호출
  elf/dl-fini.c            : _dl_fini — fini_array 호출
  elf/dl-load.c            : 라이브러리 파일 로드
  elf/dl-lookup.c          : 심볼 검색
  elf/dl-reloc.c           : relocation 수행
  elf/dl-runtime.c         : _dl_fixup — lazy binding
  elf/dl-open.c            : dlopen
  elf/dl-close.c           : dlclose
  elf/dl-sym.c             : dlsym
  
  sysdeps/<arch>/dl-machine.h     : 아키텍처별 매크로
  sysdeps/<arch>/dl-trampoline.h  : _dl_runtime_resolve (asm)

ld.so 의 _start — _dl_start

ld.so 의 시작점은 — 프로그램의 _start 와는 별개의 함수. master 의 sysdeps/x86_64/dl-machine.h 의 RTLD_START 매크로 본체:

; [user-space - glibc sysdeps/x86_64/dl-machine.h RTLD_START master 인용]

.text
    .align 16
.globl _start
.globl _dl_start_user
_start:
    movq %rsp, %rdi              ; rsp = 인자 stack 의 시작 (argc 가 있는 곳)
    call _dl_start               ; ld.so 의 C 본체 (반환값 = user entry)

_dl_start_user:
    # Save the user entry point address in %r12.
    movq %rax, %r12              ; _dl_start 의 반환값 = 프로그램의 _start 주소
    
    # See if we were run as a command with the executable file
    # name as an extra leading argument.
    movl _dl_skip_args(%rip), %eax
    # Pop the original argument count.
    popq %rdx                    ; argc → rdx
    # Adjust the stack pointer to skip _dl_skip_args words.
    leaq (%rsp,%rax,8), %rsp     ; ld.so 자신의 인자들 (--library-path 등) skip
    # Subtract _dl_skip_args from argc.
    subl %eax, %edx              ; argc 보정
    # Push argc back on the stack.
    pushq %rdx
    
    # Call _dl_init (struct link_map *main_map, int argc, char **argv, char **env)
    # argc -> rsi
    movq %rdx, %rsi              ; argc → rsi (2nd arg)
    # Save %rsp value in %r13.
    movq %rsp, %r13              ; r13 = 현재 rsp 백업 (정렬 전)
    # And align stack for the _dl_init call.
    andq $-16, %rsp              ; 16-byte align
    # _dl_loaded -> rdi
    movq _rtld_local(%rip), %rdi ; main_map → rdi (1st arg)
    # env -> rcx
    leaq 16(%r13,%rdx,8), %rcx   ; env 의 주소 → rcx (4th arg)
    # argv -> rdx
    leaq 8(%r13), %rdx           ; argv → rdx (3rd arg)
    # Clear %rbp to mark outermost frame obviously even for constructors.
    xorl %ebp, %ebp              ; frame pointer 0
    # Call the function to run the initializers.
    call _dl_init                ; .init_array 호출 (의존 라이브러리들)
    
    # Pass our finalizer function to the user in %rdx, as per ELF ABI.
    leaq _dl_fini(%rip), %rdx    ; rtld_fini = _dl_fini → rdx
                                  ; → 프로그램의 _start 가 r9 로 옮김
    # And make sure %rsp points to argc stored on the stack.
    movq %r13, %rsp              ; rsp 복원 (정렬 전으로)
    
    # Jump to the user's entry point.
    jmp *%r12                    ; 프로그램의 _start 로 점프!

매우 정교한 어셈블리다. 핵심 동작 요약:

[_dl_start_user 의 동작 핵심]

  1. _dl_start 의 반환값 (= 프로그램의 _start 주소) 을 r12 에 보관
  
  2. _dl_skip_args 처리
     ld.so 가 직접 실행될 때 (예: /lib64/ld-linux.so.2 ./hello)
     extra arg 가 있을 수 있음 → skip
  
  3. _dl_init 호출
     인자: (main_map, argc, argv, env)
     의존 라이브러리들의 .init_array 호출
     (메인 프로그램은 skip — 19장의 call_init 동작)
  
  4. _dl_fini 의 주소를 rdx 에 (rtld_fini 인자 위치)
     → 프로그램의 _start 가 이걸 r9 로 옮겨서 __libc_start_main 의 6번째 인자로
  
  5. 프로그램의 _start 로 점프

ELF ABI 가 — entry point 진입 시 %rdx 에 dynamic linker 의 정리 함수 주소가 있다고 정의한다 (15장의 kernel 의 stack 구성에서 이 부분). ld.so 가 — 자기 _dl_fini 를 rdx 에 넣고 프로그램의 _start 로 점프 → 프로그램의 _start 가 mov %RDX_LP, %R9_LP 로 받음 (16장).

핵심 흐름:

[ld.so 의 흐름]

  1. _start (ld.so 내부)
     → 프로그램의 _start 가 아니라 ld.so 의 _start 다!
     → kernel 이 PT_INTERP 따라 여기로 점프
     
  2. _dl_start
     → ld.so 자기 자신 부트스트랩
     → 매우 미묘한 코드 (자기 GOT 가 아직 reloc 안 됨)
     → 결국 dl_main 호출
     
  3. dl_main (elf/rtld.c)
     → 본격적인 동적 링크 작업
     → 의존 라이브러리들 찾아서 로드
     → relocation 수행
     → 프로그램의 entry 주소 결정
     
  4. _dl_init
     → 모든 라이브러리의 .init_array 호출
     → 의존성 graph 의 leaf 부터 (depth-first post-order)
     
  5. 프로그램의 _start 로 점프
     → 이제부터 사용자 영역 (chapter 16-17)

dl_main 의 흐름

master 의 elf/rtld.c 의 dl_main 이 가장 복잡한 함수 (수천 줄). 핵심:

// [user-space - glibc elf/rtld.c dl_main 핵심 흐름 단순화]
static void
dl_main(const ElfW(Phdr) *phdr,
        ElfW(Word) phnum,
        ElfW(Addr) *user_entry,
        ElfW(auxv_t) *auxv)
{
    /* 1. 환경 변수 파싱 (LD_*) */
    process_envvars(&state);
    
    /* 2. 프로그램의 link_map 만들기 */
    main_map = _dl_new_object(...);
    
    /* 3. LD_PRELOAD 처리 */
    if (preloadlist)
        handle_ld_preload(preloadlist);
    
    /* 4. 의존 라이브러리 검색 + 로드 (재귀적) */
    _dl_map_object_deps(main_map, ...);
    /* ↑ DT_NEEDED 의 .so 들을 모두 mmap */
    
    /* 5. 모든 라이브러리의 relocation 수행 */
    for (l = _dl_loaded; l; l = l->l_next)
        _dl_relocate_object(l, ...);
    
    /* 6. ifunc resolver 호출 (13장 참조) */
    /* 각 STT_GNU_IFUNC 심볼에 대해 resolver 호출 */
    
    /* 7. main_map 에서 entry point 추출 */
    *user_entry = main_map->l_entry;    /* = 프로그램의 _start */
    
    /* 8. _dl_starting_up = 0 (시작 끝) */
    _dl_starting_up = 0;
}

각 단계가 매우 깊다.

PLT 와 GOT — lazy binding

가장 흥미로운 부분이 PLT (Procedure Linkage Table) + GOT (Global Offset Table) 의 메커니즘. 외부 함수 호출의 동작 방식이다.

[PLT/GOT 의 lazy binding]

  문제: printf 같은 외부 함수의 주소를 매번 로드 시점에 
        모두 해결하면 시간 낭비.
        실제로 호출되지 않는 함수의 주소까지 해결할 필요는 없음.
        
  해결: lazy binding
        첫 호출 시점에만 주소 해결.
        그 다음부터는 캐시된 주소 사용.

구조:

┌─────────────────────────────────────────────────────────┐
│ PLT/GOT 의 구조                                            │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  .plt (Procedure Linkage Table):                         │
│    실행 가능한 코드. 외부 함수 호출의 wrapper.              │
│    각 외부 함수마다 entry 1 개.                            │
│                                                          │
│  .got.plt (Global Offset Table for PLT):                 │

│      [0] = .dynamic 의 주소                              │

│      [2] = _dl_runtime_resolve 의 주소                  │
│    그 뒤로 각 외부 함수마다 entry 1 개.                    │
│                                                          │
└─────────────────────────────────────────────────────────┘

첫 호출 vs 두 번째 호출

printf 의 첫 호출 시점:

[printf 의 첫 호출]

  user code:
    call printf@PLT          ← .plt 의 printf entry 호출
    
  .plt 의 printf entry:
    1. jmp *printf@GOT(%rip) ← .got.plt 의 entry 읽어 점프
                              아직 미해결: 다음 명령어를 가리킴
    2. push $printf_index     ← 어느 함수인지 식별
    3. jmp .plt0              ← PLT 의 0 번째 entry 로
    
  .plt0:
    1. push 8(GOT)            ← link_map 주소 push
    2. jmp *16(GOT)           ← _dl_runtime_resolve 호출
    
  _dl_runtime_resolve:
    1. 인자 register 들 저장 (asm)
    2. _dl_fixup 호출 (C 코드)
       → printf 의 실제 주소 검색
       → .got.plt 의 entry 를 실제 주소로 갱신
    3. register 들 복원
    4. printf 의 실제 주소로 점프
    
  → printf 실행 + 반환

두 번째 호출은 — 이미 .got.plt 가 실제 주소로 갱신되어 있으니, 첫 명령어 (jmp *printf@GOT) 가 바로 printf 로 점프. resolver 우회.

[printf 의 두 번째 호출]

  user code:
    call printf@PLT
    
  .plt 의 printf entry:
    1. jmp *printf@GOT(%rip)
       → .got.plt 가 이제 실제 printf 주소를 가짐
       → 바로 printf 로 점프
       
  → printf 실행 (resolver 안 거침)

_dl_runtime_resolve 의 본체

master 의 sysdeps/x86_64/dl-trampoline.h:

; [user-space - glibc sysdeps/x86_64/dl-trampoline.h master 단순화]
_dl_runtime_resolve:
    ; 모든 인자 register 저장 (rdi, rsi, rdx, rcx, r8, r9)
    ; SSE/AVX register 도 저장 (resolver 가 변경 가능하므로)
    push %rdi
    push %rsi
    push %rdx
    ...
    
    ; _dl_fixup(link_map, reloc_index) 호출
    ; → 진짜 함수 주소 반환
    mov 16+8(%rsp), %rdi    ; link_map (PLT 에서 push 한 것)
    mov 8+8(%rsp), %rsi     ; reloc_index
    call _dl_fixup
    
    ; rax = 진짜 함수 주소
    mov %rax, %r11          ; 점프 위해 보관
    
    ; register 복원
    pop %rdi
    pop %rsi
    ...
    
    ; 진짜 함수로 점프
    jmp *%r11

_dl_fixup (C 함수) 가 — .got.plt 의 entry 를 갱신하고, 실제 주소를 반환.

LD_BIND_NOW — lazy binding 끄기

LD_BIND_NOW=1 환경 변수를 설정하면 — ld.so 가 로드 시점에 모든 함수 주소를 미리 해결한다.

# [user-space - bash]
$ LD_BIND_NOW=1 ./myapp
# 시작 시간 더 길어짐, 하지만 첫 함수 호출이 빠름

trade-off:

[lazy binding vs LD_BIND_NOW]

  Lazy (default):
    + 시작 빠름
    - 첫 호출에 약간의 overhead
    - 안 쓰이는 함수도 메모리에 매핑됨
    
  LD_BIND_NOW=1:
    - 시작 느림 (모든 .got.plt 미리 채움)
    + 모든 호출이 빠름
    + 보안 (GOT 가 read-only 가능)

-Wl,-z,now linker flag 로 binary 안에 박을 수도 있다 (DT_BIND_NOW 또는 DT_FLAGS 의 DF_BIND_NOW).

GNU_RELRO — 보안 강화

-Wl,-z,relro (기본) + -Wl,-z,now 조합이 Full RELRO (Relocation Read-Only).

[RELRO 의 보호 효과]

  Partial RELRO (-Wl,-z,relro, 기본):
    .got 은 read-only
    .got.plt 는 여전히 writable (lazy binding 위해)
    공격자가 .got.plt 의 entry 덮어쓰면 → 임의 함수 호출
    
  Full RELRO (-Wl,-z,relro,-z,now):
    .got 와 .got.plt 모두 read-only
    모든 binding 즉시 해결 후 read-only 로 변경
    GOT 공격 불가능
    하지만 시작 시간 약간 느림

readelf -l 에 보이는 GNU_RELRO segment 가 — RELRO 영역.

# [user-space - bash]
$ readelf -l ./hello | grep RELRO
  GNU_RELRO       0x000000000000f5b0 0x00000000000205b0 ...    R   0x1

dlopen / dlsym / dlclose

런타임에 라이브러리를 로드하는 것도 ld.so 의 기능.

// [user-space - C 예제]
#include <dlfcn.h>

void *handle = dlopen("./mylib.so", RTLD_LAZY);
if (!handle) { fprintf(stderr, "%s\n", dlerror()); exit(1); }

/* 심볼 찾기 */
int (*my_func)(int) = dlsym(handle, "my_func");
int result = my_func(42);

dlclose(handle);

내부:

[dlopen 의 동작]

  dlopen("./mylib.so", RTLD_LAZY)
    → _dl_open
        → _dl_map_object: mmap 으로 .so 로드
        → _dl_open_check: 의존성 처리
        → _dl_relocate_object: relocation 수행
        → _dl_init: .init_array 호출
    → handle 반환
    
  dlsym(handle, "func")
    → _dl_sym
        → handle 의 dynamic symbol table 검색
        → 심볼 주소 반환
        
  dlclose(handle)
    → _dl_close
        → __cxa_finalize: __cxa_atexit 으로 등록된 destructor 호출
                          이 .so 의 것만 (dso_handle 매칭)
        → .fini_array 호출
        → DT_FINI (_fini) 호출
        → munmap 으로 메모리 해제

audit interface — LD_AUDIT

매우 강력한 기능 — LD_AUDIT 환경 변수로 — dynamic linking 의 모든 동작을 가로챌 수 있다.

// [user-space - audit 라이브러리 예제 - 단순화]
#include <link.h>

unsigned int la_version(unsigned int version) { return version; }

void la_preinit(uintptr_t *cookie) {
    fprintf(stderr, "preinit called\n");
}

uintptr_t la_symbind64(Elf64_Sym *sym, unsigned int ndx,
                       uintptr_t *refcook, uintptr_t *defcook,
                       unsigned int *flags, const char *symname) {
    fprintf(stderr, "binding symbol: %s\n", symname);
    return sym->st_value;
}

LD_AUDIT=./myaudit.so ./hello 로 모든 심볼 해결을 모니터.

이 장의 결론

dynamic linker 의 역할:

[ld.so 의 역할 정리]

  ✓ kernel 이 처음 점프하는 user-space 코드 (PT_INTERP)
  ✓ 의존 라이브러리들 검색 + 로드 (DT_NEEDED)
  ✓ relocation 수행 (.rela.dyn, .rela.plt)
  ✓ ifunc resolver 호출 (R_X86_64_IRELATIVE)
  ✓ .init_array 호출 (의존성 graph 순서로)
  ✓ 프로그램의 _start 로 점프
  ✓ 런타임: PLT/GOT 의 lazy binding (_dl_runtime_resolve)
  ✓ dlopen/dlsym/dlclose 의 처리
  ✓ 종료 시 _dl_fini 로 .fini_array 처리

dynamic linker 는 — Linux user-space 의 거의 모든 프로그램에서 — 시작과 끝에 관여한다. 매우 정교한 시스템.

다음 마지막 장에서는 — static 으로 link 했을 때 어떻게 달라지는지, dynamic 과 static 의 미묘한 차이들을 본다.

22static vs dynamic 링크 시 차이

Part III 의 마지막 장. dynamic linked binary 의 동작은 깊이 봤지만 — static linked binary 는 어떻게 다른가? 이 장은 그 차이의 모든 측면을 본다.

한 줄 정리

[static vs dynamic 의 핵심 차이]

  Dynamic linked:
    필요한 라이브러리들이 _별도 파일_ (.so) 로 남아 있음
    실행 시점에 dynamic linker (ld.so) 가 로드
    여러 프로세스가 같은 .so 의 .text 를 공유 (메모리 절약)
    
  Static linked:
    필요한 모든 코드를 _하나의 실행 파일_ 에 박음
    dynamic linker 불필요
    각 프로세스가 자기 사본 (메모리 더 사용)
    self-contained (디스크에 .so 없어도 실행됨)

빌드 방법

# [user-space - bash]
# Dynamic linked (기본)
$ gcc hello.c -o hello_dynamic

# Static linked
$ gcc -static hello.c -o hello_static

# 또는 특정 라이브러리만 static (glibc)
$ gcc hello.c -static-libgcc -o hello_partial

ELF 헤더의 차이

# [user-space - bash]
$ readelf -h hello_dynamic | grep -E "Type|Entry"
  Type:                              DYN (Position-Independent Executable file)
  Entry point address:               0x10c0

$ readelf -h hello_static | grep -E "Type|Entry"
  Type:                              EXEC (Executable file)
  Entry point address:               0x401b50

Type 이 다르다:

[ELF Type 의 차이]

  Dynamic linked:
    Type: DYN (= PIE)
    base 가 랜덤하게 결정 (ASLR)
    매번 다른 주소에 매핑
    
  Static linked (기본):
    Type: EXEC
    고정 base (예: 0x400000)
    매번 같은 주소
    ASLR 적용 안 됨

static binary 도 PIE 로 만들 수는 있다 (-static-pie).

# [user-space - bash]
$ gcc -static-pie hello.c -o hello_static_pie
$ readelf -h hello_static_pie | grep Type
  Type:                              DYN

PT_INTERP 의 유무

# [user-space - bash]
$ readelf -l hello_dynamic | grep INTERP
  INTERP         0x...    [Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]

$ readelf -l hello_static | grep INTERP
# 아무것도 출력 안 됨

static binary 는 — PT_INTERP 가 없다. **kernel 이 dynamic linker 를 로드하지 않고, 직접 프로그램의 _start 로 점프**.

파일 크기

# [user-space - bash]
$ ls -lh hello_dynamic hello_static
-rwxr-xr-x 1 user user  16K Apr 15 hello_dynamic
-rwxr-xr-x 1 user user 868K Apr 15 hello_static

약 54 배 차이. static binary 는 — glibc 의 일부 (필요한 함수만) 가 박혀 있다.

[크기 차이의 이유]

  Dynamic:
    실행 파일은 작음
    하지만 시스템에 libc.so (~2 MB), 기타 .so 필요
    여러 프로세스가 공유 (메모리에서)
    
  Static:
    실행 파일에 모든 코드 포함
    libc 의 필요한 함수만 추출되어 link
    self-contained

실행 흐름의 차이 — 시작점

[Dynamic linked 의 시작 흐름]

  execve("./hello_dynamic")
    │
    ▼
  kernel
    - ELF 파싱
    - PT_INTERP 발견 → /lib64/ld-linux-x86-64.so.2 로드
    - LOAD segments 매핑
    - stack 구성
    - entry = ld.so 의 _start
    │
    ▼
  ld.so 의 _start
    - _dl_start → dl_main
    - 의존 라이브러리들 로드 (libc.so 등)
    - relocation
    - _dl_init (모든 init_array 호출)
    - 프로그램의 _start 로 점프
    │
    ▼
  프로그램의 _start (crt1.o)
    - __libc_start_main 호출
    │
    ▼
  __libc_start_main (libc.so 안)
    - 초기화
    - main 호출
[Static linked 의 시작 흐름]

  execve("./hello_static")
    │
    ▼
  kernel
    - ELF 파싱
    - PT_INTERP _없음_
    - LOAD segments 매핑
    - stack 구성
    - entry = 프로그램의 _start
    │
    ▼
  프로그램의 _start (crt1.o)
    - 직접 __libc_start_main 호출
    │
    ▼
  __libc_start_main (binary 안에 박혀 있음)
    - 매우 많은 초기화 직접 수행:
      __pthread_initialize_minimal
      __libc_init_first
      _dl_relocate_static_pie (PIE 인 경우)
      _init
      .init_array 순회 호출
    - main 호출

static 의 경우 ld.so 가 하던 일을 — __libc_start_main 이 직접 한다.

__libc_start_main 의 분기 — SHARED vs static

이전에 봤듯이 — master 의 csu/libc-start.c 안에 분기가 있다.

// [user-space - glibc csu/libc-start.c master 분기 단순화]
STATIC int
LIBC_START_MAIN (int (*main)(int, char**, char**),
                 int argc, char **argv,
                 ...)
{
    /* env, stack_end 등 공통 작업 */
    
#ifndef SHARED
    /* static binary 일 때만: 직접 초기화 */
    __pthread_initialize_minimal();
    __libc_init_first(argc, argv, __environ);
    
    /* PIE 인 경우 self-relocation */
    _dl_relocate_static_pie();
    
    /* .init_array 호출 */
    _init();    /* DT_INIT */
    const size_t size = __init_array_end - __init_array_start;
    for (size_t i = 0; i < size; i++)
        (*__init_array_start[i])(argc, argv, __environ);
#endif
    /* SHARED 빌드: ld.so 가 이미 다 처리함 */
    
    /* main 호출 */
    int result = main(argc, argv, __environ);
    exit(result);
}

SHARED 빌드 = glibc 가 .so 로 빌드될 때 = dynamic linked binary 가 link 하는 버전. static 빌드 = 프로그램 자체가 static 일 때 link 되는 버전.

memory layout 의 차이

[Dynamic linked 의 메모리 layout (대표)]

  0x7ffe_xxxx_xxxx  [stack]
                   :
                   :
  0x7f44_xxxx_xxxx  [libc.so]              ← 공유 라이브러리들
  0x7f44_yyyy_yyyy  [libpthread.so]
  0x7f44_zzzz_zzzz  /lib64/ld-linux.so.2   ← dynamic linker
                   :
                   :
  0x5555_5555_5xxx  [vdso]
                   :
  0x5555_5555_5xxx  [hello_dynamic]         ← 프로그램 (PIE base)
                   :
                   :
  0x0000_0000_0000  (low)
[Static linked 의 메모리 layout (대표)]

  0x7ffe_xxxx_xxxx  [stack]
                   :
                   :
  (그 사이는 모두 비어 있음, .so 들 없음)
                   :
                   :
                   :
                   :
  0x0000_0040_0000  [hello_static]
                    (text, data, bss 모두 한 binary 안)
                   :
  0x0000_0000_0000  (low)

static binary 는 — 매우 단순한 layout. 모든 게 한 곳에 있다.

메모리 사용량

여러 프로그램이 동시에 실행될 때:

[메모리 효율의 차이]

  10 개의 dynamic binary 동시 실행:
    각 binary: 16 KB × 10 = 160 KB
    공유 libc.so: 2 MB (모두가 공유)
    공유 libpthread.so: 100 KB
    공유 ld.so: 200 KB
    총: ~2.5 MB
    
  10 개의 static binary 동시 실행 (각 약 1 MB):
    1 MB × 10 = 10 MB
    
  → static 이 4 배 더 많은 메모리 사용

다만 — kernel 의 page cache 가 같은 binary 의 여러 instance 에서는 메모리 공유. 그래도 다른 binary 들끼리는 공유 안 됨.

시작 시간

# [user-space - bash]
$ time ./hello_dynamic > /dev/null
real    0m0.003s

$ time ./hello_static > /dev/null
real    0m0.002s

미묘하게 — static 이 더 빠르다. dynamic linker 의 작업 시간이 절약.

[시작 시간 차이의 이유]

  Dynamic:
    1. kernel 이 ld.so 도 매핑
    2. ld.so 의 부트스트랩
    3. 의존 라이브러리들 (libc.so 등) 검색 + 매핑
    4. relocation 수행
    5. .init_array 호출
    → 수 ms 정도
    
  Static:
    1. binary 만 매핑
    2. 직접 _start
    → 수십 µs 만에 main

다만 ld.so 의 시작 비용은 — 실제로는 매우 작다 (몇 ms). 큰 프로그램일수록 차이가 사소.

보안 비교

[보안 측면]

  Dynamic linked:
    + ASLR (PIE 의 기본)
    + lazy binding 의 PLT/GOT 공격 가능
    - GOT 공격으로 임의 함수 호출 가능 (RELRO 안 쓸 때)
    + 라이브러리 업데이트로 보안 패치 즉시 적용
    
  Static linked:
    - 기본은 EXEC (ASLR 안 됨, -static-pie 로 가능)
    + PLT/GOT 없음 (lazy binding 없음)
    + binary 안의 모든 함수 주소가 고정
    - 보안 패치마다 재컴파일 필요
    - 알려진 취약점도 그대로 박힘

배포 측면에서 — dynamic 이 더 안전하다 (배포자가 libc 업데이트하면 모든 binary 가 혜택).

라이브러리 의존성

# [user-space - bash]
$ ldd hello_dynamic
        linux-vdso.so.1 (0x00007ffe92d65000)
        libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f4a8a72a000)
        /lib64/ld-linux-x86-64.so.2 (0x00007f4a8a91a000)

$ ldd hello_static
        not a dynamic executable

ldd 가 — static binary 에 대해 "not a dynamic executable" 을 출력. 의존성 없이 독립적.

syscall 처리의 차이

매우 미묘하지만 — syscall wrapper 가 약간 다르다.

[dynamic 의 syscall 호출]

  user 코드:
    write(1, buf, n)
    │
    ▼
  PLT (write@plt)
    │ jmp *write@GOT (해결 후)
    ▼
  libc.so 의 write
    │ INLINE_SYSCALL
    ▼
  syscall 명령어 → kernel
[static 의 syscall 호출]

  user 코드:
    write(1, buf, n)
    │
    ▼  (PLT 없이 직접 call)
  binary 안의 write (libc 의 일부)
    │ INLINE_SYSCALL
    ▼
  syscall 명령어 → kernel

static 이 — PLT 단계 한 번 절약. 매우 작은 차이.

glibc 의 static link 제약

glibc 는 — static link 를 일부 권장하지 않는다.

[glibc 의 static link 알림]

  $ gcc -static hello.c -lc
  /usr/bin/ld: warning: Using 'gethostbyname' in statically linked applications 
  requires at runtime the shared libraries from the glibc version used for linking

이유:

[glibc static link 의 한계]

  1. NSS (Name Service Switch):
     getpwnam, gethostbyname 등이 동적으로 모듈을 로드
     /etc/nsswitch.conf 의 설정에 따라 다른 .so 로드
     → static binary 도 결국 .so 가 필요
     
  2. dlopen:
     dlopen 자체가 동적 link 메커니즘
     static binary 에서는 동작 안 함 (또는 제한적)
     
  3. libpthread:
     동적 link 와 통합되어 설계됨
     static 에서는 일부 기능 제약

이런 제약 때문에 — musl 이 static linking 에 더 친화적. musl 은 처음부터 static 을 1급 시민으로 설계.

Go 와 Rust 의 static binary

Go 는 — 기본적으로 static linked binary 를 만든다 (cgo 안 쓰면).

# [user-space - bash]
$ go build hello.go
$ ldd hello
        not a dynamic executable

Rust 도 musl target 으로 static binary 가능.

# [user-space - bash]
$ rustup target add x86_64-unknown-linux-musl
$ cargo build --target x86_64-unknown-linux-musl --release
$ ldd target/x86_64-unknown-linux-musl/release/hello
        statically linked

container 의 시대에 — static binary 가 다시 인기. distroless 이미지에서 의존성 없는 binary 가 운영 편리.

언제 static, 언제 dynamic?

[선택 가이드]

  Static 권장:
    - container 의 distroless 이미지
    - rescue / recovery 도구 (libc 없는 환경)
    - 임베디드 시스템 (rootfs 작게)
    - 보안 sensitive (의존성 통제)
    - musl 기반 (Alpine Linux 등)
    - Go, Rust 의 default
    
  Dynamic 권장 (일반):
    - 데스크탑 / 서버 일반 프로그램
    - 여러 binary 가 같은 라이브러리 사용
    - 라이브러리 업데이트 시 모든 binary 가 자동 혜택
    - 메모리 효율 중요한 경우
    - plugins / dlopen 필요

Part III 의 결론

Part III 전체를 마무리한다. ./hello 의 한 줄 실행이 — 얼마나 깊은 시스템 호출의 사슬을 부르는지 봤다.

[Part III 의 전체 흐름 정리]

  user: ./hello
    │
    ▼
  shell: fork + execve("./hello", ...)
    │
    ▼
  kernel: do_execveat_common
    → ELF magic 검사
    → PT_INTERP 발견 (dynamic 의 경우)
    → begin_new_exec (메모리 교체)
    → LOAD segments mmap (ASLR)
    → dynamic linker 도 매핑
    → stack 에 argc/argv/envp/auxv 배치
    → entry point 로 점프 (iret/sysret)
    │
    ▼
  [Dynamic linked 의 경우]
  ld.so 의 _start
    → _dl_start, dl_main
    → 의존 라이브러리들 로드
    → relocation
    → ifunc resolver 호출
    → _dl_init: 의존 라이브러리들의 .init_array 호출
                (leaf 부터, 메인 프로그램은 skip)
    → rdx = _dl_fini (ELF ABI: rtld_fini 인자)
    → 프로그램의 _start 로 점프
    │
    ▼
  프로그램의 _start (crt1.o, glibc start.S)
    → frame pointer = 0
    → stack 에서 argc, argv 추출 + register 배치
    → rcx = NULL (init), r8 = NULL (fini), r9 = _dl_fini
    → 16-byte stack align
    → __libc_start_main 호출
    │
    ▼
  __libc_start_main (csu/libc-start.c)
    → __environ 설정
    → __pthread_initialize_minimal (TLS, TCB, %fs)
    → stack canary 설정 (AT_RANDOM)
    → pointer mangling 키 설정
    → rseq 등록
    → rtld_fini (= _dl_fini) 를 __cxa_atexit 으로 등록
    → call_init: 메인 프로그램의 .init_array 호출
                 (DT_INIT_ARRAY, csu 의 별도 call_init 함수)
    → setjmp + main 호출
    │
    ▼
  main(argc, argv, envp)
    → 사용자 코드 실행
    │
    ▼ (main 반환 또는 exit)
  exit(status)
    → atexit 큐 처리 (LIFO)
       → C++ 글로벌 destructor
       → _dl_fini (ld.so 의 정리)
          → 의존 라이브러리들의 .fini_array
          → 의존 라이브러리들의 _fini
       → 사용자 atexit 함수들
    → thread-local destructor
    → exit_group syscall
    │
    ▼
  kernel: 프로세스 종료
    → 열린 파일 닫기
    → 메모리 해제
    → 부모에게 SIGCHLD

main 한 줄 뒤에 — 이 모든 시스템이 있다. 이게 — Linux 의 user-space program 의 진짜 실체다.
