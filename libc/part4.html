Part IVmalloc 의 깊이 — 동적 메모리의 정체

지금까지 — 프로그램의 시작 (Part III), libc 의 구조 (Part II), kernel 과의 인터페이스 (Part I) 를 봤다. 이제 — C 프로그램이 가장 자주 사용하는 함수 중 하나로 들어간다.

malloc(size).

이 한 줄 호출 안에는 — 수십 년에 걸쳐 진화한 매우 정교한 메커니즘이 있다.

// [user-space - C]
char *buf = malloc(1024);
// 이 호출 안에서 무슨 일이 일어나는가?

답이 단순하지 않다. 할당 크기에 따라 완전히 다른 경로를 거친다.

[malloc 호출의 분기 — 미리 보기]

  malloc(size) 호출:
       │
       ▼
  ┌─────────────────────────────────────────┐
  │ tcache 에 적합한 크기의 free chunk 있는가?  │
  │ (glibc 2.26+)                            │
  └────────┬──────────────────┬─────────────┘
           │ YES              │ NO
           ▼                  ▼
       ┌───────┐    ┌────────────────────────────┐
       │ 반환! │    │ fastbin 에 있는가?           │
       └───────┘    │ (≤ 80 bytes 정도)            │
                   └────────┬───────────────┬───┘
                            │ YES           │ NO
                            ▼               ▼
                       ┌───────┐    ┌──────────────────┐
                       │ 반환! │    │ smallbin?         │
                       └───────┘    │ (< 512 bytes)     │
                                    └────────┬─────┬───┘
                                             │     │
                                             ▼     ▼
                                       ┌────────┐  ...
                                       │ unsortedbin │
                                       │ largebin    │
                                       │ top chunk   │
                                       │ heap 확장    │
                                       │ mmap        │
                                       └────────┘

이 모든 경로를 — 데이터 구조 (malloc_chunk, malloc_state) 와 알고리즘 (binning, coalescing, threshold) 로 풀어낸다.

[Part IV 의 학습 목표]

  ✓ heap 의 정체 — brk 와 mmap 의 분담
  ✓ malloc_chunk 자료구조 (boundary tag, in-use 비트 등)
  ✓ arena 의 multi-thread 분리
  ✓ bin 종류 (fast, small, large, unsorted, tcache)
  ✓ coalescing — 단편화 대응
  ✓ free 의 안전 장치 (double-free, corruption 탐지)
  ✓ glibc / musl / jemalloc / mimalloc 비교

이 Part 가 끝나면 — malloc(N) 한 줄을 보고 어떤 경로를 거치는지 머릿속에 그릴 수 있게 된다.

23heap 의 정체 — brk 와 mmap

먼저 가장 근본적인 질문부터. malloc 은 어디에서 메모리를 가져오는가?

답 — kernel 에게서. 그것도 두 가지 다른 방법으로.

두 가지 메모리 확장 방법

malloc 이 kernel 에게서 메모리를 받는 방법은 두 가지다.

┌───────────────────────────────────────────────────────────┐
│ malloc 이 kernel 에게서 메모리 받는 방법                       │
├───────────────────────────────────────────────────────────┤
│                                                            │
│  1. brk / sbrk                                             │
│     - 옛 메커니즘 (Unix V7 시절부터)                          │
│     - "program break" (data segment 의 끝) 를 위로 이동       │
│     - 연속된 영역 확장만 가능                                  │
│     - 빠르고 단순                                            │
│     - main_arena 의 heap 에 사용                             │
│                                                            │
│  2. mmap (MAP_ANONYMOUS)                                   │
│     - 새 메커니즘 (4.2BSD 이후)                              │
│     - 임의의 가상 주소 공간에 새 영역 매핑                       │
│     - 따로따로 매핑/해제 가능                                  │
│     - 큰 할당 (≥ MMAP_THRESHOLD)                            │
│     - thread arena 의 heap 에 사용                          │
│     - munmap 으로 OS 에 반환 가능                            │
│                                                            │
└───────────────────────────────────────────────────────────┘

brk 와 sbrk 의 동작

먼저 brk 와 sbrk 의 정체.

// [user-space - libc API]
#include <unistd.h>

int   brk(void *addr);    // program break 를 addr 로 설정
void *sbrk(intptr_t inc); // break 를 inc 만큼 증가, 옛 break 반환

sbrk(0) 은 — 현재 break 위치를 반환. sbrk(N) 은 — break 를 N 바이트 증가시키고, 증가하기 전 위치를 반환 (즉 새로 확보된 영역의 시작 주소).

// [user-space - C 예제]
#include <stdio.h>
#include <unistd.h>

int main(void) {
    void *initial = sbrk(0);              // 현재 break
    printf("initial break: %p\n", initial);
    
    void *new_area = sbrk(4096);          // 4KB 증가
    printf("got new area: %p\n", new_area);
    printf("new break:    %p\n", sbrk(0));
    
    /* new_area 부터 4096 바이트가 이제 우리 것 */
    *(int *)new_area = 42;
    
    sbrk(-4096);                          // 다시 줄임
    return 0;
}

출력 예:

initial break: 0x55bcd5b9d000
got new area:  0x55bcd5b9d000     ← 시작 주소 (initial 과 같음)
new break:     0x55bcd5b9e000     ← 4KB 증가됨

brk 의 의미 — "data segment 의 끝"

brk 의 의미를 그림으로 보면:

[프로세스의 메모리 layout 과 brk]

  (high)                                          
  ┌────────────────────────────┐                  
  │       stack                │ ← 위에서 아래로 자람
  │   (rsp 가 가리킴)            │                  
  ├────────────────────────────┤                  
  │                            │                  
  │    매핑되지 않은 영역         │                  
  │  (여기에 .so, mmap 영역 등)   │                  
  │                            │                  
  ├────────────────────────────┤  ← 현재 break    
  │                            │                  
  │       heap                 │ ← brk 로 자람     
  │   (malloc 영역)              │                  
  │                            │                  
  ├────────────────────────────┤                  
  │       .bss                 │                  
  │  (0 초기화 전역 변수)         │                  
  ├────────────────────────────┤                  
  │       .data                │                  
  │  (초기화된 전역 변수)         │                  
  ├────────────────────────────┤                  
  │       .rodata              │                  
  │       .text (코드)          │                  
  └────────────────────────────┘                  
  (low)

옛 Unix 의 메모리 모델에서 — data segment 의 끝이 brk. 이 break 를 위로 옮기면 — 그 사이가 새로 할당된 메모리. malloc 이 이걸 활용해서 heap 이라는 추상화를 만든다.

brk syscall — kernel 의 시각

brk 는 — libc 함수가 아니라 syscall 이다. master 의 mm/mmap.c:

// [kernel-space - mm/mmap.c master 단순화]
SYSCALL_DEFINE1(brk, unsigned long, brk)
{
    struct mm_struct *mm = current->mm;
    unsigned long origbrk = mm->brk;
    unsigned long newbrk = PAGE_ALIGN(brk);
    
    /* 페이지 단위로 align */
    
    if (newbrk == origbrk) {
        /* 변경 없음 */
        return origbrk;
    }
    
    if (newbrk < mm->start_brk) {
        /* 시작 위치 이하로는 못 줄임 */
        return mm->brk;
    }
    
    if (newbrk < origbrk) {
        /* 줄이는 경우: 그 영역 unmap */
        do_munmap(mm, newbrk, origbrk - newbrk, ...);
    } else {
        /* 늘리는 경우: 새 페이지 매핑 */
        do_brk_flags(origbrk, newbrk - origbrk, 0);
    }
    
    mm->brk = newbrk;
    return newbrk;
}

kernel 이 mm_struct->brk 라는 변수를 관리. 이걸 위로 옮기면 해당 영역이 mapping 된다. 아래로 옮기면 unmap.

[brk syscall 의 정리]

  return 값:
    성공 시: 새 break 주소
    실패 시: 옛 break 주소 (변경 안 됨)
    
  libc 의 sbrk wrapper:
    sbrk(inc) = old_break = brk();
                brk(old_break + inc);
                return old_break;

brk 의 한계 — 단편화 문제

brk 의 큰 한계가 있다 — 연속된 영역만 다룬다. 중간에 구멍 못 만듬.

[brk 의 단편화 문제]

  malloc 1 KB → heap_start ~ +1024
  malloc 4 KB → +1024 ~ +5120
  malloc 1 KB → +5120 ~ +6144
  
  중간 4 KB 를 free:
  
  ┌──────┬───────────┬──────┐
  │ A 1K │  free 4K  │ C 1K │
  └──────┴───────────┴──────┘
  
  → 이 free 4KB 를 OS 에 반환?
    못 함! brk 는 끝에서만 줄일 수 있음.
    C 가 있는 한 — 그 사이 4KB 를 OS 에 못 돌려줌.

이게 — heap 의 fragmentation (단편화) 문제. malloc 이 free 해도 OS 에게 반환되지 않을 수 있는 이유. C 를 free 하면 — top chunk 가 커지면서 그제야 brk 를 줄일 수 있다.

이 한계 때문에 — 큰 할당에는 다른 메커니즘 (mmap) 을 쓴다.

mmap (MAP_ANONYMOUS) 의 동작

mmap 은 — 파일을 메모리에 매핑하는 syscall 인데, MAP_ANONYMOUS 플래그로 — 파일 없는 익명 메모리를 매핑할 수 있다.

// [user-space - C 예제]
#include <sys/mman.h>

void *area = mmap(NULL,                                /* OS 가 위치 결정 */
                  1024 * 1024,                         /* 1 MB */
                  PROT_READ | PROT_WRITE,              /* 권한 */
                  MAP_PRIVATE | MAP_ANONYMOUS,         /* 익명, private */
                  -1, 0);                              /* fd 없음 */
/* area = 1 MB 의 새 메모리 영역 */

/* ... 사용 ... */

munmap(area, 1024 * 1024);                            /* OS 에 반환 */

MAP_ANONYMOUS 의 핵심 — kernel 이 0 으로 채워진 새 메모리를 제공. fd 가 -1, offset 도 0.

[mmap (MAP_ANONYMOUS) 의 특성]

  ✓ kernel 이 0 으로 초기화 (보안)
  ✓ 임의의 가상 주소에 매핑 (ASLR)
  ✓ 페이지 (4KB) 단위
  ✓ 독립적으로 munmap 가능 (즉시 OS 에 반환)
  ✓ MAP_PRIVATE: 다른 프로세스와 공유 안 됨
  ✗ brk 보다 느림 (page table 조작, syscall 오버헤드)

왜 두 메커니즘? — 각자의 장단점

[brk vs mmap 의 trade-off]

  brk:
    + 빠름 (단순한 포인터 조작)
    + cache friendly (연속 영역)
    + 적은 syscall 오버헤드
    - 단편화 문제 (중간 못 반환)
    - 연속된 영역만
    
  mmap:
    + 즉시 반환 가능 (munmap)
    + 단편화 없음
    + 임의 위치 (ASLR)
    + 큰 할당에 적합
    - 느림 (kernel page table 조작)
    - 페이지 정렬 강제 (4KB 미만 할당은 낭비)

glibc 의 결정 — 작은 할당 (< MMAP_THRESHOLD) 은 brk, 큰 할당은 mmap.

MMAP_THRESHOLD — 경계값

glibc 의 MMAP_THRESHOLD 가 — brk 와 mmap 의 경계.

[MMAP_THRESHOLD 의 기본값과 동작]

  기본값: 128 KB (DEFAULT_MMAP_THRESHOLD = 128*1024)
  
  malloc(size) 시:
    if (size >= mmap_threshold) → mmap 사용
    else                        → arena 의 heap (brk) 사용
  
  값 확인 / 변경:
    mallopt(M_MMAP_THRESHOLD, value)
    또는 환경 변수 MALLOC_MMAP_THRESHOLD_=value

동적 MMAP_THRESHOLD — 영리한 휴리스틱

glibc 의 ingenious 기법 — MMAP_THRESHOLD 가 동적으로 조정된다. master 의 malloc/malloc.c:

// [user-space - glibc malloc/malloc.c master 핵심 로직 단순화]
/* free 가 호출될 때, 만약 free 되는 chunk 가 mmap 된 것이고
   현재 threshold 보다 크면 — threshold 를 그 크기로 올림 */
if (chunk_is_mmapped(p)) {
    if (!mp_.no_dyn_threshold
        && chunksize(p) > mp_.mmap_threshold
        && chunksize(p) <= DEFAULT_MMAP_THRESHOLD_MAX) {
        mp_.mmap_threshold = chunksize(p);
        mp_.trim_threshold = 2 * mp_.mmap_threshold;
    }
}

이게 매우 영리하다. 동작 시나리오:

[동적 MMAP_THRESHOLD 의 시나리오]

  초기: mmap_threshold = 128 KB
  
  프로그램이 256 KB chunk 를 malloc → mmap 사용 (>128KB)
  그 chunk 를 free → munmap 으로 OS 반환
                    ★ mmap_threshold 가 256 KB 로 올라감
  
  같은 프로그램이 다시 256 KB malloc:
    → 이제는 threshold 보다 작거나 같음
    → arena heap (brk) 에서 할당!
    → mmap/munmap 의 syscall 비용 절약
  
  최대값: DEFAULT_MMAP_THRESHOLD_MAX (보통 32 MB)

이 휴리스틱이 — 반복적인 큰 할당이 있는 프로그램의 성능을 크게 개선한다. 처음엔 mmap 으로 보수적이지만 — pattern 을 학습하면 brk 로 전환.

main_arena 의 heap — brk 의 활용

이제 main_arena 의 heap 가 어떻게 작동하는지.

[main_arena 의 heap 운영 — brk]

  최초 malloc 호출 시:
    1. sbrk(0) 으로 현재 break 확인
    2. sbrk(132KB) 로 132KB 확장 (heap 시작)
    3. 그 영역을 "main_arena 의 heap" 으로 사용
    4. 작은 chunk 들을 그 안에 잘라서 할당
  
  heap 이 부족하면:
    sbrk(추가) 로 확장
    → top chunk 가 커짐
  
  heap 의 끝부분이 충분히 비면:
    sbrk(-감소) 로 OS 반환
    → trim_threshold 이상의 빈 공간이 있을 때만

/proc/PID/maps 로 실제 확인 가능.

# [user-space - bash]
$ cat /proc/self/maps | grep heap
55bcd5b9c000-55bcd5bbd000 rw-p 00000000 00:00 0    [heap]
   ↑                ↑
   heap 시작 (data 끝)    현재 break
   
# 차이 = 0x21000 = 132 KB ← main_arena heap 초기 크기

thread arena 의 heap — mmap 의 활용

multi-thread 프로그램에서는 — 각 thread 가 자기 arena 를 갖는다. brk 는 프로세스에 1개만 있으니 — thread arena 는 mmap 으로 생성.

[thread arena 의 생성]

  새 thread 가 malloc 호출 → arena 가 필요
       │
       ▼
  free arena 있나? (재사용)
    YES → 그것 사용
    NO  ▼
       new_heap() 호출
       → mmap(NULL, HEAP_MAX_SIZE, ...) 으로 64MB 매핑
       → 처음엔 PROT_NONE (read/write 권한 없음)
       → 필요할 때마다 mprotect 으로 권한 줌
  
  HEAP_MAX_SIZE: 보통 64 MB (DEFAULT_THREAD_HEAP_SIZE)
  처음 사용 가능 크기: ~64 KB

/proc/PID/maps 의 예:

# [user-space - bash, multi-thread 프로그램]
$ cat /proc/self/maps | grep -E "(heap|anon)"
55bcd5b9c000-55bcd5bbd000 rw-p 00000000 00:00 0     [heap]          ← main arena
7ffff0000000-7ffff0021000 rw-p 00000000 00:00 0                     ← thread arena 1 (사용 중)
7ffff0021000-7ffff4000000 ---p 00000000 00:00 0                     ← thread arena 1 (예약)
7fffec000000-7fffec021000 rw-p 00000000 00:00 0                     ← thread arena 2
7fffec021000-7ffff0000000 ---p 00000000 00:00 0                     ← thread arena 2 (예약)

---p (권한 없음) 의 큰 영역이 — mmap 으로 미리 reserved 된 64 MB. 사용할 때 mprotect 로 권한 줘서 활용. address space 만 예약, 실제 메모리는 안 차지.

큰 할당의 mmap — 별도 메커니즘

MMAP_THRESHOLD 이상의 큰 할당은 — arena 와 별개로, 직접 mmap 으로 처리.

[큰 할당 (≥ MMAP_THRESHOLD) 의 처리]

  malloc(256 KB):    (threshold 가 128 KB 인 경우)
       │
       ▼
  arena 안 거침
  mmap(NULL, 256 KB + 헤더, PROT_READ|PROT_WRITE,
       MAP_PRIVATE|MAP_ANONYMOUS, -1, 0)
       │
       ▼
  반환된 영역에 chunk header 붙임
  IS_MMAPPED 비트 설정
       │
       ▼
  사용자에게 반환
  
  나중에 free 시:
    chunk header 확인
    IS_MMAPPED 비트 보고
    munmap(area, size) → 즉시 OS 반환!

/proc/PID/maps 에서 — 이런 mmap 된 큰 chunk 는 별도의 영역으로 보인다 (heap 과 분리).

# [user-space - bash]
# 큰 chunk 가 free 안 된 상태:
$ cat /proc/self/maps | grep "rw-p" | head -10
55bcd5b9c000-55bcd5bbd000 rw-p ... [heap]
7ffff0040000-7ffff0081000 rw-p 00:00 0          ← 256 KB mmap 된 chunk!
...

brk vs mmap 의 성능 차이 측정

실제 두 방법의 성능 차이를 보면:

// [user-space - C 벤치마크 예시]
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

#define N 1000000

int main(void) {
    struct timespec t1, t2;
    
    /* 작은 할당 1M 번 — brk 경로 */
    clock_gettime(CLOCK_MONOTONIC, &t1);
    for (int i = 0; i < N; i++) {
        void *p = malloc(64);
        free(p);
    }
    clock_gettime(CLOCK_MONOTONIC, &t2);
    printf("small (brk): %ld ms\n", (t2.tv_sec - t1.tv_sec) * 1000 + (t2.tv_nsec - t1.tv_nsec) / 1000000);
    
    /* 큰 할당 1M 번 — mmap 경로 */
    clock_gettime(CLOCK_MONOTONIC, &t1);
    for (int i = 0; i < N; i++) {
        void *p = malloc(256 * 1024);  /* 256 KB > MMAP_THRESHOLD */
        free(p);
    }
    clock_gettime(CLOCK_MONOTONIC, &t2);
    printf("large (mmap): %ld ms\n", (t2.tv_sec - t1.tv_sec) * 1000 + (t2.tv_nsec - t1.tv_nsec) / 1000000);
    
    return 0;
}

전형적인 결과:

small (brk):  ~20 ms       ← 매우 빠름 (포인터 조작만)
large (mmap): ~2000 ms     ← 100배 느림 (syscall 마다)

**mmap 이 약 100배 느리다**. 이래서 — glibc 가 작은 할당에는 절대 mmap 안 쓰는 것.

MALLOC_* 환경 변수로 조정

malloc 의 동작을 — 환경 변수로 조정할 수 있다.

[malloc 관련 환경 변수]

  MALLOC_MMAP_THRESHOLD_   : MMAP_THRESHOLD 값 설정
  MALLOC_TRIM_THRESHOLD_   : heap trim 임계값
  MALLOC_TOP_PAD_          : sbrk 시 추가로 늘릴 양
  MALLOC_MMAP_MAX_         : 최대 mmap 개수 (기본 65536)
  MALLOC_ARENA_MAX         : 최대 arena 개수
  MALLOC_CHECK_            : malloc 디버깅 활성화
  MALLOC_PERTURB_          : free 시 메모리 패턴 채우기

예:

# [user-space - bash]
# 256 KB 부터 mmap 사용
$ MALLOC_MMAP_THRESHOLD_=262144 ./myapp

# trim 임계값 조정 (heap 줄임 빈도)
$ MALLOC_TRIM_THRESHOLD_=131072 ./myapp

# 작은 할당부터 mmap 사용 (느리지만 단편화 X)
$ MALLOC_MMAP_THRESHOLD_=0 ./myapp

MALLOC_MMAP_THRESHOLD_=0 은 — 모든 할당을 mmap 으로. heap fragmentation 회피지만 매우 느림. 보안 sensitive 한 데몬에서 가끔 사용.

이 장의 결론

malloc 의 메모리 출처:

[malloc 메모리 출처 정리]

  ✓ brk / sbrk syscall
    - main_arena 의 heap 확장
    - 작은 할당 (< MMAP_THRESHOLD)
    - 단편화 문제 있음
    
  ✓ mmap (MAP_ANONYMOUS)
    - thread arena 의 heap 매핑
    - 큰 할당 (≥ MMAP_THRESHOLD)
    - 즉시 munmap 으로 OS 반환 가능
    - 동적 threshold 휴리스틱
    
  ✓ /proc/PID/maps 로 직접 관찰 가능
  ✓ MALLOC_* 환경 변수로 조정

malloc 이 kernel 에게서 메모리를 받는 방법은 — 단순하다. 두 가지 syscall, 그게 전부. 그런데 이 위에 — 매우 정교한 자료구조와 알고리즘이 올라간다.

다음 장에서는 — malloc 이 그 영역을 어떻게 잘라서 사용자에게 나눠주는지의 핵심 단위, **malloc_chunk** 를 본다.

24malloc_chunk — 모든 것의 단위

이전 장에서 — malloc 이 kernel 에게서 한 덩어리의 메모리 영역을 받는 방법을 봤다. 그런데 사용자는 다양한 크기 (malloc(16), malloc(4096), malloc(100000)) 를 요청한다. 그 큰 덩어리를 어떻게 잘라서 나눠주는가?

답은 — **malloc_chunk** 라는 자료구조. glibc malloc 의 가장 핵심 단위.

malloc_chunk 의 정의

master 의 malloc/malloc.c 의 시작 부분에 정의되어 있다.

// [user-space - glibc malloc/malloc.c master 인용]
struct malloc_chunk {
    INTERNAL_SIZE_T mchunk_prev_size;  /* Size of previous chunk (if free).  */
    INTERNAL_SIZE_T mchunk_size;       /* Size in bytes, including overhead. */
    
    struct malloc_chunk* fd;           /* double links -- used only if free. */
    struct malloc_chunk* bk;
    
    /* Only used for large blocks: pointer to next larger size.  */
    struct malloc_chunk* fd_nextsize;  /* double links -- used only if free. */
    struct malloc_chunk* bk_nextsize;
};
typedef struct malloc_chunk* mchunkptr;

매우 흥미로운 구조. INTERNAL_SIZE_T 는 — x86_64 에서 size_t (= uint64_t). 즉 8 바이트.

[malloc_chunk 의 크기 — x86_64]

  mchunk_prev_size : 8 bytes
  mchunk_size      : 8 bytes
  fd               : 8 bytes
  bk               : 8 bytes
  fd_nextsize      : 8 bytes
  bk_nextsize      : 8 bytes
  ────────────────────────
  총                : 48 bytes
  
  단 — 이 모두를 항상 사용하는 건 아니다.
  fd, bk, fd_nextsize, bk_nextsize 는 _free chunk_ 일 때만 사용.
  in-use 인 chunk 에서는 fd 부터가 _사용자 데이터_ 영역으로 재활용.

이 "공간 재사용" 이 매우 핵심이다. chunk 가 free 일 때는 링크 정보를 담고, 사용 중일 때는 사용자 데이터를 담는다. 같은 메모리 위치, 다른 의미.

mchunk_size 의 비밀 — 하위 3 비트가 플래그

mchunk_size 의 묘미. 8 바이트 정수지만 — 하위 3 비트가 플래그로 활용된다.

// [user-space - glibc malloc/malloc.c master 매크로 인용]
#define PREV_INUSE     0x1
#define IS_MMAPPED     0x2
#define NON_MAIN_ARENA 0x4

#define SIZE_BITS (PREV_INUSE | IS_MMAPPED | NON_MAIN_ARENA)

/* 진짜 크기 (플래그 제외) */
#define chunksize(p) (chunksize_nomask(p) & ~SIZE_BITS)

왜 하위 3 비트가 남는가? — chunk 의 크기가 항상 8 바이트 (또는 16 바이트) 의 배수이기 때문이다. 정렬 강제로 하위 3-4 비트가 항상 0 이니, 그 비트들을 플래그로 활용.

[mchunk_size 의 비트 분포 — x86_64]

  63                                                  3  2  1  0
  ┌───────────────────────────────────────────────────┬──┬──┬──┐
  │                  실제 크기 (60 비트)                  │N │M │P │
  └───────────────────────────────────────────────────┴──┴──┴──┘
                                                       │  │  │
                                                       │  │  └ PREV_INUSE (P)
                                                       │  └ IS_MMAPPED (M)
                                                       └ NON_MAIN_ARENA (N)

각 플래그의 의미:

[3 개 플래그의 의미]

  PREV_INUSE (P, 0x1):
    이전 (메모리상 앞) chunk 가 in-use 인지
    P=1: 이전 chunk 가 사용 중 → mchunk_prev_size 무효
    P=0: 이전 chunk 가 free → mchunk_prev_size 가 그 크기
    
    coalescing (free 시 인접 chunk 와 합치기) 에 사용
    
  IS_MMAPPED (M, 0x2):
    이 chunk 가 arena 가 아니라 직접 mmap 된 것인지
    M=1: free 시 munmap 으로 OS 에 즉시 반환
    M=0: 일반 arena chunk
    
  NON_MAIN_ARENA (N, 0x4):
    이 chunk 가 main_arena 가 아닌 thread arena 의 것인지
    N=1: 어느 arena 인지 알아내기 위해 heap_info 확인
    N=0: main_arena 의 chunk

in-use chunk vs free chunk 의 메모리 layout

같은 자료구조가 — 상태에 따라 완전히 다른 의미를 가진다.

┌───────────────────────────────────────────────────────────┐
│ in-use chunk 의 layout                                      │
├───────────────────────────────────────────────────────────┤
│                                                            │
│  chunk 시작 →  ┌────────────────────────────────────┐         │
│              │ 이전 chunk 의 사용자 데이터 (재활용!)     │         │
│              │ (chunk 가 in-use 면 prev_size 영역      │         │
│              │  을 이전 chunk 가 사용자 데이터로 사용)   │         │
│              ├────────────────────────────────────┤         │
│              │ mchunk_size | flags (N|M|P=1)       │         │
│              ├────────────────────────────────────┤         │
│  mem (반환) →│                                     │         │
│              │      사용자 데이터                    │         │
│              │  (malloc 이 반환한 포인터부터)         │         │
│              │                                     │         │
│              │  ★ fd, bk 위치도 사용자 데이터 영역      │         │
│              │                                     │         │
│              │                                     │         │
│              ├────────────────────────────────────┤         │
│ 다음 chunk → │ 다음 chunk 의 prev_size (= 무효)       │         │
│              ┊                                     ┊         │
│              ┊  ★ 다음 chunk 의 size 의 P 비트 = 1     ┊         │
│              ┊                                     ┊         │
└───────────────────────────────────────────────────────────┘
┌───────────────────────────────────────────────────────────┐
│ free chunk 의 layout                                        │
├───────────────────────────────────────────────────────────┤
│                                                            │
│  chunk 시작 →  ┌────────────────────────────────────┐         │
│              │ mchunk_prev_size                    │         │
│              │   (이전 chunk 의 크기, 이전이 free 면)  │         │
│              │                                     │         │
│              │   ── 또는 ──                          │         │
│              │   이전 chunk 가 in-use 면 그것의 데이터   │         │
│              ├────────────────────────────────────┤         │
│              │ mchunk_size | flags                 │         │
│              ├────────────────────────────────────┤         │
│              │ fd → 다음 free chunk                  │         │
│              ├────────────────────────────────────┤         │
│              │ bk → 이전 free chunk                  │         │
│              ├────────────────────────────────────┤         │
│              │ fd_nextsize (large bin 만)            │         │
│              │   → 다음 다른 크기의 free chunk         │         │
│              ├────────────────────────────────────┤         │
│              │ bk_nextsize (large bin 만)            │         │
│              │   → 이전 다른 크기의 free chunk         │         │
│              ├────────────────────────────────────┤         │
│              │                                     │         │
│              │      (사용 안 되는 공간)               │         │
│              │                                     │         │
│              ├────────────────────────────────────┤         │
│ 다음 chunk → │ 이 chunk 의 크기 (boundary tag)        │         │
│              ┊  ★ 다음 chunk 의 prev_size 에 저장       ┊         │
│              ┊  ★ 다음 chunk 의 size 의 P 비트 = 0       ┊         │
└───────────────────────────────────────────────────────────┘

boundary tag 기법

malloc.c 의 주석이 직접 설명한다:

[boundary tag 기법 — Knuth 의 원래 아이디어]

  핵심: free chunk 의 크기를 _양쪽 끝_ 에 저장
  
  - 시작: mchunk_size 에 자기 크기
  - 끝: 다음 chunk 의 mchunk_prev_size 에 자기 크기 (복사)
  
  → 어디서든 chunk 의 앞뒤 경계를 알 수 있음
  → coalescing (29장) 에 결정적
  
  공간 절약 trick:
  in-use chunk 는 prev_size 영역을 사용자 데이터로 활용
  → 8 byte 절약
  → 그래서 다음 chunk 의 P 비트가 필요한 것
     (그 영역이 valid prev_size 인지 사용자 데이터인지 구분)

이 boundary tag 기법이 — Donald Knuth 의 *The Art of Computer Programming* (1968) 에서 처음 정의된 고전적 아이디어. 50년 넘게 살아남은 디자인.

chunk 와 mem 의 변환 — 2 워드 오프셋

사용자가 받는 포인터 (mem) 와 — 내부 chunk 포인터 (chunkptr) 의 변환.

// [user-space - glibc malloc/malloc.c master 매크로 인용]
#define chunk2mem(p)   ((void*)((char*)(p) + 2 * SIZE_SZ))
#define mem2chunk(mem) ((mchunkptr)((char*)(mem) - 2 * SIZE_SZ))

SIZE_SZ = 8 (x86_64). 즉 — 사용자가 받는 포인터는 chunk 시작점 + 16 바이트 (prev_size 와 size 를 건너뜀).

// [user-space - C 예제]
void *p = malloc(100);
/* p 가 가리키는 위치는 chunk 의 fd 자리 (사용자 데이터의 시작) */

mchunkptr chunk = (mchunkptr)((char *)p - 16);
size_t real_size = chunk->mchunk_size & ~0x7;
/* real_size 가 chunk 의 진짜 크기 (헤더 포함) */

최소 chunk 크기와 alignment

malloc 이 반환하는 모든 포인터는 — 16 바이트 정렬 (x86_64). 그래서 — 최소 chunk 크기도 16 의 배수.

[chunk 크기 규칙 — x86_64]

  MALLOC_ALIGNMENT = 16
  SIZE_SZ          = 8
  MIN_CHUNK_SIZE   = 32     ← chunk 의 최소 크기 (헤더 포함)
  MINSIZE          = 32
  
  사용자가 malloc(1) 해도 → 32 바이트 chunk 가 할당됨
    - prev_size: 8 bytes
    - size:      8 bytes  
    - 사용자 데이터: 16 bytes (1 만 사용)
  
  사용자가 malloc(24) 해도 → 32 바이트 chunk
  사용자가 malloc(25) → 48 바이트 chunk

왜 16 바이트 정렬? — SSE/SIMD 명령어가 16 바이트 정렬을 요구. _mm_load_ps 같은 명령어가 정렬 안 되면 SEGV. malloc 이 반환하는 모든 포인터를 SIMD 사용 가능하게 보장.

request2size — 사용자 크기를 chunk 크기로

사용자가 malloc(N) 호출하면 — N 을 어떻게 chunk 크기로 변환?

// [user-space - glibc malloc/malloc.c master 단순화]
#define request2size(req)                                          \
    (((req) + SIZE_SZ + MALLOC_ALIGN_MASK < MINSIZE)               \
     ? MINSIZE                                                     \
     : ((req) + SIZE_SZ + MALLOC_ALIGN_MASK) & ~MALLOC_ALIGN_MASK)

계산 흐름:

[request2size 의 변환]

  req       SIZE_SZ   MALLOC_ALIGN_MASK = 15
  user 요청    8         15
  
  내부 크기 = req + SIZE_SZ + MALLOC_ALIGN_MASK   ← 헤더 1개분만 더함 (next 의 prev_size 활용)
            그 다음 16 의 배수로 내림 (& ~15)
            MINSIZE(32) 미만이면 32
  
  예:
    malloc(1)   → req=1   → (1+8+15)&~15 = 16  → MINSIZE = 32
    malloc(8)   → (8+8+15)&~15  = 16          → MINSIZE = 32
    malloc(24)  → (24+8+15)&~15 = 32          → 32
    malloc(25)  → (25+8+15)&~15 = 48
    malloc(40)  → (40+8+15)&~15 = 48
    malloc(56)  → (56+8+15)&~15 = 64

**SIZE_SZ 만 더하고 MALLOC_ALIGNMENT(=16) 는 안 더한다**. 왜? — 다음 chunk 의 prev_size 영역 (8 bytes) 을 자기 데이터로 활용하니까. 그래서 오버헤드가 한 워드뿐.

// [user-space - C 예제]
#include <malloc.h>

printf("%zu\n", malloc_usable_size(malloc(1)));   // 24
printf("%zu\n", malloc_usable_size(malloc(24)));  // 24
printf("%zu\n", malloc_usable_size(malloc(25)));  // 40

malloc_usable_size(p) 가 — 실제 사용 가능한 크기 (chunk 크기 - 헤더). malloc(1) 도 사실 24 바이트를 쓸 수 있다.

인접 chunk 의 탐색 — next_chunk 매크로

boundary tag 의 능력 — 어느 chunk 든 인접한 (메모리상 다음/이전) chunk 를 O(1) 에 찾을 수 있다.

// [user-space - glibc malloc/malloc.c master 매크로 인용]
/* 메모리상 다음 chunk */
#define next_chunk(p)    ((mchunkptr)(((char*)(p)) + chunksize(p)))

/* 메모리상 이전 chunk (이전이 free 일 때만 valid) */
#define prev_chunk(p)    ((mchunkptr)(((char*)(p)) - prev_size(p)))

/* prev_size 가져오기 */
#define prev_size(p)     ((p)->mchunk_prev_size)

/* 이전 chunk 가 in-use 인가 */
#define prev_inuse(p)    ((p)->mchunk_size & PREV_INUSE)

이 능력이 — coalescing (단편화 합치기) 의 기반이다. free 호출 시 인접 chunk 가 free 면 합쳐서 더 큰 chunk 로.

chunk 간의 연결 — fd, bk 의 의미

free chunk 들은 — doubly-linked list 로 연결된다. fd (forward) 와 bk (backward) 가 그 링크.

┌─────────────────────────────────────────────────┐
│ free chunk 들의 doubly-linked list                 │
├─────────────────────────────────────────────────┤
│                                                  │
│  bin 헤드          chunk 1            chunk 2      │
│  ┌────┐           ┌────────┐         ┌────────┐    │
│  │ fd ├──────────→│ ...    │         │ ...    │    │
│  │ bk │←──────────│ fd     ├────────→│ fd     ├──→ │
│  └────┘           │ bk     │←────────│ bk     │←── │
│                   │ (data) │         │ (data) │    │
│                   └────────┘         └────────┘    │
│                                                  │
└─────────────────────────────────────────────────┘

bin (다음 장에서 자세히) 의 헤드부터 시작해서 — fd 따라가면 free chunk 들의 list. malloc 호출 시 이 list 에서 적합한 크기 찾음.

fd_nextsize, bk_nextsize — large bin 전용

fd_nextsize 와 bk_nextsize 는 — large bin (1 KB 이상) 에서만 사용. large bin 안에는 다양한 크기의 chunk 가 섞여 있고, 같은 크기끼리 묶는 부수적 link.

[large bin 의 구조]

  같은 크기의 chunk 들은 fd/bk 로 묶임
  다른 크기끼리는 fd_nextsize/bk_nextsize 로 묶임 (정렬됨)
  
  large bin (예: 1024~1031 바이트):
    
    head ── 1024 bytes ──┬── 1024 bytes ── 1024 bytes
                          │  (fd/bk 로 연결)
                          │
                          ▼ fd_nextsize
                         1500 bytes ─── 1500 bytes
                          │  (같은 크기끼리 fd/bk)
                          │
                          ▼ fd_nextsize
                         2048 bytes

자세한 내용은 26장 (bin 의 종류) 에서.

chunk header 의 오버헤드

chunk 의 오버헤드를 계산하면:

[chunk 오버헤드 — x86_64]

  in-use chunk:
    헤더 = mchunk_size (8 bytes)
           (mchunk_prev_size 는 이전 chunk 의 데이터로 활용)
    오버헤드 = 8 bytes per chunk
  
  free chunk:
    헤더 = mchunk_prev_size + mchunk_size + fd + bk (32 bytes)
           large bin 이면 + fd_nextsize + bk_nextsize = 48 bytes
    free 시점에만 필요 — 실제 메모리에는 영향 없음
    
  최악의 경우 — 작은 할당 1 백만 개:
    1M × 8 bytes = 8 MB 의 헤더 오버헤드

이 오버헤드가 — malloc(1) 같은 매우 작은 할당이 비효율적인 이유. 1 바이트 요청에 32 바이트 chunk → 97% 낭비.

malloc_usable_size — 실제 사용 가능 크기

// [user-space - C API]
#include <malloc.h>
size_t malloc_usable_size(void *ptr);

malloc 이 반환한 포인터에서 실제로 사용 가능한 크기를 반환. 사용자가 요청한 크기보다 클 수 있다 (alignment 와 chunk 단위 때문에).

// [user-space - C 예제]
#include <stdio.h>
#include <stdlib.h>
#include <malloc.h>

int main(void) {
    for (size_t i = 1; i <= 100; i += 17) {
        void *p = malloc(i);
        printf("malloc(%2zu) → usable %zu\n", i, malloc_usable_size(p));
        free(p);
    }
    return 0;
}

전형적 출력:

malloc( 1) → usable 24
malloc(18) → usable 24
malloc(35) → usable 40
malloc(52) → usable 56
malloc(69) → usable 72
malloc(86) → usable 88

8 바이트 단위로 늘어남. 16 바이트가 아닌 이유는 — 다음 chunk 의 prev_size 자리 활용 (8 byte 절약).

이 장의 결론

malloc_chunk 가 모든 것의 단위:

[malloc_chunk 정리]

  ✓ 모든 malloc 영역은 chunk 단위
  ✓ in-use 와 free 가 같은 구조, 다른 의미 (공간 재사용)
  ✓ mchunk_size 의 하위 3 비트가 플래그 (P, M, N)
  ✓ boundary tag 기법으로 인접 chunk 탐색 O(1)
  ✓ free chunk 끼리 doubly-linked list (fd, bk)
  ✓ large bin 은 크기별 추가 link (fd_nextsize, bk_nextsize)
  ✓ 최소 chunk 32 bytes, 16 byte 정렬
  ✓ 오버헤드: in-use 시 8 bytes (재활용 trick)

이 단위 위에서 — 다음 장의 arena 가 build 된다. arena 안에 여러 chunk 가 모이고, bin 으로 분류되고, 휴리스틱으로 빠르게 검색.

25arena — multi-thread 의 heap 분리

malloc 의 가장 큰 도전 중 하나가 — multi-thread 환경. 여러 thread 가 동시에 malloc 을 호출하면, lock 경쟁이 일어나 매우 느려진다. glibc 의 ptmalloc2 가 이걸 해결한 방법이 — arena 라는 개념.

한 줄 정리

[arena 의 핵심]

  arena = 독립적인 malloc 의 작업 공간
        = malloc_state (메타데이터) + heap (실제 메모리)
        = 하나의 lock 으로 보호되는 영역
        
  thread 들은 각자 다른 arena 를 사용
  → lock 경쟁 회피
  → 거의 lock-free 같은 성능

옛 시절 — 단일 arena

ptmalloc2 이전의 dlmalloc (Doug Lea malloc) 은 — arena 가 하나. multi-thread 가 같은 heap 을 두고 lock 경쟁.

[옛 dlmalloc — 단일 heap, 큰 lock]

  Thread 1:  malloc(64) ──→ lock(global_mutex)
                            allocate
                            unlock
                            
  Thread 2:  malloc(64) ──→ lock(global_mutex)  ← Thread 1 끝날 때까지 wait!
                            wait...
                            wait...
                            
  → 4 thread, 8 thread 늘어날수록 거의 직렬화

이 문제를 해결하기 위해 — Wolfram Gloger 가 dlmalloc 에 multi-arena 를 추가한 게 ptmalloc (1996). 현재 glibc 가 사용하는 ptmalloc2 가 그 발전형.

multi-arena 의 아이디어

[multi-arena 의 핵심]

  각 thread 가 자기만의 arena 사용:
  
  Thread 1 ─── arena_1 (lock_1)
  Thread 2 ─── arena_2 (lock_2)
  Thread 3 ─── arena_3 (lock_3)
  Thread 4 ─── arena_4 (lock_4)
  
  → 서로 다른 lock 이라 경쟁 없음
  → 거의 lock-free 한 성능
  
  단점:
  - 메모리 단편화 증가 (각 arena 마다 별도 heap)
  - 일부 메모리는 다른 arena 에서 사용 못 함

malloc_state — arena 의 메타데이터

각 arena 의 핵심은 malloc_state 구조체. master 의 malloc/malloc.c:

// [user-space - glibc malloc/malloc.c master 인용]
struct malloc_state {
    /* Serialize access. */
    __libc_lock_define(, mutex);                  /* arena 보호 lock */
    
    /* Flags (formerly in max_fast). */
    int flags;
    
    /* Set if the fastbin chunks contain recently inserted free blocks. */
    int have_fastchunks;
    
    /* Fastbins */
    mfastbinptr fastbinsY[NFASTBINS];             /* fastbin 배열 (10개) */
    
    /* Base of the topmost chunk -- not otherwise kept in a bin */
    mchunkptr top;                                /* top chunk */
    
    /* The remainder from the most recent split of a small request */
    mchunkptr last_remainder;
    
    /* Normal bins packed as described above */
    mchunkptr bins[NBINS * 2 - 2];               /* 일반 bin 들 (256개 ptr) */
    
    /* Bitmap of bins */
    unsigned int binmap[BINMAPSIZE];             /* bin 비트맵 (4개 word) */
    
    /* Linked list */
    struct malloc_state *next;                   /* 다음 arena */
    
    /* Linked list for free arenas. */
    struct malloc_state *next_free;
    
    /* Number of threads attached to this arena. */
    INTERNAL_SIZE_T attached_threads;            /* 이 arena 쓰는 thread 수 */
    
    /* Memory allocated from the system in this arena.  */
    INTERNAL_SIZE_T system_mem;
    INTERNAL_SIZE_T max_system_mem;
};

typedef struct malloc_state *mstate;

각 필드의 역할:

[malloc_state 필드 해설]

  mutex            : 이 arena 의 lock (pthread_mutex 또는 lll_lock)
  flags            : 다양한 상태 비트
  have_fastchunks  : fastbin 에 free chunk 가 있는지 (빠른 체크)
  fastbinsY[10]    : fastbin 들 (각각 single-linked LIFO)
  top              : top chunk (arena 의 끝, 미분할 영역)
  last_remainder   : 작은 분할에서 남은 chunk (locality 최적화)
  bins[254]        : 일반 bin 들의 헤드 (127 개 bin × 2 = fd/bk)
  binmap[4]        : 128 bit 비트맵 (어느 bin 이 비어있는지 빠른 체크)
  next             : 다음 arena (circular list)
  attached_threads : 이 arena 사용 중인 thread 수
  system_mem       : 이 arena 가 OS 에서 받은 메모리 양

크기는 — 약 2 KB. arena 마다 이 메타데이터가 있다.

main_arena — 특별한 arena

첫 thread (main thread) 가 사용하는 arena 는 — **main_arena** 라는 고정된 위치에 있다.

// [user-space - glibc malloc/malloc.c master]
static struct malloc_state main_arena = {
    .mutex = _LIBC_LOCK_INITIALIZER,
    .next = &main_arena,                          /* 자기 자신 가리킴 */
    .attached_threads = 1,
};

main_arena 는 — libc.so 의 .data section 안의 정적 변수. 프로세스 시작 시 그 자리에 이미 있다. 따로 할당 안 함.

[main_arena 의 위치]

  ./hello binary
       │
       └──→ libc.so.6
            │
            └──→ .data section
                  │
                  └──→ main_arena 라는 struct malloc_state (정적!)
                        │
                        ├──→ next: &main_arena (자기 자신)
                        ├──→ heap: brk 영역 (별도 위치)
                        └──→ ...

main_arena 의 heap 은 — 23장에서 본 brk 로 자라는 영역. process 의 [heap] segment.

thread arena — 동적 생성

main 외의 thread 가 처음 malloc 호출하면 — 새 arena 가 동적으로 생성된다.

// [user-space - glibc malloc/arena.c master 단순화]
static mstate
_int_new_arena(size_t size)
{
    mstate a;
    heap_info *h;
    char *ptr;
    unsigned long misalign;
    
    /* 1. 새 heap 영역 mmap (64 MB) */
    h = new_heap(size + HEAP_MIN_SIZE, mp_.top_pad);
    if (!h) {
        h = new_heap(size, mp_.top_pad);
        if (!h) return NULL;
    }
    
    /* 2. heap 안에 malloc_state 배치 (heap_info 바로 다음) */
    a = h->ar_ptr = (mstate)(h + 1);
    malloc_init_state(a);
    
    /* 3. arena list 에 추가 */
    a->next = main_arena.next;
    main_arena.next = a;
    
    /* 4. 첫 chunk 생성 (top chunk) */
    ptr = (char *)(a + 1);
    misalign = (unsigned long)chunk2mem(ptr) & MALLOC_ALIGN_MASK;
    if (misalign > 0)
        ptr += MALLOC_ALIGN - misalign;
    top(a) = (mchunkptr)ptr;
    set_head(top(a), (((char *)h + h->size) - ptr) | PREV_INUSE);
    
    return a;
}

핵심 — thread arena 는 mmap 으로 만든 64 MB 영역 안에 자체적으로 위치한다. main_arena 와 달리 별도 메모리.

[arena 들의 circular list]

         main_arena.next
              │

thread → arena 매핑 — TLS

각 thread 는 자기가 사용할 arena 를 — TLS (Thread-Local Storage) 에 저장.

// [user-space - glibc malloc/malloc.c master]
static __thread mstate thread_arena attribute_tls_model_ie = &main_arena;

새 thread 가 만들어지면 — thread_arena 는 처음엔 &main_arena. 첫 malloc 호출 시 — arena_get 함수가 적절한 arena 를 찾아 할당.

// [user-space - glibc malloc/arena.c master 단순화]
#define arena_get(ptr, size)                    \
    do {                                         \
        ptr = thread_arena;                      \
        arena_lock(ptr, size);                  \
    } while (0)

static void
arena_lock(mstate ptr, size_t size)
{
    if (ptr)
        __libc_lock_lock(ptr->mutex);
    else
        ptr = arena_get2(size, NULL);   /* 새 arena 할당 */
}

흐름:

[arena_get 의 흐름]

  thread_arena 가 이미 있나? (TLS 값)
       │
       ├── YES → lock 시도
       │         ├── 성공 → 그 arena 사용
       │         └── 실패 (다른 thread 사용 중)
       │              ↓
       │              다른 arena 시도
       │
       └── NO → arena_get2 호출
                ↓
                free arena 있나?
                ├── YES → 재사용
                └── NO → 새 arena 생성 (arena_max 미만이면)
                         또는 기존 arena 재사용

arena_max — 최대 arena 개수

arena 가 무한정 생성되면 — 메모리 낭비. glibc 가 한도를 둔다.

[arena 개수 한도]

  arena_max 의 기본값:
    32-bit: 2 × NCPUS
    64-bit: 8 × NCPUS
    
  예: 4 코어 CPU, 64-bit:
    arena_max = 32
    
  thread 수가 32 초과해도 — arena 는 32 개로 제한
  → 여러 thread 가 한 arena 공유 (lock 경쟁 다시 시작)
  → 하지만 32 개 분산이라 그래도 빠름

MALLOC_ARENA_MAX 환경변수로 조정 가능. 일부 응용 (Ruby, PostgreSQL 등) 은 — 작게 (MALLOC_ARENA_MAX=2) 설정해서 메모리 절약.

# [user-space - bash]
$ MALLOC_ARENA_MAX=2 ./myapp        # arena 를 2개로 제한
# 메모리 절약, lock 경쟁 약간 증가

arena 와 chunk 의 관계 — NON_MAIN_ARENA 비트

24장에서 본 NON_MAIN_ARENA 비트가 — 여기서 의미를 가진다.

[chunk 의 NON_MAIN_ARENA 비트]

  N=0 (main arena):
    chunk 는 main_arena 의 heap (brk 영역) 에 있음
    free 시 main_arena 사용
    
  N=1 (thread arena):
    chunk 는 thread arena 의 heap (mmap 영역) 에 있음
    free 시 — 어느 arena 인지 알아내야 함
       ↓
    chunk 의 주소를 64MB 경계로 mask
       ↓
    그 위치의 heap_info 의 ar_ptr 가 arena
// [user-space - glibc malloc/arena.c master 매크로]
/* chunk 가 main_arena 의 것이 아니면, arena 의 mstate 찾기 */
#define heap_for_ptr(ptr) \
    ((heap_info *)((unsigned long)(ptr) & ~(HEAP_MAX_SIZE - 1)))

#define arena_for_chunk(ptr) \
    (chunk_main_arena(ptr) ? &main_arena : heap_for_ptr(ptr)->ar_ptr)

매우 영리한 trick. chunk 의 주소만으로 어느 arena 인지 O(1) 에 결정. 64 MB 단위로 align 되어 있다는 사실을 활용.

per-thread chunks 의 처리 — race condition

한 thread 가 할당한 chunk 를 — 다른 thread 가 free 하는 경우는?

// [user-space - C 예제]
// Thread 1:
char *p = malloc(100);    // arena 1 에서 할당
shared_ptr = p;

// Thread 2:
free(shared_ptr);         // arena 1 에서 free 해야 함!

free 의 정확한 동작:

// [user-space - glibc malloc/malloc.c master 단순화]
void __libc_free(void *mem)
{
    mchunkptr p = mem2chunk(mem);
    mstate ar_ptr = arena_for_chunk(p);     /* ★ chunk 의 arena 찾기 */
    
    __libc_lock_lock(ar_ptr->mutex);        /* 그 arena 의 lock */
    _int_free(ar_ptr, p, 0);
    __libc_lock_unlock(ar_ptr->mutex);
}

arena_for_chunk 가 — chunk 의 주소만으로 arena 결정. Thread 가 다르더라도 정확한 arena 의 lock 을 잡고 free.

main_arena 의 특수성

main_arena 와 thread arena 의 차이:

[main_arena vs thread arena]

  main_arena:
    - libc.so 의 .data 에 정적 위치
    - heap_info 없음 (heap 이 brk 영역, 별도 위치)
    - heap 이 brk 로 자람 (연속적)
    - 단편화 위험 (중간 못 반환)
    - main thread 시작 시 이미 존재
    
  thread arena:
    - mmap 으로 새 64MB 영역
    - heap_info + malloc_state 가 영역 시작에
    - 영역 안에서 chunk 들 할당
    - 다 비면 munmap 으로 OS 반환 가능
    - thread 첫 malloc 시 동적 생성

[arena 정리]

  ✓ thread 간 lock 경쟁 회피의 핵심 아이디어
  ✓ main_arena = libc.so 의 .data 에 정적, brk heap 사용
  ✓ thread arena = mmap 으로 64MB 영역, 그 안에 모든 것
  ✓ heap_info + malloc_state 가 arena 의 메타데이터
  ✓ arena 끼리 circular linked list
  ✓ chunk 의 NON_MAIN_ARENA 비트로 arena 식별
  ✓ heap_for_ptr 매크로로 64MB 정렬 이용한 O(1) arena lookup
  ✓ arena_max 로 최대 개수 제한 (보통 8 × NCPUS)

이제 각 arena 안에 — chunk 들이 어떻게 정리되어 있는지를 봐야 한다. 다음 장의 주제 — bin 의 종류. small bin, large bin, unsorted bin, 그리고 fastbin 과 tcache 의 구조.

26bin 의 종류 — small / large / unsorted

이전 장에서 — malloc_state 의 bins[NBINS * 2 - 2] 라는 큰 배열을 봤다. 이게 — 모든 free chunk 가 분류되는 곳, bin. 이 장은 그 bin 의 종류와 각자의 역할을 본다.

(이 장은 일반 bin 들을 본다. fastbin 과 tcache 는 별도의 메커니즘이라 다음 두 장에서 따로 다룬다.)

│     - single-linked LIFO                               │
│     - 10 size class (16~80 bytes 정도)                  │
│     - coalescing 안 됨 (속도 우선)                       │
│                                                         │
│  3. unsorted bin                                       │
│     - 임시 정리장                                        │
│     - free 시 처음 들어가는 곳                            │
│     - malloc 시 정리해서 small/large 로 분배              │
│                                                         │
│  4. small bins                                         │
│     - 32~1024 bytes (62 개 bin)                         │
│     - 각 bin 안에 같은 크기만 (정확히 16 byte 간격)        │
│                                                         │
│  5. large bins                                         │
│     - 1024 bytes 이상 (63 개 bin)                       │
│     - 각 bin 안에 다양한 크기 (정렬됨)                    │
│                                                         │
└────────────────────────────────────────────────────────┘

이 장에서는 unsorted, small, large 의 3 종류를 본다.

bins[] 배열의 구조

malloc_state 의 bins 배열이 모든 일반 bin 의 헤드들을 담는다.

// [user-space - glibc malloc/malloc.c master]
#define NBINS              128
#define NSMALLBINS         64
#define SMALLBIN_WIDTH     MALLOC_ALIGNMENT   /* 16 (x86_64) */
#define MIN_LARGE_SIZE     (NSMALLBINS * SMALLBIN_WIDTH)

/* malloc_state 의 일부 */
mchunkptr bins[NBINS * 2 - 2];     /* 128 * 2 - 2 = 254 개 포인터 */

bins[] 가 — 각 bin 의 fd/bk 쌍을 담는 배열. doubly-linked list 의 sentinel 들.

[bins[] 배열의 layout]

  bin index   배열 위치           용도
  ─────────────────────────────────────────────
  0           bins[0]/bins[1]    unsorted bin
  1           bins[2]/bins[3]    smallbin (32)
  2           bins[4]/bins[5]    smallbin (48)
  ...         ...                ...
  63          bins[126]/bins[127] smallbin (1024)
  64          bins[128]/bins[129] largebin (1024~1088)
  ...         ...                ...
  127         bins[254]/bins[255] largebin (largest)
  
  bin index 0 은 사용 안 됨
  bin index 1 부터 시작 (smallest smallbin)

각 bin index 마다 2 개의 포인터 (fd, bk). 그래서 NBINS * 2 = 256 인데 -2 인 이유 — 시작 부분의 unused 영역 제외.

bin index 와 size 의 매핑

각 bin 이 어떤 크기를 담당하는가?

[bin index → size 매핑 — x86_64]

  unsorted (1):
    크기 무관 (임시 정리장)
    
  smallbin (2 ~ 63):
    16 byte 간격, 정확히 한 크기
    
    smallbin idx → chunk size
        2          32
        3          48
        4          64
        5          80
        ...
       63        1008
       
    공식: chunk_size = 2 * SIZE_SZ + (idx - 1) * MALLOC_ALIGNMENT
                     = 16 + (idx - 1) * 16
    
    예: idx=2 → 16 + 1*16 = 32 ✓
       idx=63 → 16 + 62*16 = 1008 ✓
       
  largebin (64 ~ 126):
    각 bin 이 _범위_ 를 담당 (정확히 한 크기 X)
    
    largebin idx → size range
       64          1024 ~ 1087
       65          1088 ~ 1151
       ...
      126         (very large)

매크로로 표현:

// [user-space - glibc malloc/malloc.c master 매크로]
#define smallbin_index(sz)  \
    ((SMALLBIN_WIDTH == 16 ? (((unsigned)(sz)) >> 4) : (((unsigned)(sz)) >> 3)) \
     + SMALLBIN_CORRECTION)

#define largebin_index_64(sz)                            \
    (((((unsigned long)(sz)) >> 6) <= 48) ?  48 + (((unsigned long)(sz)) >> 6) :\
     ((((unsigned long)(sz)) >> 9) <= 20) ?  91 + (((unsigned long)(sz)) >> 9) :\
     ((((unsigned long)(sz)) >> 12) <= 10) ? 110 + (((unsigned long)(sz)) >> 12) :\
     ((((unsigned long)(sz)) >> 15) <=  4) ? 119 + (((unsigned long)(sz)) >> 15) :\
     ((((unsigned long)(sz)) >> 18) <=  2) ? 124 + (((unsigned long)(sz)) >> 18) :\
                                            126)

largebin 의 index 가 — 로그 스케일. 작은 large 는 64 byte 간격, 큰 large 는 더 큰 간격. 이래야 — 매우 큰 chunk 까지 작은 수의 bin 으로 커버.

small bin — 같은 크기끼리

small bin 의 핵심 — 각 bin 안에는 정확히 한 크기의 chunk 만.

[smallbin (예: idx=4, size=64) 의 구조]

  bin[4] 의 헤드
       │
       fd ──→ chunk_A (64 bytes free)
                │
                fd ──→ chunk_B (64 bytes)
                       │
                       fd ──→ chunk_C (64 bytes)
                              │
                              fd ──→ bin[4] (자기 자신)
                              
                              bk ←── 다시 거꾸로
                              
  모든 chunk 가 정확히 64 bytes
  FIFO (실제는 미묘 - free 한 게 head 에, malloc 한 게 head 에서)

이 단순함 덕에 — small bin 에서 chunk 찾기가 O(1).

// [user-space - glibc malloc/malloc.c _int_malloc 의 smallbin 경로 단순화]
if (in_smallbin_range(nb)) {
    idx = smallbin_index(nb);
    bin = bin_at(av, idx);
    
    if ((victim = last(bin)) != bin) {       /* bin 비어있지 않으면 */
        bck = victim->bk;
        if (__glibc_unlikely(bck->fd != victim))
            malloc_printerr("malloc(): smallbin double linked list corrupted");
        
        set_inuse_bit_at_offset(victim, nb);
        bin->bk = bck;
        bck->fd = bin;
        
        /* 반환! */
        return chunk2mem(victim);
    }
}

매우 단순. last() 가 doubly-linked list 의 마지막 chunk 를 가져오고, 그걸 list 에서 떼서 반환. O(1).

large bin — 다양한 크기, 정렬

large bin 은 다르다. 한 bin 안에 다양한 크기가 있고 — 크기로 정렬됨.

[largebin 의 구조 — fd, bk, fd_nextsize, bk_nextsize]

  largebin head
       │
       fd ──→ 1500 bytes ──→ 1500 bytes ──→ 1500 bytes
                │              (같은 크기끼리 fd/bk)
                │
                fd_nextsize
                │
                ▼
              1800 bytes ──→ 1800 bytes
                │
                fd_nextsize
                │
                ▼
              2500 bytes ──→ 2500 bytes ──→ 2500 bytes ──→ 2500 bytes
                │
                fd_nextsize
                │
                ▼
              ... (큰 크기 순으로)
              
  크기별 그룹의 _첫 chunk_ 만 fd_nextsize/bk_nextsize 로 연결
  같은 크기 안에서는 fd/bk 로 연결

이 구조의 묘미 — best-fit 검색을 빠르게. 사용자가 1700 byte 요청하면:

[largebin best-fit 검색]

  대상: 1700 byte 이상의 가장 작은 chunk
  
  1. fd_nextsize 따라가며 size 비교
     1500 < 1700, skip
     1800 ≥ 1700, 후보!
     
  2. 그 그룹 (1800 짜리들) 중 _마지막_ chunk 선택
     (가장 오래된 chunk, locality 고려)
     
  3. 1800 - 1700 = 100 bytes 가 남으면 split
     남는 100 byte 는 unsorted bin 에 넣음

fd_nextsize 가 — 정렬된 link 라 빠른 검색 가능. O(N) 이지만 — N 이 크기 차이 정도로 적음.

unsorted bin — 임시 정리장

unsorted bin 이 가장 흥미롭다. 모든 free 된 chunk 가 처음 들어가는 곳.

[unsorted bin 의 역할]

  free(chunk):
    1. 인접 chunk 와 coalesce 시도 (29장)
    2. 결과 chunk 를 unsorted bin 에 추가
       (small bin, large bin 으로 직접 안 감!)
    
  malloc(N):
    1. tcache 체크 (28장)
    2. fastbin 체크 (27장)
    3. smallbin 체크 (정확한 크기 있나)
    4. ★ unsorted bin 처리:
       - unsorted bin 의 chunk 들을 _하나씩 꺼내서_
       - 적합한 chunk 면 → 사용
       - 부적합하면 → 적절한 small/large bin 에 넣음
       - 이렇게 unsorted bin 을 비우면서 분배
    5. largebin 검색 (best-fit)
    6. top chunk 사용 또는 heap 확장

unsorted bin 의 묘미 — lazy sorting. free 시점에 정렬하지 않고, 다음 malloc 호출 시 정리. 빈번한 malloc/free 패턴에서 빠른 캐시 역할.

[unsorted bin 의 캐시 효과]

  free(p1)    : unsorted bin 에 들어감
       │
  malloc(같은 크기 N):
       │
       └──→ unsorted bin 의 p1 발견 → 즉시 사용!
            (small/large bin 으로 분류 안 거침)
            
  → 같은 크기를 반복적으로 alloc/free 하는 패턴 (예: 임시 buffer) 매우 빠름

unsorted bin 의 검색 — _int_malloc 의 핵심 loop

master 의 unsorted bin 처리 루프:

// [user-space - glibc malloc/malloc.c _int_malloc 단순화]
for (;;) {
    int iters = 0;
    while ((victim = unsorted_chunks(av)->bk) != unsorted_chunks(av)) {
        bck = victim->bk;
        size = chunksize(victim);
        nextchunk = chunk_at_offset(victim, size);
        
        /* unsorted bin 에서 떼냄 */
        unsorted_chunks(av)->bk = bck;
        bck->fd = unsorted_chunks(av);
        
        /* 1. 마지막 남은 작은 chunk 가 우리한테 맞나? — last_remainder */
        if (in_smallbin_range(nb) &&
            bck == unsorted_chunks(av) &&
            victim == av->last_remainder &&
            (unsigned long)(size) > (unsigned long)(nb + MINSIZE)) {
            /* split 해서 일부 반환, 나머지는 last_remainder */
            return chunk2mem(victim);
        }
        
        /* 2. 정확히 맞으면 사용 */
        if (size == nb) {
            set_inuse_bit_at_offset(victim, size);
            return chunk2mem(victim);
        }
        
        /* 3. 아니면 적절한 bin (small 또는 large) 에 넣음 */
        if (in_smallbin_range(size)) {
            victim_index = smallbin_index(size);
            ...                                 /* smallbin 에 추가 */
        } else {
            victim_index = largebin_index(size);
            ...                                 /* largebin 에 정렬 추가 */
        }
        
        if (++iters >= MAX_ITERS) break;       /* 안전 한계 */
    }
    
    /* unsorted bin 비웠으면 large bin 검색 */
    ...
}

매우 정교한 루프. 각 unsorted chunk 가 적합한지 검사하고, 아니면 분류. 100 번 iteration 후엔 강제 종료 (DoS 방지).

last_remainder — locality 최적화

malloc_state 의 last_remainder 필드가 흥미롭다. 마지막에 split 된 chunk 의 남은 부분.

[last_remainder 의 동작]

  malloc(100) 호출
       ↓
  large chunk (256 bytes) 를 split:
    100 bytes 반환
    156 bytes 가 last_remainder 가 됨
       ↓
  다음 malloc(100) 호출
       ↓
  last_remainder 가 100+MINSIZE 이상이면
  → 거기서 또 split
  → cache 친화적 (같은 영역에서 계속 할당)

이게 — 반복적인 같은 크기 할당의 locality 최적화. 메모리상 인접한 위치를 계속 사용 → CPU cache hit 율 향상.

binmap — 빠른 bin 탐색

malloc_state 의 binmap[4] 필드가 — 128 비트 비트맵.

// [user-space - glibc malloc/malloc.c master]
#define BINMAPSIZE         4
#define BITSPERMAP         32
#define MAX_FAST_SIZE     (80 * SIZE_SZ / 4 - 1)

unsigned int binmap[BINMAPSIZE];     /* 4 × 32 = 128 비트 */

각 비트가 — bin 1개의 상태 (chunk 있는지). 큰 bin 들을 순회할 때 — 비트맵으로 비어있는 bin 들 빠르게 건너뜀.

[binmap 의 활용]

  malloc(500) 호출:
    smallbin idx = 32 (적합한 smallbin)
    
  smallbin[32] 비어있으면 → smallbin[33], smallbin[34], ...
  
  순회 방법:
    binmap[idx/32] 의 idx%32 비트 검사
    set 된 비트만 검사 (set 안 된 bin 은 비어있음)
    → 128 비트 한 워드로 32 bin 한 번에 검사 가능
    → 매우 빠른 탐색
// [user-space - glibc malloc/malloc.c 비트맵 매크로]
#define mark_bin(m, i)    ((m)->binmap[idx2block(i)] |= idx2bit(i))
#define unmark_bin(m, i)  ((m)->binmap[idx2block(i)] &= ~(idx2bit(i)))
#define get_binmap(m, i)  ((m)->binmap[idx2block(i)] & idx2bit(i))

bin 에 chunk 가 추가되면 — mark_bin. 비워지면 — unmark_bin. 검색 시 — get_binmap 으로 빠른 체크.

_int_malloc 의 전체 흐름

malloc 의 _큰_ 흐름이 이제 그려진다.

┌─────────────────────────────────────────────────┐
│ _int_malloc(av, nb) 의 전체 흐름                    │
└──────────┬──────────────────────────────────────┘
           │
           ▼
  ┌─────────────────────┐
  │ tcache 체크 (28장)   │ YES → 반환
  └──────────┬──────────┘
           │ NO
           ▼
  ┌─────────────────────┐
  │ fastbin 체크 (27장)  │ YES → 반환
  └──────────┬──────────┘
           │ NO
           ▼
  ┌─────────────────────┐
  │ smallbin 적합 크기?    │ YES → 사용 → 반환
  └──────────┬──────────┘
           │ NO (또는 smallbin 비어있음)
           ▼
  ┌─────────────────────────┐
  │ unsorted bin 처리:        │
  │ 각 chunk 검사:             │
  │  - last_remainder 맞나?    │ YES → split, 반환
  │  - 정확한 크기?            │ YES → 반환
  │  - 아니면 적절한 bin 에 분배  │
  └──────────┬──────────────┘
           │
           ▼
  ┌─────────────────────┐
  │ largebin 검색 (best-fit) │ YES → split, 반환
  └──────────┬──────────┘
           │ NO
           ▼
  ┌─────────────────────┐
  │ 더 큰 binmap 검색       │ YES → 반환
  └──────────┬──────────┘
           │ NO
           ▼
  ┌─────────────────────┐
  │ top chunk 사용         │ 가능하면 → 반환
  └──────────┬──────────┘
           │ 부족하면
           ▼
  ┌─────────────────────┐
  │ heap 확장 (brk/mmap)  │ → 반환
  └─────────────────────┘

이 흐름이 — **사용자가 malloc(N) 하면 일어나는 일** 의 모든 것.

bin 의 검사 — malloc_consolidate

흥미로운 마지막 메커니즘 — **malloc_consolidate**. 가끔 fastbin 의 chunk 들을 모두 일반 bin 으로 합치는 함수.

[malloc_consolidate 의 트리거]

  - 큰 malloc 호출 (fastbin 으로 못 처리)
  - mallopt 호출
  - 명시적 호출
  
  동작:
    fastbin 의 모든 chunk 를:
      1. fastbin 에서 꺼냄
      2. coalesce (인접 free 와 합침)
      3. unsorted bin 에 넣음
      
  목적: 단편화 회복
        fastbin 은 coalesce 안 함 → 가끔 정리 필요

이게 — 메모리 단편화의 주기적 청소. 자세한 내용은 27장 (fastbin) 에서.

이 장의 결론

bin 의 세계 정리:

[bin 정리]

  ✓ malloc_state.bins[] 가 모든 일반 bin 의 헤드
  ✓ unsorted bin: 모든 free 가 처음 들어가는 임시 정리장
  ✓ small bin: 16~1024 bytes, 각 bin 에 같은 크기만 (62개)
  ✓ large bin: 1024+ bytes, 다양한 크기 정렬 (63개)
  ✓ fd_nextsize/bk_nextsize: large bin 의 크기별 추가 link
  ✓ binmap: 128 비트 비트맵으로 빠른 bin 탐색
  ✓ last_remainder: locality 최적화
  ✓ malloc_consolidate: fastbin 청소

다음 두 장에서 — 일반 bin 보다 더 빠른 두 메커니즘을 본다. 27장은 fastbin (single-linked LIFO 의 빠른 길), 28장은 tcache (lock-free thread cache).

27fastbin — single-linked LIFO 의 빠른 길

자주 일어나는 작은 할당과 해제 — malloc(64); free(p) — 를 매우 빠르게 처리하기 위한 별도 캐시가 fastbin 이다. 일반 bin 의 복잡함을 우회해서, 최단 경로로 chunk 를 재사용한다.

fastbin 의 디자인 결정

fastbin 의 모든 디자인은 — "빠름이 최우선" 이라는 한 가지 원칙에서 나왔다.

[fastbin 의 디자인 결정과 trade-off]

  1. single-linked list (fd 만, bk 없음)
     → 메모리 절약 + 더 빠른 삽입/삭제
     → 단점: 중간에서 못 뺌, 양방향 탐색 불가
     
  2. LIFO (FIFO 가 아님)
     → 가장 최근 free 된 것 먼저 재사용
     → CPU cache hot → 빠름
     → 단점: 오래된 chunk 가 안 쓰일 수도
     
  3. coalescing 안 함
     → free 시 인접 chunk 와 안 합침
     → PREV_INUSE 비트를 1 로 유지 → 합치기 비활성화
     → 단점: 단편화 위험
     → 보완: malloc_consolidate 가 가끔 청소
     
  4. 정확한 크기 매칭만 (split 안 함)
     → 64 바이트 요청 → 정확히 64 byte fastbin 만 봄
     → split 비용 없음
     → 단점: 약간 다른 크기는 fastbin 사용 안 됨

fastbin 의 구조 — master 정의

master 의 malloc/malloc.c:

// [user-space - glibc malloc/malloc.c master 인용]
typedef struct malloc_chunk *mfastbinptr;

/* malloc_state 의 일부 */
mfastbinptr fastbinsY[NFASTBINS];

/* fastbin 의 최대 크기 (사용자 데이터 기준 ~80 bytes) */
#define MAX_FAST_SIZE     (80 * SIZE_SZ / 4)
/* x86_64: 80 * 8 / 4 = 160 (chunk 헤더 포함) */
/* 사용자 데이터: 160 - 16 = 144 → 실제 기본 limit 은 80 byte 정도 */

#define NFASTBINS         (fastbin_index(request2size(MAX_FAST_SIZE)) + 1)
/* x86_64: 10 */

#define fastbin_index(sz)                                          \
    ((((unsigned int)(sz)) >> (SIZE_SZ == 8 ? 4 : 3)) - 2)

#define fastbin(ar_ptr, idx)  ((ar_ptr)->fastbinsY[idx])

fastbin 의 크기 분포

x86_64 에서 — fastbin index 와 chunk 크기의 관계:

[fastbin index ↔ chunk size — x86_64]

  fastbin_index(sz) = (sz >> 4) - 2
  
  idx  chunk size  사용자 데이터
  ───────────────────────────────
   0     32         24
   1     48         40
   2     64         56
   3     80         72
   4     96         88
   5    112        104
   6    128        120
   7    144        ──┐ 기본은 안 씀
   8    160        ──┤ (global_max_fast 가 128 으로 제한)
   9    176        ──┘
   
  기본 동작: idx 0~6 (128 bytes 까지) 만 사용
  글로벌 변수 global_max_fast 로 조정 가능

global_max_fast 는 — 런타임에 fastbin 의 limit 을 조정. 기본은 chunk size 기준 약 128. mallopt(M_MXFAST, N) 으로 변경.

fastbin 의 single-linked LIFO

각 fastbin 의 구조 — 매우 단순.

[fastbin (예: idx=2, 64 byte chunk) 의 구조]

  malloc_state.fastbinsY[2] (head pointer)
        │
        ▼
   ┌──────────┐
   │ chunk A  │ ← 가장 최근에 free 된 chunk
   │ fd  ─────┼──→
   │ ...      │
   └──────────┘
        │
        ▼
   ┌──────────┐
   │ chunk B  │
   │ fd  ─────┼──→
   │ ...      │
   └──────────┘
        │
        ▼
   ┌──────────┐
   │ chunk C  │
   │ fd  ─────┼──→ NULL  ← 가장 오래된
   │ ...      │
   └──────────┘
  
  free(D) 호출:
    head[2] = D, D->fd = A
    → D 가 새로 맨 앞
  
  malloc(64) 호출:
    chunk = head[2]
    head[2] = chunk->fd
    → 맨 앞에서 빼냄 (LIFO)

이게 — stack 처럼 동작. push/pop 이 O(1).

safe-linking — glibc 2.32 의 보안 강화

위 코드의 PROTECT_PTR 와 REVEAL_PTR 가 — glibc 2.32 (2020.8) 부터 추가된 보안 메커니즘.

// [user-space - glibc malloc/malloc.c master safe-linking 매크로]
/* Safe-Linking:
   Use randomness from ASLR (mmap_base) to protect single-linked lists
   of Fast-Bins and TCache.  That is, mask the "next" pointers of the
   lists' chunks, and also perform allocation alignment checks on them.
   This mechanism makes graceful symmetric attacks much harder. */

#define PROTECT_PTR(pos, ptr) \
    ((__typeof(ptr))((((size_t)pos) >> 12) ^ ((size_t)ptr)))

#define REVEAL_PTR(ptr)  PROTECT_PTR(&ptr, ptr)

동작:

[safe-linking 의 동작]

  fastbin / tcache 의 fd 포인터 저장 시:
    저장값 = (실제 주소) XOR (현재 위치 >> 12)
    
  읽을 때:
    실제 주소 = (저장값) XOR (현재 위치 >> 12)
    
  공격자가 fd 를 임의 주소로 덮어쓰려면:
    원하는 주소 XOR (해당 위치 >> 12) 를 정확히 알아야 함
    → ASLR base 도 알아야 함 (어렵게 만듦)
    → 또한 fd 의 결과가 page 정렬되어야 통과 → 추가 제약
    
  결과: fastbin/tcache attack 의 난이도 크게 상승

이 메커니즘이 — glibc 2.32 이후 heap exploit 의 게임 체인저. House of Force 같은 옛 공격 기법 일부 무력화.

fastbin 의 PREV_INUSE 비트 — coalescing 우회

fastbin 의 흥미로운 trick — PREV_INUSE 비트를 1 로 유지.

[fastbin chunk 의 PREV_INUSE 비트]

  일반 free 시:
    chunk 의 다음 chunk 의 size 의 P 비트 = 0
    → 다음 free 시 이 chunk 가 free 임을 알 수 있음 (coalesce 가능)
    
  fastbin free 시:
    ★ chunk 의 다음 chunk 의 size 의 P 비트 = 1 유지!
    → 다음 free 시 이 chunk 가 _사용 중_ 으로 보임
    → coalesce 안 일어남
    → 그래서 단편화 발생 가능

이게 — 의도된 단편화. 속도를 위해 단편화를 감수.

                        nextinuse = inuse_bit_at_offset(nextchunk, nextsize);

                            size += nextsize;
                            unlink_chunk(av, nextchunk);
                        } else
                            clear_inuse_bit_at_offset(nextchunk, 0);

                        first_unsorted = unsorted_bin->fd;
                        unsorted_bin->fd = p;
                        first_unsorted->bk = p;

                    else {

                    }

    }
}

fastbin 의 보안 이슈와 대응

fastbin 은 — CTF / heap exploit 의 단골 표적. 단순함이 공격 표면이 됨.

[fastbin 의 주요 공격 기법]

  1. Fastbin Dup (double-free):
     같은 chunk 를 두 번 free
     → fastbin 에 같은 chunk 가 두 번 들어감
     → 다음 malloc 으로 그 chunk 재할당받으면서 fd 조작
     → 임의 주소 할당 가능
     
  2. Fastbin AAR (Arbitrary Address Read):
     fastbin 의 fd 를 임의 주소로 덮어씀
     → 다음 malloc 이 그 주소 반환
     → 그 주소의 데이터 읽기/쓰기 가능
     
  3. House of Force (옛):
     top chunk 의 size 덮어써서 임의 큰 할당
     → 임의 메모리 접근
     (현대 glibc 는 top chunk 크기 검증으로 방지)

이 공격들에 대응하는 보안 메커니즘들:

[fastbin 의 보안 메커니즘]

  ✓ double-free 체크 (head 와 직전 free 가 같은지)
  ✓ size sanity 검사 (fastbin 범위인지)
  ✓ alignment 검사
  ✓ ★ safe-linking (glibc 2.32+)
       fd 포인터를 XOR 으로 mangling
  ✓ ★ tcache safe-linking (glibc 2.32+) — 다음 장

코드 측면에서 fastbin 은 — 사용자에게 투명. 그냥 malloc(64); free(p) 하면 — 알아서 fastbin 활용. 사용자가 fastbin 을 의식할 필요 없음.

이 장의 결론

fastbin 의 본질:

[fastbin 정리]

  ✓ 작은 chunk (16~80 bytes) 의 빠른 캐시
  ✓ single-linked LIFO (fd 만 사용)
  ✓ NFASTBINS (10), 기본 사용 7개
  ✓ coalescing 안 함 (PREV_INUSE = 1 유지)
  ✓ split 안 함 (정확한 크기만)
  ✓ malloc_consolidate 가 가끔 청소
  ✓ have_fastchunks 플래그로 빠른 체크
  ✓ safe-linking (glibc 2.32+) 보안 강화
  ✓ ~12 ns/op, CPU cache 친화적

fastbin 은 — 속도와 보안과 단편화 사이의 정교한 균형. 다음 장의 tcache 가 — 이걸 한 단계 더 발전시킨다. lock-free per-thread cache.

28tcache — thread 별 캐시 (glibc 2.26+)

fastbin 이 빠르긴 하지만 — 여전히 arena lock 이 필요하다. multi-thread 에서 arena 를 공유하면 — fastbin 접근도 lock 경쟁. glibc 2.26 (2017.8) 이 도입한 해결책이 tcache (thread cache) — lock 없이 동작하는 per-thread 캐시.

이게 — 현대 glibc malloc 의 가장 큰 성능 개선.

tcache 의 핵심 아이디어

[tcache 의 디자인 — 한 줄 요약]

  각 thread 가 _자기만의_ 캐시를 가짐
  → arena 의 lock 도 안 필요
  → arena 의 fastbin 보다도 빠름
  → first-line cache 역할
  
  malloc 호출 시:
    1. tcache 체크 (가장 먼저)
    2. fastbin
    3. smallbin / unsortedbin / largebin
    4. top chunk / heap 확장
  
  free 호출 시:
    1. tcache 가 안 차 있으면 → tcache 에 추가
    2. tcache 가 차 있으면 → fastbin / unsortedbin

tcache 의 자료구조

master 의 malloc/malloc.c:

// [user-space - glibc malloc/malloc.c master 인용]

   when the chunk is stored in the per-thread cache. */
typedef struct tcache_entry {
    struct tcache_entry *next;
    /* This field exists to detect double frees. */
    struct tcache_perthread_struct *key;        /* glibc 2.29+ */

   per-thread cache (hence "tcache_perthread_struct"). */

    uint16_t num_slots[TCACHE_MAX_BINS];        /* 옛: char counts[] */

핵심 — tcache 자체가 **__thread** 변수. TLS 에 저장되어 각 thread 마다 별도.

TCACHE_MAX_BINS 의 진화

tcache 의 bin 개수는 — glibc 버전에 따라 다르다.

[TCACHE_MAX_BINS 의 진화]

  glibc 2.26 (2017.8) ~ 2.41 (2025.1):
    # define TCACHE_MAX_BINS 64
    → 24 ~ 1032 bytes 의 chunk 만 캐시 (x86_64)
    
  glibc 2.42 (2025.7.28) — 큰 변경:
    # define TCACHE_SMALL_BINS 64
    # define TCACHE_LARGE_BINS 12     /* Up to 4M chunks */
    # define TCACHE_MAX_BINS (TCACHE_SMALL_BINS + TCACHE_LARGE_BINS)  // 76
    → large bin 까지 확장!
    → 최대 ~4 MB chunk 까지 캐시
    
  변경 이유:
    매우 큰 할당 패턴에서도 tcache 의 lock-free 혜택
    Java/Python 같은 GC 언어 응용에 유리

각 bin 에 들어갈 수 있는 chunk 개수:

기본 7 개. 그 이상은 fastbin 또는 unsortedbin 으로.

[tcache 의 한도]

  bin 개수    : 64 (옛) / 76 (새)
  bin 당 chunk: 최대 7 개 (기본, tunable 로 변경 가능)
  
  총 캐시 크기:
    64 bin × 7 chunk × 평균 ~256 byte = 약 115 KB

tcache_put — free 의 tcache 분기

master (glibc 2.42+) 의 tcache_put:

// [user-space - glibc malloc/malloc.c tcache_put 단순화]
static __always_inline void
tcache_put(mchunkptr chunk, size_t tc_idx)
{
    tcache_put_n(chunk, tc_idx, &tcache->entries[tc_idx], false);
}

/* Caller must ensure that we know tc_idx is valid and there's room
   for more chunks. */
static __always_inline void
tcache_put_n(mchunkptr chunk, size_t tc_idx, tcache_entry **ep, bool mangled)
{
    tcache_entry *e = (tcache_entry *)chunk2mem(chunk);
    
    /* Mark this chunk as "in the tcache" so the test in __libc_free will
       detect a double free. */
    e->key = tcache_key;                          /* ★ 별도 변수, double-free 탐지 */
    
    if (!mangled) {
        e->next = PROTECT_PTR(&e->next, *ep);     /* safe-linking */
        *ep = e;
    } else {
        e->next = PROTECT_PTR(&e->next, REVEAL_PTR(*ep));
        *ep = PROTECT_PTR(ep, e);
    }
    
    --(tcache->num_slots[tc_idx]);                /* ★ 감소 — 남은 slot 줄어듦 */
}

매우 단순. TLS 의 tcache 에 prepend, num_slots 감소 (slot 개수가 줄어드는 의미). lock 전혀 없음.

num_slots 의 의미가 미묘하다 — 남은 slot 개수. 옛 counts[] (현재 chunk 개수) 와는 _반대 방향_:

[counts (옛, ≤glibc 2.41) vs num_slots (새, glibc 2.42+)]

  counts[i] = 0:    bin i 가 비어 있음 (chunk 0개)
  counts[i] = 7:    bin i 가 가득 참 (chunk 7개)
  추가 가능 조건:    counts < tcache_count
  
  num_slots[i] = 7: bin i 가 비어 있음 (slot 7개 남음)
  num_slots[i] = 0: bin i 가 가득 참 (slot 0개 남음)
  추가 가능 조건:    num_slots != 0
  
  같은 정보를 _반대_ 로 표현. uint16_t 로 커진 것은
  TCACHE_FILL_COUNT 가 tunable 로 더 큰 값 가능해서.

tcache_key 는 — 옛 tcache (스레드의 tcache 주소) 대신 사용하는 별도 변수. 모든 thread 가 공유하는 random 값이 아니라, 각 thread 의 TLS 의 한 변수 — 다른 thread 의 tcache 와 confuse 방지 + ASLR-friendly 랜덤성.

tcache_get — malloc 의 tcache 분기

// [user-space - glibc malloc/malloc.c tcache_get 단순화]
static __always_inline void *
tcache_get_n(size_t tc_idx, tcache_entry **ep)
{
    tcache_entry *e;
    if (ep == &(tcache->entries[tc_idx]))
        e = *ep;
    else
        e = REVEAL_PTR(*ep);
    
    if (__glibc_unlikely(!aligned_OK(e)))
        malloc_printerr("malloc(): unaligned tcache chunk detected");
    
    if (ep == &(tcache->entries[tc_idx]))
        *ep = REVEAL_PTR(e->next);                  /* safe-linking 해제 */
    else
        *ep = PROTECT_PTR(ep, REVEAL_PTR(e->next));
    
    ++(tcache->num_slots[tc_idx]);                  /* ★ 증가 — 남은 slot 늘어남 */
    e->key = 0;                                     /* "no longer in tcache" 마커 */
    
    return (void *)e;
}

static __always_inline void *
tcache_get(size_t tc_idx)
{
    return tcache_get_n(tc_idx, &tcache->entries[tc_idx]);
}

마찬가지로 매우 단순. head 에서 빼고 num_slots 증가 (다시 사용 가능한 slot 늘어남). O(1), lock 없음.

tcache 의 첫 초기화

각 thread 의 첫 malloc 호출 시 — tcache 자체를 할당.

// [user-space - glibc malloc/malloc.c tcache_init 단순화]
static void
tcache_init(void)
{
    mstate ar_ptr;
    void *victim = 0;
    const size_t bytes = sizeof(tcache_perthread_struct);
    
    if (tcache_shutting_down)
        return;
    
    arena_get(ar_ptr, bytes);                     /* arena 잡고 */
    victim = _int_malloc(ar_ptr, bytes);          /* tcache_perthread_struct 할당 */
    
    if (!victim && ar_ptr != NULL) {
        ar_ptr = arena_get_retry(ar_ptr, bytes);
        victim = _int_malloc(ar_ptr, bytes);
    }
    
    if (ar_ptr != NULL)
        __libc_lock_unlock(ar_ptr->mutex);
    
    if (victim) {
        tcache = (tcache_perthread_struct *)victim;
        memset(tcache, 0, sizeof(tcache_perthread_struct));
    }
}

#define MAYBE_INIT_TCACHE()  \
    if (__glibc_unlikely(tcache == NULL))  \
        tcache_init();

흥미로운 점 — **tcache_perthread_struct 자체가 일반 malloc 으로 할당된다**. heap 의 한 chunk. 그래서 tcache 도 사실은 heap 의 일부.

  Thread 시작
       │
       ▼

       │
       ▼

       ▼
  arena 잡고 sizeof(tcache_perthread_struct) 크기 chunk 할당
       │
       ▼

       │
       ▼
  이제 본 작업: tcache 가 비어있으니 일반 경로로 N 할당
       │
       ▼
  사용자에게 반환

tcache 의 보안 — key 필드와 safe-linking

tcache 는 — 초기 도입 시 보안에 미약했지만, glibc 2.29 부터 개선.

typedef struct tcache_entry {
    struct tcache_entry *next;
    struct tcache_perthread_struct *key;          /* glibc 2.29+ */
} tcache_entry;

key 필드의 동작:

[key 필드의 double-free 탐지]

  free(p) 시:
    e->key = tcache       (= 현재 thread 의 tcache 주소)
    tcache 에 추가
    
  같은 chunk 를 또 free 시:
    e->key 가 이미 tcache 주소
    → tcache 안 검색해서 같은 chunk 있는지 확인
    → 있으면 "double free detected in tcache 2" 에러
    
  malloc(p) 시:
    e->key = 0  (마커 제거)

또한 safe-linking (glibc 2.32+) 도 tcache 에 적용:

e->next = PROTECT_PTR(&e->next, tcache->entries[tc_idx]);   /* free */
tcache->entries[tc_idx] = REVEAL_PTR(e->next);              /* malloc */

같은 XOR 트릭. tcache 의 fd 도 mangling.

tcache 의 tunable

런타임에 tcache 동작 조정 가능.

[tcache tunable (GLIBC_TUNABLES)]

  glibc.malloc.tcache_max:
    각 bin 의 최대 chunk 크기 (기본 1032 bytes)
    
  glibc.malloc.tcache_count:
    bin 당 최대 chunk 개수 (기본 7)
    
  glibc.malloc.tcache_unsorted_limit:
    malloc 시 unsorted bin 에서 가져올 최대 chunk (기본 0 = 무제한)

tcache 의 thread 종료 — tcache_thread_shutdown

thread 가 종료될 때 — tcache 의 chunk 들은 어떻게 되나? __libc_thread_freeres 가 청소.

// [user-space - glibc malloc/malloc.c master 단순화]
static void
tcache_thread_shutdown(void)
{
    int i;
    tcache_perthread_struct *tcache_tmp = tcache;
    
    if (!tcache)
        return;
    
    /* Disable the tcache and prevent further allocation */
    tcache = NULL;
    
    /* tcache 의 모든 chunk 를 (free 처리해서) 다른 thread 가 쓸 수 있게 */
    for (i = 0; i < TCACHE_MAX_BINS; ++i) {
        while (tcache_tmp->entries[i]) {
            tcache_entry *e = tcache_tmp->entries[i];
            if (__glibc_unlikely(!aligned_OK(e)))
                malloc_printerr("tcache_thread_shutdown(): unaligned tcache chunk detected");
            tcache_tmp->entries[i] = REVEAL_PTR(e->next);
            __libc_free(e);     /* 일반 free 로 unsorted bin 등에 반환 */
        }
    }
    
    __libc_free(tcache_tmp);    /* tcache 구조체 자체도 free */
}

thread 종료 시 — tcache 의 모든 chunk 가 일반 bin 으로 흘러감. 다른 thread 가 활용 가능.

tcache 의 실제 영향 — 응용

tcache 가 등장한 이후 — 많은 응용의 성능 패턴이 바뀜.

[tcache 가 효과적인 패턴]

  ✓ multi-thread + 작은 할당 (전형적 서버)
  ✓ 같은 크기 반복 alloc/free (임시 buffer)
  ✓ thread-local 자료구조 (cache 친화적)
  
  ✗ 큰 할당 (옛 버전 tcache 범위 밖)
  ✗ thread 가 자주 죽고 생기는 패턴 (tcache 초기화 비용)
  ✗ 매우 다양한 크기 (캐시 hit 율 낮음)

옛 버전과 최신 master 의 차이

[tcache 의 진화]

  glibc 2.26 (2017.8):
    초기 도입, TCACHE_MAX_BINS = 64
    key 필드 없음 (double-free 보호 약함)
    
  glibc 2.29 (2019.1):
    tcache_entry 에 key 필드 추가
    double-free 탐지 (한 bin 검사)
    
  glibc 2.30 (2019.8):
    MAX_TCACHE_COUNT = 127 추가 (BZ#24531)
    char counts 의 한도 명시
    
  glibc 2.32 (2020.8):
    safe-linking (XOR 보호)
    fastbin/tcache 의 fd mangling
    
  glibc 2.34 (2021.8):
    tcache_init 의 lock 처리 개선
    __libc_csu_init 제거 (별개 변경)
    
  glibc 2.42 (2025.7) — 가장 큰 변경:
    TCACHE_LARGE_BINS 추가 (large bin 까지 캐시, 4MB chunks)
    counts → num_slots (uint16_t)
    의미 반전: 현재 chunk 수 → 남은 slot 수
    tcache_put_large / tcache_get_large 함수 분리
    tcache_double_free_verify — 전체 bin 순회로 검사 강화 (BZ#32422)
    tcache_key 별도 변수 도입 (옛: e->key = tcache)

이 진화는 — fastbin 의 영광을 tcache 가 점진적으로 흡수해온 과정. 옛날엔 fastbin 이 작은 할당의 빠른 길이었는데, 이제는 tcache 가 그 자리. fastbin 은 fallback.

이 장의 결론

tcache 의 본질:

[tcache 정리]

  ✓ glibc 2.26 (2017.8) 도입
  ✓ per-thread (TLS), lock-free
  ✓ first-line cache: malloc/free 의 첫 검사
  ✓ single-linked LIFO (fastbin 처럼)
  ✓ 64 bin (옛) / 76 bin (새), bin 당 7 chunk
  ✓ ~1 KB (옛) / ~4 MB (새) 까지 캐시
  ✓ key 필드로 double-free 탐지 (2.29+)
  ✓ safe-linking 으로 fd mangling (2.32+)
  ✓ tunable 로 동작 조정 가능
  ✓ multi-thread 작은 할당 성능 1.5~2배 개선

지금까지 5 종류의 bin 을 모두 봤다 (unsorted, small, large, fast, tcache). 다음 장에서는 — bin 들 사이를 chunk 가 움직이는 메커니즘의 핵심, coalescing (단편화 합치기) 을 본다.

29coalescing — 단편화와의 싸움

malloc/free 가 반복되면 — heap 이 잘게 나뉘어 사용 가능한 큰 연속 영역이 사라진다. 이게 외부 단편화 (external fragmentation). 해결책이 — coalescing (인접한 free chunk 들을 합치는 것).

이 장은 그 핵심 메커니즘을 본다.

단편화의 두 종류

먼저 두 가지 단편화를 구분.

[단편화의 두 종류]

  내부 단편화 (internal fragmentation):
    chunk 안의 사용 안 되는 공간
    예: malloc(17) → 32 byte chunk 받음 → 15 byte 낭비
    원인: 정렬 강제, 최소 크기, chunk 단위
    
  외부 단편화 (external fragmentation):
    free 영역들이 작은 조각으로 흩어짐
    큰 할당 요청 못 함 (총 free 는 충분해도)
    예: 10 KB free 가 있어도 1 KB 짜리 10 개로 흩어지면
        2 KB 요청 못 받음
    원인: alloc/free 의 순서, 인접 free 안 합치면

malloc 의 가장 큰 적이 — 외부 단편화. 시간이 갈수록 쌓이는 누적적 문제.

외부 단편화의 예

[외부 단편화의 발생]

  초기:
  ┌─────────────────────────────────────────────────┐
  │                  free 32 KB                       │
  └─────────────────────────────────────────────────┘
  
  malloc(8KB) × 4:
  ┌────────┬────────┬────────┬────────┬─────────────┐
  │  A 8K  │  B 8K  │  C 8K  │  D 8K  │  free 0     │
  └────────┴────────┴────────┴────────┴─────────────┘
  
  free(B), free(D):
  ┌────────┬────────┬────────┬────────┬─────────────┐
  │  A 8K  │ free 8K│  C 8K  │ free 8K│  free 0     │
  └────────┴────────┴────────┴────────┴─────────────┘
  
  malloc(16KB) 호출:
    총 16 KB free 인데 — 16 KB 연속 영역이 없음!
    8 KB + 8 KB 로 흩어짐
    → 새로 heap 확장 필요 → 메모리 낭비

이 문제를 해결하려면 — 인접한 free chunk 들을 합쳐야. 그게 coalescing.

coalescing 의 핵심 — boundary tag

24장에서 본 boundary tag 가 — coalescing 을 가능하게 한다.

[boundary tag 의 활용]

  chunk P 를 free 할 때:
    1. P 의 _다음_ chunk 가 free 인가?
       → P 의 mchunk_size 로 다음 chunk 위치 계산
       → 그 chunk 의 P 비트 확인
       → free 면 합치기
       
    2. P 의 _이전_ chunk 가 free 인가?
       → P 의 mchunk_size 의 P 비트 확인
       → P 비트 = 0 이면 이전 chunk 가 free
       → P 의 mchunk_prev_size 로 이전 chunk 위치 계산
       → 그 chunk 와 합치기

24장에서 본 PREV_INUSE 비트가 — 여기서 결정적 역할.

_int_free 의 coalescing

master 의 _int_free 의 coalescing 부분:

// [user-space - glibc malloc/malloc.c _int_free coalescing 단순화]
static void
_int_free(mstate av, mchunkptr p, int have_lock)
{
    INTERNAL_SIZE_T size = chunksize(p);
    mchunkptr nextchunk = chunk_at_offset(p, size);
    INTERNAL_SIZE_T nextsize = chunksize(nextchunk);
    
    /* (tcache, fastbin 분기는 위에서) */
    
    /* 메모리상 _이전_ chunk 가 free 면 합치기 */
    if (!prev_inuse(p)) {
        prevsize = prev_size(p);
        size += prevsize;
        p = chunk_at_offset(p, -((long)prevsize));    /* p 가 합쳐진 chunk 의 시작 */
        if (__glibc_unlikely(chunksize(p) != prevsize))
            malloc_printerr("corrupted size vs. prev_size while consolidating");
        unlink_chunk(av, p);                          /* 이전 chunk 를 bin 에서 제거 */
    }
    
    /* 메모리상 _다음_ chunk 가 free 면 합치기 */
    if (nextchunk != av->top) {
        nextinuse = inuse_bit_at_offset(nextchunk, nextsize);
        
        if (!nextinuse) {
            unlink_chunk(av, nextchunk);              /* 다음 chunk 를 bin 에서 제거 */
            size += nextsize;
        } else {
            clear_inuse_bit_at_offset(nextchunk, 0); /* 다음 chunk 의 P 비트 클리어 */
        }
        
        /* 합쳐진 chunk 를 unsorted bin 에 넣기 */
        bck = unsorted_chunks(av);
        fwd = bck->fd;
        ...
        p->fd = fwd;
        p->bk = bck;
        ...
        set_head(p, size | PREV_INUSE);
        set_foot(p, size);
    }
    else {
        /* 다음이 top chunk 면 합쳐서 top 으로 */
        size += nextsize;
        set_head(p, size | PREV_INUSE);
        av->top = p;
    }
}

핵심 흐름:

[free 시 coalescing 의 4 경우]

  case 1: 이전 free, 다음 in-use
    → 이전과 합침
    
  case 2: 이전 in-use, 다음 free
    → 다음과 합침
    
  case 3: 이전 free, 다음 free
    → 셋 모두 합침 (이전 + p + 다음)
    
  case 4: 이전 in-use, 다음 in-use
    → 합치기 없음, p 만 unsorted bin 에

이렇게 — free 시점에 즉시 합치기 (eager coalescing). 단편화 누적 방지.

unlink_chunk — 보안 검증

unlink_chunk 가 매우 중요 — bin 의 doubly-linked list 에서 chunk 를 제거.

// [user-space - glibc malloc/malloc.c master 인용]
static void
unlink_chunk(mstate av, mchunkptr p)
{
    if (chunksize(p) != prev_size(next_chunk(p)))
        malloc_printerr("corrupted size vs. prev_size");
    
    mchunkptr fd = p->fd;
    mchunkptr bk = p->bk;
    
    if (__glibc_unlikely(fd->bk != p || bk->fd != p))
        malloc_printerr("corrupted double-linked list");
    
    fd->bk = bk;
    bk->fd = fd;
    if (!in_smallbin_range(chunksize_nomask(p)) && p->fd_nextsize != NULL) {
        if (p->fd_nextsize->bk_nextsize != p
            || p->bk_nextsize->fd_nextsize != p)
            malloc_printerr("corrupted double-linked list (not small)");
        
        if (fd->fd_nextsize == NULL) {
            if (p->fd_nextsize == p)
                fd->fd_nextsize = fd->bk_nextsize = fd;
            else {
                fd->fd_nextsize = p->fd_nextsize;
                fd->bk_nextsize = p->bk_nextsize;
                p->fd_nextsize->bk_nextsize = fd;
                p->bk_nextsize->fd_nextsize = fd;
            }
        } else {
            p->fd_nextsize->bk_nextsize = p->bk_nextsize;
            p->bk_nextsize->fd_nextsize = p->fd_nextsize;
        }
    }
}

3 가지 보안 검사가 들어 있다:

[unlink_chunk 의 보안 검사]

  1. size 일관성:
     chunksize(p) == prev_size(다음 chunk)
     → boundary tag 위조 탐지
     
  2. linked list 일관성:
     fd->bk == p && bk->fd == p
     → list 손상 탐지 (House of Spirit 등 방어)
     
  3. (large bin 만) fd_nextsize/bk_nextsize 일관성

이 검사들이 — 옛 unlink attack (heap exploit 의 고전) 을 방지.

top chunk — 단편화의 끝

top chunk 가 — heap 의 가장 끝 (top) 에 있는 chunk. 미분할 영역. 매우 특별한 chunk.

[top chunk 의 특수성]

  malloc_state.top 이 가리킴
  
  특성:
    - bin 에 들어가지 않음
    - 항상 존재 (heap 의 끝)
    - 임의의 큰 크기 가능
    - 다른 chunk 와 인접 (메모리상 마지막)
    
  malloc 시 다른 모든 옵션 실패하면:
    → top chunk 의 일부를 split 해서 반환
    → top 의 크기 줄어듦
    
  top chunk 가 너무 작아지면:
    → sbrk 또는 mmap 으로 heap 확장
    → top 을 새 영역으로 이동

free 가 top 과 인접한 경우

흥미로운 경우 — free 되는 chunk 가 top 과 바로 인접.

[free chunk 가 top 과 인접]

  ┌───────────┬─────────┬──────────────┐
  │  chunk A  │  top    │   미매핑       │
  │ (in-use)  │ (big!)  │              │
  └───────────┴─────────┴──────────────┘
  
  free(A):
    A 와 top 이 인접 → 합침
    → top 이 더 커짐 (A 크기만큼)
    
  ┌─────────────────────┬──────────────┐
  │        top          │   미매핑       │
  │     (더 커짐)         │              │
  └─────────────────────┴──────────────┘

이게 — heap 의 자연스러운 정리. free 가 누적되면 top 이 커지면서 — 결국 brk 를 줄일 수 있는 상태가 됨.

M_TRIM_THRESHOLD — heap 축소

top chunk 가 충분히 커지면 — sbrk(-N) 으로 OS 에 메모리 반환.

// [user-space - glibc malloc/malloc.c systrim 단순화]
static int
systrim(size_t pad, mstate av)
{
    long top_size = chunksize(av->top);
    long extra = ((top_size - pad - MINSIZE + (pagesize - 1)) / pagesize - 1) * pagesize;
    
    if (extra < (long)pagesize)
        return 0;
    
    /* sbrk(-extra) 로 줄임 */
    char *current_brk = sbrk(0);
    if ((char *)av->top + top_size == current_brk) {
        /* heap 끝과 break 가 일치 → 줄일 수 있음 */
        if (sbrk(-extra) != (void *)-1) {
            set_head(av->top, (top_size - extra) | PREV_INUSE);
            return 1;
        }
    }
    
    return 0;
}

M_TRIM_THRESHOLD (기본 128 KB) 이상이면 — heap 자동 축소. 명시적으로 malloc_trim(0) 호출도 가능.

coalescing 의 효과 — 측정

coalescing 의 효과는 — 단편화 측정으로 확인 가능.

// [user-space - C 예제]
#include <stdio.h>
#include <stdlib.h>
#include <malloc.h>

int main(void) {
    const int N = 1000;
    void *ptrs[N];
    
    /* 1. 1000 개 alloc */
    for (int i = 0; i < N; i++)
        ptrs[i] = malloc(64);
    
    /* 2. 짝수 인덱스만 free → 짝수 인덱스가 빈 구멍 */
    for (int i = 0; i < N; i += 2)
        free(ptrs[i]);
    
    /* 단편화 측정 */
    struct mallinfo2 mi = mallinfo2();
    printf("After fragment: arena=%zu, free=%zu\n",
           mi.arena, mi.fordblks);
    printf("largest free block: %zu\n", mi.fsmblks);  /* ★ 가장 큰 free chunk */
    
    /* 3. 홀수 인덱스도 free → coalescing 발동 */
    for (int i = 1; i < N; i += 2)
        free(ptrs[i]);
    
    mi = mallinfo2();
    printf("After all free: arena=%zu, free=%zu\n",
           mi.arena, mi.fordblks);
    printf("largest free block: %zu\n", mi.fsmblks);
}

예상 출력:

After fragment:
  arena=70000, free=32000        ← 500 chunk free
  largest free block: ~64        ← 모두 64 byte 조각
  
After all free:
  arena=70000, free=64000
  largest free block: ~64000     ← coalesce 되어 하나의 큰 chunk!

free 의 순서 — 짝수 → 홀수 — 가 흥미롭다. 짝수만 free 일 때는 — 각 짝수 chunk 의 양옆이 in-use 라 합쳐지지 않음. 홀수까지 free 하면 — 그제야 줄줄이 합쳐짐.

unsorted bin 의 역할 재조명

26장에서 본 unsorted bin 이 — coalescing 의 임시 정리장.

[unsorted bin 과 coalescing]

  free 시:
    1. 인접 chunk 와 coalesce 시도
    2. 결과 chunk 를 unsorted bin 에 넣음
       (small/large bin 으로 직접 안 감)
       
  → unsorted bin 은 "방금 coalesce 된 chunk 들의 임시 보관소"

malloc 시 unsorted bin 처리 로직에서 — 각 chunk 가 적절한 small/large bin 으로 분배. 이게 — lazy classification.

fastbin/tcache 의 coalescing 우회

24장에서 본 fastbin/tcache 의 특수성 — coalescing 안 됨.

[fastbin/tcache 와 coalescing]

  fastbin/tcache chunk:
    PREV_INUSE 비트를 1 유지
    → 인접 chunk free 시에도 이 chunk 는 "in-use" 로 보임
    → coalesce 안 됨
    → 속도 우선 (alloc/free 의 fast path)
    
  단점: 단편화 누적 가능
  해결: malloc_consolidate 가 가끔 정리

malloc_consolidate 가 — fastbin 의 모든 chunk 를 진짜로 free 화하면서 coalesce. unsorted bin 으로 옮김.

internal vs external 단편화의 trade-off

malloc 디자인의 큰 trade-off.

[chunk 크기 분류의 trade-off]

  더 큰 chunk 단위 사용 (예: 32, 64, 128, ...):
    + 빠른 lookup (적은 size class)
    + bin 개수 적음
    - 내부 단편화 증가
    - 작은 할당이 큰 chunk 차지
    
  더 작은 chunk 단위 사용 (예: 16 byte 단위):
    + 내부 단편화 적음
    - bin 개수 많음
    - lookup 시간 증가
    
  glibc 의 선택:
    smallbin 은 16 byte 단위 (62 bin)
    largebin 은 로그 스케일 (63 bin)
    fastbin 은 16 byte 단위 (10 bin, max 80~160)
    tcache 는 16 byte 단위 (64~76 bin)

이 장의 결론

coalescing 의 본질:

[coalescing 정리]

  ✓ 외부 단편화의 핵심 대응
  ✓ free 시 즉시 인접 chunk 와 합침 (eager)
  ✓ boundary tag (PREV_INUSE, prev_size) 활용
  ✓ unlink_chunk 의 3 보안 검사 (linked list 일관성)
  ✓ unsorted bin = coalesce 된 chunk 의 임시 보관
  ✓ top chunk 와 인접하면 top 에 흡수
  ✓ fastbin/tcache 는 coalesce 안 함 (속도 우선)
  ✓ malloc_consolidate 가 가끔 fastbin 정리
  ✓ M_TRIM_THRESHOLD 로 heap 축소

다음 장에서는 — MMAP_THRESHOLD 이상의 큰 할당의 별도 경로를 본다. 작은 chunk 의 세계와는 완전히 다른 메커니즘.

30mmap_threshold — 큰 할당의 별도 길

지금까지 본 모든 메커니즘 (arena, bin, fastbin, tcache, coalescing) 은 — 작은 chunk 의 세계. 큰 할당은 완전히 다른 길을 간다. 23장에서 잠깐 본 MMAP_THRESHOLD 의 정확한 동작을 본다.

큰 할당의 별도 경로

[큰 할당이 별도 경로를 가는 이유]

  작은 chunk:
    arena 안에서 잘게 나눠 씀
    coalescing 으로 단편화 대응
    free 해도 OS 에 안 돌려줌 (다음 alloc 위해 캐싱)
    
  큰 chunk (≥ MMAP_THRESHOLD):
    arena 와 무관, 별도 mmap
    free 시 즉시 munmap → OS 에 즉시 반환
    단편화 없음 (각자 격리)
    chunk header 가 다름

이 분리 디자인이 — 큰 할당의 메모리를 정확히 관리하고, 작은 할당의 빠른 경로를 보호.

sysmalloc_mmap — 큰 chunk 만들기

master 의 mmap chunk 생성:

// [user-space - glibc malloc/malloc.c sysmalloc_mmap 단순화]
static void *
sysmalloc_mmap(INTERNAL_SIZE_T nb, size_t pagesize, int extra_flags, mstate av)
{
    /* 페이지 align 한 크기 */
    size = ALIGN_UP(nb + SIZE_SZ, pagesize);
    
    /* mmap 호출 */
    char *mm = (char *)MMAP(NULL, size, mtag_mmap_flags | PROT_READ | PROT_WRITE,
                            extra_flags);
    if (mm == MAP_FAILED)
        return mm;
    
    sysmadvise_thp(mm, size);
    
    /* chunk 로 설정 */
    mchunkptr p = (mchunkptr)mm;
    
    /* 정렬 보정 (memalign 등의 경우) */
    INTERNAL_SIZE_T front_misalign = (INTERNAL_SIZE_T)chunk2mem(mm) & MALLOC_ALIGN_MASK;
    if (front_misalign > 0) {
        INTERNAL_SIZE_T correction = MALLOC_ALIGNMENT - front_misalign;
        p = (mchunkptr)(mm + correction);
        set_prev_size(p, correction);
        set_head(p, (size - correction) | IS_MMAPPED);
    } else {
        set_prev_size(p, 0);
        set_head(p, size | IS_MMAPPED);
    }
    
    /* 통계 갱신 */
    ++mp_.n_mmaps;
    sum = mp_.mmapped_mem += size;
    if (sum > (unsigned long)mp_.max_mmapped_mem)
        mp_.max_mmapped_mem = sum;
    
    check_chunk(av, p);
    
    return chunk2mem(p);
}

핵심:

[mmap chunk 의 특별한 점]

  1. arena 와 무관 — av 인자는 통계용
  2. IS_MMAPPED 비트 설정 (size 의 0x2)
  3. mchunk_prev_size 의 의미가 다름:
     일반 chunk: 이전 chunk 의 크기 (or 사용자 데이터로 재활용)
     mmap chunk: ★ mmap 영역의 _시작 offset_
  4. fd, bk 같은 link 없음 (어떤 bin 에도 안 들어감)
  5. coalescing 없음 (격리됨)

mmap chunk 의 layout

┌────────────────────────────────────────────────────────┐
│ mmap chunk 의 메모리 layout                                 │
├────────────────────────────────────────────────────────┤
│                                                         │
│  mmap 영역 시작 → ┌──────────────────────────┐              │
│  ┌─────────────│  (정렬 보정 padding)         │              │
│  │ front_misalign│  (memalign 의 경우)        │              │
│  │             ├──────────────────────────┤              │
│  │  prev_size →│ mchunk_prev_size = offset  │ ← ★ 특별!      │
│  │             │ (mmap 시작부터 chunk 까지)    │              │
│  │             ├──────────────────────────┤              │
│  │             │ mchunk_size | IS_MMAPPED  │              │
│  mem(반환) →   ├──────────────────────────┤              │
│                │                          │              │
│                │   사용자 데이터              │              │
│                │   (요청한 N bytes)         │              │
│                │   (실제로는 페이지 단위 반올림)  │              │
│                │                          │              │

│  mmap 영역 끝 →  └──────────────────────────┘              │
│                                                         │

│   - 다른 chunk 와 인접하지 않음 (격리됨)                      │

│   - top, fd, bk 없음                                    │

prev_size 의 의미가 다른 점이 묘하다. memalign 등에서 정렬 보정이 들어가면 — chunk 의 시작이 mmap 영역의 시작과 다를 수 있음. prev_size 가 그 offset 을 기록.

[front_misalign 의 경우]

  memalign(16384, 1000) 호출:
    16 KB 정렬된 1000 byte 영역 필요
    
  mmap 으로 큰 영역 받은 후:
    mmap 시작이 16 KB 정렬 안 됐을 수 있음
    → 16 KB 정렬되는 위치까지 skip
    → 그 offset 을 prev_size 에 기록
    
  free 시:
    실제 munmap 시작 = chunk - prev_size
    munmap 크기 = size + prev_size

_int_free 의 mmap chunk 분기

free(p) 의 흐름:

// [user-space - glibc malloc/malloc.c __libc_free 단순화]
void
__libc_free(void *mem)
{
    mchunkptr p;
    
    if (mem == NULL)
        return;
    
    p = mem2chunk(mem);
    
    /* ★ mmap chunk 인지 먼저 검사 */
    if (chunk_is_mmapped(p)) {
        /* 동적 threshold 조정 */
        if (!mp_.no_dyn_threshold
            && chunksize_nomask(p) > mp_.mmap_threshold
            && chunksize_nomask(p) <= DEFAULT_MMAP_THRESHOLD_MAX) {

munmap_chunk — OS 에 즉시 반환

// [user-space - glibc malloc/malloc.c munmap_chunk 단순화]
static void
munmap_chunk(mchunkptr p)
{
    size_t pagesize = GLRO(dl_pagesize);
    INTERNAL_SIZE_T size = chunksize(p);
    
    assert(chunk_is_mmapped(p));
    
    /* size 검증 */
    uintptr_t mem = (uintptr_t)chunk2mem(p);
    uintptr_t block = (uintptr_t)p - prev_size(p);
    size_t total_size = prev_size(p) + size;
    
    /* 페이지 정렬 검증 */
    if (__glibc_unlikely((block | total_size) & (pagesize - 1)) != 0
        || __glibc_unlikely(!powerof2(mem & (pagesize - 1))))
        malloc_printerr("munmap_chunk(): invalid pointer");
    
    /* 통계 갱신 */
    atomic_decrement(&mp_.n_mmaps);
    atomic_fetch_add_relaxed(&mp_.mmapped_mem, -total_size);
    
    /* 실제 munmap */
    __munmap((char *)block, total_size);
}

핵심 동작:

[munmap_chunk 의 동작]

  block = p - prev_size              ← mmap 영역의 실제 시작
  total_size = prev_size + size      ← 매핑된 총 크기
  
  munmap(block, total_size)          ← OS 에 즉시 반환!
  
  → 가상 주소 공간에서 해당 영역 unmap
  → kernel 이 페이지를 회수
  → 즉시 RSS (Resident Set Size) 감소

일반 chunk vs mmap chunk 의 free 비교

[free 의 두 경로 비교]

  일반 chunk free:
    1. arena 의 lock 잡음
    2. coalesce 시도 (인접 free 합치기)
    3. unsorted bin 에 추가
    4. → 같은 프로세스에 메모리 보유 (다음 alloc 위해)
    
  mmap chunk free:
    1. lock 안 필요 (arena 무관)
    2. coalesce 안 함
    3. bin 에 안 들어감
    4. → munmap 으로 즉시 OS 반환
       → RSS 즉시 감소

**free 의 동작이 — chunk 종류에 따라 매우 다르다**. mmap chunk 의 free 가 — 더 비싸지만 메모리 즉시 반환.

malloc 의 단순한 디자인 — chunk 의 header 가 사용자 영역 옆에 있다 — 은 공격 표면이 된다. 사용자가 (의도적 또는 실수로) chunk header 를 손상시키면 — heap 전체가 망가질 수 있다. 그래서 glibc malloc 은 — 곳곳에 무결성 검사 가 박혀 있다.

/* tcache_double_free_verify — glibc 2.42 (2025.5 patch) 의 더 엄격한 검사 */
static __attribute__((noinline)) void
tcache_double_free_verify(tcache_entry *e)
{
    tcache_entry *tmp;
    
    /* ★ 모든 tcache bin 을 _전체_ 순회 (옛: 한 bin 만)
       이유: 인접 chunk overflow 가 size 를 바꿔 검사 우회 가능 → 전체 검색으로 강화 */
    for (size_t tc_idx = 0; tc_idx < TCACHE_MAX_BINS; ++tc_idx) {
        size_t cnt = 0;
        LIBC_PROBE(memory_tcache_double_free, 2, e, tc_idx);
        for (tmp = tcache->entries[tc_idx];
             tmp;
             tmp = REVEAL_PTR(tmp->next), ++cnt) {
            if (cnt >= mp_.tcache_count)
                malloc_printerr("free(): too many chunks detected in tcache");
            if (__glibc_unlikely(!aligned_OK(tmp)))
                malloc_printerr("free(): unaligned chunk detected in tcache 2");
            if (tmp == e)
                malloc_printerr("free(): double free detected in tcache 2");
        }
    }
}

검사 흐름:

[tcache double-free 탐지 흐름]

  1. tcache 에 넣을 chunk 의 e->key 확인
     이미 tcache_key 값이면 의심!
     (tcache_put 이 e->key = tcache_key 로 설정함)
  
  2. tcache_double_free_verify 호출 (glibc 2.42+):
     ★ 모든 tcache bin 을 전체 순회 (옛: 한 bin 만 검사)
     같은 chunk 가 list 에 이미 있나?
     
  3. 있으면 → "double free detected in tcache 2"

e->key 가 — tcache 에 있는지의 빠른 marker. 99% 정상 케이스는 이 체크 한 번에 통과.

2: fastbin 의 double-free 탐지

fastbin 의 head 가 같은지 확인.

// [user-space - glibc malloc/malloc.c _int_free fastbin 분기 단순화]
mchunkptr old = *fb;

if (SINGLE_THREAD_P) {
    /* Check that the top of the bin is not the record we are going to add
       (i.e., double free). */
    if (__builtin_expect(old == p, 0))
        malloc_printerr("double free or corruption (fasttop)");
    p->fd = old;

} else {
    /* 다중 thread: atomic CAS */
    do {
        if (__builtin_expect(old == p, 0))

  검사 내용: 직전에 free 된 chunk 와 같은가

    free(p);            ← 즉시 탐지
    
  탐지 안 됨:
    free(p);
    free(q);
    free(p);            ← q 를 사이에 끼우면 우회 가능 (옛 공격 기법)
    
  → tcache 의 전체 list 검사보다 약함
  → 하지만 매우 빠름 (한 비교)

3: unlink 의 linked list 일관성 검사

29장에서 본 unlink_chunk 의 보안 검사 — 가장 중요.

// [user-space - glibc malloc/malloc.c unlink_chunk 인용]
static void
unlink_chunk(mstate av, mchunkptr p)
{
    if (chunksize(p) != prev_size(next_chunk(p)))
        malloc_printerr("corrupted size vs. prev_size");
    
    mchunkptr fd = p->fd;
    mchunkptr bk = p->bk;
    
    if (__glibc_unlikely(fd->bk != p || bk->fd != p))
        malloc_printerr("corrupted double-linked list");
    
    fd->bk = bk;
    bk->fd = fd;
    
    if (!in_smallbin_range(chunksize_nomask(p)) && p->fd_nextsize != NULL) {
        if (p->fd_nextsize->bk_nextsize != p
            || p->bk_nextsize->fd_nextsize != p)
            malloc_printerr("corrupted double-linked list (not small)");
        ...
    }
}

각 검사의 의미:

[unlink_chunk 의 3 단계 검사]

  1. size 일관성:
     chunksize(p) == prev_size(다음 chunk)
     → boundary tag 가 위조되지 않았나
     → off-by-one overflow 등 탐지
     
  2. forward/backward 일관성:
     fd->bk == p && bk->fd == p
     → linked list 가 망가지지 않았나
     → 옛 unlink attack (House of Spirit 등) 방어
     
  3. large bin 의 nextsize 일관성:
     fd_nextsize->bk_nextsize == p && bk_nextsize->fd_nextsize == p
     → large bin 의 추가 link 검증

이 검사들의 역사 — 2003 년 처음 도입. 그 전엔 unlink attack 이 매우 강력했음.

    INTERNAL_SIZE_T size = chunksize(p);
    
    /* 검사 1: chunk 가 적절한 경계에 있나 */
    if (__builtin_expect((uintptr_t)p > (uintptr_t)-size, 0)
        || __builtin_expect(misaligned_chunk(p), 0))
        malloc_printerr("free(): invalid pointer");
    
    /* 검사 2: chunk size 가 너무 작거나 정렬 안 됐나 */
    if (__glibc_unlikely(size < MINSIZE || !aligned_OK(size)))
        malloc_printerr("free(): invalid size");
    
    /* 검사 3: 다음 chunk 의 size 가 정상 범위인가 */
    if (__glibc_unlikely(chunksize_nomask(chunk_at_offset(p, size)) <= 2 * SIZE_SZ)
        || __glibc_unlikely(chunksize(chunk_at_offset(p, size)) >= av->system_mem))
        malloc_printerr("free(): invalid next size (normal)");
    
    /* (fastbin 으로 가는 경우 별도 검사) */
    
    /* 검사 4: 일반 free 의 경우 — 다음 chunk 의 PREV_INUSE 비트가 1 인가
       (1 이면 우리는 in-use 상태였음. 0 이면 이미 free 였다는 뜻 → double free) */
    if (__glibc_unlikely(!prev_inuse(nextchunk)))
        malloc_printerr("double free or corruption (!prev)");
    
    /* 검사 5: 다음 chunk 의 size 가 너무 작거나 큰가 */
    if (__glibc_unlikely(nextsize < MINSIZE || nextsize >= av->system_mem))
        malloc_printerr("free(): invalid next size (normal)");
    
    /* 검사 6: 이전 chunk 와 합치기 전, 이전 size 검사 */
    if (__glibc_unlikely(chunksize(p) != prevsize))
        malloc_printerr("corrupted size vs. prev_size while consolidating");
    
    ...
}

이 모든 검사가 — 거의 비용 없음 (CPU 입장에서). compare + jump 2~3 개. 그러나 — heap corruption 탐지엔 매우 효과적.

            /* sbrk 로 확장 */
            ...

    }
}

또한 top chunk 의 size 검증을 _이전_ 단계에서도:

8: safe-linking — fastbin/tcache 의 fd mangling

27, 28장에서 본 safe-linking (glibc 2.32+).

#define PROTECT_PTR(pos, ptr) \
    ((__typeof(ptr))((((size_t)pos) >> 12) ^ ((size_t)ptr)))

#define REVEAL_PTR(ptr) PROTECT_PTR(&ptr, ptr)
[safe-linking 의 보안 효과]

  공격자가 fastbin/tcache 의 fd 를 임의 주소로 덮어쓰려면:
    1. ASLR 의 mmap_base 알아야 함 (어렵게)
    2. 정확한 위치 (pos) 알아야 함
    3. XOR 한 결과가 16-byte align 되어야 통과
    
  → fastbin attack / tcache poisoning 의 난이도 크게 상승
  
  glibc 2.32 (2020.8) 이후 — 옛 exploit 들 일부 무력화

디버그 도구 2: MALLOC_PERTURB_

free/malloc 시 메모리를 특정 패턴으로 채워서 — uninitialized read 와 use-after-free 를 탐지하기 쉽게.

# [user-space - bash]
$ MALLOC_PERTURB_=85 ./myapp

85 가 byte 값 (0~255).

[MALLOC_PERTURB_ 의 동작]

  85 (= 0x55) 설정 시:
  
  free 호출:
    chunk 의 모든 byte 를 0x55 로 채움
    → use-after-free 시 0x55 가 나오면 의심
    
  malloc 호출:
    chunk 의 모든 byte 를 ~0x55 = 0xAA 로 채움
    → uninitialized read 시 0xAA 가 나오면 의심
    
  → 0x55, 0xAA 같은 패턴은 사용자 데이터에 거의 없음
  → 디버깅 단서

디버그 도구 4: mcheck() / mprobe()

heap 무결성을 명시적으로 확인.

// [user-space - C API]
#include <mcheck.h>
int mcheck(void (*abortfunc)(enum mcheck_status status));
int mcheck_pedantic(void (*abortfunc)(enum mcheck_status status));
enum mcheck_status mprobe(void *ptr);

/* mcheck_status: */
MCHECK_DISABLED   /* mcheck not active */
MCHECK_OK         /* No inconsistency detected */
MCHECK_HEAD       /* Memory preceding an allocated block was clobbered */
MCHECK_TAIL       /* Memory following an allocated block was clobbered */
MCHECK_FREE       /* Block freed twice */
// [user-space - C 예제]
#include <stdio.h>
#include <stdlib.h>
#include <mcheck.h>

void check_status(enum mcheck_status status) {
    fprintf(stderr, "Heap corruption: ");
    switch (status) {
        case MCHECK_HEAD: fprintf(stderr, "head overwrite\n"); break;
        case MCHECK_TAIL: fprintf(stderr, "tail overwrite\n"); break;
        case MCHECK_FREE: fprintf(stderr, "double free\n"); break;
        default: fprintf(stderr, "other\n"); break;
    }
    abort();
}

int main(void) {
    mcheck(check_status);
    
    char *p = malloc(10);
    p[10] = 'X';                    /* overflow! */
    
    if (mprobe(p) != MCHECK_OK)
        printf("corruption detected\n");
    
    return 0;
}

mcheck 가 — 각 chunk 양옆에 sentinel byte 추가. mprobe 가 그 sentinel 확인.

[heap 디버깅 외부 도구]

  Valgrind / memcheck:

비교 측면 1: 데이터 구조의 기본 단위

각자가 사용하는 "기본 단위" 가 다르다.

[기본 단위의 비교]

  ptmalloc2 (glibc):
    chunk (가변 크기, boundary tag)
    bin (chunk 들의 list)
    arena (chunk + bin + lock)
    
  mallocng (musl):
    group (같은 size class 의 slot 들)
    meta (group 의 메타데이터, 분리됨)
    in-band 메타 없음 (보안!)
    
  jemalloc:
    extent (큰 영역)
    slab (small size 의 영역)
    bin (slab 들의 list)
    arena (bin 들의 모음)
    
  tcmalloc:
    span (페이지의 모음)
    central free list (size class 별)
    per-thread (또는 per-CPU) cache
    
  mimalloc:
    page (4 KB, 같은 size class 의 block 모음)
    segment (큰 영역, 페이지들의 모음)
    heap (per-thread)

핵심 차이 — 메타데이터의 위치.

[메타데이터 위치 비교]

  ptmalloc2:
    in-band (chunk 안에 prev_size, size, fd, bk 등)
    → 빠르지만 공격에 취약 (chunk header 손상 가능)
    
  mallocng:
    핵심 메타데이터는 out-of-band (meta object)
    + group header (in-band, 가벼움)
    → 사용자 영역 overflow 가 핵심 메타에 영향 못 줌
    → secret cookie 로 무결성 검증
    
  jemalloc, tcmalloc, mimalloc:
    혼합 (small 은 separate metadata, large 는 mmap)

mallocng 가 — 핵심 메타데이터 분리를 강하게 밀어붙인 디자인. heap-based exploit 의 거의 모든 기법 무력화.

비교 측면 2: thread 경쟁 처리

multi-thread 환경에서의 lock 경쟁 최소화 방법.

[multi-thread 처리 비교]

  ptmalloc2:
    - 여러 arena (8 × NCPUS)
    - thread 마다 TLS 로 arena 지정
    - 추가: tcache (per-thread, lock-free)
    - lock 경쟁: arena 부족 시 발생
    
  mallocng:
    - 모든 thread 가 공유 (global lock)
    - 단순하지만 lock 경쟁 가능
    - musl 의 철학: "단순함 > 성능"
    
  jemalloc:
    - per-thread arena (자동 할당)
    - thread 가 사용 빈도에 따라 arena 변경
    - background thread 로 purging
    - 매우 효율적
    
  tcmalloc:
    - per-thread cache (소량) + central free list
    - 또는 per-CPU mode (최근)
    - rseq (restartable sequences) 활용
    - Google scale 에서 최적화
    
  mimalloc:
    - per-thread heap
    - 다른 thread 가 free 하면 thread-safe 큐로 전달
    - 매우 적은 atomic 연산

비교 측면 3: 단편화 처리

외부 단편화에 대한 각자의 접근.

[단편화 처리 비교]

  ptmalloc2:
    - free 시 eager coalescing
    - fastbin/tcache 는 coalesce 안 함
    - malloc_consolidate 가 가끔 정리
    - top chunk 의 trim
    
  mallocng:
    - 고정 size class
    - 같은 group 안에서만 재사용
    - free group 은 즉시 munmap
    - 외부 단편화 거의 없음
    
  jemalloc:
    - size class 기반 (단편화 통계적)
    - extent 의 coalesce/split
    - background purge (decay-based)
    
  tcmalloc:
    - size class 기반 (170 개 정도)
    - span level 의 coalesce
    - heap profile 기반 정리
    
  mimalloc:
    - page 단위 (4 KB)
    - 같은 page 안에서 free list 재사용
    - empty page 는 segment 로 반환

size class 기반 vs boundary tag 기반 의 큰 차이:

[size class vs boundary tag]

  size class (jemalloc, tcmalloc, mimalloc, mallocng):
    + 단편화 적음 (고정 크기끼리만 모임)
    + 빠름 (분류 비용 적음)
    + 메타데이터 적음
    - 정확한 크기 못 맞춤 (내부 단편화 약간 증가)
    
  boundary tag (ptmalloc2):
    + 유연 (임의 크기)
    + 내부 단편화 적음
    - 외부 단편화 위험
    - 메타데이터 많음 (각 chunk 마다)
    - 공격 표면 큼

비교 측면 4: 보안 디자인

각자의 보안 철학.

[보안 비교]

  ptmalloc2 (가장 약함):
    - in-band 메타데이터 → overflow 로 공격
    - 검사들 점진적 추가 (역사적 누적)
    - safe-linking (2.32+) — 큰 진전
    - 단순함이 공격 표면
    
  mallocng (가장 강함):
    - out-of-band 메타데이터
    - 모든 size class 격리
    - guard page 사이 (option)
    - OpenBSD 영감 (보안 우선)
    
  jemalloc:
    - in-band + out-of-band 혼합
    - canary 옵션 (--enable-fill)
    - heap profile 로 leak 탐지
    - 보안보다 성능 중시
    
  tcmalloc:
    - in-band 메타데이터 위주
    - GWP-ASan 통합 (random sampling)
    - production 의 random 검사
    
  mimalloc:
    - secure mode (--with-secure)
    - free list 의 encrypted pointer (XOR key)
    - guard page (secure mode)
    - 일반 mode 는 성능 중심

이 측면에서 — mallocng 와 mimalloc-secure 가 가장 강함. 둘 다 보안을 명시적 디자인 목표.

비교 측면 5: 성능 특성

벤치마크에서의 일반적 경향.

[성능 경향 (벤치마크 기준)]

  단일 thread, 작은 할당 (예: cfrac):
    mimalloc ≥ tcmalloc ≥ jemalloc > glibc > musl
    
  multi-thread, 균일 할당:
    mimalloc ≈ tcmalloc > jemalloc > glibc > musl
    
  large server workload:
    mimalloc ≥ jemalloc ≈ tcmalloc > glibc > musl
    (worst case latency 측면에서)
    
  메모리 사용량 (RSS):
    mimalloc ≈ mallocng ≤ jemalloc ≤ tcmalloc < glibc

mimalloc 이 대체로 우수한 경향. 다만 — 모든 workload 에서 절대 우위는 없음. workload 특성에 따라 다름.

musl mallocng 의 상세

가장 신선한 디자인이라 좀 더 자세히.

[mallocng 의 디자인]

  Rich Felker (musl 의 저자) 가 musl 1.2.1 (2020.8.4) 에 도입
  
  핵심 자료구조:
    group: 같은 size class 의 1~32 slot 모음
      - group header (in-band): slot 의 status 비트맵 등
      - slot 들은 사용자 데이터 (헤더 없음)
      - 48 size class (작은 것 linear, 큰 것 geometric)
      
    meta: group 의 _별도_ 메타데이터 (out-of-band)
      - meta_area 페이지에 따로 저장 (브 끝의 guard page 다음)
      - group 의 위치, slot stride, size class 등
      - 사용자 데이터와 분리됨
      - secret cookie 로 무결성 검증
      
  free 시:
    1. 사용자 pointer → meta 찾기 (역 매핑)
    2. secret cookie 검증
    3. meta 의 비트맵에서 해당 slot 해제
    4. group 의 모든 slot 비어있으면 → munmap
    
  malloc 시:
    1. size class 결정 (48 개 중 하나)
    2. 그 class 의 group 중 빈 slot 찾기 (비트맵)
    3. 없으면 새 group 만들기 (mmap)
    4. MMAP_THRESHOLD (~128 KB) 초과는 single-slot group

이 디자인의 강점:

[mallocng 의 보안 강점]

  ✓ 핵심 메타데이터 (meta) 가 사용자 영역과 분리
    → buffer overflow 로 핵심 메타데이터 손상 불가
    → unlink attack, fastbin attack 등 _모두_ 무력
    → glibc 의 in-band chunk header 와 대비
    
  ✓ secret cookie 로 무결성 검증
    → meta_area 의 모든 access 시 cookie 검사
    → forge 한 meta 사용 불가
    
  ✓ size class 격리 (48개)
    → 다른 크기로 type confusion 안 됨
    → free 시 사용자 ptr 의 size 검증
    
  ✓ slot 의 정확한 크기 (stride)
    → off-by-one overflow 도 인접 slot 만 영향
    → group header 도 reserved=6 같은 invalid 값으로 표시 가능 (forge 무력)
    
  → glibc 의 heap exploit 기법 대부분 안 통함
  → CVE-2022-24834 (Redis Lua) 도 mallocng 에선 제한적

jemalloc 의 특징

Firefox, FreeBSD, MongoDB 에서 사용.

[jemalloc 의 특징]

  ✓ small/large/huge 의 3 단계 size class
    - small: < 4 KB (slab)
    - large: 4 KB ~ 4 MB (extent)
    - huge: ≥ 4 MB (mmap)
    
  ✓ background thread 의 decay-based purging
    - free 후 일정 시간 보관 (dirty)
    - 안 쓰면 muzzy 상태
    - 더 오래되면 OS 반환
    → "lazy munmap" 같은 효과
    
  ✓ thread-cached small allocations
    - tcache (glibc 의 그 이름이 여기서 유래!)
    
  ✓ profiling
    - jeprof 도구로 heap profile 분석
    - massive 서비스 디버깅 가능
    
  ✓ MALLOC_CONF 환경 변수
    - tunable 매우 많음
    - run-time 조정

Mozilla Firefox 가 — jemalloc 으로 표준 glibc 보다 메모리 사용량 대폭 절감했다는 유명한 케이스.

tcmalloc 의 특징

Google 내부에서 시작. Chrome 의 기본.

[tcmalloc 의 특징]

  ✓ per-thread cache (이름의 유래)
    - 작은 할당은 thread cache 에서
    - lock 없음
    
  ✓ central free list (size class 별)
    - thread cache miss 시 여기서
    - 큰 lock 한 번만
    
  ✓ page heap
    - 페이지 단위로 OS 와 소통
    - span 의 split/coalesce
    
  ✓ per-CPU mode (최신)
    - rseq syscall 활용
    - per-thread 보다 더 효율적
    - Linux 4.18+
    
  ✓ HEAPPROFILE 도구
    - 매우 정밀한 heap profile
    - Google 의 빅 데이터 디버깅 도구
    
  ✓ Google scale 검증
    - search, GMail 등 운영
    - 매우 안정적

흥미로운 점 — **tcache 라는 이름**이 glibc 에 도입된 게 — 사실 tcmalloc 의 "thread cache" 에서 영감. 2017 년 glibc 2.26 이 — tcmalloc 의 핵심 아이디어를 받아들였다.

mimalloc 의 특징

Microsoft Research 가 개발. 2019 년 공개.

[mimalloc 의 특징]

  ✓ 매우 단순한 디자인
    - 코어 ~10K LOC (glibc malloc 의 1/3)
    - 이해하기 쉬움
    - 안정적
    
  ✓ free list multi-sharding
    - per-page free list (local)
    - per-page free list (delayed, atomic)
    - per-thread heap
    - 매우 적은 atomic
    
  ✓ "secure" mode (별도 빌드)
    - encrypted free list (XOR key)
    - guard page 사이
    - randomized allocation
    - ~10% 성능 페널티
    
  ✓ huge OS page 지원
    - 2 MB / 1 GB 페이지
    - TLB miss 감소
    
  ✓ 일관된 성능
    - 다양한 workload 에서 우수
    - 최악의 경우도 좋음
    
  ✓ Windows 친화적 (개발 환경)
    - Microsoft 가 후원
    - .NET runtime 에 통합

mimalloc 의 디자인 철학 — "단순함이 곧 성능". 복잡한 휴리스틱 없이도 우수한 성능 가능을 증명.

응용에서 선택하기

각 응용의 특성에 따라 선택.

[응용별 추천 가이드]

  일반 Linux 응용:
    → glibc ptmalloc2 (기본, 호환성 우수)
    
  보안 sensitive (예: 웹 서버, sandbox):
    → musl mallocng (Alpine) 또는 mimalloc-secure
    
  웹 브라우저, DB:
    → jemalloc (Firefox, MongoDB) 또는 mimalloc
    
  Google scale 서버:
    → tcmalloc
    
  multi-thread 고성능 응용:
    → mimalloc 또는 tcmalloc
    
  embedded / small footprint:
    → musl mallocng (적은 코드 크기)
    
  Windows 서버:
    → mimalloc

동적으로 malloc 교체 — LD_PRELOAD

응용을 다시 빌드 안 하고도 — malloc 을 교체 가능.

# [user-space - bash]
# jemalloc 으로 교체
$ LD_PRELOAD=/usr/lib/libjemalloc.so.2 ./myapp

# tcmalloc 으로
$ LD_PRELOAD=/usr/lib/libtcmalloc.so ./myapp

# mimalloc 으로
$ LD_PRELOAD=/usr/lib/libmimalloc.so ./myapp

# 효과 확인
$ /usr/bin/time -v ./myapp                              # 기본
$ /usr/bin/time -v env LD_PRELOAD=libjemalloc.so.2 ./myapp
$ /usr/bin/time -v env LD_PRELOAD=libmimalloc.so ./myapp

LD_PRELOAD 가 — 실행 시점에 그 라이브러리의 malloc/free 가 우선 link. 응용 코드 변경 없이 malloc 만 교체.

정적으로 link — -l 옵션

영구적으로 다른 malloc 을 link 하려면.

# [user-space - bash, compile]
# jemalloc 으로 빌드
$ gcc -o myapp main.c -ljemalloc

# 또는 직접 mimalloc 통합
$ gcc -o myapp main.c -lmimalloc

런타임에 다른 malloc 도 시도 가능 — 단 호환성 주의 (특정 라이브러리는 특정 malloc 가정).

성능 측정의 함정

malloc 의 벤치마크는 — 매우 까다롭다.

[벤치마크의 함정]

  ✗ 단일 thread benchmark 만 보고 판단
    → 실제 응용은 multi-thread
    
  ✗ 짧은 benchmark
    → fragmentation 은 시간 지나야 누적
    
  ✗ 동일한 패턴 반복
    → 실 응용의 다양한 패턴 모방 안 됨
    
  ✗ benchmark 자체가 malloc 의 영향 받음
    → 측정 오버헤드
    
  → 실제 응용으로 측정해야
  → mimalloc 의 bench 결과는 reference 일 뿐

내 응용에 어떤 malloc 이 좋은지는 — 실제 응용을 다양한 malloc 으로 돌려서 비교 외엔 답 없음.

mallinfo2 와 비교 — 구현체마다 다른 API

각 구현체가 — 통계 API 도 다르다.

[구현체별 통계 API]

  glibc:
    mallinfo() / mallinfo2()
    malloc_stats() — stderr 출력
    malloc_info() — XML 출력
    
  jemalloc:
    je_malloc_stats_print() — stderr
    jemalloc.stats_print MALLCTL 명령
    
  tcmalloc:
    MallocExtension::GetStats()
    HEAPPROFILE 환경변수 → /tmp/profile.<pid>.<n>.heap
    
  mimalloc:
    mi_stats_print() — stderr
    MIMALLOC_SHOW_STATS=1 환경변수
    
  mallocng:
    제한적 (단순함 추구)

각자 자기 API 가 있어서 — portable 한 코드 작성 시 주의.

Hyrum's Law — malloc 의 호환성

흥미로운 사실 — malloc 은 표준 함수이지만, 응용들은 종종 구현 세부 사항에 의존.

[malloc 의 Hyrum's Law]

  예 1: realloc 의 in-place 확장
    glibc 는 가끔 in-place, 가끔 copy
    응용이 "항상 copy" 가정하면 안 됨
    
  예 2: malloc 의 정렬 보장
    표준은 max_align_t 정렬 보장
    실제는 16 byte 정렬 (현대 64-bit)
    SSE 응용이 16 byte 정렬에 의존
    
  예 3: free 후의 메모리 패턴
    응용이 use-after-free 후의 값 의존하면 ←... (버그!)
    MALLOC_PERTURB_ 으로 패턴 바뀌면 깨짐
    
  예 4: arena 의 개수
    응용이 특정 thread count 에서 특정 메모리 사용량 가정
    MALLOC_ARENA_MAX 변경하면 깨짐

이런 의존 — 표준이 보장하지 않는 동작에 의존한 응용 — 이 malloc 교체 시 깨지는 이유.

Part IV 의 결론

이 장으로 — Part IV (malloc 의 깊이) 가 마무리.

[Part IV 마무리 — malloc 의 모든 것]

  ✓ 23장: heap 의 정체 — brk 와 mmap
  ✓ 24장: malloc_chunk — 단위
  ✓ 25장: arena — multi-thread 분리
  ✓ 26장: bin 의 종류 (5 가지)
  ✓ 27장: fastbin — 빠른 single-linked LIFO
  ✓ 28장: tcache — lock-free per-thread cache
  ✓ 29장: coalescing — 단편화와의 싸움
  ✓ 30장: mmap_threshold — 큰 할당의 별도 길
  ✓ 31장: free 의 안전 장치
  ✓ 32장: malloc 구현체 비교

malloc(N) 한 줄 호출이 — 이 모든 메커니즘을 거친다. 작은 수십 byte 의 할당에서도 — tcache, fastbin, smallbin, unsorted bin, coalescing, safe-linking 등의 세계가 동작.

이 깊이를 이해하면 — 응용의 메모리 동작이 보인다. 메모리 누수, 성능 병목, 보안 취약점이 모두 — 이 메커니즘의 어디선가 발생.
