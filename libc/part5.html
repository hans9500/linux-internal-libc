Part Vstdio 의 깊이 — 입출력의 정체

Part IV 에서 — malloc(N) 한 줄 호출의 깊이를 봤다. 비슷하게 — **printf("hello\n")** 한 줄도 사실 매우 정교한 메커니즘을 거친다.

// [user-space - C]
printf("hello %d\n", 42);

이 한 줄의 안에는:

[printf 한 줄의 일생]

  printf 호출
       │
       ▼
  format string "hello %d\n" 파싱
       │
       ▼
  va_list 에서 인자 (42) 꺼냄
       │
       ▼
  "hello 42\n" 문자열 구성
       │
       ▼

       ▼

       ▼

       │
       ▼
  kernel 의 file descriptor 처리
       │
       ▼

이 모든 메커니즘이 — printf 한 줄 뒤에 숨어있다.

33FILE 구조체 — opaque pointer 의 정체

C 의 stdio API 는 — 사용자에게 FILE * 를 준다. 사용자는 그 포인터를 받고, fread, fwrite, fclose 등에 넘긴다. 그런데 FILE 자체가 무엇인지는 — opaque (불투명). 사용자는 모른다.

// [user-space - C 표준]
FILE *fp = fopen("data.txt", "r");      // FILE * 받음
char buf[100];
fread(buf, 1, 100, fp);                 // 그냥 사용
fclose(fp);

이 FILE 의 정체가 — 이 장의 주제.

FILE 의 표준적 정의

C 표준은 — FILE 을 stdio.h 의 정의되지 않은 type 으로 둔다.

// [user-space - C 표준의 모호함]
// stdio.h:
typedef struct ??? FILE;

표준은 — 구체적 구조를 명시하지 않는다. 각 libc 구현체가 자유롭게 정의. 사용자는 FILE * 만 다루고, 내부는 신경 쓰지 말라는 메시지.

glibc 의 _IO_FILE

glibc 는 — FILE 을 struct _IO_FILE 의 alias 로 정의.

// [user-space - glibc libio/bits/types/struct_FILE.h]
typedef struct _IO_FILE FILE;

_IO_FILE 의 정의 — bits/types/struct_FILE.h 에. master 의 본체:

// [user-space - glibc libio/bits/types/struct_FILE.h master 인용]
struct _IO_FILE {
    int _flags;                  /* High-order word is _IO_MAGIC; rest is flags. */
    
    /* The following pointers correspond to the C++ streambuf protocol. */
    char *_IO_read_ptr;          /* Current read pointer */
    char *_IO_read_end;          /* End of get area. */
    char *_IO_read_base;         /* Start of putback+get area. */
    char *_IO_write_base;        /* Start of put area. */
    char *_IO_write_ptr;         /* Current put pointer. */
    char *_IO_write_end;         /* End of put area. */
    char *_IO_buf_base;          /* Start of reserve area. */
    char *_IO_buf_end;           /* End of reserve area. */
    
    /* The following fields are used to support backing up and undo. */
    char *_IO_save_base;         /* Pointer to start of non-current get area. */
    char *_IO_backup_base;       /* Pointer to first valid character of backup area */
    char *_IO_save_end;          /* Pointer to end of non-current get area. */
    
    struct _IO_marker *_markers;
    
    struct _IO_FILE *_chain;     /* 다음 FILE (linked list) */
    
    int _fileno;                 /* 실제 file descriptor */
    int _flags2;
    __off_t _old_offset;
    
    /* 1 + column number of pbase(); 0 is unknown. */
    unsigned short _cur_column;
    signed char _vtable_offset;
    char _shortbuf[1];
    
    _IO_lock_t *_lock;
    
    __off64_t _offset;
    /* Wide character stream stuff. */
    struct _IO_codecvt *_codecvt;
    struct _IO_wide_data *_wide_data;
    struct _IO_FILE *_freeres_list;
    void *_freeres_buf;
    size_t __pad5;
    int _mode;
    /* Make sure we don't get into trouble again. */
    char _unused2[15 * sizeof(int) - 4 * sizeof(void *) - sizeof(size_t)];
};

매우 큰 구조체. 약 200 바이트 (x86_64). 사용자에게는 이 모든 게 숨겨져 있다.

핵심 필드들의 의미

_IO_FILE 의 필드는 — 크게 4 그룹으로 나뉜다.

[_IO_FILE 의 4 그룹]

  1. 상태 (status):
     _flags          : magic + 상태 비트
     _fileno         : 실제 file descriptor (kernel 의 정수)
     _mode           : narrow / wide char 구분
     
  2. read buffer (입력):
     _IO_read_base   : buffer 의 시작
     _IO_read_ptr    : 현재 읽기 위치
     _IO_read_end    : 유효 데이터의 끝
     
  3. write buffer (출력):
     _IO_write_base  : buffer 의 시작
     _IO_write_ptr   : 현재 쓰기 위치
     _IO_write_end   : buffer 의 끝
     
  4. backup / 메타데이터:
     _IO_save_*      : ungetc 처리용
     _chain          : 모든 FILE 의 linked list
     _lock           : thread 안전성

read buffer 의 동작 — _IO_read_*

읽기의 핵심은 — _IO_read_base, _IO_read_ptr, _IO_read_end 의 세 포인터.

[read buffer 의 상태]

  ┌──────────────────────────────────────────┐
  │ _IO_buf_base ~ _IO_buf_end (전체 buffer)    │
  ├──────────────────────────────────────────┤
  │                                           │
  │   _IO_read_base                           │
  │       │                                   │
  │       ▼                                   │
  │   ┌───────────────────┬─────────────┐     │
  │   │ 이미 읽은 데이터       │ 안 읽은 데이터  │     │
  │   └───────────────────┴─────────────┘     │
  │       ↑                ↑               ↑  │
  │   _IO_read_base   _IO_read_ptr    _IO_read_end  │
  │                                           │
  │   ┌────────────────────────────┐         │
  │   │ buffer 의 나머지 영역 (미사용)  │         │
  │   └────────────────────────────┘         │
  │                              ↑           │
  │                          _IO_buf_end     │
  │                                           │
  └──────────────────────────────────────────┘
  
  사용자가 fread:
    → _IO_read_ptr 부터 _IO_read_end 까지 데이터 복사
    → _IO_read_ptr 가 그만큼 전진
    → 만약 buffer 부족하면 → read syscall 로 refill

이 디자인이 — C++ streambuf protocol 에서 유래. glibc 의 libio 가 — 옛날 C++ stream class 와 같이 발전한 역사.

write buffer 의 동작 — _IO_write_*

비슷하게 쓰기도 세 포인터.

[write buffer 의 상태]

  ┌──────────────────────────────────────────┐
  │ _IO_buf_base ~ _IO_buf_end (전체 buffer)    │
  ├──────────────────────────────────────────┤
  │                                           │
  │   ┌──────────────────┬──────────────┐     │
  │   │  쓴 데이터 (대기)   │   빈 공간      │     │
  │   └──────────────────┴──────────────┘     │
  │       ↑              ↑               ↑    │
  │   _IO_write_base  _IO_write_ptr   _IO_write_end │
  │                                           │
  └──────────────────────────────────────────┘
  
  사용자가 fwrite:
    → 데이터를 _IO_write_ptr 부터 복사
    → _IO_write_ptr 가 그만큼 전진
    → 만약 buffer 꽉 차면 → flush (write syscall)
    
  fflush:
    → _IO_write_base 부터 _IO_write_ptr 까지를 write syscall
    → 그 다음 _IO_write_ptr = _IO_write_base 로 reset

이 buffer 가 — stdio 의 성능 이득의 핵심. 한 번에 큰 chunk 로 syscall.

_flags 비트들

_flags 는 — 매우 많은 정보를 담는 비트맵.

// [user-space - glibc libio/libio.h master 인용]
#define _IO_MAGIC        0xFBAD0000    /* High-order bits */
#define _OLD_STDIO_MAGIC 0xFABC0000    /* Emulate old stdio */
#define _IO_MAGIC_MASK   0xFFFF0000
#define _IO_USER_BUF     0x0001        /* User-supplied buffer */
#define _IO_UNBUFFERED   0x0002        /* No buffering */
#define _IO_NO_READS     0x0004        /* Reading not allowed */
#define _IO_NO_WRITES    0x0008        /* Writing not allowed */
#define _IO_EOF_SEEN     0x0010        /* EOF reached */
#define _IO_ERR_SEEN     0x0020        /* Error occurred */
#define _IO_DELETE_DONT_CLOSE 0x0040
#define _IO_LINKED       0x0080        /* In _IO_list_all */
#define _IO_IN_BACKUP    0x0100
#define _IO_LINE_BUF     0x0200        /* Line buffered */
#define _IO_TIED_PUT_GET 0x0400
#define _IO_CURRENTLY_PUTTING 0x0800
#define _IO_IS_APPENDING 0x1000
#define _IO_IS_FILEBUF   0x2000
#define _IO_BAD_SEEN     0x4000        /* dead */
#define _IO_USER_LOCK    0x8000

상위 16 비트는 — 매직 넘버 0xFBAD. 잘못된 포인터 (random memory) 가 들어왔는지 빠르게 검증.

// [user-space - C 검증 예]
FILE *fp = fopen("test", "r");
if ((fp->_flags & _IO_MAGIC_MASK) != _IO_MAGIC) {
    /* fp 가 진짜 FILE * 가 아니다 */
    abort();
}

사용자에게는 opaque 이유

C 표준이 FILE 을 opaque 로 둔 이유:

[opaque pointer 의 이유]

  ✓ ABI 안정성:
     구현체가 내부 변경해도 사용자 코드 영향 X
     예: glibc 가 _IO_FILE 의 필드 순서 바꿔도 API 동일
     
  ✓ 구현체 다양성:
     glibc, musl, FreeBSD libc 가 각자 다른 구조 사용 가능
     사용자 코드는 모든 구현체에서 동작
     
  ✓ 보안:
     사용자가 직접 필드 조작 어렵게
     예: _fileno 를 직접 바꿔서 다른 fd 로 redirect 같은 공격 어렵게
     
  ✓ 추상화:
     stream 추상화 — file, pipe, socket, memory buffer 모두 같은 API

_IO_list_all — 모든 FILE 의 linked list

_chain 필드를 통해 — 현재 열려 있는 모든 FILE 이 linked list 로 연결된다.

// [user-space - glibc libio/genops.c master]
extern struct _IO_FILE_plus *_IO_list_all;
[_IO_list_all 의 구조]

  _IO_list_all
       │
       ▼
   ┌────────┐  _chain    ┌────────┐  _chain    ┌────────┐
   │ stderr ├──────────→│ stdout ├──────────→│ stdin  │
   │ FILE   │           │ FILE   │           │ FILE   │
   └────────┘           └────────┘           └────────┘
                                                 │
                                                 _chain
                                                 │
                                                 ▼
                                            ┌────────┐
                                            │ fp1    │ ← fopen 으로 연 파일
                                            │ FILE   │
                                            └────────┘
                                                 │
                                                 ...
  
  새 fopen 시: _chain 의 앞에 추가
  fclose 시: list 에서 제거
  exit 시: 모든 FILE 을 순회하며 flush

이 list 가 — exit 시 자동 flush 의 기반. atexit 의 핸들러가 list 를 순회.

stdin, stdout, stderr — 정적 FILE

C 의 표준 stream 들은 — libc.so 의 .data 에 정적으로 존재.

// [user-space - glibc libio/stdfiles.c master]
static struct _IO_FILE_plus _IO_2_1_stdin_;
static struct _IO_FILE_plus _IO_2_1_stdout_;
static struct _IO_FILE_plus _IO_2_1_stderr_;

FILE *stdin = (FILE *)&_IO_2_1_stdin_;
FILE *stdout = (FILE *)&_IO_2_1_stdout_;
FILE *stderr = (FILE *)&_IO_2_1_stderr_;

_IO_FILE_plus 는 — _IO_FILE + vtable 의 wrapper.

// [user-space - glibc libio/libioP.h master]
struct _IO_FILE_plus {
    FILE file;
    const struct _IO_jump_t *vtable;     /* 가상 함수 테이블 */
};

각 FILE 이 — 자기만의 vtable 을 가진다. 일반 파일, pipe, memory stream 등 동작이 다르므로.

vtable — 가상 함수 테이블

_IO_jump_t 가 — FILE 의 동작을 추상화하는 함수 포인터 모음.

// [user-space - glibc libio/libioP.h master 단순화]
struct _IO_jump_t {
    JUMP_FIELD(_IO_finish_t, __finish);          /* fclose 시 */
    JUMP_FIELD(_IO_overflow_t, __overflow);      /* buffer 꽉 찼을 때 */
    JUMP_FIELD(_IO_underflow_t, __underflow);    /* buffer 비었을 때 */
    JUMP_FIELD(_IO_underflow_t, __uflow);        /* underflow + 1 char */
    JUMP_FIELD(_IO_pbackfail_t, __pbackfail);    /* ungetc */
    JUMP_FIELD(_IO_xsputn_t, __xsputn);          /* fwrite 핵심 */
    JUMP_FIELD(_IO_xsgetn_t, __xsgetn);          /* fread 핵심 */
    JUMP_FIELD(_IO_seekoff_t, __seekoff);        /* fseek */
    JUMP_FIELD(_IO_seekpos_t, __seekpos);
    JUMP_FIELD(_IO_setbuf_t, __setbuf);          /* setvbuf */
    JUMP_FIELD(_IO_sync_t, __sync);              /* fflush */
    JUMP_FIELD(_IO_doallocate_t, __doallocate);  /* buffer 할당 */
    JUMP_FIELD(_IO_read_t, __read);              /* 실제 read */
    JUMP_FIELD(_IO_write_t, __write);            /* 실제 write */
    JUMP_FIELD(_IO_seek_t, __seek);              /* 실제 lseek */
    JUMP_FIELD(_IO_close_t, __close);            /* 실제 close */
    JUMP_FIELD(_IO_stat_t, __stat);
    JUMP_FIELD(_IO_showmanyc_t, __showmanyc);
    JUMP_FIELD(_IO_imbue_t, __imbue);
};

각 stream 종류마다 — 다른 vtable.

[stream 종류별 vtable]

  일반 파일 (fopen):
    _IO_file_jumps
    __xsputn → 일반 buffered write
    __read   → read(2) syscall
    
  pipe (popen):
    _IO_proc_jumps
    __close  → pclose, wait 도 호출
    
  memory stream (fmemopen):
    _IO_mem_jumps
    __read  → 메모리 복사
    
  string stream (sprintf 의 내부):
    _IO_str_jumps

이 vtable 메커니즘이 — C++ class 의 virtual function 흉내. glibc libio 의 C++ 영향.

vtable 의 보안 이슈 — FILE attack

vtable 이 — 보안 측면에서 단골 공격 표면.

[FILE structure attack]

  공격자가 _IO_FILE 의 vtable 포인터를 임의 주소로 덮으면:
    fclose(fp) 호출 시 vtable 의 __finish 호출
    → 공격자가 지정한 함수 실행
    → arbitrary code execution
    
  완화 메커니즘 (glibc 2.24+):
    _IO_vtable_check 가 vtable 이 libc 안의 합법적 vtable 인지 검증
    libc.so 의 vtable 영역 (특정 주소 범위) 밖이면 abort

FILE * 와 file descriptor 의 관계

[FILE * vs int fd]

  int fd:
    kernel 의 정수 식별자 (0, 1, 2, ...)
    read(fd, buf, n), write(fd, buf, n) 직접 syscall
    buffering 없음 (사용자가 관리)
    
  FILE *:
    user-space 의 stream 객체
    fread(buf, n, m, fp), fwrite(buf, n, m, fp)
    내부에 fd 있음 (fp->_fileno)
    buffering 자동 처리
    
  변환:
    fd → FILE *: fdopen(fd, "r")
    FILE * → fd: fileno(fp)
    
  관계: FILE * 는 fd 의 wrapper + buffer + 메타데이터

이 추상화 덕에 — 사용자는 buffer 직접 관리 안 하고 편하게 I/O.

__pad5, _unused2 등 — ABI 안정성

_IO_FILE 의 끝에 — __pad5, _unused2[...] 같은 묘한 필드가 있다.

size_t __pad5;
int _mode;
char _unused2[15 * sizeof(int) - 4 * sizeof(void *) - sizeof(size_t)];

이 padding 의 의미:

[padding 의 의미 — ABI 안정성]

  _IO_FILE 의 _크기_ 가 ABI 의 일부:
    sizeof(struct _IO_FILE) = 고정 (특정 값)
    → 사용자 binary 가 컴파일 시 고정된 offset 으로 필드 접근
    
  새 필드 추가 필요:
    옛: 새 필드를 끝에 append → 크기 변경 → ABI 깨짐
    새: padding 영역을 새 필드로 사용 → 크기 유지 → ABI 보존
    
  _unused2 의 크기 공식:
    "여기 15 * int - 4 * ptr - size_t 만큼 padding 둠"
    → 새 ptr 추가하면 4*ptr 부분이 늘어남 → unused2 가 줄어듦
    → 전체 크기는 변함없음

이 영리한 trick 으로 — glibc 가 _IO_FILE 의 새 필드 추가 가능, 옛 binary 호환성 유지.

의외의 단순함 — 구조체 한 줄

이 모든 복잡함에도 — 사용자가 보는 건:

FILE *fp = fopen("data.txt", "r");

FILE * 한 줄. 모든 복잡함이 — 이 opaque pointer 뒤에 숨겨져 있다. 이게 — 추상화의 가치. 사용자는 buffer, vtable, magic, linked list 신경 안 써도 됨.

FILE 구조체 정리:

[FILE 구조체 정리]

  ✓ C 표준은 opaque (구체적 정의 없음)
  ✓ glibc: struct _IO_FILE 의 alias
  ✓ 약 200 바이트, 매우 큰 구조체
  ✓ 핵심 필드:
    - _flags (magic + 상태)
    - _IO_read_*, _IO_write_*, _IO_buf_* (buffer pointers)
    - _fileno (실제 file descriptor)
    - _chain (linked list)
    - _lock (thread 안전성)
  ✓ vtable 로 stream 종류별 동작 추상화
  ✓ 보안: _IO_vtable_check (2.24+) 으로 vtable hijacking 방어
  ✓ padding 으로 ABI 안정성 유지

다음 장에서는 — FILE * 의 핵심 기능 중 하나, buffering 의 세 종류 (line, full, none) 를 본다. 왜 printf("hello\n") 가 즉시 보이고, printf("hello") 는 안 보일 수도 있는지의 답.

34buffering 의 3 종류 — line / full / none

이전 장에서 — FILE * 안에 buffer (_IO_buf_base ~ _IO_buf_end) 가 있다는 걸 봤다. 이 buffer 의 flush 시점을 결정하는 게 — buffering mode. 3 가지 모드가 있고, 이 차이가 — 신기한 stdio 동작들을 만든다.

신기한 현상부터

먼저 — 사람들이 stdio 에서 헷갈리는 전형적인 경우.

// [user-space - C 예제 1]
#include <stdio.h>
int main(void) {
    printf("hello");        // \n 없음
    sleep(10);              // 10 초 대기
    return 0;
}

./a.out 실행 시 — 10 초 동안 "hello" 안 보임. 그러다 종료 시점에 갑자기 나옴.

// [user-space - C 예제 2]
printf("hello");            // \n 없음
fprintf(stderr, "world");   // \n 없음

출력 순서가 — "world" 가 먼저 보일 수도 있음. stdout 와 stderr 의 buffering 이 다르므로.

// [user-space - C 예제 3]
// terminal 에서:
./a.out
hello                       ← 즉시 보임 (line buffer)

// 그러나 pipe 로:
./a.out | cat
                           ← stdout 가 full buffer 가 됨, "hello" 안 보임

이 모든 현상의 답이 — buffering mode.

3 가지 buffering mode

[buffering mode 3 종류]

  1. _IOFBF (Full Buffered, block buffered):
     buffer 가 _꽉 차거나_ fflush 호출 시 flush
     → 가장 적은 syscall, 가장 빠름
     → 일반 file 의 기본값
     
  2. _IOLBF (Line Buffered):
     buffer 가 꽉 차거나, _\n 출력 시_, 또는 fflush
     → 사용자에게 적당히 보임
     → terminal 의 stdout 의 기본값
     
  3. _IONBF (No buffering):
     매 write 마다 즉시 flush (syscall 매번)
     → 가장 느림
     → stderr 의 기본값 (에러는 즉시 보여야)

기본 설정 — POSIX 의 규칙

POSIX 가 — 각 stream 의 기본 buffering 규칙을 정의.

[표준 stream 의 기본 buffering]

  stderr:
    항상 unbuffered (_IONBF)
    → 에러 메시지는 즉시 보여야 함
    → 프로그램 abort 직전에도 보임
    
  stdout:
    terminal (isatty(1) == 1) 이면:
      → line buffered (_IOLBF)
    아니면 (pipe, file 등):
      → full buffered (_IOFBF)
      
  stdin:
    terminal 이면:
      → line buffered
    아니면:
      → full buffered
      
  fopen 으로 연 일반 파일:
    → full buffered

이 규칙이 — ./a.out | cat 에서 stdout 가 full buffered 가 되는 이유. pipe 는 terminal 이 아니니까.

결정 시점 — isatty 검사

stdio 가 — FILE 의 첫 사용 시점에 isatty 호출.

// [user-space - glibc libio/filedoalloc.c master 단순화]
int
_IO_file_doallocate(FILE *fp)
{
    size_t size;
    char *p;
    struct stat64 st;
    
    /* terminal 인지 검사 */
    if (fp->_fileno >= 0 && __builtin_expect(__fxstat64(_STAT_VER, fp->_fileno, &st), 0) >= 0) {
        if (S_ISCHR(st.st_mode)) {
            /* terminal-like device */
            fp->_flags |= _IO_LINE_BUF;
        }
    }
    
    /* buffer 할당 */
    size = BUFSIZ;
    p = malloc(size);
    _IO_setb(fp, p, p + size, 1);
    return 1;
}

fstat 으로 file 타입 확인 → character device 면 line buffered.

BUFSIZ — 기본 buffer 크기

buffer 의 기본 크기는 — BUFSIZ 매크로.

// [user-space - stdio.h]
#define BUFSIZ 8192     /* 또는 4096, system 의존 */
[BUFSIZ 의 결정]

  보통 4096 ~ 8192 byte
  C 표준은 ≥ 256 만 보장
  
  glibc 의 결정:
    페이지 크기 (4 KB) 의 배수
    cache friendly
    syscall 효율
    
  너무 작으면 → syscall 자주
  너무 크면 → 메모리 낭비, latency

/* mode: _IOFBF, _IOLBF, _IONBF */

예:

각자 — setvbuf 의 단순한 wrapper.

glibc 의 _IO_setvbuf 본체

master 의 정확한 구현:

// [user-space - glibc libio/iosetvbuf.c master 단순화]
int
_IO_setvbuf(FILE *fp, char *buf, int mode, size_t size)
{
    int result;
    CHECK_FILE(fp, EOF);
    _IO_acquire_lock(fp);
    
    switch (mode) {
    case _IOFBF:
        fp->_flags &= ~(_IO_LINE_BUF | _IO_UNBUFFERED);
        if (buf == NULL) {
            if (fp->_IO_buf_base == NULL) {
                /* tty 라도 line buffer 안 함 (사용자가 명시적으로 full 요청) */
                if (_IO_DOALLOCATE(fp) != 0)
                    goto unlock_return;
                fp->_flags &= ~(_IO_LINE_BUF | _IO_UNBUFFERED);
            }
            break;
        }
        break;
        
    case _IOLBF:
        fp->_flags &= ~_IO_UNBUFFERED;
        fp->_flags |= _IO_LINE_BUF;
        break;
        
    case _IONBF:
        fp->_flags &= ~_IO_LINE_BUF;
        fp->_flags |= _IO_UNBUFFERED;
        buf = NULL;
        size = 0;
        break;
        
    default:
        result = EOF;
        goto unlock_return;
    }
    
    result = _IO_SETBUF(fp, buf, size) == NULL ? EOF : 0;
    
unlock_return:
    _IO_release_lock(fp);
    return result;
}

flag 비트 (_IO_LINE_BUF, _IO_UNBUFFERED) 조작 + buffer 교체.

line buffered 의 flush 트리거

line buffered 모드의 flush 시점:

[line buffered 의 flush]

  fwrite(data, ..., fp):
    각 byte 를 buffer 에 복사
    '\n' 발견 시 → flush 까지 buffer 모두 비움
    buffer 꽉 차도 → flush
    
  fread(buf, ..., stdin):
    stdin 이 line buffer 면 — fflush(stdout) 호출 후 read
    → "tied stream" 동작
    → 입력 받기 전에 prompt 가 보이도록
    
  exit, fclose:
    그때 남은 buffer 도 flush

이 마지막 — fread 가 stdout 를 flush — 가 흥미로운 디자인. printf("prompt: "); fgets(...) 가 — 자동으로 prompt 보임.

buffer 종류와 flush 시점 표

[모든 buffer 종류의 flush 시점 정리]

  ┌─────────────┬──────────────────────────────────┐
  │ mode        │ flush 시점                         │
  ├─────────────┼──────────────────────────────────┤
  │ _IOFBF      │ buffer 꽉 참 / fflush / fclose    │
  │ (full)      │ exit                              │
  ├─────────────┼──────────────────────────────────┤
  │ _IOLBF      │ buffer 꽉 참 / '\n' 출력 /         │
  │ (line)      │ 다른 line buffered stream 의 read │
  │             │ fflush / fclose / exit            │
  ├─────────────┼──────────────────────────────────┤
  │ _IONBF      │ 매 write 마다 즉시 (syscall)        │
  │ (none)      │                                   │
  └─────────────┴──────────────────────────────────┘

fflush — 명시적 flush

fflush(fp) 가 — 특정 stream 의 buffer 강제 flush.

// [user-space - C API]
#include <stdio.h>
int fflush(FILE *stream);          // stream 의 buffer flush
                                   // NULL 이면 모든 output stream flush
// [user-space - C 예제]
printf("Loading");
fflush(stdout);                    // 강제로 보이게
sleep(2);
printf(" done\n");

fflush(NULL) — 모든 output stream 의 buffer 비움. 디버깅 시 유용. (input stream 의 fflush 는 C 표준상 undefined, glibc 에서는 buffer 무효화).

exit 시의 자동 flush

exit(0) 또는 main 의 return 0 시 — 모든 FILE 의 buffer 가 자동 flush.

[exit 시 FILE 정리 과정]

  exit() 호출:
    1. atexit 의 등록된 함수들 호출
    2. ★ _IO_cleanup() — 모든 FILE 의 buffer flush
    3. fclose 호출 (close syscall)
    4. _exit() syscall — kernel 에게

_IO_cleanup 의 내부:

// [user-space - glibc libio/genops.c master 단순화]
int
_IO_cleanup(void)
{
    /* 모든 FILE 을 _IO_list_all 의 chain 따라 순회 */
    int result = _IO_flush_all_lockp(0);
    
    return result;
}

**abort 나 _exit 호출 시는 자동 flush 안 됨**. 그래서 — segfault 등으로 죽으면 stdout 의 buffer 가 소실.

보통 안 보이는 자동 flush 동작

stdio 가 — 사용자 모르게 flush 하는 경우가 더 있다.

[자동 flush 의 트리거]

  ✓ \n 출력 (line buffered 만)
  ✓ buffer 꽉 참
  ✓ fflush 호출
  ✓ fclose, exit
  ✓ scanf / fread 호출 (tied stream)
  ✓ system, popen 호출 전
  ✓ fork 후 child 의 첫 stdio 호출 시

마지막 — fork 후의 흥미로운 경우.

// [user-space - C 예제, fork 의 함정]
#include <stdio.h>
#include <unistd.h>

int main(void) {
    printf("hello");        // \n 없음, buffer 에 있음
    
    if (fork() == 0) {
        /* child */
        exit(0);             // ★ "hello" 다시 나옴!
    }
    
    /* parent */
    exit(0);                 // ★ "hello" 또 나옴
}

결과: hellohello 출력. 왜? — fork 시 buffer 도 같이 복사. 부모와 자식 둘 다 같은 buffer 를 가짐. 각자 exit 시 flush 하면서 두 번 출력.

해결 — fork 전에 fflush(NULL).

다음 장에서는 — stdio 의 가장 유명한 함수, **printf** 의 format string 해석 메커니즘을 본다. %d, %s, %5.2f 등이 어떻게 파싱되고 실행되는지의 깊이.

35printf 의 format string 해석

printf("%5.2f\n", 3.14) 한 줄. 사용자는 이 안의 정교한 메커니즘을 보통 의식하지 않는다. 이 장은 — format string 파싱과 변환의 깊이를 본다.

printf 가족 — 한 핵심 함수

printf, fprintf, sprintf, snprintf, ... 가 모두 하나의 internal 함수로 귀결.

// [user-space - glibc stdio-common/printf.c master 단순화]
int
__printf(const char *format, ...)
{
    va_list arg;
    int done;
    
    va_start(arg, format);
    done = __vfprintf_internal(stdout, format, arg, 0);
    va_end(arg);
    
    return done;
}

전체 family 의 관계:

[printf family 의 관계]

  printf       → __vfprintf_internal(stdout, format, va, 0)
  fprintf      → __vfprintf_internal(fp, format, va, 0)
  vprintf      → __vfprintf_internal(stdout, format, va, 0)
  vfprintf     → __vfprintf_internal(fp, format, va, 0)
  
  sprintf      → __vsprintf_internal(buf, ..., format, va, 0)
  snprintf     → __vsnprintf_internal(buf, n, format, va, 0)
  vsprintf     → __vsprintf_internal(...)
  vsnprintf    → __vsnprintf_internal(...)
  
  asprintf     → __asprintf_internal(...)   /* malloc 으로 buffer 할당 */
  dprintf      → __dprintf(fd, ...)         /* fd 직접 (FILE * 없이) */
  
  중심: __vfprintf_internal (~3000 줄, glibc 의 가장 긴 함수 중 하나)

__vfprintf_internal 가 — 모든 일을 한다. format 파싱, 인자 추출, 변환, 출력.

format string 의 문법

format string 의 정확한 syntax:

[format string syntax]

  %[flags][width][.precision][length]specifier
   │     │      │            │      │
   │     │      │            │      └ 변환 종류 (d, s, f, x, ...)
   │     │      │            └ 인자 크기 (hh, h, l, ll, L, ...)
   │     │      └ 정밀도 (소수점 자리 또는 최대 문자 수)
   │     └ 최소 너비
   └ -, +, space, #, 0

각 부분을 차례로.

specifier — 변환 종류

[printf specifier]

  정수:
    %d, %i  : signed decimal
    %u      : unsigned decimal
    %o      : unsigned octal
    %x, %X  : unsigned hex (소문자/대문자)
    %b      : unsigned binary (C23+, glibc 2.35+)
    
  부동소수점:
    %f, %F  : decimal float (3.14)
    %e, %E  : exponential (3.14e+00)
    %g, %G  : 자동 선택 (짧은 쪽)
    %a, %A  : hex float (0x1.91eb851eb851fp+1)
    
  문자/문자열:
    %c      : 단일 문자
    %s      : null-terminated string
    
  포인터:
    %p      : 주소 (0x7ffe...)
    
  특수:
    %n      : 지금까지 출력한 문자 수를 인자에 저장 (★ 보안 위험)
    %%      : 리터럴 '%' 출력

flags — 5 가지

[printf flags]

  '-' : 왼쪽 정렬 (기본은 오른쪽)
        예: %-10d → "42        "
        
  '+' : 양수에도 부호 ('+' 또는 '-')
        예: %+d  → "+42"
        
  ' ' : 양수 앞에 공백
        예: % d  → " 42"
        
  '#' : alternate form
        - %#o → "052" (앞에 0)
        - %#x → "0x2a" (앞에 0x)
        - %#X → "0X2A"
        - %#f → 소수점 항상 (3.0 대신 "3.000000")
        
  '0' : 빈 자리를 0 으로 채움 (왼쪽 정렬과 충돌, 그땐 무시)
        예: %05d → "00042"

flags 는 여러 개 동시 사용 가능. %+05d, %-#10x 등.

width — 최소 너비

너비 지정 방법:

[width]

  숫자:
    %10d → 최소 10 자 (오른쪽 정렬)
           42 → "        42"
           1234567890 → "1234567890" (그대로)
           
  '*' (인자에서):
    printf("%*d", 10, 42) → "        42"
    → width 가 인자로 전달됨

너비는 — 최소 너비. 값이 더 크면 그대로 출력 (잘리지 않음).

precision — 정밀도

. 다음에 숫자. 의미가 type 따라 달라짐.

[precision]

  정수 (%d, %u, ...):
    최소 자릿수. 값이 짧으면 앞에 0 채움.
    %.5d → 42 → "00042"
    
  부동소수점 (%f, %e):
    소수점 _아래_ 자릿수.
    %.2f → 3.14159 → "3.14"
    
  %g:
    유효 숫자 자릿수.
    %.3g → 3.14159 → "3.14"
    
  문자열 (%s):
    _최대_ 출력 문자 수.
    %.5s → "hello world" → "hello"
    
  '.*' (인자에서):
    printf("%.*s", 5, "hello world") → "hello"

length — 인자 크기

C 의 default integer promotion 때문에 — 인자가 항상 int 또는 double 로 promote. 더 작은 타입 출력하려면 length 지정.

[length modifier]

  hh : char (signed char, unsigned char)
  h  : short
  (none) : int
  l  : long
  ll : long long
  j  : intmax_t
  z  : size_t
  t  : ptrdiff_t
  L  : long double (only with f, e, g, a)
  
  예:
    long long x = 42LL;
    printf("%lld\n", x);    /* long long 명시 */
    
    size_t n = 100;
    printf("%zu\n", n);     /* size_t */

**%zu 가 매우 중요** — 64-bit 환경에서 size_t 는 64 bit, 32-bit 에서는 32 bit. %zu 가 — 양쪽 모두 정확.

파싱의 핵심 — switch 의 거대한 미로

__vfprintf_internal 의 파싱:

// [user-space - glibc stdio-common/vfprintf-internal.c master 매우 단순화]
int
__vfprintf_internal(FILE *fp, const char *format, va_list ap, unsigned int mode_flags)
{
    const char *f = format;
    
    while (*f) {
        /* 일반 문자면 그냥 출력 */
        if (*f != '%') {
            outchar(fp, *f++);
            continue;
        }
        
        /* '%' 발견 — 변환 시작 */
        f++;
        
        /* 1. flags 파싱 */
        int flags = 0;
        while (1) {
            switch (*f) {
            case '-': flags |= LEFT;    f++; continue;
            case '+': flags |= PLUS;    f++; continue;
            case ' ': flags |= SPACE;   f++; continue;
            case '#': flags |= HASH;    f++; continue;
            case '0': flags |= ZEROPAD; f++; continue;
            }
            break;
        }
        
        /* 2. width 파싱 */
        int width = 0;
        if (*f == '*') {
            width = va_arg(ap, int);
            f++;
        } else {
            while (isdigit(*f))
                width = width * 10 + (*f++ - '0');
        }
        
        /* 3. precision 파싱 */
        int precision = -1;
        if (*f == '.') {
            f++;
            precision = 0;
            if (*f == '*') {
                precision = va_arg(ap, int);
                f++;
            } else {
                while (isdigit(*f))
                    precision = precision * 10 + (*f++ - '0');
            }
        }
        
        /* 4. length modifier */
        int length = LEN_DEFAULT;
        switch (*f) {
        case 'h':
            f++;
            if (*f == 'h') { length = LEN_CHAR; f++; }
            else            length = LEN_SHORT;
            break;
        case 'l':
            f++;
            if (*f == 'l') { length = LEN_LLONG; f++; }
            else            length = LEN_LONG;
            break;
        case 'j': length = LEN_INTMAX; f++; break;
        case 'z': length = LEN_SIZE;   f++; break;
        case 't': length = LEN_PTRDIFF; f++; break;
        case 'L': length = LEN_LDOUBLE; f++; break;
        }
        
        /* 5. specifier 처리 */
        switch (*f) {
        case 'd': case 'i':
            print_signed_int(fp, flags, width, precision, length, ap);
            break;
        case 'u':
            print_unsigned_int(fp, flags, width, precision, length, ap, 10);
            break;
        case 'x':
            print_unsigned_int(fp, flags, width, precision, length, ap, 16);
            break;
        case 'X':
            print_unsigned_int_upper(fp, flags, width, precision, length, ap, 16);
            break;
        case 'o':
            print_unsigned_int(fp, flags, width, precision, length, ap, 8);
            break;
        case 's':
            print_string(fp, flags, width, precision, va_arg(ap, char *));
            break;
        case 'c':
            print_char(fp, flags, width, va_arg(ap, int));
            break;
        case 'p':
            print_pointer(fp, va_arg(ap, void *));
            break;
        case 'f': case 'F':
            print_float(fp, flags, width, precision, length, ap, 'f');
            break;
        case 'e': case 'E':
            print_float(fp, flags, width, precision, length, ap, 'e');
            break;
        case 'g': case 'G':
            print_float(fp, flags, width, precision, length, ap, 'g');
            break;
        case 'n':
            *(int *)va_arg(ap, int *) = chars_output_so_far;
            break;
        case '%':
            outchar(fp, '%');
            break;
        }
        
        f++;
    }
    
    return chars_output;
}

실제 코드는 수천 줄. 위는 가장 단순화된 버전.

va_arg — 인자 추출

printf 는 가변 인자 (variadic). va_arg 매크로로 — 인자를 type 별로 꺼낸다.

// [user-space - C 표준]
#include <stdarg.h>

void example(int first, ...) {
    va_list ap;
    va_start(ap, first);
    
    int i = va_arg(ap, int);       /* int 꺼냄 */
    double d = va_arg(ap, double); /* double 꺼냄 */
    char *s = va_arg(ap, char *);  /* string 꺼냄 */
    
    va_end(ap);
}

va_arg 의 type 인자가 결정적 — 잘못된 type 으로 꺼내면 undefined behavior. 그래서 — printf("%d", 3.14) 같은 mismatch 가 위험.

type-format mismatch 의 위험

// [user-space - C 위험한 예제]
printf("%d\n", 3.14);          /* %d 인데 double 전달 */
printf("%s\n", 42);            /* %s 인데 int 전달 → 주소로 해석 → SEGV */
printf("%lld\n", (int)42);     /* %lld 인데 int 전달 → 8 byte 읽음 */

이 mismatch 가 — format string vulnerability 의 핵심.

[format string 의 보안 이슈]

  CWE-134: Format String Vulnerability
  
  printf(user_input);            /* user_input 이 "%s%s%s%s" 같으면 */
                                  /* 임의 메모리 읽기 (info leak) */
                                  /* 또는 "%n" 으로 임의 메모리 쓰기 */
  
  안전한 방법:
    printf("%s", user_input);    /* format 은 항상 고정 string */
    
  gcc 의 -Wformat 가 도와줌

%n — 가장 위험한 specifier

%n 이 — printf 의 가장 이상한 specifier. 출력 안 함, 대신 메모리에 쓴다.

// [user-space - C 예제]
int count;
printf("hello world%n\n", &count);
// 출력: "hello world\n"
// 동시에: count = 11 (지금까지 출력한 문자 수)

이게 — format string attack 의 핵심 도구. 임의 메모리 쓰기 가능.

**현대 glibc 는 %n 을 제한**:

[%n 의 제한 (glibc 2.20+)]

  read-only memory 의 format string 에서만 %n 허용
  
  안전한 케이스:
    printf("...%n", &count)       /* OK */
    
  위험한 케이스:
    char fmt[] = "...%n";          /* writable */
    printf(fmt, &count)            /* ★ glibc 가 abort! */
    
  메시지: "*** %n in writable segment detected ***"

이게 — _FORTIFY_SOURCE 의 일부.

표준의 묘한 기능 — 위치 지정.

//       ^^^^^ ^^^^^

부동소수점 출력 — 가장 복잡한 부분

printf("%.2f", 3.14) 같은 부동소수점 변환이 — printf 의 가장 복잡한 부분.

[float printing 의 어려움]

  3.14 (double) 를 "3.14" 로 변환?
  
  사실 3.14 는 정확히 표현 안 됨:
    binary: 0x40091EB851EB851F
    decimal: 3.140000000000000124344978758017532527446746826171875
  
  printf 가 어떻게 정확히 "3.14" 라고 출력?
    → 매우 정교한 알고리즘 (Steele & White's Dragon4, Grisu3, Ryu 등)
    → glibc 는 mpn (multi-precision) library 사용
    → 정확한 round-trip 보장

glibc 의 부동소수점 출력 — printf_fp.c 만 약 1500 줄. 매우 복잡.

printf 의 성능

printf 가 — 사실 꽤 느린 함수.

// [user-space - C 벤치마크]
struct timespec t1, t2;

clock_gettime(CLOCK_MONOTONIC, &t1);

300 ns 가 — 일반 함수 호출 (~1 ns) 대비 _매우 느린_. 이유:

[printf 가 느린 이유]

  1. format string 파싱 (한 글자씩 검사)
  2. va_arg 의 type 처리
  3. 정수 → 문자열 변환 (10 으로 나누기 반복)
  4. buffer 처리, flag 처리
  5. mutex lock (thread safety)
  6. 부동소수점은 더 느림 (10~50배)

성능 critical 응용은 — fmt::format (C++), printf 의 알려진 format 만 직접 처리, syscall 직접 등으로 대체.

printf 의 흥미로운 corner case

// [user-space - C 예제]
// 1. NULL string

// C 표준: undefined behavior
// glibc: "(null)" 출력 (graceful, 항상 그래왔음)

36fread / fwrite 의 깊이

printf 가 텍스트의 세계라면 — fread / fwrite 는 이진 (binary) 의 세계. format string 없이, 메모리의 바이트를 그대로 읽고 쓴다. 단순해 보이지만, 그 안에 함정이 여럿.

표면적인 API

// [user-space - C API]
#include <stdio.h>

size_t fread(void *ptr, size_t size, size_t nmemb, FILE *stream);
size_t fwrite(const void *ptr, size_t size, size_t nmemb, FILE *stream);

size * nmemb 만큼의 바이트를 — buffer 와 stream 사이로 옮긴다. 반환값은 — 성공한 항목 수 (바이트 수 아님).

size × nmemb 의 이상한 분리

왜 size 와 nmemb 가 별도 인자? 이게 — fread 의 가장 헷갈리는 부분.

// [user-space - C 두 가지 방식]
char buf[1000];

/* 방식 1: 한 byte 씩 1000 개 */
size_t n = fread(buf, 1, 1000, fp);
/* 반환: 실제 읽은 byte 수 */

/* 방식 2: 1000 byte 짜리 1 개 */
size_t n = fread(buf, 1000, 1, fp);
/* 반환: 0 (실패) 또는 1 (모두 성공) */

같은 1000 바이트지만 — 반환값 의미가 다르다.

[size × nmemb 의 의미]

  size = 한 _항목_ 의 크기 (예: struct 의 크기)
  nmemb = 항목의 _개수_
  
  사용 의도:
    fread(records, sizeof(struct rec), 100, fp)
    → struct rec 100 개 읽기
    → 반환: 성공한 struct 개수 (0~100)
    
  반환 의미:
    부분 읽기 시 — _완전한 항목_ 만 카운트
    item 의 일부만 읽혔으면 → 그 item 은 카운트 안 됨
    
  특수 케이스:
    size=0 또는 nmemb=0 → 반환 0, errno 변경 X

master 의 _IO_fread

glibc 의 정확한 구현:

// [user-space - glibc libio/iofread.c master 인용]
_IO_size_t
_IO_fread(void *buf, _IO_size_t size, _IO_size_t count, _IO_FILE *fp)
{
    _IO_size_t bytes_requested = size * count;
    _IO_size_t bytes_read;
    
    CHECK_FILE(fp, 0);
    
    if (bytes_requested == 0)
        return 0;
    
    _IO_acquire_lock(fp);
    bytes_read = _IO_sgetn(fp, (char *)buf, bytes_requested);
    _IO_release_lock(fp);
    
    return bytes_requested == bytes_read ? count : bytes_read / size;
}
weak_alias(_IO_fread, fread)

핵심: 1. **size * count 계산 — overflow 안 검사 (!) 2. lock 잡고 — thread-safe 3. _IO_sgetn 호출 — vtable 의 __xsgetn 으로 위임 4. 반환값 계산** — 완전 성공이면 count, 부분이면 bytes_read / size

_IO_file_xsgetn — buffer 의 핵심 동작

master 의 libio/fileops.c 의 _IO_file_xsgetn 단순화:

// [user-space - glibc libio/fileops.c master 단순화]
size_t
_IO_file_xsgetn(FILE *fp, void *data, size_t n)
{
    size_t want, have;
    ssize_t count;
    char *s = data;
    
    want = n;
    
    /* buffer 가 비어 있고 큰 요청이면 — buffer 우회, 직접 read */
    if (fp->_IO_buf_base == NULL) {
        /* 큰 요청은 buffer 거치지 않음 */
        if (want >= WANT_BUFFER_SIZE) {
            count = _IO_SYSREAD(fp, s, want);
            if (count <= 0)
                return n - want;
            s += count;
            want -= count;
            return n - want;
        }
        /* 작은 요청이면 buffer 할당 */
        _IO_doallocbuf(fp);
    }
    
    /* main loop */
    while (want > 0) {
        have = fp->_IO_read_end - fp->_IO_read_ptr;
        
        if (want <= have) {
            /* 충분히 buffer 에 있음 */
            memcpy(s, fp->_IO_read_ptr, want);
            fp->_IO_read_ptr += want;
            want = 0;
        } else {
            if (have > 0) {
                /* buffer 의 남은 데이터 모두 복사 */
                memcpy(s, fp->_IO_read_ptr, have);
                fp->_IO_read_ptr += have;
                s += have;
                want -= have;
            }
            
            /* buffer 비었음 — 다시 채우기 */
            if (_IO_UNDERFLOW(fp) == EOF)
                break;
        }
    }
    
    return n - want;
}

핵심 흐름:

[fread 의 buffer 처리 흐름]

  요청: N 바이트
  
  loop:
    buffer 에 있는 바이트 수 (have) 계산
    
    ┌────────────────────────────────────┐
    │ want ≤ have ?                       │
    └─────────┬───────────────┬──────────┘
              │ YES           │ NO
              ▼               ▼
         buffer 에서 want    buffer 의 have 모두 복사
         만큼 복사            (그러나 부족)
              │               │
              │               buffer 다시 채우기
              │               (read syscall)
              │               │
              │               EOF 면 break
              │               │
              ▼               ▼
         완료              loop 다시

이 디자인이 — buffer 의 효율성의 핵심. 큰 요청이 와도, buffer 단위로 차근차근.

refill — _IO_UNDERFLOW (read syscall)

buffer 가 비면 — _IO_UNDERFLOW 호출. 이게 — 실제 read syscall 발생 시점.

// [user-space - glibc libio/fileops.c master 단순화]
int
_IO_file_underflow(FILE *fp)
{
    ssize_t count;
    
    if (fp->_flags & _IO_NO_READS) {
        fp->_flags |= _IO_ERR_SEEN;
        __set_errno(EBADF);
        return EOF;
    }
    
    /* buffer 가 없으면 할당 */
    if (fp->_IO_buf_base == NULL) {
        _IO_doallocbuf(fp);
    }
    
    /* buffer reset */
    fp->_IO_read_base = fp->_IO_buf_base;
    fp->_IO_write_base = fp->_IO_write_ptr = fp->_IO_write_end = fp->_IO_buf_base;
    
    /* ★ 실제 read syscall! */
    count = _IO_SYSREAD(fp, fp->_IO_buf_base, fp->_IO_buf_end - fp->_IO_buf_base);
    
    if (count <= 0) {
        if (count == 0)
            fp->_flags |= _IO_EOF_SEEN;
        else {
            fp->_flags |= _IO_ERR_SEEN;
            count = 0;
        }
        return EOF;
    }
    
    /* read 성공 — buffer 의 pointer 갱신 */
    fp->_IO_read_end = fp->_IO_buf_base + count;
    fp->_IO_read_ptr = fp->_IO_buf_base;
    
    /* 한 글자 반환 (underflow 의 의미) */
    return *(unsigned char *)fp->_IO_read_ptr;
}

_IO_SYSREAD 가 — vtable 의 __read (보통 read(2) syscall).

fwrite 의 동작

fwrite 는 거의 대칭.

// [user-space - glibc libio/iofwrite.c master 인용]
_IO_size_t
_IO_fwrite(const void *buf, _IO_size_t size, _IO_size_t count, _IO_FILE *fp)
{
    _IO_size_t request = size * count;
    _IO_size_t written;
    
    CHECK_FILE(fp, 0);
    if (request == 0)
        return 0;
    
    _IO_acquire_lock(fp);
    
    if (_IO_vtable_offset(fp) != 0 || _IO_fwide(fp, -1) == -1)
        written = _IO_sputn(fp, (const char *)buf, request);
    else
        written = 0;
    
    _IO_release_lock(fp);
    
    return request == written ? count : written / size;
}

_IO_sputn → vtable 의 __xsputn → _IO_file_xsputn → buffer 에 복사 또는 write syscall.

partial read 의 함정

fread 의 가장 위험한 점 — 부분 읽기.

// [user-space - C 위험한 예제]
struct record records[100];
size_t n = fread(records, sizeof(struct record), 100, fp);

if (n == 100) {
    /* 다 읽음 */
} else {
    /* 부분 — n 개만 valid */
    /* records[n] ~ records[99] 는 _부정의 상태_ */
}
[fread 부분 읽기 처리]

  fread 가 100 개 요청 → 73 개만 읽음:
    반환값: 73
    records[0]~records[72]: valid
    records[73]: 부분만 채워졌을 수도 (UB)
    records[74]~records[99]: 미정
    
  EOF 일 수도, error 일 수도:
    if (n != 100) {
        if (feof(fp))   { /* 정상 EOF */ }
        if (ferror(fp)) { /* I/O 에러 */ }
    }

size × nmemb 의 overflow 함정

glibc 의 fread 가 — size * count 의 overflow 검사 안 함.

// [user-space - C 위험 예제]
size_t huge = SIZE_MAX / 2 + 1;
fread(buf, huge, 4, fp);
// size * count 가 overflow → 작은 수가 됨
// fread 는 그 작은 양 만큼 읽고 4 반환 (성공으로 보임)
// → buffer overflow!

C 표준은 — 이런 overflow 동작 미정의. glibc 는 검사 안 함 (성능 우선). 사용자가 알아서 검증.

// 안전한 사용:
if (size > 0 && nmemb > SIZE_MAX / size) {
    /* overflow 위험 — 거부 */
    return -1;
}
fread(buf, size, nmemb, fp);

binary I/O 의 endian 이슈

fread 가 — 그냥 바이트를 그대로 읽는다. endian 변환 없음.

// [user-space - C 예제]
uint32_t value;
fread(&value, sizeof(value), 1, fp);

// fp 에 0x01 0x02 0x03 0x04 바이트가 있으면:
// x86_64 (little-endian): value = 0x04030201
// PowerPC (big-endian):   value = 0x01020304

network protocol 이나 file format 다룰 때 매우 주의해야 한다. portable 한 파일 포맷은:

// portable 한 방법: bytes 직접
uint8_t bytes[4];
fread(bytes, 1, 4, fp);
uint32_t value = ((uint32_t)bytes[0] << 24)    /* big-endian decode */
               | ((uint32_t)bytes[1] << 16)
               | ((uint32_t)bytes[2] <<  8)
               | ((uint32_t)bytes[3]      );

또는 htonl/ntohl 같은 endian 변환 함수.

fread vs read 의 trade-off

stdio 의 fread vs raw read(2) 의 비교.

[fread vs read]

  fread:
    + buffering 자동 (작은 read 빨라짐)
    + lock 자동 (thread safety)
    + EOF / error 추상화
    + portable (Windows 등에서도)
    - 약간의 오버헤드
    - buffer 동기화 신경
    
  read (syscall):
    + 빠름 (직접 syscall)
    + memory copy 없음 (큰 read 일 때 유리)
    - 작은 read 매번 syscall (느림)
    - 사용자가 buffer 관리
    - EAGAIN, EINTR 직접 처리

성능 측정:

// [user-space - C 벤치마크]
// 1 byte 씩 1M 번 read:
read(fd, &c, 1) × 1M    → ~3000 ms     (syscall 매번)
fread(&c, 1, 1, fp) × 1M → ~30 ms      (buffer)

// 큰 chunk 한 번:
read(fd, buf, 1MB)      → ~5 ms        (직접)
fread(buf, 1, 1MB, fp)  → ~7 ms        (buffer 복사)

작은 I/O 가 많으면 — fread 가 압도적. 큰 I/O 한 번이면 — read 가 약간 빠름.

setvbuf 로 buffer 조정

성능 critical 응용은 — setvbuf 로 buffer 크기 조정.

// [user-space - C 예제]
FILE *fp = fopen("huge.bin", "rb");

/* 큰 buffer (1 MB) 로 */
char *bigbuf = malloc(1024 * 1024);
setvbuf(fp, bigbuf, _IOFBF, 1024 * 1024);

/* 이제 fread 가 1MB 단위 syscall */
while (fread(record, sizeof(record), 100, fp)) {
    process(record);
}

buffer 크면 — syscall 더 적음, 더 빠름. 단 메모리 사용 증가.

mmap 으로 더 빠른 대안

매우 큰 파일은 — mmap 으로 더 빠를 수도.

// [user-space - C 예제, mmap 으로 파일 read]
int fd = open("huge.bin", O_RDONLY);
struct stat st;
fstat(fd, &st);

void *map = mmap(NULL, st.st_size, PROT_READ, MAP_PRIVATE, fd, 0);
/* map 으로 파일을 직접 메모리처럼 접근 */

uint8_t byte_42 = ((uint8_t *)map)[42];   /* 그냥 메모리 접근! */

munmap(map, st.st_size);
close(fd);
[fread vs mmap]

  mmap:
    + memory copy 없음 (zero-copy)
    + 큰 파일에 매우 빠름
    + random access 효율적
    - 작은 파일 overhead
    - SIGBUS (디스크 에러 시)
    - 페이지 fault overhead
    
  fread:
    + 작은 파일에 무난
    + 안정적 error 처리
    - memory copy 있음

대규모 server 응용 (DB, search engine) 들은 — 보통 mmap 사용.

fread 의 thread safety

fread 는 — lock 사용 (FILE 의 _lock 필드).

// libio/iofread.c 의 의사 코드:
_IO_acquire_lock(fp);
bytes_read = _IO_sgetn(fp, buf, bytes);
_IO_release_lock(fp);

multi-thread 환경에서도 안전. 단 lock 의 비용. 더 빠른 변종:

// [user-space - C unlocked 버전]
size_t fread_unlocked(void *ptr, size_t size, size_t nmemb, FILE *stream);
size_t fwrite_unlocked(...);

lock 안 잡음 — 사용자가 보장해야 함. 한 thread 에서만 사용하거나, 직접 lock 처리.

feof, ferror, clearerr

fread 반환값이 요청보다 작으면 — feof 또는 ferror 로 원인 확인.

// [user-space - C 예제]
size_t n = fread(buf, sizeof(struct), 100, fp);
if (n < 100) {
    if (feof(fp))
        printf("EOF reached after %zu items\n", n);
    if (ferror(fp))
        perror("I/O error");
}

/* 오류 상태 reset */
clearerr(fp);

_flags 의 _IO_EOF_SEEN, _IO_ERR_SEEN 비트 검사. 자세한 내용 40 장 (error handling).

빈번한 잘못된 패턴들

// [user-space - C 잘못된 패턴들]

/* ✗ 잘못 1: while feof */
while (!feof(fp)) {
    fread(buf, 1, sizeof(buf), fp);    /* feof 는 _다음_ 시도 후에야 set */
    process(buf);                       /* 마지막 partial read 도 처리해버림 */
}

/* ✓ 옳은 패턴 */
while ((n = fread(buf, 1, sizeof(buf), fp)) > 0) {
    process(buf, n);
}
if (ferror(fp)) handle_error();

/* ✗ 잘못 2: fread 후 fwrite (같은 stream 에) */
fread(buf, 1, 100, fp);
fwrite(buf, 1, 100, fp);    /* UB! seek 필요 */

/* ✓ 옳은 패턴 */
fread(buf, 1, 100, fp);
fseek(fp, 0, SEEK_CUR);     /* 또는 fflush */
fwrite(buf, 1, 100, fp);

마지막 — read 와 write 사이에 fseek 또는 fflush 필요. C 표준의 미묘한 요구사항.

이 장의 결론

fread / fwrite 의 본질:

[fread/fwrite 정리]

  ✓ size × nmemb 의 분리 — 항목 단위 처리
  ✓ 반환값 = 성공한 항목 수 (byte 수 아님)
  ✓ _IO_sgetn / _IO_sputn → vtable 의 __xsgetn / __xsputn
  ✓ buffer 가 있으면 memcpy, 부족하면 underflow / overflow 호출
  ✓ _IO_UNDERFLOW → 실제 read syscall
  ✓ _IO_OVERFLOW → 실제 write syscall
  ✓ partial read 의 함정 (반환값 < 요청 → 일부만 valid)
  ✓ size * count overflow 검사 없음 (사용자 책임)
  ✓ endian 변환 없음 (사용자 책임)
  ✓ thread-safe (lock), fread_unlocked 는 빠르지만 위험
  ✓ feof, ferror 로 원인 구분

다음 장에서는 — 더 어려운 동생, **scanf** 의 input parsing 을 본다. printf 보다 훨씬 복잡한 이유와 흔한 함정.

37scanf 와 input parsing

scanf 는 — printf 의 반대처럼 보인다. format string 으로 입력을 파싱. 그런데 — 훨씬 더 어렵다. printf 는 우리가 알고 있는 값을 출력하는 것이지만, scanf 는 알지 못하는 입력을 해석해야 한다. 이 차이가 — 많은 함정의 원인.

표면적인 API

// [user-space - C API]
int scanf(const char *format, ...);
int fscanf(FILE *stream, const char *format, ...);
int sscanf(const char *str, const char *format, ...);

내부는 모두 __vfscanf_internal 로 귀결 (printf 와 같은 패턴).

// [user-space - C 예제]
int n;
scanf("%d", &n);              // 정수 하나 읽기

printf 와 다른 점

[scanf vs printf 의 차이]

  printf:
    값 → 텍스트 출력
    값이 명확함 (사용자가 안다)
    
  scanf:
    텍스트 → 값 파싱
    텍스트가 _알 수 없음_
    잘못된 형식, 누락, 너무 큰 값 등 가능
    → 매우 어려운 문제

scanf 의 어려움은 — 입력의 불확실성에서 온다.

scanf 의 specifier

[scanf specifier]

  정수:
    %d  : signed decimal
    %u  : unsigned decimal
    %o  : octal
    %x  : hexadecimal
    %i  : 자동 감지 (0x → hex, 0 → oct, else dec)
    
  부동소수점:
    %f, %e, %g : float
    %lf, %le, %lg : double (★ printf 와 다름!)
    %Lf : long double
    
  문자/문자열:
    %c  : 단일 문자 (whitespace 건너뜀 안 함!)
    %s  : 공백까지의 단어
    %[...] : character set
    
  포인터:
    %p  : 주소
    
  특수:
    %n  : 지금까지 읽은 _문자 수_ 저장
    %%  : 리터럴 '%'

printf 와 다른 핵심 — %f vs %lf

매우 중요한 차이.

[printf 와 scanf 의 %f 차이]

  printf:
    %f → float 또는 double 모두 (default argument promotion)
    %lf → 같음 (l 무시)
    
  scanf:
    %f → float* 요구
    %lf → double* 요구 (★ 다름!)
    %Lf → long double*
    
  잘못 쓰면 UB:
    double d;
    scanf("%f", &d);    // ★ 위험! %f 는 float* 요구
                        // double 의 일부만 채워짐 또는 stack 손상

이 차이가 — 초보자가 가장 많이 빠지는 함정.

whitespace 의 묘한 처리

format string 의 공백 문자가 — 매우 특별한 의미.

[scanf 의 whitespace 규칙]

  format 의 ' ' (공백):
    입력의 _0개 이상_ 의 공백 문자와 매치
    → 공백, 탭, newline 모두 건너뜀
    
  format 의 다른 문자:
    입력에서 _정확히_ 그 문자와 매치
    매치 안 되면 → 그 위치에서 중단
    
  %d, %u, %f, %x 같은 변환 specifier:
    먼저 _whitespace 건너뜀_ (자동)
    그 다음 변환
    
  ★ %c, %[, %n 의 특수성:
    whitespace 건너뜀 안 함
    있는 그대로 처리
// [user-space - C 예제]
char a, b;
scanf("%c%c", &a, &b);
// "x y" 입력:
//   a = 'x', b = ' ' (공백도 문자!)

// 공백 건너뛰려면 명시:
scanf(" %c %c", &a, &b);    // 공백을 명시적으로
//   a = 'x', b = 'y'

%s 의 위험성

%s 가 — scanf 의 가장 위험한 specifier.

// [user-space - C 위험!]
char buf[10];
scanf("%s", buf);
// "AAAAAAAAAAAAAAAAAAAAAA..." 입력 → buffer overflow!

%s 는 — whitespace 만나기 전까지 모두 읽음. 길이 제한 없음. 사용자 입력 길이 모르면 — 항상 width 명시:

// [user-space - C 안전]
char buf[10];
scanf("%9s", buf);    /* 최대 9 글자 + null = 10 byte */

또는 — fgets 가 더 안전.

// [user-space - C 더 안전]
char buf[100];
fgets(buf, sizeof(buf), stdin);
sscanf(buf, "%d", &n);    /* 줄 단위 입력 후 parse */

%[...] — character set

%s 의 더 정교한 버전.

[%[...] 의 문법]

  %[abc]     : a, b, c 중 하나면 매치
  %[^abc]    : a, b, c _가 아니면_ 매치 (negation)
  %[a-z]     : 범위
  %[^\n]     : newline 이 아닌 모든 문자
  %[a-zA-Z0-9_] : C identifier 같은 것
// [user-space - C 예제]
char line[256];
scanf("%255[^\n]", line);   /* 한 줄 읽기 (newline 까지) */

int year;
char month[10];
scanf("%d-%9[a-zA-Z]", &year, month);
// "2026-January" → year=2026, month="January"

반환값과 에러 처리

scanf 의 반환값이 — 매우 특별.

[scanf 반환값]

  성공 시: _성공적으로 변환된 항목 수_
    scanf("%d %d", &a, &b)
      → 둘 다 성공: 2
      → a 만 성공: 1
      → 둘 다 실패: 0
      
  EOF 면: EOF (-1)
    입력 전에 EOF 발생
    
  matching failure:
    format 과 입력이 안 맞음 → 부분 반환
    
  conversion failure:
    숫자 변환 등 실패 → 부분 반환
// [user-space - C 예제]
int a, b, c;
int n = scanf("%d %d %d", &a, &b, &c);
if (n == 3) {
    /* 다 성공 */
} else if (n == EOF) {
    /* 입력 끝 */
} else {
    /* 일부 실패 — a~n-1 만 유효 */
    /* a[n] ~ a[2] 는 _건드리지 않음_ (변경 안 됨) */
}

overflow 의 함정

%d 가 — 정수 overflow 검사 안 함.

// [user-space - C 위험]
int n;
scanf("%d", &n);
// 입력: "9999999999999999999999"
// → n 값 unpredictable (UB)

C99 부터는 — overflow 시 INT_MAX 또는 INT_MIN, errno = ERANGE 로 약간 개선. 그러나 사용자가 직접 검증:

// [user-space - C 안전한 방법]
char line[100];
fgets(line, sizeof(line), stdin);
errno = 0;
char *end;
long n = strtol(line, &end, 10);
if (errno == ERANGE) {
    /* overflow */
}
if (end == line) {
    /* 숫자 아님 */
}

strtol 이 — scanf("%d", ...) 보다 훨씬 안전. 명확한 에러 처리.

sscanf — 더 안전한 변종

문자열에서 파싱.

// [user-space - C 예제]
char input[] = "hello 42 3.14";
char word[20];
int num;
double pi;

sscanf(input, "%19s %d %lf", word, &num, &pi);
/* word="hello", num=42, pi=3.14 */

sscanf 가 — stream 에서 직접 읽기보다 안전. buffer 의 끝이 명확 (null byte). EOF 처리 단순.

scanf 의 깊은 문제 — input 위치

scanf 가 입력의 어느 부분까지 처리했는지 — 알기 어렵다.

// [user-space - C 문제 예제]
int n;
char rest[100];

scanf("%d", &n);            /* "42abc" → n=42, "abc" 가 stream 에 남음 */
scanf("%s", rest);          /* "abc" 읽음 */
[scanf 의 위치 추적]

  scanf 는 자기가 어디까지 읽었는지 알려주지 않음
  → fseek/ftell 로 위치 알 수 있음 (stream 의 경우)
  → sscanf 는 %n 으로 추적 가능
  
  대안: sscanf 의 %n 활용
  
  int consumed;
  sscanf(input, "%d%n", &n, &consumed);
  /* consumed = 입력에서 읽은 문자 수 */
  /* input + consumed 부터 다음 처리 */

fgets + sscanf 패턴

스트리밍 입력의 안전한 패턴 — 줄 단위로 fgets, 각 줄을 sscanf.

// [user-space - C 권장 패턴]
char line[256];

while (fgets(line, sizeof(line), stdin)) {
    int n;
    if (sscanf(line, "%d", &n) == 1) {
        process(n);
    } else {
        fprintf(stderr, "Invalid line: %s", line);
    }
}

장점:

  • 부분 입력 시 다음 줄에서 다시 시도 가능
  • 줄 단위로 명확한 에러 처리
  • buffer overflow 방지 (fgets 의 size 인자)

getline — POSIX 의 더 안전한 fgets

POSIX 의 getline — 동적 buffer 로 임의 길이 라인 처리.

// [user-space - POSIX API]
#include <stdio.h>
ssize_t getline(char **lineptr, size_t *n, FILE *stream);
// [user-space - C 예제]
char *line = NULL;
size_t cap = 0;
ssize_t len;

while ((len = getline(&line, &cap, stdin)) != -1) {
    /* line: \n 포함된 한 줄 (마지막 줄은 \n 없을 수도) */
    process(line, len);
}
free(line);
  • buffer 가 부족하면 — getline 이 자동 realloc.
  • 임의 길이 라인 처리 가능.
  • 사용자가 free 해야 함.

scanf 의 보안 이슈

scanf 는 — 보안에서 매우 조심해야 함.

[scanf 의 보안 이슈]

  ✗ %s 없이 width — buffer overflow
     scanf("%s", buf)            → 위험
     scanf("%99s", buf)          → 안전
     
  ✗ format string injection (printf 처럼)
     scanf(user_input, ...)      → 위험!
     
  ✗ 정수 overflow
     scanf("%d", &n)             → INT_MAX 초과 시 UB
     → strtol 사용
     
  ✗ 부족한 인자
     scanf("%d %d", &a)          → b 가 없음, UB
     
  ✗ pointer mismatch
     int n;
     scanf("%ld", &n)            → long 으로 8 bytes 쓰는데
                                   int 는 4 bytes → 인접 메모리 손상

scanf 가 사실 잘 안 쓰이는 이유

전문가들은 — scanf 를 거의 안 쓴다. 대안:

[scanf 의 대안]

  텍스트 입력:
    fgets + strtol/strtod/strtoul + 명시적 파싱
    또는 getline (POSIX)
    
  CSV 파싱:
    전용 library (libcsv, csv-parser)
    
  JSON 파싱:
    json-c, jansson, cJSON
    
  명령행 옵션:
    getopt, getopt_long
    
  설정 파일:
    libconfig, INI parser
    
  → scanf 의 "한 줄 입력" 가치는 — 학습용 예제 정도

scanf 의 흔한 사용처는 — 학교 과제, 경쟁 프로그래밍, 간단한 디버깅 도구. 실제 production 에서는 거의 안 쓴다.

getchar, gets — 옛 친구들

stdio 의 가장 단순한 입력 함수들.

// [user-space - C 표준]
int getchar(void);                /* stdin 에서 한 글자 */
int fgetc(FILE *fp);              /* fp 에서 한 글자 */
char *fgets(char *s, int n, FILE *fp);   /* 한 줄 */
char *gets(char *s);              /* ★ 사용 금지! */

gets 는 — C11 에서 표준에서 제거됨. 매우 위험 (length 인자 없음). 절대 사용 금지.

ungetc — 되돌리기

읽은 문자를 다시 stream 에 넣기.

// [user-space - C 표준]
int ungetc(int c, FILE *fp);
// [user-space - C 예제: 숫자 vs 문자 구별]
int c = getchar();
if (isdigit(c)) {
    ungetc(c, stdin);              /* 되돌리기 */
    int n;
    scanf("%d", &n);                /* 정수로 다시 읽기 */
} else {
    /* 다른 처리 */
}

ungetc 는 — 1 글자만 보장. 표준상 더 많은 ungetc 는 UB.

scanf 의 buffer 동작

scanf 가 FILE * 의 buffer 사용.

[scanf 와 buffer]

  scanf("%d", &n) 호출:
    stdin 의 buffer 에 있는 글자 읽음
    필요하면 read syscall 로 refill
    
  buffer 에 더 있는 글자 — 다음 scanf 가 사용
  → "1 2\n" 입력 → 첫 scanf("%d") 는 "1" 읽고 " 2\n" 남김

이 buffer 동작이 — 인터랙티브 prompt 의 미묘한 동작 원인.

// [user-space - C 예제]
printf("Enter number: ");
scanf("%d", &n);                  /* 사용자가 "42\n" 입력 */
printf("Enter char: ");
scanf("%c", &c);                  /* 사용자 입력 없이도 c = '\n' ! */
                                  /* "\n" 가 buffer 에 남아있었음 */

해결 — 사이에 공백 추가:

scanf("%d", &n);
scanf(" %c", &c);                 /* 공백 → whitespace 건너뜀 */

이 장의 결론

scanf 의 본질:

[scanf 정리]

  ✓ printf 의 반대 — 텍스트 → 값
  ✓ 입력의 불확실성으로 매우 어려움
  ✓ printf 와 핵심 차이:
    - %f vs %lf (scanf 는 double 에 %lf 필수)
    - %c, %[ 는 whitespace 안 건너뜀
    - %s 는 width 없으면 매우 위험
  ✓ whitespace 처리 규칙 미묘
  ✓ 반환값 = 성공한 변환 수 (EOF 가능)
  ✓ overflow, format mismatch 검사 약함
  ✓ 권장 패턴: fgets + strtol/strtod/sscanf
  ✓ 또는 POSIX 의 getline
  ✓ gets 는 절대 금지 (C11 에서 제거)
  ✓ ungetc 로 1 글자 되돌리기
  ✓ 인터랙티브 prompt 의 buffer 함정

다음 장에서는 — stream 의 생애 전체, **fopen, fclose, fflush** 의 정확한 동작을 본다.

38fopen / fclose / fflush 의 동작

stream 의 생애 — 만들어지고, 사용되고, 정리되는 과정. 이 장에서 — fopen, fclose, fflush 의 깊이를 본다.

fopen 의 API

// [user-space - C 표준]
FILE *fopen(const char *pathname, const char *mode);
FILE *freopen(const char *pathname, const char *mode, FILE *stream);
FILE *fdopen(int fd, const char *mode);          /* POSIX */

3 가지 변종:

  • fopen — 파일명에서 stream
  • freopen — 기존 stream 을 다른 파일로 재할당
  • fdopen — file descriptor 에서 stream

mode 문자열

[fopen 의 기본 mode]

  "r"   : 읽기 전용. 파일 없으면 NULL 반환.
  "w"   : 쓰기 전용. 파일 _덮어씀_ (truncate 0).
  "a"   : 추가 쓰기. 파일 끝부터.
  "r+"  : 읽기/쓰기. 파일 있어야 함.
  "w+"  : 읽기/쓰기. 파일 truncate.
  "a+"  : 읽기/쓰기. 추가 모드.
  
  "b" 추가:
    "rb", "wb", "rb+" 등.
    Linux: 무시 (binary/text 구분 없음)
    Windows: CRLF 변환 비활성

glibc 의 확장

POSIX/glibc 추가 mode flags:

[glibc 확장 mode 문자]

  "ce" — c (cancellation point 안 됨) + e (O_CLOEXEC)
  "x"  — O_EXCL (이미 있으면 실패)
  "m"  — mmap 사용 시도 (읽기만)
  
  예:
    fopen("file", "re")    /* O_CLOEXEC — exec 시 자동 close */
    fopen("file", "wx")    /* 이미 있으면 실패, 새 파일만 */
    fopen("file", "rm")    /* mmap 시도 */

O_CLOEXEC 가 매우 중요 — race condition 없는 fd 보호.

fopen 내부 동작

master 의 __fopen_internal 단순화:

// [user-space - glibc libio/iofopen.c master 단순화]
FILE *
__fopen_internal(const char *filename, const char *mode, int is32)
{
    struct locked_FILE {
        struct _IO_FILE_plus fp;
#ifdef _IO_MTSAFE_IO
        _IO_lock_t lock;
#endif
        struct _IO_wide_data wd;
    } *new_f = (struct locked_FILE *)malloc(sizeof(struct locked_FILE));
    
    if (new_f == NULL)
        return NULL;
    
#ifdef _IO_MTSAFE_IO
    new_f->fp.file._lock = &new_f->lock;
#endif
    
    /* 1. _IO_FILE 초기화 */
    _IO_no_init(&new_f->fp.file, 0, 0, &new_f->wd, &_IO_wfile_jumps);
    _IO_JUMPS(&new_f->fp) = &_IO_file_jumps;
    _IO_new_file_init_internal(&new_f->fp);
    
    /* 2. 실제 open(2) syscall */
    if (_IO_file_fopen((FILE *)new_f, filename, mode, is32) != NULL)
        return __fopen_maybe_mmap(&new_f->fp.file);
    
    /* 실패 시 정리 */
    _IO_un_link(&new_f->fp);
    free(new_f);
    return NULL;
}

흐름:

[fopen 의 단계]

  1. malloc — FILE 구조체 + lock 영역 통합 할당
  
  2. _IO_no_init — _IO_FILE 의 필드 초기화
     - _flags 에 magic + 기본 비트
     - buffer pointer 들 모두 NULL
     - _chain 에 _IO_list_all 추가
     
  3. vtable 설정 → _IO_file_jumps
  
  4. _IO_file_fopen → mode 파싱 + open(2) syscall
     - "r" → O_RDONLY
     - "w" → O_WRONLY | O_CREAT | O_TRUNC
     - "a" → O_WRONLY | O_CREAT | O_APPEND
     - "+" → O_RDWR
     - "e" → O_CLOEXEC
     - "x" → O_EXCL
     
  5. _fileno 에 fd 저장
  
  6. 반환

mode 파싱의 trick

glibc 가 — mode string 의 처음 7 글자만 검사.

[mode 파싱 한계]

  glibc 2.14 이전: 처음 6 글자
  glibc 2.14 이상: 처음 7 글자
  
  fdopen: 처음 5 글자
  
  이유: "rb+cmxe" 같은 가능한 모든 조합 수용
  
  주의: 7+ 글자 mode 는 무시됨!

fdopen — fd 에서 stream 만들기

// [user-space - C 예제]
int fd = open("file.txt", O_RDONLY);
FILE *fp = fdopen(fd, "r");
/* fp 는 fd 를 wrap 한 FILE * */

/* fclose(fp) 가 close(fd) 도 호출 */
fclose(fp);

용도:

[fdopen 의 활용]

  ✓ open 의 더 정교한 flag 활용 후 stream 화
     int fd = open(..., O_NONBLOCK | O_CLOEXEC | ...);
     FILE *fp = fdopen(fd, "r");
     
  ✓ socket 을 stream 으로
     int s = socket(...);
     FILE *fp = fdopen(s, "r+");
     /* fread/fwrite 로 네트워크 데이터 처리 */
     
  ✓ pipe 의 end 를 stream 으로
     int pipefd[2];
     pipe(pipefd);
     FILE *reader = fdopen(pipefd[0], "r");
     FILE *writer = fdopen(pipefd[1], "w");
     
  ✓ 표준 stream 의 재할당
     /* stdout 를 file 로 redirect */
     int fd = open("output.log", O_WRONLY|O_CREAT, 0644);
     dup2(fd, STDOUT_FILENO);
     close(fd);

freopen — stream 재할당

기존 FILE* 을 다른 파일로 재사용.

// [user-space - C 예제]
freopen("output.txt", "w", stdout);
/* 이후 모든 stdout 출력이 output.txt 로 */

printf("hello\n");          /* output.txt 에 기록 */

내부 동작:

[freopen 의 동작]

  1. 기존 stream flush (fflush)
  2. 기존 fd close
  3. 새 파일 open
  4. 같은 FILE * 에 새 fd 와 mode 설정
  5. _IO_list_all 의 위치 그대로
  
  → 포인터 그대로, 내용은 다른 파일
  → stdout, stderr 의 redirect 에 매우 유용

흥미로운 사용 — dup2 와의 차이:

// dup2: file descriptor 레벨에서 redirect
dup2(fd, STDOUT_FILENO);
/* C level: stdout 의 _fileno 가 fd 가 됨 */
/* 그러나 stdout 의 buffer 는 그대로 (옛 데이터 있을 수도) */

// freopen: stream 레벨에서 완전 재할당
freopen("file", "w", stdout);
/* buffer 도 reset */

fclose 의 동작

// [user-space - glibc libio/iofclose.c master 인용]
int
_IO_new_fclose(FILE *fp)
{
    int status;
    
    CHECK_FILE(fp, EOF);
    
#if SHLIB_COMPAT(libc, GLIBC_2_0, GLIBC_2_1)
    /* 호환성 검사 — 옛 stream 인지 */
    if (_IO_vtable_offset(fp) != 0)
        return _IO_old_fclose(fp);
#endif
    
    /* 1. _IO_list_all 의 chain 에서 제거 */
    if (fp->_flags & _IO_IS_FILEBUF)
        _IO_un_link((struct _IO_FILE_plus *)fp);
    
    _IO_acquire_lock(fp);
    
    /* 2. buffer flush + fd close */
    if (fp->_flags & _IO_IS_FILEBUF)
        status = _IO_file_close_it(fp);
    else
        status = fp->_flags & _IO_ERR_SEEN ? -1 : 0;
    
    _IO_release_lock(fp);
    
    /* 3. vtable 의 __finish 호출 (cleanup) */
    _IO_FINISH(fp);
    
    /* 4. wide character 의 codecvt 해제 */
    if (fp->_mode > 0) {
        struct _IO_codecvt *cc = fp->_codecvt;
        __libc_lock_lock(__gconv_lock);
        __gconv_release_step(cc->__cd_in.__cd.__steps);
        __gconv_release_step(cc->__cd_out.__cd.__steps);
        __libc_lock_unlock(__gconv_lock);
    }
    
    /* 5. 정적 stream (stdin/stdout/stderr) 가 아니면 free */
    if (_IO_have_backup(fp))
        _IO_free_backup_area(fp);
    if (fp != _IO_stdin && fp != _IO_stdout && fp != _IO_stderr) {
        fp->_IO_file_flags = 0;
        free(fp);
    }
    
    return status;
}

흐름:

[fclose 의 5 단계]

  1. _IO_list_all 에서 제거 (linked list)
  
  2. buffer flush:
     write buffer 의 남은 데이터 → write syscall
     read buffer 무시 (drop)
     fd close(2) syscall
     
  3. vtable 의 __finish 호출:
     stream 종류별 추가 정리
     (예: pipe 면 wait 호출)
     
  4. wide character 자원 해제 (있으면)
  
  5. FILE 구조체 free
     단 stdin/stdout/stderr 는 안 함 (정적)

fclose 의 에러 처리 — 매우 중요

대부분이 무시하는 사실 — **fclose 도 에러를 반환**한다.

// [user-space - C 일반적이지만 위험]
fclose(fp);    /* 반환값 무시 */

// [user-space - C 안전]
if (fclose(fp) != 0) {
    perror("fclose");
    /* ★ 데이터 손실 가능! */
}

fclose 가 — buffer 의 flush 시 write 에 실패하면 EOF 반환. 그 데이터는 영원히 사라짐.

[fclose 의 흔한 에러]

  ✗ ENOSPC: 디스크 꽉 참 (가장 흔함!)
     buffer 의 데이터가 flush 안 됨
     
  ✗ EIO: I/O 에러
     디스크 손상 등
     
  ✗ EFBIG: 파일 크기 한도 초과
  
  특히 write-heavy 프로그램은 fclose 검사 필수
  (rsync, tar, 등은 모두 검사)

fflush 의 정확한 동작

// [user-space - C API]
int fflush(FILE *stream);
int fflush(NULL);                /* 모든 output stream */
[fflush 의 동작]

  output stream:
    buffer 의 데이터를 write syscall 로 flush
    write 가 끝나면 buffer reset
    fd 의 데이터는 _kernel buffer_ 에
    (디스크에 도달은 fsync 가 보장)
    
  input stream:
    C 표준상 UB
    glibc: buffer 무효화 (다음 read 는 새로 read syscall)
    
  fflush(NULL):
    모든 output stream 의 buffer flush
    _IO_list_all 순회

fflush 와 fsync 의 차이

매우 헷갈리는 부분.

[fflush vs fsync]

  fflush(fp):
    user-space buffer → kernel buffer
    아직 디스크에 안 갔을 수 있음
    프로세스 죽어도 데이터는 kernel 에 → OS 가 flush
    
  fsync(fd):
    kernel buffer → 실제 디스크
    매우 느림 (수 ms)
    전원 끊겨도 안전
    
  안전한 저장:
    fflush(fp);              /* user → kernel */
    fsync(fileno(fp));       /* kernel → disk */
    fclose(fp);

DB 의 commit 같은 — 데이터 영구 보장이 필요한 곳은 fsync 필수.

exit 시의 자동 처리

[프로세스 종료 시 stream 정리]

  exit():
    1. atexit 의 등록 함수 호출
    2. _IO_cleanup() — 모든 stream flush
    3. 각 stream 의 __finish 호출
    4. _exit syscall
    
  abort():
    이런 정리 안 함
    → buffer 의 데이터 손실 가능
    → 시그널 핸들러에서 abort 호출 시 주의
    
  _exit():
    아무 정리도 안 함
    → 빠른 종료
    → fork 후 child 에서 자주 사용

fcloseall — 비표준 확장

// [user-space - glibc 확장]
int fcloseall(void);

_IO_list_all 의 모든 stream 을 한 번에 close. POSIX 가 아님, glibc 확장.

// [user-space - C 예제]
#define _GNU_SOURCE
#include <stdio.h>

int main(void) {
    /* ... 여러 fopen ... */
    
    fcloseall();     /* 모두 close (stdin/stdout/stderr 도) */
    
    return 0;
}

조심: stdin, stdout, stderr 도 close됨. 이후 printf 안 됨.

tmpfile — 임시 파일 stream

// [user-space - C 표준]
FILE *tmpfile(void);

자동 삭제되는 임시 파일을 stream 으로 연다.

[tmpfile 의 동작]

  내부:
    1. /tmp 에 unique 이름 생성
    2. O_CREAT | O_EXCL | O_RDWR 로 open
    3. unlink 즉시 호출 — 파일이 디렉토리에서 사라짐
       (그러나 fd 가 열려 있는 동안은 데이터 존재)
    4. fdopen 으로 stream 만듬
    
  fclose 시:
    fd close → 마지막 reference → 파일 영구 삭제
    
  장점:
    - 자동 정리 (프로세스 죽어도 OS 가 삭제)
    - 다른 프로세스가 못 봄 (unlinked)
    - race condition 없음

setvbuf 의 시점 제약

setvbuf 는 — fopen 직후, 첫 I/O 전에만 호출 가능.

// [user-space - C 옳은 패턴]
FILE *fp = fopen("file", "r");
setvbuf(fp, buf, _IOFBF, BUFSIZ);    /* ✓ */
fread(...);                           /* 이후 I/O */

// [user-space - C 잘못된 패턴]
FILE *fp = fopen("file", "r");
fread(...);                           /* I/O 시작 */
setvbuf(fp, buf, _IOFBF, BUFSIZ);    /* ✗ UB! */

이유 — 첫 I/O 시 buffer 가 할당됨. 그 뒤에는 변경하면 일관성 깨짐.

fopen vs open — 비교

[fopen vs open(2)]

  fopen (stdio):
    ✓ buffer 자동 (성능)
    ✓ portable (Windows 등에서도)
    ✓ mode 문자열 (간편)
    ✓ printf/scanf 같이 사용
    ✗ 작은 read/write 의 약간 오버헤드
    ✗ 부분적인 동기화 (fsync 필요)
    
  open (syscall):
    ✓ 정밀한 flag (O_NONBLOCK, O_DIRECT, 등)
    ✓ 빠른 큰 I/O
    ✓ Linux 의 특수 기능 (epoll, mmap 등)
    ✗ buffer 직접 관리
    ✗ 작은 I/O 매번 syscall

대부분의 응용에서 — fopen 으로 충분. 고성능 server, DB 등은 — open 직접.

이 장의 결론

stream 의 생애 정리:

[fopen/fclose/fflush 정리]

  ✓ fopen: malloc + _IO_FILE 초기화 + open(2) + chain 에 추가
  ✓ mode: r/w/a + + + b (binary) + 확장 (e, c, m, x)
  ✓ glibc mode: 처음 7 글자만 검사
  ✓ fdopen: 기존 fd 를 stream 으로 wrap
  ✓ freopen: 기존 FILE* 재할당 (포인터 그대로)
  ✓ fclose: 5 단계 (unlink, flush+close, FINISH, codecvt 해제, free)
  ✓ fclose 반환값 검사 필수! (write 실패 시 데이터 손실)
  ✓ fflush: buffer → kernel (디스크 안 보장)
  ✓ fsync: kernel → 디스크 (영구 보장)
  ✓ exit 시 모든 stream 자동 flush
  ✓ tmpfile: 자동 삭제 임시 파일
  ✓ setvbuf 는 첫 I/O 전에만

다음 장에서는 — stdio 가 결국 어떻게 syscall 까지 도달하는지의 정확한 메커니즘을 본다. read/write syscall 의 wrapper 로서의 stdio 의 정체.

39stdio 와 syscall — read/write 의 wrapper

여기까지 본 모든 stdio 메커니즘 — buffer, format, FILE* — 가 결국 도달하는 곳은 **read(2) 와 write(2) syscall**. 이 장은 — stdio 의 시각에서 syscall 의 정체를 본다.

stdio 의 최종 목적지

[stdio 의 호출 체인]

  user code:
    fread(buf, ...)
       │
       ▼
  glibc:
    _IO_fread → _IO_sgetn → _IO_XSGETN → _IO_file_xsgetn
       │
       ▼
    _IO_file_underflow (buffer 비면)
       │
       ▼
    _IO_SYSREAD → fp->vtable->__read
       │
       ▼
    __read(fd, buf, size)
       │
       ▼
  kernel (syscall):
    sys_read → 실제 디스크/소켓/...

stdio 의 모든 buffer 와 추상화의 끝 — read(2) 와 write(2) syscall.

_IO_SYSREAD, _IO_SYSWRITE

master 의 매크로:

// [user-space - glibc libio/libioP.h master]
#define _IO_SYSREAD(FP, DATA, LEN)  JUMP2(__read, FP, DATA, LEN)
#define _IO_SYSWRITE(FP, DATA, LEN) JUMP2(__write, FP, DATA, LEN)
#define _IO_SYSSEEK(FP, OFFSET, MODE) JUMP2(__seek, FP, OFFSET, MODE)
#define _IO_SYSCLOSE(FP) JUMP1(__close, FP)

vtable 의 함수 호출. 일반 파일이면 — _IO_file_read, _IO_file_write 가 실제 syscall.

_IO_file_read — read syscall wrapper

// [user-space - glibc libio/fileops.c master 단순화]
ssize_t
_IO_file_read(FILE *fp, void *buf, ssize_t size)
{
    return (__builtin_expect(fp->_flags2 & _IO_FLAGS2_NOTCANCEL, 0)
            ? __read_nocancel(fp->_fileno, buf, size)
            : __read(fp->_fileno, buf, size));
}

핵심 — 단순한 read syscall wrapper. 하나의 read 호출.

  • _IO_FLAGS2_NOTCANCEL 가 set 이면 (mode "c") — pthread cancellation point 가 아님.
  • 그렇지 않으면 일반 read.

_IO_file_write — write syscall wrapper

write 는 약간 다르다 — short write 처리.

// [user-space - glibc libio/fileops.c master 단순화]
ssize_t
_IO_file_write(FILE *fp, const void *buf, ssize_t n)
{
    ssize_t to_do = n;
    while (to_do > 0) {
        ssize_t count = (__builtin_expect(fp->_flags2 & _IO_FLAGS2_NOTCANCEL, 0)
                         ? __write_nocancel(fp->_fileno, buf, to_do)
                         : __write(fp->_fileno, buf, to_do));
        if (count < 0) {
            fp->_flags |= _IO_ERR_SEEN;
            break;
        }
        to_do -= count;
        buf = (const char *)buf + count;
    }
    n -= to_do;
    return n;
}

핵심 차이 — loop 으로 short write 처리. write 가 요청한 만큼 다 쓰지 못할 수 있다 (signal interrupt, pipe full 등).

short read / short write 의 정체

read 와 write 가 — 요청한 양보다 적게 처리할 수 있다.

[short read 의 원인]

  ✓ pipe / socket / tty:
     상대방이 아직 안 보냄 → 일부만
     
  ✓ EOF:
     파일 끝 → 그 지점까지만
     
  ✓ signal interrupt (EINTR):
     read 중 signal → 일부만 읽고 반환
     
  ✓ O_NONBLOCK 인데 데이터 없음:
     EAGAIN / EWOULDBLOCK 반환, 0 byte
     
  ✓ kernel buffer 한도 (소켓 등):
     buffer 의 크기만큼만

[short write 의 원인]

  ✓ pipe full:
     상대방이 안 읽음 → PIPE_BUF 만큼만
     
  ✓ disk full:
     ENOSPC, 일부만 쓰고 멈춤
     
  ✓ signal:
     EINTR, 일부만 쓰고 반환
     
  ✓ socket buffer:
     송신 buffer 의 크기만큼만

stdio 가 — write 의 short write 를 자동 처리 (loop). read 는 안 함 (한 번만).

EINTR — signal 의 interrupt

[EINTR 의 발생]

  read syscall 중 signal 도착:
    옛 동작: read 가 -1 반환, errno = EINTR
    SA_RESTART flag: syscall 자동 재시작
    
  POSIX 의 안전한 처리:
    while ((n = read(fd, buf, size)) == -1 && errno == EINTR)
        ; /* 재시도 */
    
  glibc 의 stdio:
    SA_RESTART 가 보통 set → 알아서 재시도됨
    그러나 보장은 안 됨 → loop 필요

O_NONBLOCK 과 stdio 의 부조화

O_NONBLOCK (non-blocking I/O) 가 stdio 와 잘 안 맞는다.

// [user-space - C 위험한 패턴]
int fd = open("file", O_RDONLY | O_NONBLOCK);
FILE *fp = fdopen(fd, "r");
fread(buf, 1, 100, fp);    /* O_NONBLOCK 이라 즉시 반환 가능 */
                            /* 100 안 채워졌을 수 있음 */
[O_NONBLOCK 의 stdio 문제]

  stdio buffer 의 가정:
    "underflow 호출하면 데이터 옴"
    
  O_NONBLOCK 의 동작:
    "데이터 없으면 EAGAIN, 즉시 반환"
    
  부조화 결과:
    fread 가 partial 반환 (EOF 도 EAGAIN 도 아닌 묘한 상태)
    ferror 검사하면 EAGAIN
    
  현실:
    blocking I/O 가 stdio 와 잘 맞음
    non-blocking 은 raw read/write 사용
    또는 select/poll/epoll 로 ready 확인 후 stdio

write(2) 의 atomicity

POSIX 가 — write 의 atomicity 를 보장하는 경우:

[write atomicity]

  pipe (PIPE_BUF 이하):
    PIPE_BUF (보통 4096) 이하의 write 는 atomic
    → 다른 writer 의 데이터와 안 섞임
    /usr/include/limits.h 의 PIPE_BUF
    
  socket TCP:
    atomicity 보장 안 됨 (split, merge 가능)
    
  regular file:
    POSIX 표준상 atomic
    그러나 Linux 는 _과거_ 에 부정확 — kernel 5.0+ 부터 보장

multi-process 가 같은 파일/pipe 에 쓸 때 — PIPE_BUF 이하면 안전. 큰 데이터는 lock 필요.

pwrite, pread — 위치 지정

// [user-space - POSIX]
ssize_t pread(int fd, void *buf, size_t count, off_t offset);
ssize_t pwrite(int fd, const void *buf, size_t count, off_t offset);
[pread / pwrite 의 이점]

  ✓ atomic 한 seek + read/write
     lseek + read 의 race condition 없음
     
  ✓ multi-thread 에서 같은 fd 동시 사용 가능
     각 thread 가 다른 offset 으로
     
  ✓ DB, file server 같은 응용에 필수
  
  stdio 는 이걸 활용 안 함 — raw fd 만

O_DIRECT — kernel buffer 우회

매우 특수한 flag — kernel 의 page cache 우회.

// [user-space - C 예제]
int fd = open("data.bin", O_RDONLY | O_DIRECT);
/* read/write 가 kernel page cache 거치지 않고 직접 디스크 */
[O_DIRECT 의 trade-off]

  ✓ DB 처럼 자체 cache 가진 응용:
     double-buffering 회피
     I/O 동작 예측 가능
     
  ✗ 제약:
     buffer alignment 필수 (512 byte, page 등)
     size 도 alignment
     보통 응용엔 매우 느림
     
  stdio 의 buffer 와 충돌:
     setvbuf 의 buffer 가 align 안 되면 동작 안 함
     일반적으로 raw read/write 와만 사용

sendfile — zero-copy

큰 파일을 효율적으로 복사하는 방법.

// [user-space - Linux API]
#include <sys/sendfile.h>
ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);
[sendfile 의 효과]

  일반적인 file copy:
    read(in_fd, buf, n)  → user buffer
    write(out_fd, buf, n) → kernel 다시
    → 데이터가 user space 거침 (memory copy)
    
  sendfile:
    kernel 안에서 직접 in_fd → out_fd
    → memory copy 없음 (zero-copy)
    → 매우 빠름 (특히 큰 파일)
    
  web server (nginx 등) 의 핵심 기법
  → 파일을 socket 으로 효율 전송

stdio 는 — sendfile 추상화 없음. 직접 syscall.

splice, tee, vmsplice — 더 정교한 zero-copy

[Linux 의 고급 I/O syscall]

  splice(in_fd, ..., out_fd, ..., n):
    두 fd 사이 (둘 중 하나는 pipe) zero-copy 이동
    
  tee(in_fd, out_fd, n):
    pipe → pipe 의 복제 (consume 없이)
    
  vmsplice(fd, iov, ...):
    user memory → pipe (zero-copy)
    
  → 모두 매우 특수 용도
  → 일반 응용은 사용 안 함

iovec 와 vectored I/O

여러 buffer 를 한 번에 read/write.

// [user-space - POSIX]
#include <sys/uio.h>

struct iovec {
    void  *iov_base;
    size_t iov_len;
};

ssize_t readv(int fd, const struct iovec *iov, int iovcnt);
ssize_t writev(int fd, const struct iovec *iov, int iovcnt);
// [user-space - C 예제]
struct iovec iov[3];
iov[0].iov_base = header;    iov[0].iov_len = sizeof(header);
iov[1].iov_base = body;      iov[1].iov_len = body_len;
iov[2].iov_base = footer;    iov[2].iov_len = sizeof(footer);

writev(fd, iov, 3);          /* 한 번의 syscall 로 3 buffer 다 씀 */
[writev 의 이점]

  ✓ HTTP response 처럼 header + body 한 번에
  ✓ syscall 한 번 → 컨텍스트 스위치 절감
  ✓ Atomic 한 write (PIPE_BUF 이하면)

stdio 는 — iovec 추상화 없음. format string 으로 부분적 효과만.

posix_fadvise — kernel 에게 힌트

// [user-space - POSIX]
#include <fcntl.h>
int posix_fadvise(int fd, off_t offset, off_t len, int advice);

/* advice:
   POSIX_FADV_NORMAL
   POSIX_FADV_SEQUENTIAL    — 순차 읽기
   POSIX_FADV_RANDOM        — 랜덤 접근
   POSIX_FADV_NOREUSE       — 한 번만 (cache 안 함)

// [user-space - C 예제]
int fd = open("huge.log", O_RDONLY);
posix_fadvise(fd, 0, 0, POSIX_FADV_SEQUENTIAL);
/* kernel 이 더 적극적 readahead */

FILE *fp = fdopen(fd, "r");
/* 빠른 순차 처리 */

stdio 의 명확한 한계

stdio 가 잘 안 맞는 경우:

[stdio 를 안 쓰는 경우]

  ✗ 비동기 I/O (epoll, io_uring)

Linux 5.1 (2019) 의 io_uring — 비동기 I/O 의 혁신.

[io_uring 의 특징]

  ✓ syscall 거의 없음 (shared ring buffer)

40error handling — errno, ferror, feof

stdio 의 함수들이 실패 시 — 어떻게 알리고, 어떻게 검사하나? errno, ferror, feof 의 메커니즘과 이들 사이의 미묘한 관계.

두 가지 에러 통보 방식

stdio 가 — 에러를 두 곳에 기록.

[stdio 의 두 가지 에러 통보]

  1. errno (thread-local 변수):
     모든 syscall, 많은 libc 함수가 사용
     마지막 실패의 errno 값
     성공 시에는 변경 _안 됨_ (단 그 보장은 함수마다 다름)
     
  2. FILE * 의 _flags (per-stream):
     _IO_ERR_SEEN  : 에러 발생
     _IO_EOF_SEEN  : EOF 도달
     ferror, feof 로 검사
     clearerr 로 reset

두 곳이 — 다른 정보. errno 는 _마지막_ 에러의 원인. ferror 는 _그 stream_ 에 에러가 _있었는지_.

errno 의 정체

errno 는 — 사용자에겐 글로벌 변수처럼 보이지만, 실제는 thread-local.

// [user-space - C 표준]
#include <errno.h>

extern int errno;

사실은 — __errno_location() 함수의 reference.

// [user-space - glibc bits/errno.h master]
#ifndef __ASSEMBLER__
extern int *__errno_location(void) __THROW __attribute_const__;
# define errno (*__errno_location())
#endif
[errno 의 thread-local 처리]

  옛 C (single thread):
    extern int errno;     /* 진짜 글로벌 */
    
  현대 POSIX (multi-thread):
    #define errno (*__errno_location())
    
    __errno_location() 가 thread-local 변수의 주소 반환
    각 thread 가 자기만의 errno
    
  glibc 의 구현:
    TLS (Thread Local Storage) 의 변수
    __thread int __libc_errno;

이 디자인이 — multi-thread 안전성의 기반. 다른 thread 의 errno 가 안 섞임.

errno 의 미묘한 규칙

성공 시 errno 가 변경되는 경우가 있다 — 함수의 _내부_ 에서 다른 syscall 이 실패해도.

// [user-space - C 흔한 함정]
errno = 0;
printf("hello\n");
if (errno != 0) {
    /* 실패? — _아닐 수도_ ! */
    /* printf 가 내부에서 isatty 같은 검사를 하는데
       그게 ENOTTY 반환할 수 있음
       printf 자체는 성공해도 errno = ENOTTY 가 남음 */
}

ERR01-C (CMU SEI) 의 권고:

[errno 의 올바른 사용]

  ✗ 잘못된 패턴:
    errno = 0;
    library_function();
    if (errno != 0) /* error */
    
  ✓ 올바른 패턴:
    int ret = library_function();
    if (ret == -1) {              /* 함수 반환값이 에러 표시 */
        /* errno 검사 */
    }
    
  → errno 는 _함수가 명시적으로 실패 표시한 후에만_ 의미
  → 성공 케이스에 errno 검사 X

stdio 의 함수는 — 실패 시 보통 NULL (fopen), EOF (fclose, putc), 또는 부분 반환 (fread). 그 시점에 errno 의미.

stdio 의 stream 에러 비트

stream 의 _flags 에:

// [user-space - glibc libio/libio.h master]
#define _IO_EOF_SEEN  0x0010
#define _IO_ERR_SEEN  0x0020

이 비트들이 — 그 stream 에 EOF 또는 에러가 한 번이라도 있었는지 표시. 누적된다 (clearerr 까지).

feof — EOF 검사

// [user-space - C API]
int feof(FILE *stream);
// [user-space - glibc libio/feof.c master 단순화]
int
_IO_feof(FILE *fp)
{
    int result;
    CHECK_FILE(fp, EOF);
    _IO_acquire_lock(fp);
    result = _IO_feof_unlocked(fp);
    _IO_release_lock(fp);
    return result;
}

/* libio/libio.h 의 macro */
#define _IO_feof_unlocked(__fp) (((__fp)->_flags & _IO_EOF_SEEN) != 0)

단순한 비트 검사. _IO_EOF_SEEN 비트가 set 이면 1.

ferror — 에러 검사

// [user-space - C API]
int ferror(FILE *stream);
// [user-space - glibc 매크로]
#define _IO_ferror_unlocked(__fp) (((__fp)->_flags & _IO_ERR_SEEN) != 0)

마찬가지로 비트 검사.

clearerr — 비트 reset

// [user-space - C API]
void clearerr(FILE *stream);
// [user-space - glibc 매크로]
#define _IO_clearerr(__fp) ((__fp)->_flags &= ~(_IO_ERR_SEEN | _IO_EOF_SEEN))

두 비트 모두 reset.

feof 의 흔한 오류

가장 흔한 패턴 — **while (!feof(fp))** 이 거의 항상 잘못된 패턴.

// [user-space - C 잘못된 패턴]
while (!feof(fp)) {
    fscanf(fp, "%d", &n);     /* feof 는 _다음_ 시도 후 set */
    process(n);                /* 마지막 partial 도 처리 */
}
[feof 의 함정]

  핵심: feof 는 _과거에 EOF 만났는지_ 표시
  → fread 가 EOF 만나기 _전_ 에는 false
  → loop 의 첫 시도는 항상 진행
  → 마지막 fread 가 부분 또는 실패해도, 그 다음 feof 만 true
  → 이미 invalid 데이터를 처리한 후
  
  올바른 패턴:
    while (fread(buf, 1, sizeof(buf), fp) > 0) {
        process(buf);
    }
    /* loop 끝 시점:
       fread 가 0 또는 -1 반환했음 — feof 또는 ferror 검사 */
    if (ferror(fp)) handle_error();
    /* feof 면 정상 종료 */

같은 원리가 — getchar:

// [user-space - C 잘못]
char c;
while ((c = getchar()) != EOF) { ... }
/* c 가 char (signed). EOF (-1) 를 char 에 저장 시 정보 손실 */
/* 0xFF byte 도 EOF 로 인식 가능 */

// [user-space - C 올바름]
int c;
while ((c = getchar()) != EOF) { ... }
/* int 로 받기 — getchar 의 반환형이 int 인 이유 */

strerror 와 perror — 사람 친화적 에러 메시지

// [user-space - C API]
char *strerror(int errnum);                   /* errno → 영문 메시지 */
void perror(const char *s);                    /* "s: <errno 메시지>" 를 stderr */
// [user-space - C 예제]
FILE *fp = fopen("/nonexistent", "r");
if (fp == NULL) {
    perror("fopen");
    /* "fopen: No such file or directory" */
    
    /* 또는: */
    fprintf(stderr, "fopen failed: %s\n", strerror(errno));
}

strerror 는 — locale 의존. 한국어 locale 이면 한국어 메시지.

strerror_r — thread-safe 변종

strerror 는 — 정적 buffer 사용 (thread-unsafe 가능). POSIX 의 thread-safe 변종:

// [user-space - POSIX]
int strerror_r(int errnum, char *buf, size_t buflen);   /* XSI 버전 */
char *strerror_r(int errnum, char *buf, size_t buflen);  /* GNU 버전 */

복잡한 GNU vs XSI 의 차이 — feature_test_macros 따라 다름.

errno 의 thread safety

errno 가 thread-local 이라 — 다른 thread 의 errno 영향 안 받음.

// [user-space - C 예제]
void *worker(void *arg) {
    fopen("/nonexistent", "r");
    /* 이 thread 의 errno = ENOENT */
    return NULL;
}

int main(void) {
    pthread_t t;
    pthread_create(&t, NULL, worker, NULL);
    pthread_join(t, NULL);
    
    /* main thread 의 errno 는 영향 없음 */
    printf("%d\n", errno);    /* worker 의 errno 아님 */
}

ferror vs errno — 어느 것을 검사?

긴 일련의 stdio 호출 후 — ferror 가 더 신뢰할 만.

[ferror 가 더 좋은 이유]

  errno:
    각 syscall 마다 변경 가능 (성공 시도)
    여러 호출 후엔 마지막 호출의 _부수효과_ errno 일 수 있음
    
  ferror:
    그 stream 에 에러 났는지 누적 표시
    명확한 stream 수준 검사
    
  권장:
    fread/fwrite 등 stdio 함수 반환값 검사
    ferror 로 에러 여부 확인
    errno 는 _바로 그 호출_ 직후만 신뢰

setbuf 의 무서운 사실

setbuf 는 — 에러 반환 안 함!

// [user-space - C 함정]
char buf[100];
setbuf(fp, buf);
/* 만약 실패해도 알 방법 없음 */

같은 이유로 setvbuf 권장 — 반환값으로 에러 표시.

EOF 의 묘한 정의

EOF 가 — int 의 -1.

// [stdio.h]
#define EOF (-1)
[EOF 의 의미]

  반환 의미:
    getchar() == EOF  → 입력 끝
    putc() == EOF     → 출력 에러
    fclose() == EOF   → close 에러
    
  주의:
    char 변수에 받지 말 것 (signed char 의 0xFF == EOF 와 충돌)
    int 로 받아야 함
    
  실제 byte 와 EOF 구분:
    if (c == EOF) {
        if (feof(stdin)) /* 정상 EOF */
        else if (ferror(stdin)) /* I/O 에러 */
    }

인터럽트 시 EOF — 묘한 경계 케이스

// [user-space - C 예제]
int c;
while ((c = getchar()) != EOF) {
    /* signal (예: SIGINT) 도착 시 — read 가 EINTR 반환 */
    /* glibc 이 자동 retry 또는 EOF 반환 (config 따라) */
}

/* 안전한 처리: */
clearerr(stdin);
while ((c = getchar()) != EOF) {
    ...
}
if (ferror(stdin)) {
    if (errno == EINTR) /* signal */
    else if (errno == EIO) /* 진짜 I/O error */
}

exit 시 자동 에러 처리

[종료 시 누적 에러 검사]

  exit() 의 _IO_cleanup 가 모든 FILE flush
  fclose 의 반환값 — exit 가 _무시_ ★
  → ENOSPC 같은 마지막 에러가 사라짐!
  
  대안:
    atexit 에 등록한 함수에서 ferror(stdout), fclose 검사
    또는 main 에서 명시적 fclose
    
  best practice:
    if (fclose(stdout) != 0) {
        fprintf(stderr, "stdout: %s\n", strerror(errno));
        exit(EXIT_FAILURE);
    }

coreutils (예: cat, cp) 들은 — 명시적으로 stdout 의 close 검사. 데이터 손실 방지.

__libc_errno 와 syscall

syscall 의 errno 처리 메커니즘:

[syscall 의 errno 설정]

  syscall 의 반환값:
    음수면 -errno (예: -ENOENT = -2)
    glibc 가 음수 보고 errno 설정 + -1 반환
    
  glibc 의 syscall wrapper:
    ssize_t read(int fd, void *buf, size_t n) {
        ssize_t r = SYSCALL(read, fd, buf, n);
        if (r < 0) {
            errno = -r;
            return -1;
        }
        return r;
    }

따라서 — **syscall 직접 호출 (예: syscall(SYS_read, ...)) 에서는 errno 가 자동 설정 안 됨**. 사용자가 직접 검사.

이 장의 결론

stdio 에러 처리 정리:

[error handling 정리]

  ✓ errno: thread-local (실제로는 __errno_location)
  ✓ stream 의 _flags 에 _IO_ERR_SEEN, _IO_EOF_SEEN
  ✓ feof, ferror: 비트 검사 (누적)
  ✓ clearerr: 비트 reset
  ✓ ferror 가 errno 보다 신뢰 (긴 호출 후)
  ✓ 함정:
    - while (!feof(fp)) 안티패턴
    - char c = getchar() 의 EOF 손실
    - 성공 케이스 errno 검사
  ✓ strerror, perror 로 사람 친화적 메시지
  ✓ strerror_r 는 thread-safe (GNU vs XSI 차이)
  ✓ fclose 반환값 검사 필수 (ENOSPC 등 마지막 에러)
  ✓ syscall 직접 호출 시 errno 직접 설정

다음 장에서는 — stdio 의 가장 복잡한 부분, locale 와 wide character I/O 의 세계를 본다.

41locale 의 영향과 wide character I/O

stdio 가 처리하는 것 — 바이트 stream. 그런데 사용자는 — 글자를 다룬다. ASCII 면 byte = 글자, 그런데 한글, 일본어, 이모지 같은 건? 한 글자가 여러 byte. 이 차이가 — locale 와 wide character I/O 의 세계.

locale — 글자, 숫자, 시간의 표현 방식

// [user-space - C API]
#include <locale.h>
char *setlocale(int category, const char *locale);

locale 이 — 프로그램의 culture-aware 동작을 결정.

[locale 의 categories]

  LC_COLLATE  : 문자열 정렬 순서
                "Ä" 가 "A" 와 같은 등급인지, "Z" 다음인지
                
  LC_CTYPE    : 문자 분류 (isalpha, isdigit 등)
                multi-byte encoding (UTF-8 등)
                
  LC_MONETARY : 통화 표시 ($1,234.56 vs 1.234,56 €)
  
  LC_NUMERIC  : 숫자 표시 (1,234.56 vs 1.234,56)
                printf %f 의 소수점 문자
                
  LC_TIME     : 시간 형식 (1/15 vs 15.1.)
                strftime, 요일/월 이름
                
  LC_MESSAGES : 에러 메시지의 언어
                "No such file" vs "파일 또는 디렉터리 없음"
                
  LC_ALL      : 위 모두 한번에

setlocale 의 동작

// [user-space - C 예제]
#include <locale.h>
#include <stdio.h>

int main(void) {
    /* 시작 시 항상 "C" locale */
    
    /* 환경변수 LANG, LC_* 에 따라 설정 */
    setlocale(LC_ALL, "");
    
    /* 명시적 locale */
    setlocale(LC_NUMERIC, "de_DE.UTF-8");
    
    printf("%f\n", 3.14);       /* "3,140000" — comma 가 소수점! */
    
    return 0;
}
[setlocale 의 매개변수]

  category: LC_ALL, LC_CTYPE, LC_NUMERIC, ...
  locale:
    NULL:    현재 locale 반환만 (변경 X)
    "":      환경변수 따라 (LANG, LC_*)
    "C":     기본 "C" locale (ASCII)
    "POSIX": "C" 의 alias
    "en_US.UTF-8": 명시적 locale
    
  반환값:
    성공 시: 설정된 locale 이름 (char *)
    실패 시: NULL (그 locale 이 system 에 없음)

"C" locale 의 기본

프로그램 시작 시 — 항상 "C" locale.

["C" locale 의 특징]

  ✓ ASCII 만 (0~127)
  ✓ isalpha 가 a-z, A-Z 만 (한글, 이모지 X)
  ✓ 정렬은 byte 순서
  ✓ 소수점 = '.'
  ✓ 통화 표시 X
  ✓ 시간 형식 = 영어 미국식 (간단)
  ✓ 메시지 영문
  
  단점:
    - 비ASCII 문자열 처리 안 됨
    - "한글" 같은 string 은 wchar 변환 안 됨 (mbtowc 실패)
    
  대부분의 응용은 시작 시 setlocale(LC_ALL, "") 호출

locale 의 printf 영향

// [user-space - C 예제]
setlocale(LC_NUMERIC, "de_DE.UTF-8");
printf("%f\n", 3.14);            /* "3,140000" */

setlocale(LC_NUMERIC, "C");
printf("%f\n", 3.14);            /* "3.140000" */

/* "'" flag — thousands separator */
printf("%'d\n", 1234567);        /* "C" locale: 그냥 "1234567"
                                    "en_US": "1,234,567"
                                    "de_DE": "1.234.567" */

이런 locale 의존 — 종종 버그의 원인. JSON 같은 portable format 만들 때:

// [user-space - C 함정]
setlocale(LC_ALL, "");          /* en_US 환경 */
char json[100];
sprintf(json, "{\"value\": %f}", 3.14);  /* {"value": 3.140000} ✓ */

setlocale(LC_ALL, "de_DE");
sprintf(json, "{\"value\": %f}", 3.14);  /* {"value": 3,140000} ✗ */
                                          /* invalid JSON! */

해결 — 임시로 "C" locale 설정 또는 printf_l (POSIX 2008 의 locale-specific):

// [user-space - POSIX 2008]
locale_t c_locale = newlocale(LC_ALL_MASK, "C", (locale_t)0);
sprintf_l(json, c_locale, "{\"value\": %f}", 3.14);  /* 항상 '.' */
freelocale(c_locale);

multi-byte 문자 — UTF-8

UTF-8 의 인코딩 규칙:

[UTF-8 인코딩]

  ASCII (0x00 ~ 0x7F):
    0xxxxxxx  (1 byte)
    
  Latin extended (0x80 ~ 0x7FF):
    110xxxxx 10xxxxxx  (2 bytes)
    
  Basic CJK (0x800 ~ 0xFFFF):
    1110xxxx 10xxxxxx 10xxxxxx  (3 bytes)
    
  Supplementary (0x10000 ~ 0x10FFFF):
    11110xxx 10xxxxxx 10xxxxxx 10xxxxxx  (4 bytes)
    
  예:
    '한' = U+D55C
    UTF-8: 0xED 0x95 0x9C (3 bytes)
    
  '🎉' = U+1F389
    UTF-8: 0xF0 0x9F 0x8E 0x89 (4 bytes)

UTF-8 의 장점 — ASCII 와 호환, byte 처리 라이브러리들이 대체로 동작 (단 byte != 글자).

wchar_t — wide character

C 에 — 글자를 표현하는 타입.

// [user-space - C 표준]
typedef int wchar_t;     /* glibc 의 정의 */
[wchar_t 의 크기]

  Linux/glibc: 4 bytes (UTF-32 / UCS-4 code point)
  Windows:    2 bytes (UTF-16 code unit)
  
  → 글이 잘 안 맞춤. 이식성 문제
  
  대안 (C11):
    char16_t (UTF-16 code unit)
    char32_t (UTF-32 code point)

byte ↔ wide 변환

// [user-space - C 표준 API]
#include <wchar.h>
size_t mbrtowc(wchar_t *pwc, const char *s, size_t n, mbstate_t *ps);
size_t wcrtomb(char *s, wchar_t wc, mbstate_t *ps);

mbstate_t — 변환 상태 (multi-byte sequence 의 중간 상태).

// [user-space - C 예제]
#include <locale.h>
#include <wchar.h>
#include <stdio.h>

int main(void) {
    setlocale(LC_CTYPE, "en_US.UTF-8");
    
    const char *s = "한글";      /* UTF-8: 6 bytes */
    wchar_t wc;
    mbstate_t state = {0};
    
    while (*s) {
        size_t consumed = mbrtowc(&wc, s, MB_CUR_MAX, &state);
        printf("U+%04X (%zd bytes)\n", (unsigned)wc, consumed);
        s += consumed;
    }
    /* 출력:
       U+D55C (3 bytes)
       U+AE00 (3 bytes) */
}

wide character I/O 함수들

[wide character I/O 함수들]

  fputwc(wc, fp)      ←→ fputc
  fgetwc(fp)          ←→ fgetc
  fputws(ws, fp)      ←→ fputs
  fgetws(buf, n, fp)  ←→ fgets
  
  wprintf(fmt, ...)   ←→ printf
  fwprintf(fp, ...)   ←→ fprintf
  swprintf(buf, ...)  ←→ snprintf
  
  wscanf(fmt, ...)    ←→ scanf
  fwscanf(fp, ...)    ←→ fscanf
  swscanf(s, ...)     ←→ sscanf
  
  ungetwc(wc, fp)     ←→ ungetc
  putwchar(wc)        ←→ putchar
  getwchar()          ←→ getchar

사용 예

// [user-space - C 예제]
#include <stdio.h>
#include <wchar.h>
#include <locale.h>

int main(void) {
    setlocale(LC_ALL, "");
    
    /* wide string literal — L 접두사 */
    wchar_t *greeting = L"안녕하세요";
    
    /* wide character 출력 */
    wprintf(L"메시지: %ls\n", greeting);
    
    /* wide character 입력 */
    wchar_t buf[100];
    fgetws(buf, 100, stdin);
    
    /* wprintf 의 %lc, %ls 가 wide */
    /* printf 의 %lc, %ls 도 작동 (자동 변환) */
    
    return 0;
}

L"..." 가 — wide string literal. wchar_t 의 배열.

stream orientation — byte 또는 wide

각 FILE * 가 — orientation 을 가진다.

[stream orientation]

  fopen 직후: orientation 없음 (undecided)
  
  첫 사용 시 결정:
    fread, fwrite, fprintf 등 byte 함수 → byte oriented
    fputwc, fwprintf 등 wide 함수 → wide oriented
    
  ★ 한 번 정해지면 변경 불가!
  ★ 혼합 사용 시 UB
  
  fwide(fp, mode) 로 명시적 설정:
    mode > 0 → wide 시도
    mode < 0 → byte 시도
    mode == 0 → 현재 상태만 반환
// [user-space - C 예제]
FILE *fp = fopen("out.txt", "w");

fprintf(fp, "hello\n");      /* byte oriented */

fwprintf(fp, L"안녕\n");      /* ★ UB! 이미 byte oriented */

이 제약 때문에 — 응용은 보통 한 종류만 사용. mixing 안 함.

fwide 의 미묘함

// [user-space - C 예제]
FILE *fp = fopen("file", "w");
int ori = fwide(fp, 0);          /* 0 → 안 바꿈, 현재 상태 */
/* ori == 0 (아직 안 정해짐) */

fwide(fp, 1);                     /* wide 로 시도 */
ori = fwide(fp, 0);
/* ori > 0 (wide oriented) */

fwide(fp, -1);                    /* byte 로 바꾸려 시도 */
ori = fwide(fp, 0);
/* ori 는 여전히 > 0 — _못 바꿈_ */

_IO_FILE 의 wide 관련 필드

33장에서 본 wide 관련 필드:

struct _IO_FILE {
    ...
    struct _IO_codecvt *_codecvt;       /* 변환 함수 */
    struct _IO_wide_data *_wide_data;    /* wide buffer */
    ...
    int _mode;                            /* orientation */
};
[_mode 의 값]

  0  : orientation 없음
  > 0: wide oriented
  < 0: byte oriented
  
  내부적으로 _wide_data 의 buffer 사용 (별도 buffer)
  byte buffer 와 wide buffer 는 독립적

wide I/O 의 buffer

wide I/O 의 buffer 는 — byte buffer 와 다르다.

[wide buffer 의 동작]

  fwrite_unlocked(wide string):
    1. _wide_data 의 buffer 에 wchar_t 그대로 저장
    2. flush 시점에:
       - codecvt 호출 (wide → multi-byte)
       - multi-byte 의 결과를 _IO_buf_base 로
       - byte buffer 의 normal write
       
  → wide buffer + byte buffer 두 단계
  → 약간의 오버헤드

locale-aware 정렬 — strcoll

// [user-space - C API]
int strcoll(const char *s1, const char *s2);
size_t strxfrm(char *dest, const char *src, size_t n);

strcmp 는 — byte 순서 비교. strcoll 은 — locale 의 collation 순서.

// [user-space - C 예제]
setlocale(LC_COLLATE, "de_DE.UTF-8");

strcmp("Ä", "B");      /* > 0 (Ä = 0xC3 0x84, B = 0x42, byte 비교) */
strcoll("Ä", "B");     /* < 0 (de 에서 Ä 가 A 와 같이 정렬, B 앞에) */

strxfrm — 정렬 가능한 key 로 변환. 대량 정렬 시 효율적.

시간 형식 — strftime

// [user-space - C 예제]
#include <time.h>

struct tm *tm = localtime(&now);
char buf[100];

// "Friday, May 29"

환경변수와 locale

[locale 결정 환경변수 우선순위]

  1. LC_ALL (있으면 모두 override)
  2. LC_<category> (LC_CTYPE, LC_NUMERIC, ...)
  3. LANG (default fallback)
  4. "C" (default)
  
  예:
    LANG=en_US.UTF-8 → 모두 en_US.UTF-8
    LC_NUMERIC=de_DE LANG=en_US → LC_NUMERIC 만 de_DE, 나머지 en_US
    LC_ALL=C LANG=ko_KR → 모두 C

locale 의 함정

[locale 관련 흔한 함정]

  ✗ setlocale 안 하고 wide I/O
    → "C" locale 의 ASCII 만 처리
    → multi-byte string 의 mbrtowc 가 실패
    
  ✗ printf 의 %f 가 locale 따라 다름
    → JSON, CSV 등 portable format 시 위험
    
  ✗ strcmp 대신 strcoll 필요할 때 안 씀
    → 다국어 정렬 잘못
    
  ✗ thread 마다 다른 locale?
    → 옛 setlocale 은 thread-unsafe
    → POSIX 2008: uselocale + locale_t (thread-local)
    
  ✗ stream orientation 변경 시도
    → 한 번 정해지면 못 바꿈
    
  ✗ ASCII 만 가정한 코드
    → "café" 의 길이가 4 글자, 5 byte
    → strlen("café") = 5

42stdio 구현체 비교 — glibc vs musl

Part IV 의 마지막 장이 — malloc 구현체들의 비교였다. Part V 도 같은 패턴으로 마무리 — stdio 의 구현체 비교. 같은 표준 API 가 — 각 libc 에서 어떻게 다르게 구현되어 있는지.

비교할 구현체들

[주요 libc 의 stdio 구현체]

  1. glibc (GNU C Library)
     사용처: 대부분의 Linux 배포판
     크기: 매우 큼 (libio 가 약 30K LOC)
     역사: 1987 ~ (40 년)
     라이센스: LGPL
     
  2. musl libc
     사용처: Alpine, embedded Linux
     크기: 작음 (stdio 가 약 5K LOC)
     역사: 2011 ~
     라이센스: MIT
     
  3. FreeBSD libc / macOS libSystem
     사용처: FreeBSD, macOS
     크기: 중간
     역사: BSD 의 직계 (1970s)
     라이센스: BSD
     
  4. Bionic
     사용처: Android
     크기: 중간
     역사: 2008 ~
     라이센스: BSD
     
  5. Microsoft UCRT
     사용처: Windows
     크기: 큼
     역사: 1980s ~
     라이센스: proprietary

이 장은 — 주로 glibc vs musl 의 차이를 본다. 가장 자주 비교되는 페어.

비교 측면 1: 디자인 철학

[디자인 철학 비교]

  glibc:
    - 기능 완벽성 우선
    - C++ streambuf 의 영향 (libio)
    - 확장 기능 많음 (register_printf_specifier, fopencookie, ...)
    - 큰 코드 베이스
    - "있는 것은 다 지원"
    
  musl:
    - 단순함 우선
    - "표준만 구현, 더는 안 함"
    - 정확성 우선
    - 작은 코드 베이스
    - "복잡하면 버그 많음"

이 차이가 — 모든 측면에서 드러난다.

비교 측면 2: FILE struct 의 크기

[FILE 구조체 비교]

  glibc 의 _IO_FILE:
    필드 약 30 개
    크기 약 200 bytes (x86_64)
    vtable, _wide_data, codecvt 등
    
  musl 의 FILE (struct _IO_FILE):
    필드 약 15 개
    크기 약 130 bytes
    훨씬 단순
    
  주요 차이:
    glibc: backup buffer, vtable, wide data, ABI padding
    musl: read/write buffer, lock, fd 만

musl 의 FILE 의 단순함이 — 코드 크기와 보안 두 측면에서 이점.

비교 측면 3: vtable 사용

[vtable 사용 비교]

  glibc:
    각 FILE 이 vtable 가짐
    file, pipe, memstream, strstream 등 각자 다른 vtable
    fopen 의 추상화 강함
    그러나 — vtable hijacking 공격 표면
    
  musl:
    vtable 없음
    각 stream 의 read/write/close 가 직접 함수 포인터로
    더 단순, 공격 표면 작음
    그러나 — fopencookie 같은 확장은 없음

glibc 의 vtable 이 — __overflow, __underflow, __xsputn, __xsgetn 등을 추상화. musl 은 직접 호출.

비교 측면 4: buffer 알고리즘

[buffering 알고리즘 비교]

  glibc:
    - 옛 디자인 (C++ streambuf 의 6 pointer)
    - read_ptr / read_end / read_base / write_base / write_ptr / write_end
    - putback buffer 별도
    - 복잡한 상태 머신
    
  musl:
    - 단순 (3 pointer)
    - buf / pos / end
    - putback 은 별도 1 글자만
    - 명확한 의도

musl 의 독특한 트릭 — readv/writev 활용:

[musl 의 영리한 디자인]

  fread 의 큰 요청:
    glibc: buffer 채우고 → 사용자 buffer 로 복사
           buffer 부족 시 다시 read
           multiple memcpy + multiple read
           
    musl: readv 로 한 번에:
          iov[0] = 남은 buffer 의 데이터
          iov[1] = 사용자 buffer
          → 한 syscall 로 buffer 비우기 + 사용자 buffer 채우기
          → memcpy 없음, syscall 1 번

이 디자인이 — musl 이 큰 I/O 에서 glibc 보다 빠를 수 있는 이유 중 하나.

비교 측면 5: EOF 처리의 정확성

매우 흥미로운 차이 — EOF 의 sticky 동작.

[ISO C 와 POSIX 의 요구]

  EOF status 는 sticky 이어야 함:
    한 번 EOF 가 set 되면, 
    다음 read 도 EOF 반환 (새 입력 없어야)
    clearerr 로만 reset 가능
    
  musl: 항상 준수
  glibc: 2.28 이전 — 무시! (새 입력 있으면 반환)
         2.28 이후 — 표준 준수

이게 — 인터랙티브 입력 처리에 미묘한 영향. tty 에서 Ctrl+D 후 다시 입력해도 — glibc 옛 버전은 새 입력 받음, musl 은 안 받음.

비교 측면 6: 코드 크기와 binary 크기

[binary 크기 비교 — hello world]

  정적 링크:
    glibc:  ~750 KB (printf 한 줄)
    musl:   ~25 KB  (printf 한 줄)
    
  동적 링크:
    glibc:  ~10 KB  + libc.so 의 부담
    musl:   ~10 KB  + ld-musl.so 의 부담
    
  → embedded 시스템 (Alpine 의 컨테이너 등) musl 매우 유리
  → musl Docker 이미지 (alpine:latest) 가 5 MB 정도
  → glibc 기반 (debian:slim) 은 70 MB 이상

비교 측면 7: 확장 기능

glibc 가 — 매우 많은 확장.

[glibc 만의 stdio 확장]

  ✓ register_printf_specifier — custom format
  ✓ fopencookie — 임의 stream 구현 (read/write callback)
  ✓ open_memstream — 동적 메모리 buffer stream
  ✓ open_wmemstream — wide character 메모리 stream
  ✓ fmemopen 의 확장 ('w' mode 등)
  ✓ dprintf, vdprintf — fd 직접
  ✓ asprintf, vasprintf — 동적 buffer
  ✓ obstack_printf
  ✓ "%a" format (hexadecimal float)
  ✓ printf 의 ' flag (thousands separator)
  ✓ "%I" flag (locale-specific digits)
  
  musl 의 stance:
    "POSIX 와 일부 GNU 확장 (asprintf 등) 만"
    "register_printf_specifier 같은 위험한 건 안 함"

비교 측면 8: 성능 (벤치마크)

[stdio 성능 비교 (대략)]

  printf "%d\n" × 1M:
    glibc:  ~350 ms
    musl:   ~250 ms       ← 단순함이 성능
    
  fread (1 byte × 1M):
    glibc:  ~30 ms
    musl:   ~25 ms
    
  fread (1 MB block × 100):
    glibc:  ~150 ms       (memcpy + read)
    musl:   ~100 ms       (readv 한 번)
    
  fopen + fclose × 100K:
    glibc:  ~200 ms
    musl:   ~150 ms
    
  → musl 이 대체로 약간 빠름
  → 일부 응용 (cpython 등) 은 glibc 가 빠른 경우도

musl 의 빠른 fread — readv 의 영리한 활용. 그러나 — 정확한 영향은 workload 따라 다름.

비교 측면 9: 보안

[stdio 보안 비교]

  glibc:
    ✗ FILE * vtable hijacking 공격 표면 큼
       _IO_vtable_check (2.24+) 가 완화
       그러나 여전히 House of Orange 등 가능
    ✗ %n 의 위험 (현재는 차단되지만 역사적)
    ✓ FORTIFY_SOURCE 강력
    ✓ ASLR 잘 지원
    
  musl:
    ✓ vtable 없음 → 그 공격 자체가 없음
    ✓ 단순한 구조 → 공격 표면 작음
    ✓ 메모리 관리 안전 (mallocng)
    ✗ FORTIFY_SOURCE 제한적 (구현 안 함)
    ✗ 어떤 검사들은 안 함

비교 측면 10: 호환성

[응용 호환성]

  glibc 가정 응용:
    - GNU 확장 사용 (fopencookie, register_printf, ...)
    - %a, %p 의 glibc 특정 형식
    - errno 의 자세한 의미
    - 옛 (POSIX 외) extension
    
  musl 의 응용:
    - POSIX 만 가정
    - 더 portable
    - Linux 가 아닌 시스템에서도 동작 (Alpine 의 일부 포팅)
    
  실제 영향:
    - Python, Ruby 등 인터프리터 — 보통 잘 동작
    - 일부 closed-source 응용 — glibc 가정으로 musl 에 안 됨
    - 정적 링크된 응용 — 모두 동작

fopencookie — glibc 의 매우 유용한 확장

glibc 의 매력적인 기능 중 하나.

// [user-space - glibc API]
#include <stdio.h>

typedef struct _IO_cookie_io_functions_t {
    cookie_read_function_t  *read;
    cookie_write_function_t *write;
    cookie_seek_function_t  *seek;
    cookie_close_function_t *close;
} cookie_io_functions_t;

FILE *fopencookie(void *cookie, const char *mode,
                  cookie_io_functions_t io_funcs);
// [user-space - C 예제, custom stream]
struct mybuf { char *data; size_t len; size_t pos; };

ssize_t my_read(void *cookie, char *buf, size_t size) {
    struct mybuf *m = cookie;
    /* m->data 에서 buf 로 복사 */
    ...
}

int main(void) {
    struct mybuf m = {data, len, 0};
    cookie_io_functions_t fns = {my_read, NULL, NULL, NULL};
    FILE *fp = fopencookie(&m, "r", fns);
    
    /* 이제 fread 가 my_read 호출 */
    fread(buf, 1, 100, fp);
}

매우 강력 — 임의의 source 를 stream 으로. zlib, encryption 등 wrapper 만들 수 있음. musl 에는 — 없음.

이런 의존이 — musl 로 옮길 때 깨지는 응용의 원인.
