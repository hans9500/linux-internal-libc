Part VIpthread 의 깊이 — 동시성의 정체

Part I~V 에서 — 단일 thread 의 모든 메커니즘을 봤다. 이제 — multi-thread 의 세계로.

// [user-space - C 예제]
#include <pthread.h>

void *worker(void *arg) {
    /* 다른 thread 가 동시에 실행 */
    return NULL;
}

int main(void) {
    pthread_t t;
    pthread_create(&t, NULL, worker, NULL);
    pthread_join(t, NULL);
    return 0;
}

이 단순한 코드 — pthread_create 한 줄 뒤에 정교한 메커니즘이.

[pthread_create 한 줄의 일생]

  pthread_create 호출
       │
       ▼
  thread stack 할당 (mmap)
       │
       ▼
  TLS (Thread-Local Storage) 설정
       │
       ▼
  pthread_attr_t 처리 (size, detach, sched policy 등)
       │
       ▼
  clone(2) syscall — 새 task_struct 생성
       │
       ▼
  kernel 이 새 thread 추가 (같은 PID 의 다른 TID)
       │
       ▼
  새 thread 가 worker() 진입
       │
       ▼
  worker return → pthread_exit
       │
       ▼
  cleanup (TSD destructor, cancellation 처리, stack 해제)
       │
       ▼
  exit syscall — kernel 의 task 종료

43pthread 의 정체 — kernel thread 와의 관계

C 의 thread 가 — kernel 의 어떤 entity 와 매핑되는가? Linux 의 답은 — task_struct. process 도 thread 도 같은 자료구조.

두 가지 관점 — kernel vs POSIX

[두 가지 thread 관점]

  POSIX 의 관점:
    process = 자원 컨테이너
    thread = 실행 context (한 process 안에 여러 개)
    pthread_create → 새 thread (자원 공유)
    fork → 새 process (자원 복사)
    
  Linux kernel 의 관점:
    task = 실행 단위 (실제 자료구조: task_struct)
    process vs thread 의 구분 없음 — 둘 다 task
    각자 자원 공유 정도가 다를 뿐
    clone(2) 의 flags 가 공유 정도 결정

이 차이가 — Linux 의 독특한 디자인. 흔히 — "Linux 의 thread 는 process 다" 라고 표현.

task_struct — 모든 것의 단위

Linux kernel 의 핵심 자료구조 task_struct 가 — process 도 thread 도 표현.

// [kernel-space - linux/include/linux/sched.h master 단순화]
struct task_struct {
    /* state */
    unsigned int     __state;
    
    /* identifiers */
    pid_t            pid;         /* ★ thread id (TID) */
    pid_t            tgid;        /* ★ thread group id = process id (PID) */
    
    /* parent / children */
    struct task_struct *real_parent;
    struct task_struct *parent;
    struct list_head children;
    struct list_head sibling;
    
    /* address space */
    struct mm_struct *mm;          /* 가상 메모리 — thread 들 공유 */
    
    /* file system */
    struct fs_struct *fs;          /* cwd, umask */
    struct files_struct *files;    /* open fd table */
    
    /* signal handling */
    struct sighand_struct *sighand;
    struct signal_struct *signal;
    
    /* scheduling */
    int prio, static_prio, normal_prio;
    struct sched_class *sched_class;
    
    /* ... 수백 개의 필드 */
};

핵심 — pid (TID) 와 tgid (PID) 의 분리.

[pid vs tgid 의 차이]

  Linux kernel 내부:
    pid  = 개별 task 의 id (실제로는 TID 의미)
    tgid = thread group id (같은 process 의 thread 들이 공유)
    
  POSIX 의 호환:
    getpid() → tgid 반환 (사용자에게 보이는 process id)
    gettid() → pid 반환 (실제 thread id)
    
  같은 process 의 thread 들:
    각자 pid 다름
    그러나 tgid 같음 (대표 thread 의 pid 와 같음)
    
  process 단일 thread 일 때:
    pid = tgid (대표 thread 자체)

clone(2) — 모든 것의 시작

새 task 생성의 핵심 syscall.

// [user-space - Linux API]
#include <sched.h>

int clone(int (*fn)(void *), void *stack, int flags, void *arg, ...);

/* 또는 syscall 직접: */
long clone(unsigned long flags, void *stack, int *parent_tid,
           int *child_tid, unsigned long tls);

flags 가 — 공유 정도 결정. 매우 많은 flag.

[주요 clone flags]

  자원 공유:
    CLONE_VM       : 같은 가상 메모리 (★ thread 의 핵심)
    CLONE_FS       : 같은 file system info (cwd, umask)
    CLONE_FILES    : 같은 fd table
    CLONE_SIGHAND  : 같은 signal handler
    CLONE_SYSVSEM  : 같은 SysV semaphore
    
  thread 의미:
    CLONE_THREAD   : 같은 thread group (tgid 공유, POSIX thread)
    CLONE_PARENT   : 같은 parent task
    
  TLS / 동기화:
    CLONE_SETTLS         : TLS 설정 (tls 인자)
    CLONE_PARENT_SETTID  : parent 에게 child tid 전달
    CLONE_CHILD_SETTID   : child 의 tid 메모리에 저장
    CLONE_CHILD_CLEARTID : ★ child 종료 시 tid 위치 clear + futex wake
                          (pthread_join 의 메커니즘!)
    
  기타:
    CLONE_UNTRACED, CLONE_PTRACE, CLONE_FORK, ...

fork() vs pthread_create() — clone 의 두 면

[fork() 의 clone 호출]

  flags = SIGCHLD  (자식 종료 시 signal)
         + CLONE_CHILD_SETTID
         + CLONE_CHILD_CLEARTID
  
  → 거의 모든 자원 _복사_ (CLONE_VM, CLONE_FILES 등 안 set)
  → child 의 mm_struct, files_struct 새로
  → 다른 PID (= 다른 tgid)
  → 새 process

[pthread_create() 의 clone 호출]

  flags = CLONE_VM       (가상 메모리 공유)
        | CLONE_FS       (cwd 공유)
        | CLONE_FILES    (fd 공유)
        | CLONE_SIGHAND  (signal handler 공유)
        | CLONE_THREAD   (★ 같은 thread group)
        | CLONE_SYSVSEM
        | CLONE_SETTLS   (TLS 설정)
        | CLONE_PARENT_SETTID
        | CLONE_CHILD_CLEARTID (★ join 메커니즘)
  
  → 자원 _공유_
  → 같은 PID (= 같은 tgid)
  → 다른 TID
  → 새 thread

이 차이가 — 단지 flags 의 차이. 같은 syscall 로 둘 다 만들어진다.

thread 의 PID — gettid()

// [user-space - Linux API]
#include <sys/types.h>
#include <unistd.h>

pid_t gettid(void);              /* Linux 의 thread id */
pid_t getpid(void);              /* process id (= tgid) */
// [user-space - C 예제]
#include <pthread.h>
#include <unistd.h>
#include <sys/syscall.h>
#include <stdio.h>

void *worker(void *arg) {
    pid_t tid = syscall(SYS_gettid);     /* 또는 gettid() */
    pid_t pid = getpid();
    printf("Thread: tid=%d, pid=%d\n", tid, pid);
    return NULL;
}

int main(void) {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    return 0;
}

출력 예:

Thread: tid=12345, pid=12340      ← main 의 pid
Thread: tid=12346, pid=12340      ← 같은 pid, 다른 tid
Thread: tid=12347, pid=12340

pid 가 같음 — 같은 process. tid 다름 — 각자의 thread.

pthread_t — 사용자 측의 식별자

POSIX 의 pthread_t 는 — kernel 의 tid 가 _아님_.

// [user-space - glibc pthreadtypes-arch.h]
typedef unsigned long int pthread_t;     /* 보통 64-bit */
[pthread_t 의 정체]

  glibc 의 구현:
    pthread_t 는 _struct pthread_ 의 주소
    (thread descriptor)
    
  주의:
    pthread_t 와 tid 는 _완전히 다른 식별자_
    
  변환:
    pthread_t → tid: glibc 내부에서 td_desc->tid
    tid → pthread_t: 직접 불가 (glibc 가 매핑 안 함)
    
  비교:
    pthread_equal(t1, t2) 로만 비교 가능
    == 는 표준 외 (단 glibc 는 동작)

이 추상화 덕에 — 다른 OS (Windows, macOS) 에서도 같은 pthread API 사용 가능.

thread 의 stack

각 thread 는 — 자기만의 stack.

[thread stack 의 메모리 layout]

  main thread:
    초기 stack — kernel 이 줌 (보통 8 MB, ulimit 의존)
    
  pthread_create:
    glibc 가 mmap 으로 새 stack 할당
    기본 크기: 8 MB (glibc), 128 KB (musl)
    또는 pthread_attr_setstacksize 로 지정
    
  stack layout:
    +-- guard page (PROT_NONE) --+   ← overflow 탐지
    | red zone                    |
    +-- stack (성장 방향: 위→아래) --+
    | thread descriptor (struct pthread) |
    | (TCB)                       |
    +-- TLS area                  +

stack 위에 — thread control block (TCB) 가 함께. struct pthread 라는 자료구조.

struct pthread — thread descriptor

glibc 의 thread descriptor.

// [user-space - glibc nptl/descr.h master 단순화]
struct pthread {
    /* TCB head (커널/CPU 가 사용) */
    union {
        tcbhead_t header;
        void *__padding[24];
    };
    
    /* multiple_threads, cancelhandling, ... */
    int multiple_threads;
    int gscope_flag;
    
    /* tid — kernel 의 thread id */
    pid_t tid;
    
    /* signal mask, cancellation state */
    sigset_t cancel_jmp_buf;
    int canceltype;
    int cancelstate;
    
    /* TSD (Thread Specific Data) */
    struct pthread_key_data {
        uintptr_t seq;
        void *data;
    } specific_1stblock[PTHREAD_KEY_2NDLEVEL_SIZE];
    
    /* cleanup handlers (pthread_cleanup_push) */
    struct _pthread_cleanup_buffer *cleanup;
    
    /* exit value, joining thread */
    void *result;
    
    /* 매우 많은 필드... (수백 byte) */
};

이 descriptor 가 — thread 의 모든 메타데이터. pthread_t 가 가리키는 곳.

Linux thread 모델의 역사

[Linux thread 의 진화]

  옛 (Linux 2.0 ~):
    LinuxThreads (Xavier Leroy, 1996)
    각 thread 가 _진짜_ 새 process (CLONE_VM 만)
    signal 처리 문제, getpid 가 thread 마다 다름 등 비표준
    
  과도기 (Linux 2.4):
    LinuxThreads 의 개선
    그러나 여전히 POSIX 비호환
    
  현대 (Linux 2.6+, 2003):
    NPTL (Native POSIX Thread Library)
    Red Hat 이 주도
    glibc 2.4+ 의 기본
    
    kernel 의 협조:
      - CLONE_THREAD (tgid 공유)
      - futex (효율적 동기화)
      - TLS 지원 (CLONE_SETTLS)
      - tid + cleartid (join 메커니즘)

NPTL 의 성공이 — 현대 Linux 의 thread 가 POSIX 표준 그대로 동작하는 이유.

POSIX vs Linux 의 미묘한 차이

POSIX 표준이 — Linux 의 thread 모델과 완전히 같지 않음.

[POSIX 와 Linux 의 차이]

  POSIX:
    process 가 자원의 단위
    thread 는 단지 실행 컨텍스트
    
  Linux:
    task 가 단위 (process 와 thread 구분 없음)
    공유 정도가 다를 뿐
    
  결과적 차이:
    ✓ Linux: 같은 process 의 thread 가 _다른 user_ 로 setuid 가능
       (옛 LinuxThreads 의 흔적, 현재는 막혀 있음)
    ✓ Linux: thread 마다 다른 nice 값 가능
       (POSIX: process 단위)
    ✓ Linux: thread 마다 다른 cgroup, namespace 가능
       (Docker container 의 일부)
    ✓ getpid() 가 LinuxThreads 시절엔 thread 마다 다름
       (NPTL 부터 POSIX 호환)

thread 와 process 의 자원 비교

[process vs thread 자원 비교]

  자원                   process       thread
  ─────────────────────────────────────────────
  task_struct           각자          각자
  mm_struct (가상 메모리)   다름        ★ 공유
  files_struct (fd 표)    다름        ★ 공유
  fs_struct (cwd, umask)  다름        ★ 공유
  sighand_struct        다름        ★ 공유
  stack                  각자        각자
  TLS                    각자        각자
  registers (CPU)        각자        각자
  pid (kernel)          다름         다름
  tgid                   다름        ★ 공유
  scheduling             각자        각자
  errno                  각자        ★ 각자 (TLS)
  
  공유 자원:
    - 가상 메모리 (heap, .data, .bss, .text)
    - file descriptor (open file 의 view)
    - signal handler
    - cwd, umask
    
  각자 가진 자원:
    - stack
    - register 상태
    - TLS
    - errno
    - kernel 의 task_struct

이 차이가 — thread 의 효율성과 위험성의 근원.

thread 의 효율성

[thread 의 성능 우위]

  fork() 비용:
    mm_struct 복사 (page table copy-on-write)
    files_struct 복사
    fs_struct 복사
    sighand 복사
    → 약 50~100 μs
    
  pthread_create() 비용:
    stack mmap (8 MB lazy)
    TLS 설정
    clone syscall (자원 공유라 빠름)
    → 약 5~20 μs
    
  → thread 가 5~10 배 빠른 생성
  → 응용에서 thread pool 으로 더 빠름

thread 의 위험성

[thread 의 위험성]

  데이터 race:
    공유 메모리 → 동기화 없이 동시 접근 시 race
    → mutex, atomic 필요
    
  signal 처리:
    sighand 공유 → 어느 thread 가 signal 처리?
    → 복잡한 규칙 (POSIX 의 specification)
    
  errno:
    TLS 라 thread 간 안 섞임 ✓
    
  자원 누수:
    fd table 공유 → 어느 thread 가 close 해야?
    
  cancellation:
    중간에 thread 강제 종료 → 자원 누수 가능
    → 매우 위험

첫 thread (main) 의 특별함

main thread 가 — 약간 특별하다.

[main thread 의 특별함]

  ✓ kernel 이 직접 만듬 (execve syscall)
  ✓ stack 도 kernel 이 줌 (mmap 아님)
  ✓ 다른 thread 들의 기준 (tgid 의 대표)
  ✓ main() 의 return → exit() → 모든 thread 종료
  
  반면:
  ✓ 자원은 다른 thread 와 같음
  ✓ stack 크기 ulimit -s 따라 (기본 8 MB)
  ✓ TID == PID (== TGID)

main thread 의 종료 — 전체 process 종료.

task_struct 가 보여주는 통일된 모델

[Linux 의 통일된 task 모델]

  kernel 입장에선 process / thread 구분 없음:
    "task_struct 하나가 task 하나"
    
  공유 정도:
    fully shared → "thread" 라 부름
    nothing shared → "process" 라 부름
    부분 공유도 가능 (CLONE_VM 만, 등)
    
  스케줄링:
    task 단위로 스케줄 (process vs thread 무관)
    
  signal:
    task 단위로 전달 가능 (tgkill)
    또는 thread group 전체 (kill)
    
  /proc:
    /proc/PID/ — process 의 root
    /proc/PID/task/TID/ — 그 안의 thread 들

이 통일된 모델이 — Linux 의 단순함과 강력함의 비결.

이 장의 결론

pthread 의 본질:

[pthread 정리]

  ✓ Linux 에서 thread = task = process (자료구조 같음)
  ✓ task_struct: pid (TID) + tgid (PID 의 대표)
  ✓ clone(2) syscall 이 모든 task 생성
  ✓ flags 의 차이가 fork vs pthread 의 차이
    - CLONE_VM, CLONE_FILES, CLONE_THREAD 등이 thread 의미
  ✓ pthread_t 는 struct pthread 의 주소 (kernel tid 아님)
  ✓ TCB (thread control block) 가 stack 위에
  ✓ thread 생성 비용: process 의 5~10 배 빠름
  ✓ 공유 자원: mm, files, sighand, cwd
  ✓ 각자 자원: stack, register, TLS, errno
  ✓ /proc/PID/task/TID/ 로 thread 시각화
  ✓ NPTL (glibc 2.4+) 이 현재 표준 구현

다음 장에서는 — pthread_create 와 pthread_join 의 정확한 내부 동작을 본다. clone syscall 의 인자, futex 의 join, cleanup handler 의 처리.

44pthread_create / pthread_join

이 장에서 — pthread_create 가 실제로 무엇을 하는지, pthread_join 이 어떻게 thread 의 종료를 기다리는지 본다.

pthread_create 의 API

// [user-space - POSIX]
#include <pthread.h>

int pthread_create(pthread_t *thread,
                   const pthread_attr_t *attr,
                   void *(*start_routine)(void *),
                   void *arg);
[인자]

  thread       : 생성된 thread 의 식별자 (pthread_t)
  attr         : 속성 (stack size, detach 등). NULL = 기본
  start_routine: thread 가 실행할 함수
  arg          : start_routine 의 인자

반환값:

  • 0: 성공
  • 양수 (errno 값): 실패 — EAGAIN, EINVAL, EPERM 등

glibc 의 구현 — 7 단계

master 의 __pthread_create_2_1 의 단계:

[pthread_create 의 7 단계]

  1. pthread_attr_t 의 확인
     - stack size, guard size, scheduling policy
     - NULL 이면 기본값
     
  2. stack 할당
     - 사용자 지정 (attr->stackaddr) 아니면 자체 할당
     - mmap 으로 PROT_READ|PROT_WRITE
     - guard page (PROT_NONE) 추가
     
  3. struct pthread (TCB) 초기화
     - stack 위에 위치
     - TLS 영역 같이
     - canceltype, cancelstate 등 기본값
     
  4. start_thread wrapper 설정
     - 사용자의 start_routine 을 직접 호출 않음
     - glibc 의 wrapper (start_thread) 가 호출
     - exception 처리, cleanup 등 처리
     
  5. clone(2) syscall
     - 매우 많은 flags
     - 새 task 생성
     - child 가 start_thread 진입
     
  6. parent 는 tid 받음
     - pthread_t 에 struct pthread 의 주소 저장
     - return
     
  7. child 는 start_thread 실행
     - 사용자의 start_routine 호출
     - 반환 시 pthread_exit 자동 호출

실제 clone 호출 — 매우 많은 flags

master 의 clone flags 의 정확한 set:

// [user-space - glibc nptl/pthread_create.c master 단순화]
int
create_thread(struct pthread *pd, const struct pthread_attr *attr,
              bool *stopped_start, STACK_VARIABLES_PARMS, bool *thread_ran)
{
    const int clone_flags = (CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SYSVSEM
                           | CLONE_SIGHAND | CLONE_THREAD
                           | CLONE_SETTLS | CLONE_PARENT_SETTID
                           | CLONE_CHILD_CLEARTID
                           | 0);
    
    TLS_DEFINE_INIT_TP(tp, pd);
    
    if (__clone_internal(&args, &start_thread, pd) == -1)
        return errno;
    
    return 0;
}

각 flag 의 의미:

[pthread_create 의 clone flags 의미]

  CLONE_VM:
    가상 메모리 (mm_struct) 공유
    → 모든 thread 가 같은 heap, .data, .text
    
  CLONE_FS:
    file system info (cwd, umask) 공유
    
  CLONE_FILES:
    file descriptor table 공유
    → open(2) 한 fd 를 모든 thread 가 사용
    
  CLONE_SIGHAND:
    signal handler 테이블 공유
    → signal() 의 효과가 모든 thread 에
    
  CLONE_THREAD:
    같은 thread group (tgid 공유)
    → getpid() 가 모든 thread 에서 같은 값
    → POSIX 의 process 의미
    
  CLONE_SYSVSEM:
    SysV semaphore 공유 (drop_semundo 의미)
    
  CLONE_SETTLS:
    TLS 의 location 설정 (tls 인자 사용)
    → 새 thread 의 fs/gs register 가 새 TLS 가리킴
    
  CLONE_PARENT_SETTID:
    parent (pd->tid) 에 child 의 tid 저장
    → parent 가 child 의 tid 즉시 알 수 있음
    
  CLONE_CHILD_CLEARTID:
    ★ child 종료 시 — 그 메모리 위치를 0 으로 clear
    → 그리고 _futex wake_ (그 위치로)
    → pthread_join 의 메커니즘 핵심!

마지막 — CLONE_CHILD_CLEARTID 가 매우 영리한 디자인. 다음 절에서 설명.

start_thread — glibc 의 wrapper

clone 의 child 가 — 사용자의 start_routine 을 _직접_ 호출 않음. 먼저 glibc 의 wrapper.

// [user-space - glibc nptl/pthread_create.c master 단순화]
static void *
start_thread(void *arg)
{
    struct pthread *pd = arg;
    
    /* TLS 초기화 (이미 부분적으로 done) */
    ...
    
    /* signal mask 설정 (parent 의 mask 상속) */
    ...
    
    /* cleanup buffer 의 시작 위치 */
    not_first_call = setjmp((struct __jmp_buf_tag *)pd->cancelbuf);
    
    if (__glibc_likely(!not_first_call)) {
        /* ★ 사용자의 start_routine 호출! */
        void *ret = pd->start_routine(pd->arg);
        
        /* 반환값 저장 (pthread_join 가 가져갈 것) */
        pd->result = ret;
    } else {
        /* cancellation 의 longjmp */
        pd->result = PTHREAD_CANCELED;
    }
    
    /* cleanup handlers 실행 */
    while (pd->cleanup != NULL) {
        struct _pthread_cleanup_buffer *this = pd->cleanup;
        pd->cleanup = this->prev;
        this->routine(this->arg);
    }
    
    /* TSD destructors */
    __nptl_deallocate_tsd();
    
    /* clone 의 CLONE_CHILD_CLEARTID 가 작동:
       이 thread 의 tid 가 든 위치를 0 으로 clear
       + 그 위치에 futex wake 
       → pthread_join 깨움 */
    
    /* exit syscall */
    __exit_thread();
}

이 wrapper 가 — 여러 중요한 일 을 한다.

[start_thread 의 역할]

  ✓ TLS 의 최종 초기화
  ✓ signal mask 처리
  ✓ setjmp 으로 cancellation 의 longjmp 기준점
  ✓ 사용자의 start_routine 호출 + 반환값 저장
  ✓ cleanup handlers 실행 (pthread_cleanup_push 등록)
  ✓ TSD destructors 실행
  ✓ exit syscall — kernel 에게 task 종료 알림

사용자가 pthread_exit 호출하면?

// [user-space - C 예제]

    pthread_exit(NULL);     /* 명시적 종료 */
    /* 여기 도달 안 함 */
}

pthread_exit 의 동작:

[pthread_exit 의 동작]

  1. cleanup handlers 실행
  2. TSD destructors 실행
  3. result 저장
  4. longjmp 으로 start_thread 의 setjmp 위치로
  5. start_thread 가 마무리 (exit syscall)
  
  effectively:
    worker return == pthread_exit(return_value)

worker 가 그냥 return 해도 — 같은 효과. start_thread 가 결과 처리.

pthread_join — 종료 기다림

// [user-space - POSIX]
int pthread_join(pthread_t thread, void **retval);

thread 가 종료할 때까지 _대기_. retval 에 반환값 받음.

pthread_join 의 내부 — futex 사용

매우 영리한 메커니즘:

// [user-space - glibc nptl/pthread_join_common.c master 단순화]
int
__pthread_join(pthread_t threadid, void **thread_return)
{
    struct pthread *pd = (struct pthread *)threadid;
    
    /* 자기 자신 join 검사 */
    if (pd == THREAD_SELF)
        return EDEADLK;
    
    /* 이미 detached 검사 */
    if (IS_DETACHED(pd))
        return EINVAL;
    
    /* 다른 join 진행 중 검사 */
    pthread_t self = THREAD_SELF;
    if (atomic_compare_exchange_strong(&pd->joinid, NULL, self) == 0)
        return EINVAL;     /* 이미 다른 thread 가 join 중 */
    
    /* ★ 핵심: tid 가 0 이 될 때까지 대기 */
    pid_t tid;
    while ((tid = atomic_load_acquire(&pd->tid)) != 0) {
        /* tid 가 아직 set 되어 있음 = thread 살아있음 */
        /* futex_wait 으로 대기 */
        lll_futex_wait(&pd->tid, tid, LLL_PRIVATE);
        /* 깨면 다시 loop */
    }
    
    /* thread 종료됨! */
    if (thread_return != NULL)
        *thread_return = pd->result;
    
    /* stack, TLS, descriptor 정리 */
    __free_tcb(pd);
    
    return 0;
}

핵심:

[pthread_join 의 메커니즘 — 매우 영리]

  thread 생성 시:
    clone 의 CLONE_CHILD_CLEARTID 사용
    pd->tid 위치를 child_tidptr 로 지정
    
  child 실행 중:
    pd->tid = (kernel 의 tid)
    
  child 종료 시:
    kernel 이 pd->tid 를 0 으로 clear (★ CLONE_CHILD_CLEARTID 효과)
    kernel 이 그 주소에 futex wake (모든 waiter 깨움)
    
  pthread_join:
    while (pd->tid != 0):
        futex_wait(&pd->tid, tid)        /* tid 가 변할 때까지 대기 */
    /* 변하면 깨어남 = thread 종료 */
    
  ★ 이게 매우 efficient
    → mutex, signal 안 씀
    → kernel 의 단일 futex
    → race-free (atomic operation)

pthread_detach — join 안 할 thread

join 하지 않을 thread 는 — detach 하면 자동 정리.

[detached thread]

  joined thread:
    종료 시 → 자원 안 풀림 (pthread_join 까지 보관)
    pthread_join 가 자원 해제
    
  detached thread:
    종료 시 → 즉시 자원 해제 (stack, TLS, descriptor 자동 free)
    join 불가
    
  사용처:
    "fire and forget" 패턴
    background worker, signal handler thread 등

자원 누수 — join 또는 detach 필수

joined 도 detached 도 아닌 thread 가 종료 안 되고 — main 이 끝나도 자원 누수 가능.

for (int i = 0; i < 1000; i++) {
    pthread_t t;
    pthread_create(&t, NULL, worker, NULL);
    /* join 도 detach 도 안 함! */
    /* worker 끝나도 stack 안 풀림 → 점점 메모리 누적 */
}

// [user-space - C 안전 패턴]
for (int i = 0; i < 1000; i++) {
    pthread_t t;
    pthread_create(&t, NULL, worker, NULL);
    pthread_detach(t);     /* 끝나면 자동 정리 */
}

thread 종료의 두 경로

[thread 종료 경로]

  1. start_routine 의 return
     → start_thread wrapper 가 처리
     → return value 저장
     → cleanup, TSD destructors
     → exit syscall
     
  2. pthread_exit() 명시적 호출
     → longjmp 으로 start_thread 의 setjmp 으로
     → return value 저장
     → cleanup 등
     
  3. main thread 의 종료
     → 전체 process 종료 (exit_group syscall)
     → 모든 thread 즉시 죽음 (cleanup 없이)
     
  4. pthread_cancel — 다른 thread 가 종료시킴
     → cancellation point 에서 longjmp 발동
     → 매우 위험 (50 장 참고)
     
  5. exit(0) — 전체 process 종료
     → main thread 와 같음

마지막 두 가지 — cleanup handlers 실행 안 함 또는 부분 실행. 자원 누수 가능.

stack 의 자동 처리

[thread stack 의 생애]

  생성:
    pthread_create:
      mmap(stacksize) 으로 stack 할당
      guard page 추가
      TCB 영역 같이 (stack 위)
      
  사용 중:
    각 thread 가 자기 stack 사용
    main thread 만 kernel 이 준 stack

    joined: pthread_join 가 munmap

thread-safety 의 시작

새 thread 가 시작되면 — 동기화 문제 시작.

// [user-space - C 흔한 race condition]
int counter = 0;

void *worker(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        counter++;          /* ★ race! */
    }
    return NULL;
}

int main(void) {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    printf("counter = %d\n", counter);    /* 2000000 아님! */
}

counter++ 가 — atomic 아님. read-modify-write 3 step. race condition 발생. 해결 — mutex 또는 atomic.

다음 장부터 — 이 동기화 메커니즘을 다룬다.

  → 없으면 즉시 반환

  ✓ clone 의 정확한 flags set (12 개)

45mutex — 가장 기본의 lock

// [user-space - C 표준 예제]

pthread_mutex_lock(&lock);
/* critical section */
pthread_mutex_unlock(&lock);

이 두 줄 사이의 코드 — 한 thread 만 실행. 너무도 단순해 보이지만, 그 안에 futex 와 atomic operation 의 정교한 협업.

mutex 의 본질

[mutex 의 본질]

  목적:
    여러 thread 가 동시 접근 _금지_
    한 번에 한 thread 만 critical section
    
  구현:
    user-space 의 atomic 변수 (lock state)
    contention 시 kernel 의 futex 로 대기
    
  비유:
    화장실 잠금 + 줄
    문 잠긴지 확인 (atomic)
    잠겨있으면 줄에 섬 (futex_wait)
    문 열리면 깨움 (futex_wake)

pthread_mutex_t 의 구조

// [user-space - glibc sysdeps/x86/bits/struct_mutex.h master]
struct __pthread_mutex_s {
    int __lock;                  /* lock state — futex word */
    unsigned int __count;        /* recursive lock 의 count */
    int __owner;                 /* 소유 thread 의 tid */
    int __nusers;                /* waiter 수 */
    int __kind;                  /* mutex 종류 */
    short __spins;
    short __elision;
    __pthread_list_t __list;     /* robust mutex 의 list */
};

typedef union {
    struct __pthread_mutex_s __data;
    char __size[__SIZEOF_PTHREAD_MUTEX_T];
    long int __align;
} pthread_mutex_t;

핵심 필드는 — __lock (futex word) 와 __kind (type).

lock 의 3 상태

[__lock 의 3 상태]

  0: unlocked (자유)
     아무도 안 잡음
     
  1: locked, no waiter
     한 thread 가 잡음
     기다리는 thread 없음
     → unlock 시 syscall 불필요!
     
  2: locked, with waiter
     한 thread 가 잡음
     기다리는 thread 있음
     → unlock 시 futex_wake 필요

이 영리한 디자인이 — uncontended fast path (kernel 안 가는 lock/unlock) 의 핵심.

pthread_mutex_lock 의 정확한 흐름

master 의 단순화:

// [user-space - glibc nptl/pthread_mutex_lock.c master 단순화]
int
__pthread_mutex_lock(pthread_mutex_t *mutex)
{
    unsigned int type = PTHREAD_MUTEX_TYPE(mutex);
    
    /* fast path 가능한 case */
    if (__glibc_likely(type == PTHREAD_MUTEX_TIMED_NP)) {
        /* PTHREAD_MUTEX_NORMAL — 가장 단순 */
        LLL_MUTEX_LOCK(mutex);
        assert(mutex->__data.__owner == 0);
    } else if (...) {
        /* RECURSIVE, ERRORCHECK 등 다른 type */
        return __pthread_mutex_lock_full(mutex);
    }
    
    /* lock 획득 — owner 기록 */
    mutex->__data.__owner = THREAD_GETMEM(THREAD_SELF, tid);
    ++mutex->__data.__nusers;
    
    return 0;
}

핵심은 — LLL_MUTEX_LOCK 매크로. 이게 fast path + slow path 분기.

lll_lock — fast path 의 핵심

// [user-space - glibc sysdeps/nptl/lowlevellock.h master 단순화]
#define lll_lock(futex, private) \
    do { \
        int *__futex = &(futex); \
        if (__glibc_unlikely \
            (atomic_compare_and_exchange_bool_acq(__futex, 1, 0))) \
            __lll_lock_wait(__futex, private); \
    } while (0)

흐름:

[lll_lock 의 동작]

  1. atomic CAS: __lock 이 0 이면 1 로 변경
     → 성공: lock 획득, syscall 없음 (fast path) ✓
     → 실패: 누군가 이미 lock 잡음, slow path 로
     
  2. slow path: __lll_lock_wait 호출
     → futex syscall 사용
     → kernel 의 wait queue 에 추가

uncontended case 의 lock — 한 atomic operation. 매우 빠름. 대략 10~30 ns.

__lll_lock_wait — slow path

contention 시 호출.

// [user-space - glibc sysdeps/unix/sysv/linux/lowlevellock.c master 단순화]
void
__lll_lock_wait(int *futex, int private)
{
    /* 현재 lock 상태 확인 */
    if (atomic_load_relaxed(futex) == 2)
        goto futex;
    
    /* lock state 를 2 로 (waiter 있음) */
    while (atomic_exchange_acquire(futex, 2) != 0) {
    futex:
        /* futex_wait: futex 가 2 면 대기, 변하면 깨어남 */
        lll_futex_wait(futex, 2, private);
    }
    /* 깨어남 → lock 획득 시도 (atomic_exchange) */
}

핵심 흐름:

[__lll_lock_wait 의 동작]

  1. __lock 을 2 로 만듬 (waiter 있음 표시)
     이전 값이 0 이면 우리가 잡았음! 끝
     아니면 누가 잡고 있음
     
  2. futex_wait(futex, 2):
     "__lock 이 2 면 대기"
     atomic 비교 — race 없음
     변하면 깨어남
     
  3. 깨어남 → 다시 1 로 시도
     loop

이 futex_wait 가 — kernel 의 wait queue 에 추가. 다른 thread 가 깨울 때까지 sleep.

     1 → waiter 없었음, syscall 불필요 ✓
     2 → waiter 있었음, futex_wake 로 한 명 깨움

[futex 의 주요 operation]

  FUTEX_WAIT:
    *uaddr == val 이면 대기

[pthread_mutex 의 4 type]

  PTHREAD_MUTEX_NORMAL (= TIMED_NP):

Priority Inheritance (PI) mutex

실시간 (real-time) 응용의 필수 기능.

[Priority Inheritance 의 의미]

  문제:
    낮은 우선순위 thread A 가 lock 잡음
    높은 우선순위 thread B 가 lock 대기
    중간 우선순위 thread C 가 A 를 preempt
    → A 안 끝남 → B 도 안 됨 (priority inversion)
    
  해결:
    A 가 lock 잡고 있는 동안 — A 의 우선순위를 _B 의 수준으로 임시 상승_
    → C 가 A 를 preempt 못 함
    → A 가 빨리 끝남 → B 가 lock 획득
    
  내부:
    futex 의 FUTEX_LOCK_PI / FUTEX_UNLOCK_PI
    kernel 이 우선순위 자동 조정

이 메커니즘이 — 화성 탐사선 같은 실시간 시스템의 안정성에 매우 중요.

lock contention 측정

// [user-space - C 예제, 측정]
#include <time.h>

struct timespec t1, t2;
clock_gettime(CLOCK_MONOTONIC, &t1);
pthread_mutex_lock(&mutex);
clock_gettime(CLOCK_MONOTONIC, &t2);
long ns = (t2.tv_sec - t1.tv_sec) * 1e9 + (t2.tv_nsec - t1.tv_nsec);
/* uncontended: ~20 ns
   contended: 수 μs ~ 수 ms */
[lock 의 latency 분포]

  uncontended fast path:
    ~20 ns (한 atomic CAS)
    
  contended (다른 thread 가 잡음):
    ~수 μs (futex_wait + scheduler 동작)
    
  contended + scheduler latency:
    ~수 ms (다른 thread 가 critical section 길 때)
    
  → contention 회피가 성능의 핵심
  → critical section 짧게
  → lock granularity 신중

mutex 의 흔한 함정

[mutex 함정]

  ✗ pthread_mutex_init 안 함 (정적 init 아니면)
     → undefined behavior
     
  ✗ pthread_mutex_destroy 안 함
     → 메모리 누수 + kernel resource (robust list)
     
  ✗ recursive lock 시도 (NORMAL type)
     → deadlock
     
  ✗ 다른 thread 가 unlock (NORMAL type)
     → undefined (ERRORCHECK 면 EPERM)
     
  ✗ critical section 너무 큼
     → contention 심함, 성능 저하
     
  ✗ lock 순서 다름 → 데드락
     thread A: lock1, lock2
     thread B: lock2, lock1
     → 두 thread 가 서로 기다림
     해결: 항상 정해진 순서로 lock
     
  ✗ lock 안 풀고 return / exit
     → 다른 thread 영원히 대기
     
  ✗ signal handler 에서 mutex 사용
     → deadlock 위험 (handler 가 같은 thread)
     → async-signal-safe 안 됨

scope 의 RAII (C++ 의 영향)

mutex 의 본질:

[mutex 정리]

  ✓ 3 상태: 0 (unlocked), 1 (locked, no waiter), 2 (locked + waiter)
  ✓ fast path: atomic CAS (uncontended ~20 ns)
  ✓ slow path: futex_wait/wake (contended)
  ✓ unlock fast path: atomic_exchange + waiter 있으면 futex_wake
  ✓ 4 type: NORMAL, RECURSIVE, ERRORCHECK, ADAPTIVE
  ✓ PTHREAD_MUTEX_INITIALIZER 의 정적 초기화
  ✓ trylock, timedlock 변종
  ✓ PROCESS_SHARED: mmap MAP_SHARED 영역에서
  ✓ ROBUST: owner 죽으면 EOWNERDEAD
  ✓ PRIO_INHERIT: priority inversion 방지
  ✓ 함정: critical section 길이, lock 순서, recursive, signal handler

다음 장에서는 — mutex 와 짝을 이루는 condition variable — wait/signal 메커니즘을 본다.

46condition variable — wait/signal 메커니즘

mutex 가 — "한 명만" 의 제어라면, condition variable 은 — "조건이 충족될 때까지 대기" 의 제어. 둘이 함께 — 거의 모든 동기화 패턴을 표현.

동기화의 두 패턴

[동기화의 기본 패턴]

  Mutual exclusion (mutex):
    "이 자원은 한 명만"
    → pthread_mutex_lock / unlock
    
  Wait / Notify (cond variable):
    "조건이 될 때까지 대기"
    "조건이 충족됐다고 알림"
    → pthread_cond_wait / signal
    
  거의 모든 동시성 패턴은 이 둘 의 조합

condition variable 의 API

// [user-space - POSIX]
#include <pthread.h>

int pthread_cond_init(pthread_cond_t *cond, const pthread_condattr_t *attr);
int pthread_cond_destroy(pthread_cond_t *cond);

int pthread_cond_wait(pthread_cond_t *cond, pthread_mutex_t *mutex);
int pthread_cond_timedwait(pthread_cond_t *cond, pthread_mutex_t *mutex,
                           const struct timespec *abstime);

int pthread_cond_signal(pthread_cond_t *cond);
int pthread_cond_broadcast(pthread_cond_t *cond);

기본 패턴

// [user-space - C 예제, producer-consumer]
pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  cv  = PTHREAD_COND_INITIALIZER;
int data_ready = 0;

void *producer(void *arg) {
    pthread_mutex_lock(&mtx);
    /* 데이터 준비 */
    data_ready = 1;
    pthread_cond_signal(&cv);          /* "조건 충족!" */
    pthread_mutex_unlock(&mtx);
    return NULL;
}

void *consumer(void *arg) {
    pthread_mutex_lock(&mtx);
    while (!data_ready)
        pthread_cond_wait(&cv, &mtx);  /* 조건 될 때까지 대기 */
    /* 데이터 처리 */
    pthread_mutex_unlock(&mtx);
    return NULL;
}

pthread_cond_wait 의 정확한 동작

매우 미묘한 함수. 3 단계가 atomic 해야 한다.

[pthread_cond_wait 의 3 단계]

  1. mutex unlock
  2. wait (sleep)
  3. mutex relock (깨어난 후)
  
  ★ 1 과 2 가 atomic 해야 함!
    아니면 race: unlock 직후 다른 thread 가 signal 보내면 missing
    
  POSIX 표준이 atomic 보장
  → futex 의 magic 으로 구현

왜 mutex 와 함께?

condition variable 은 mutex 와 함께만 사용. 이유:

[mutex 가 필요한 이유]

  공유 데이터 (예: data_ready) 의 보호
    
  producer: data_ready = 1   ← write
  consumer: read data_ready
  → race condition 없이 하려면 mutex
  
  더 깊은 이유:
    cond_wait 가 atomic unlock + wait 보장
    → 사용자가 mutex 를 _이미 잡고_ 호출해야 함
    → cond_signal 도 mutex 안에서
    → 안 그러면 lost wakeup

lost wakeup 의 함정

// [user-space - C 잘못된 예제]
/* mutex 없이 */

void *consumer(void *arg) {
    if (!data_ready)
        pthread_cond_wait(&cv, ???);    /* 무엇을 unlock? */
    /* 데이터 처리 */
}

void *producer(void *arg) {
    data_ready = 1;
    pthread_cond_signal(&cv);
    /* lost wakeup: signal 이 wait 보다 _먼저_ 도착하면 missed */
}
[lost wakeup 시나리오]

  1. consumer: !data_ready 검사 (false, 처음엔)
  2. consumer: pthread_cond_wait 진입 _직전_
  3. producer: data_ready = 1
  4. producer: pthread_cond_signal — _아무도 대기 안 함, 무시됨_
  5. consumer: pthread_cond_wait 진입 — _영원히 대기_
  
  해결:
    mutex 로 consumer 의 검사 + wait 를 atomic 묶음
    producer 의 set + signal 도 mutex 로 묶음

while loop 의 필수

if 가 _아니라_ **while** 로 검사.

// [user-space - C 옳은 패턴]
pthread_mutex_lock(&mtx);
while (!data_ready)             /* ★ while, not if */
    pthread_cond_wait(&cv, &mtx);
/* ... */
pthread_mutex_unlock(&mtx);
[while 이 필수인 이유]

  1. spurious wakeup:
     futex 의 구현 특성, 시그널 등으로
     condition 충족 안 됐는데 wait 가 깨어날 수 있음
     → 다시 검사 필요
     
  2. broadcast 의 경우:
     여러 thread 가 깨어남
     첫 thread 가 데이터 가져가면 나머지는 다시 못 가져감
     → 검사 후 다시 wait
     
  3. POSIX 표준이 spurious wakeup 허용
     "wait 가 실패할 수 있음, 다시 시도"
     
  → if 는 잠재 버그, while 가 항상 안전

pthread_cond_t 의 구조

// [user-space - glibc sysdeps/nptl/bits/struct_cond.h master 단순화]
struct __pthread_cond_s {
    __atomic_wide_counter __wseq;          /* wait sequence number */
    __atomic_wide_counter __g1_start;      /* group 1 의 start */
    unsigned int __g_refs[2];              /* g1 / g2 의 ref count */
    unsigned int __g_size[2];              /* 각 group 의 크기 */
    unsigned int __g1_orig_size;
    unsigned int __wrefs;                  /* waiter ref */
    unsigned int __g_signals[2];           /* signal count per group */
};

매우 복잡한 자료구조 — 효율적이고 race-free 한 wait/signal 처리.

pthread_cond_wait 의 내부

대단히 복잡한 알고리즘 (glibc 의 새 algorithm — 2017 년 도입).

[pthread_cond_wait 의 흐름 (단순화)]

  1. cond 의 sequence number 기록
  2. mutex unlock
  3. futex_wait — sequence 변할 때까지 대기
  4. 깨어남 (signal/broadcast/spurious)
  5. mutex relock
  6. return
  
  glibc 의 새 알고리즘:
    - 2 개의 group (g1, g2)
    - 새 waiter 는 g2 에
    - signal/broadcast 는 g1 에
    - group switch 로 starvation 방지

이 정교한 디자인이 — fair scheduling, broadcast 의 정확성, 동시 wait/signal 의 효율성을 모두 달성.

pthread_cond_signal vs pthread_cond_broadcast

// [user-space - POSIX]
int pthread_cond_signal(pthread_cond_t *cond);     /* 한 명 깨움 */
int pthread_cond_broadcast(pthread_cond_t *cond);  /* 모두 깨움 */
[signal vs broadcast]

  signal:
    waiter 중 _하나_ 만 깨움
    어느 thread 인지는 unspecified
    → "한 자원이 가능" 일 때 적합
    예: 큐에 한 item 추가 → 한 consumer 만 받아감
    
  broadcast:
    모든 waiter 깨움
    → "조건이 모두에 해당" 일 때
    예: producer 가 끝남 → 모든 consumer 알림
    예: read-write lock 의 writer unlock
    
  주의: broadcast 는 thundering herd 가능
    모든 thread 가 깨어나 mutex 잡으려 contention

pthread_cond_timedwait — 시간 제한

// [user-space - C 예제]
struct timespec ts;
clock_gettime(CLOCK_REALTIME, &ts);
ts.tv_sec += 5;            /* 5 초 후 timeout */

pthread_mutex_lock(&mtx);
int ret;
while (!condition) {
    ret = pthread_cond_timedwait(&cv, &mtx, &ts);
    if (ret == ETIMEDOUT) {
        /* 시간 초과 */
        break;
    }
}
pthread_mutex_unlock(&mtx);

abstime 이 절대 시간 — 매우 중요. timeout 이 아니라 시간점.

[abstime 의 의미]

  pthread_cond_timedwait(cv, mtx, abstime):
    abstime 까지 대기 (절대 시간)
    
  잘못된 패턴:
    while (!condition)
        timedwait(cv, mtx, &ts);    /* 매번 새 ts 안 만들면 OK */
    
  올바른 패턴:
    struct timespec deadline;
    clock_gettime(CLOCK_REALTIME, &deadline);
    deadline.tv_sec += 5;
    while (!condition) {
        if (timedwait(cv, mtx, &deadline) == ETIMEDOUT)
            break;
    }
    /* deadline 까지 spurious wakeup 들이 깨워도 — 같은 deadline 으로 다시 wait */

pthread_condattr — clock 지정

기본 — CLOCK_REALTIME. wallclock 시간이라 — NTP, system clock 조정 시 영향.

// [user-space - C 예제, MONOTONIC clock]

producer-consumer 의 완전한 예제

// [user-space - C 완전한 패턴]
#include <pthread.h>
#define QUEUE_SIZE 100

typedef struct {
    int items[QUEUE_SIZE];
    int head, tail, count;
    pthread_mutex_t mtx;
    pthread_cond_t not_empty;
    pthread_cond_t not_full;
} bounded_queue_t;

void enqueue(bounded_queue_t *q, int item) {
    pthread_mutex_lock(&q->mtx);
    while (q->count == QUEUE_SIZE)
        pthread_cond_wait(&q->not_full, &q->mtx);
    
    q->items[q->tail] = item;
    q->tail = (q->tail + 1) % QUEUE_SIZE;
    q->count++;
    
    pthread_cond_signal(&q->not_empty);
    pthread_mutex_unlock(&q->mtx);
}

int dequeue(bounded_queue_t *q) {
    pthread_mutex_lock(&q->mtx);
    while (q->count == 0)
        pthread_cond_wait(&q->not_empty, &q->mtx);
    
    int item = q->items[q->head];
    q->head = (q->head + 1) % QUEUE_SIZE;
    q->count--;
    
    pthread_cond_signal(&q->not_full);
    pthread_mutex_unlock(&q->mtx);
    return item;
}

이게 — 모든 producer-consumer 의 캐논. 두 개의 condition variable (full/empty) + 하나의 mutex.

read-write lock — 읽기는 동시, 쓰기는 단독

mutex 의 일반화 — 읽기는 여러 thread 동시 허용, 쓰기는 한 thread.

// [user-space - POSIX API]
#include <pthread.h>

pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

int pthread_rwlock_rdlock(pthread_rwlock_t *rwlock);    /* read lock */
int pthread_rwlock_wrlock(pthread_rwlock_t *rwlock);    /* write lock */
int pthread_rwlock_unlock(pthread_rwlock_t *rwlock);    /* unlock */

int pthread_rwlock_tryrdlock(pthread_rwlock_t *rwlock);

rwlock 의 의미

[rwlock 의 의미론]

  read lock (여러 가능):
    여러 thread 가 동시 보유 가능
    데이터 _읽기_ 만 (변경 X)
    다른 read lock 과 호환
    write lock 과는 _아님_
    
  write lock (단독):
    한 thread 만 보유
    데이터 변경 가능
    다른 read/write lock 과 _아님_
    
  → 읽기 위주 응용에 매우 효율적
  → DB의 select vs update 같은 패턴

예제 — read-heavy 패턴

// [user-space - C 예제]
typedef struct {
    int *data;
    size_t size;
    pthread_rwlock_t lock;
} cache_t;

int cache_get(cache_t *c, int key) {
    pthread_rwlock_rdlock(&c->lock);   /* read lock */
    int value = lookup(c->data, key);
    pthread_rwlock_unlock(&c->lock);
    return value;
}

void cache_set(cache_t *c, int key, int value) {
    pthread_rwlock_wrlock(&c->lock);   /* write lock */
    insert(c->data, key, value);
    pthread_rwlock_unlock(&c->lock);
}

100 thread 가 cache_get 하면 — 모두 동시 실행. cache_set 만 — 단독.

rwlock 의 정확한 동작

[rwlock state machine]

  states:
    free        — 아무도 안 잡음
    read_locked — read lock 보유 (1 명 이상)
    write_locked — write lock 보유 (단독)
    
  transitions:
    read 시도:
      free → read_locked (1)        : 즉시 획득
      read_locked → read_locked (++): 추가 reader
      write_locked → wait           : write 끝날 때까지
      
    write 시도:
      free → write_locked            : 즉시 획득
      read_locked → wait             : 모든 reader 끝날 때까지
      write_locked → wait            : 단독이므로
      
    unlock:
      reader 가 0 되면 → free
      writer 가 풀면 → free
      waiter 깨움

writer starvation 의 문제

[writer starvation]

  문제:
    reader 가 계속 옴 → rwlock 이 계속 read_locked 상태
    waiting writer 가 영원히 기다림
    
  대응:
    fairness 정책:
      - reader preference: 단순, 그러나 writer starvation 위험
      - writer preference: writer 우선, 그러나 reader 도 starve 가능
      - fair / FIFO: 도착 순서 (가장 안전, 약간 느림)
    
  POSIX 표준:
    어느 정책 사용할지 unspecified

    또는 pthread_rwlockattr_setkind_np 로 변경

rwlock 의 성능

read 비율이 매우 높을 때만 — mutex 보다 빠름.

[rwlock vs mutex 의 성능]

  100% read:
    rwlock 압도적 (mutex 의 N 배)
    
  90% read, 10% write:
    rwlock 약간 우수
    
  50/50:
    mutex 가 더 빠를 수 있음
    (rwlock 의 오버헤드)
    
  10% read, 90% write:
    mutex 가 우수
    rwlock 의 readers 카운팅 비용
    
  → 정확한 비율 측정 필요
  → 추정으로 rwlock 쓰지 말 것

spinlock 의 동작

[spinlock 의 본질]

  mutex:
    contended 시 — sleep (futex_wait, syscall)
    수 μs 의 latency
    
  spinlock:
    contended 시 — busy wait (CPU 계속 쓰면서 재시도)
    sleep 안 함, syscall 없음
    ~ns 의 latency
    
  trade-off:
    매우 짧은 lock: spinlock 이득
    조금이라도 길면: CPU 낭비, mutex 가 나음

내부 — 단순한 atomic CAS loop.

// [user-space - 단순화된 구현]
void spin_lock(spinlock_t *lock) {
    while (atomic_exchange(&lock->val, 1))    /* 1 이면 다시 시도 */
        cpu_relax();                          /* pause instruction */
}

void spin_unlock(spinlock_t *lock) {
    atomic_store(&lock->val, 0);
}

cpu_relax() 가 x86 의 pause instruction — hyper-threading 협조, power saving.

spinlock 의 적절한 사용 케이스

[spinlock 의 적절한 use case]

  ✓ critical section 이 _매우_ 짧음 (수 ns ~ 수십 ns)
  ✓ 다중 코어에서 사용 (단일 코어는 spinlock 의미 없음)
  ✓ kernel 모드 (sleep 불가능 한 컨텍스트)
  ✓ scheduler 못 부르는 환경
  
  ✗ critical section 길면:
     다른 thread 의 CPU 시간 낭비
     성능 매우 안 좋음
     
  ✗ 우선순위 다른 thread:
     priority inversion 위험
     spin 하는 thread 가 lock holder 를 preempt 못 함

spinlock 의 함정

[spinlock 함정]

  ✗ 단일 코어에서:
     spin 하는 thread 가 CPU 독점
     lock holder 가 영원히 실행 못 함
     → deadlock 같은 효과
     
  ✗ critical section 이 sleep 부르는 동작:
     malloc, syscall, I/O 등
     → 다른 thread 의 busy wait 시간 매우 김
     → 절대 금지
     
  ✗ recursive lock 시도:
     spinlock 은 보통 non-recursive
     → 자기 자신의 lock 기다림 → deadlock
     
  ✗ scheduler 가 lock holder 를 다른 코어로 옮김:
     spin 하는 thread 가 다른 코어의 lock 기다림
     → 캐시 미스, cache coherence 트래픽

glibc 의 spinlock 구현

// [user-space - glibc nptl/pthread_spin_lock.c master 단순화]
int
pthread_spin_lock(pthread_spinlock_t *lock)
{
    /* atomic 으로 시도 */
    if (LIBC_PROBE(...), atomic_exchange_acquire(lock, 1) == 0)
        return 0;
    
    /* spin */
    do {
        do {
            atomic_spin_nop();    /* cpu_relax */
        } while (atomic_load_relaxed(lock) != 0);
    } while (atomic_exchange_acquire(lock, 1) != 0);
    
    return 0;
}

영리한 점 — load 와 exchange 의 분리. exchange 는 cache line 의 write — cache coherence 트래픽 매우 많음. load 는 share — 트래픽 적음. load 로 먼저 확인, 0 보이면 exchange — TTAS (Test-Test-And-Set) 패턴.

  rwlock        | read 다수 / write 단독 | ~50 ns
  spinlock      | 매우 짧은 critical  | ~5 ns (단 contention 시 spin)
  semaphore     | counting, signaling | ~30 ns
  cond variable | wait / notify       | ~30 ns (wait 시 mutex 비용 추가)
  barrier       | N 명의 동기 진행     | ~100 ns (N 명 분 wait)
  pthread_once  | 한 번 초기화         | ~10 ns (init 후 거의 무비용)
  futex (직접)   | 모든 위의 기반       | ~10 ns
  ─────────────────────────────────────────────
  
  contention 시:
    모두 비용 매우 증가 (μs ~ ms)
    spinlock 의 spin 은 CPU 낭비
    mutex 의 syscall 은 latency

선택 가이드

[동기화 primitive 선택 가이드]

  단독 접근 보장:
    mutex (대부분의 경우)
    
  매우 짧은 critical (< 1μs, kernel 안 가도 됨):
    spinlock — 단 multi-core 에서만
    
  read 가 대부분, write 가 가끔:
    rwlock — 단 측정 후 결정
    
  조건 대기:
    mutex + cond variable (대부분)
    semaphore — counting 필요할 때
    eventfd — file descriptor 통합 필요
    
  N 개 동기:
    barrier
    
  한 번 초기화:
    pthread_once (또는 C11 의 once_flag)
    
  thread 마다 다른 값:
    pthread_key + setspecific (49 장)
    또는 __thread keyword

동기화 primitive 들 정리:

[rwlock, spinlock, barrier 정리]

  rwlock:
    ✓ read 여러, write 단독
    ✓ read-heavy 응용에 유리 (90%+ read)
    ✓ writer starvation 가능 (정책 선택)
    ✓ 성능 측정 후 사용 결정
    
  spinlock:
    ✓ kernel 안 가는 lock (busy wait)
    ✓ 매우 짧은 critical section 전용
    ✓ multi-core 에서만 의미
    ✓ critical section 길면 절대 금지
    ✓ TTAS 패턴 (cache friendly)
    
  barrier:
    ✓ N 명의 동기 진행

다음 장에서는 — POSIX semaphore 를 본다. cond variable 의 단순한 대안.

48semaphore — counting 동기화

semaphore 가 — Dijkstra (1965) 의 고전적 동기화 primitive. mutex 와 cond variable 보다 _더 단순한 인터페이스_ 로 많은 동기화 패턴을 표현. 단 하나의 unique 한 능력 — async-signal-safe (signal handler 에서 호출 가능).

semaphore 의 본질

[semaphore 의 본질]

  개념:
    정수 counter (≥ 0)
    sem_wait: counter -= 1
              0 이면 block (양수 될 때까지)
    sem_post: counter += 1
              waiter 있으면 깨움
              
  비유:
    헬스장의 신발장 열쇠 (5 개)
    sem_wait = 열쇠 가져가기 (5 → 4, 없으면 대기)
    sem_post = 열쇠 반납 (4 → 5)
    
  → counting semaphore
  → binary semaphore (값이 0 또는 1 만) = mutex 비슷

  sem_getvalue: 현재 counter 값

glibc 의 sem_t 구조

// [user-space - glibc nptl/internaltypes.h master 단순화]
struct new_sem {
#if __HAVE_64B_ATOMICS
    /* 64-bit 환경: value + nwaiters 를 64 bit 안에 */
    __atomic_wide_counter data;
#else
    /* 32-bit: 분리 */

    unsigned int nwaiters;
#endif
    int private;     /* PTHREAD_PROCESS_PRIVATE 또는 SHARED */
    int pad;
};

data 안에 — value (token 수) 와 nwaiters (대기 thread 수) 가 함께. atomic update 로 race-free.

sem_wait 의 fast path

master 의 단순화:

// [user-space - glibc nptl/sem_waitcommon.c master 단순화]
static int
__new_sem_wait_fast(struct new_sem *sem, int definitive_result)
{
    /* atomic load — value 확인 */
    uint64_t d = atomic_load_relaxed(&sem->data);
    
    do {
        if ((d & SEM_VALUE_MASK) == 0)
            return -1;     /* counter 가 0, slow path 로 */
    } while (!atomic_compare_exchange_weak_acquire(&sem->data, &d, d - 1));
    
    return 0;     /* 성공 — counter 감소 */
}

uncontended (counter > 0) 케이스 — 한 atomic CAS 만. syscall 없음. mutex 의 fast path 와 같이 빠름 (~20 ns).

sem_wait 의 slow path

counter 가 0 이면 — futex_wait 으로 sleep.

// [user-space - glibc nptl/sem_waitcommon.c master 단순화]
static int
__new_sem_wait_slow(struct new_sem *sem, ...)
{
    /* nwaiters += 1 */
    atomic_fetch_add_relaxed(&sem->data, SEM_NWAITERS_INC);
    
    while (1) {
        /* 다시 시도 */
        if (__new_sem_wait_fast(sem, 0) == 0)
            break;     /* token 획득 */
        
        /* futex_wait: data 가 변할 때까지 대기 */
        __futex_abstimed_wait(&sem->data, SEM_NWAITERS_MASK, ...);
        /* 깨면 다시 시도 */
    }
    
    /* nwaiters -= 1 */
    atomic_fetch_sub_relaxed(&sem->data, SEM_NWAITERS_INC);
    
    return 0;
}

sem_post 의 동작

// [user-space - glibc nptl/sem_post.c master 단순화]
int
__new_sem_post(sem_t *sem)
{
    /* atomic add — value += 1 */
    uint64_t d = atomic_fetch_add_release(&sem->data, 1);
    
    if ((d & SEM_NWAITERS_MASK) != 0) {
        /* waiter 있음 — futex_wake */
        __futex_wake(&sem->data, 1, sem->private);
    }
    
    return 0;
}

흐름:

[sem_post 의 동작]

  1. atomic add — value += 1
     fetch_add 가 이전 값 반환
     
  2. 이전 값에서 nwaiters 부분 검사:
     0: waiter 없었음, syscall 불필요 ✓ (fast path)
     0 아님: waiter 있었음, futex_wake 한 명 깨움
     
  → uncontended sem_post 도 syscall 없음

sem_wait 의 async-signal-safe 능력

매우 unique 한 특성 — signal handler 에서 안전하게 호출 가능.

[async-signal-safe 의 의미]

  signal handler 안에서 호출 가능한 함수는 매우 제한됨:
    - signal-safe list 만 (man 7 signal-safety)
    - malloc, printf 등 _대부분_ 불가
    
  pthread 의 동기화 primitive 중:
    pthread_mutex_lock     ✗ (NOT safe)
    pthread_cond_signal    ✗
    pthread_cond_broadcast ✗
    
    sem_post              ✓ (★ POSIX 명시)
    
  → signal handler 에서 main thread 와 동기화 시 sem_post 만 옵션

이게 — semaphore 의 가장 unique 한 능력. 다른 동기화 primitive 와 차별점.

signal handler 와 main 의 동기화 패턴

// [user-space - C 예제]
#include <signal.h>
#include <semaphore.h>

sem_t sem;

void handler(int sig) {
    sem_post(&sem);                /* ★ signal-safe! */
    /* main thread 가 처리하도록 알림 */
}

int main(void) {
    sem_init(&sem, 0, 0);
    
    signal(SIGUSR1, handler);
    
    while (1) {
        sem_wait(&sem);            /* signal 받을 때까지 대기 */
        /* signal 처리 */
        process_signal();
    }
}

이 패턴 — self-pipe trick 의 대안. 더 효율적.

producer-consumer with semaphores

cond variable 없이 — semaphore 둘로 producer-consumer 구현.

// [user-space - C 예제, semaphore 기반 producer-consumer]
#define QUEUE_SIZE 100

typedef struct {
    int items[QUEUE_SIZE];
    int head, tail;
    pthread_mutex_t mtx;     /* 데이터 보호 */
    sem_t empty;              /* 빈 슬롯 수 */
    sem_t full;               /* 채워진 슬롯 수 */
} queue_t;

void queue_init(queue_t *q) {
    q->head = q->tail = 0;
    pthread_mutex_init(&q->mtx, NULL);
    sem_init(&q->empty, 0, QUEUE_SIZE);     /* 처음엔 다 빔 */
    sem_init(&q->full, 0, 0);                /* 처음엔 0 개 */
}

void enqueue(queue_t *q, int item) {
    sem_wait(&q->empty);     /* 빈 슬롯 있을 때까지 대기 */
    
    pthread_mutex_lock(&q->mtx);
    q->items[q->tail] = item;
    q->tail = (q->tail + 1) % QUEUE_SIZE;
    pthread_mutex_unlock(&q->mtx);
    
    sem_post(&q->full);      /* 채워진 슬롯 += 1 */
}

int dequeue(queue_t *q) {
    sem_wait(&q->full);      /* 채워진 슬롯 있을 때까지 대기 */
    
    pthread_mutex_lock(&q->mtx);
    int item = q->items[q->head];
    q->head = (q->head + 1) % QUEUE_SIZE;
    pthread_mutex_unlock(&q->mtx);
    
    sem_post(&q->empty);     /* 빈 슬롯 += 1 */
    return item;
}
[semaphore 버전의 producer-consumer 장점]

  ✓ cond variable 보다 코드 단순
  ✓ while loop 의 spurious wakeup 처리 불필요
  ✓ counting semaphore 가 자연스럽게 슬롯 수 추적
  
  단점:
    mutex 와 semaphore 의 잘못된 ordering 시 deadlock 위험
    예: 안에서 sem_wait 하면 lock 잡은 채로 대기

mutex 와 semaphore 의 차이

[mutex vs semaphore]

  mutex:
    - 소유 개념 (owner thread)
    - lock 한 thread 만 unlock
    - 가장 빠름 (fast path)
    - 한 번에 한 명
    
  binary semaphore (value=1):
    - 소유 개념 _없음_ (anyone can post)
    - signal handler 에서 사용 가능
    - 약간 더 느림
    - 한 번에 한 명
    
  counting semaphore (value>1):
    - N 명 동시 허용
    - 자원 pool (예: connection pool)
    - rate limiting

binary semaphore 가 — mutex 와 _기능적으로 비슷_ 하지만, owner 개념이 없어서 다른 thread 가 post 가능. mutex 와 semaphore 의 핵심 차이.

connection pool 예제

// [user-space - C 예제, 자원 풀]
#define MAX_CONNECTIONS 10

sem_t pool;
sem_init(&pool, 0, MAX_CONNECTIONS);     /* 10 개 connection */

connection_t connections[MAX_CONNECTIONS];
pthread_mutex_t pool_mtx = PTHREAD_MUTEX_INITIALIZER;

connection_t *acquire_connection(void) {
    sem_wait(&pool);     /* connection 사용 가능할 때까지 대기 */
    
    pthread_mutex_lock(&pool_mtx);
    /* free connection 찾기 */
    connection_t *c = find_free(connections);
    c->in_use = 1;
    pthread_mutex_unlock(&pool_mtx);
    
    return c;
}

void release_connection(connection_t *c) {
    pthread_mutex_lock(&pool_mtx);
    c->in_use = 0;
    pthread_mutex_unlock(&pool_mtx);
    
    sem_post(&pool);     /* 자원 반납 */
}

semaphore 가 — connection 수 자연스럽게 추적. 11 번째 thread 는 sem_wait 에서 대기.

49TLS — Thread-Local Storage

40 장에서 — errno 가 글로벌 변수처럼 보이지만 사실 thread-local 이라고 했다. 그 메커니즘이 — TLS (Thread-Local Storage). 모든 thread 가 _자기만의 errno_ 를 보는 비밀.

TLS 의 본질

// [user-space - C 예제, __thread keyword]
__thread int counter = 0;

void *worker(void *arg) {
    counter++;
    /* 다른 thread 의 counter 와 _완전히 독립_ */
    printf("my counter: %d\n", counter);
    return NULL;
}

int main(void) {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    
    /* main 의 counter 도 독립 */
    printf("main counter: %d\n", counter);     /* 0 */
}

각 thread 가 — **자기만의 counter** 변수. mutex 없이도 race 없음. 비밀은 — 각 thread 의 메모리에 별도 instance.

TLS 의 두 방식

[TLS 의 두 방식]

  1. __thread keyword (정적 TLS)
     컴파일 시 결정
     C11 의 _Thread_local 도 같은 의미
     매우 빠름 (register + offset)
     
  2. pthread_key_create (동적 TLS / TSD)
     실행 시 key 만들고 value 저장
     약간 느림 (함수 호출)
     destructor 등록 가능 (자동 cleanup)
     POSIX 표준

[__thread 의 특성]

  ✓ 매우 빠름 (~1 ns access)
  ✓ 단순한 syntax
  ✓ 컴파일러 + linker 가 처리
  
  ✗ 정적으로 결정된 변수만
  ✗ destructor 없음 (수동 정리)
  ✗ POSIX 표준 아님 (그러나 거의 모든 컴파일러 지원)

x86_64 의 TLS 구현 — %fs register

x86_64 에서 — %fs (segment register) 가 thread 의 메모리 영역 가리킴.

[x86_64 TLS layout]

  %fs:0 부터 시작 — struct pthread (TCB)
  
  ┌────────────────────────────┐ ← %fs
  │ tcbhead_t (TCB header)      │
  │  ├─ tcb (self pointer)       │
  │  ├─ dtv (DTV pointer)        │
  │  ├─ self (struct pthread*)   │
  │  ├─ stack_guard              │
  │  ├─ pointer_guard            │
  │  └─ ...                      │
  ├────────────────────────────┤
  │ struct pthread 의 나머지     │
  │  ├─ tid                      │
  │  ├─ cancelhandling           │
  │  └─ ...                      │
  ├────────────────────────────┤ ← %fs:(static_tls_size)
  │ static TLS (negative offset) │
  │  __thread 변수들             │
  └────────────────────────────┘

__thread int x; 접근의 어셈블리:

; x86_64 — IE (Initial Exec) 모델
movq  %fs:x@gottpoff(%rip), %rax    ; x 의 offset 을 GOT 에서
addq  %fs:0, %rax                    ; %fs base + offset = x 의 주소
movl  (%rax), %eax                   ; x 의 값 읽기

; 또는 LE (Local Exec):
movl  %fs:x@tpoff, %eax              ; 한 instruction!

매우 빠름 — register + offset 으로 직접 접근.

/* x86_64: */

pthread_key_t — 동적 TLS

POSIX 의 동적 변종 — 실행 시 key 만들고 value 저장.

// [user-space - POSIX API]
#include <pthread.h>

int pthread_key_create(pthread_key_t *key, void (*destructor)(void *));
int pthread_key_delete(pthread_key_t key);

int pthread_setspecific(pthread_key_t key, const void *value);
void *pthread_getspecific(pthread_key_t key);
// [user-space - C 예제]
static pthread_key_t key;
static pthread_once_t key_once = PTHREAD_ONCE_INIT;

void cleanup(void *value) {
    free(value);     /* thread 종료 시 자동 호출 */
}

void init_key(void) {
    pthread_key_create(&key, cleanup);
}

void *worker(void *arg) {
    pthread_once(&key_once, init_key);
    
    /* thread 별 buffer 할당 */
    char *buf = malloc(1024);
    pthread_setspecific(key, buf);
    
    /* 사용 */
    buf = pthread_getspecific(key);
    strcpy(buf, "hello");
    
    /* thread 종료 시 cleanup 자동 호출 → free(buf) */
    return NULL;
}

pthread_key 의 destructor — 자동 정리

매우 unique 한 기능 — thread 종료 시 자동 cleanup.

[pthread_key destructor]

  pthread_key_create(&key, destructor):
    각 thread 마다 key 의 value 가 있을 수 있음
    
  thread 종료 시:
    각 key 의 destructor 가 그 thread 의 value 로 호출됨
    NULL 이면 호출 안 함
    
  사용처:
    thread 별 buffer, connection 등 자동 해제
    
  주의:
    destructor 안에서 또 다른 setspecific 가능
    → 반복 (최대 PTHREAD_DESTRUCTOR_ITERATIONS = 4 회)

__thread vs pthread_key 비교

[__thread vs pthread_key]

  __thread:
    + 매우 빠름 (~1 ns)
    + 단순한 syntax
    + 컴파일 시 결정
    - destructor 없음 (수동 정리)
    - 정적 변수만
    
  pthread_key:
    + destructor 자동 cleanup
    + 동적 (runtime 결정)
    + POSIX 표준
    - 약간 느림 (~10-30 ns, 함수 호출)
    - 키 수 제한 (보통 1024)
    - setspecific/getspecific 호출 필요

대부분은 — **__thread 우선**. cleanup 필요한 자원이면 — pthread_key.

__tls_get_addr — 동적 라이브러리의 TLS

dlopen 으로 로드된 라이브러리의 __thread 변수 — 다른 경로 접근.

// [user-space - glibc 단순화]
typedef struct dl_tls_index {
    uint64_t ti_module;     /* 어느 모듈 */
    uint64_t ti_offset;     /* 모듈 내 offset */
} tls_index;

void *__tls_get_addr(tls_index *ti) {
    dtv_t *dtv = THREAD_DTV();
    void *p = dtv[ti->ti_module].pointer.val;
    return (char *)p + ti->ti_offset;
}
[__tls_get_addr 의 역할]

  dlopen 으로 로드된 .so 의 __thread 변수 접근:
    컴파일 시 모듈 ID 와 offset 모름 (런타임 결정)
    → __tls_get_addr 호출
    → DTV (Dynamic Thread Vector) 조회
    → 실제 주소 반환
    
  성능:
    initial-exec 모델: 빠름 (직접 fs:offset)
    general-dynamic 모델: __tls_get_addr 호출 (~10 ns)

DTV — Dynamic Thread Vector

각 thread 가 — 모든 모듈의 TLS 영역 주소를 가지는 배열.

[DTV 의 구조]

  dtv[0] = generation count (변경 시 +1)
  dtv[1] = main executable 의 TLS
  dtv[2] = .so #1 의 TLS
  dtv[3] = .so #2 의 TLS
  ...
  
  dlopen 시:
    DTV 의 크기 늘림
    새 모듈의 TLS 영역 할당
    DTV[N] 에 등록
    
  dlclose 시:
    그 entry NULL 로
    메모리 해제

TLS access model 4 가지

x86_64 의 컴파일러가 — TLS 접근에 4 가지 model 중 하나 선택.

[TLS access model 4 가지]

  1. Local Exec (LE) — 가장 빠름
     사용처: 메인 실행 파일의 __thread 변수
     코드: movl %fs:x@tpoff, %eax  (한 instruction)
     컴파일 옵션: -ftls-model=local-exec
     
  2. Initial Exec (IE) — 빠름
     사용처: 시작 시 로드된 .so 의 __thread 변수
     코드: movq x@gottpoff(%rip), %rax
           movl %fs:(%rax), %eax  (2 instructions)
     컴파일 옵션: -ftls-model=initial-exec
     
  3. Local Dynamic (LD) — 보통
     사용처: 같은 .so 안의 여러 __thread 변수 (한 번 lookup 으로 묶기)
     코드: __tls_get_addr 한 번 호출 + 여러 offset 사용
     컴파일 옵션: -ftls-model=local-dynamic
     
  4. General Dynamic (GD) — 가장 일반적
     사용처: dlopen 으로 로드 가능한 .so
     코드: __tls_get_addr 호출
     컴파일 옵션: -ftls-model=global-dynamic

기본은 — General Dynamic (가장 호환성 좋음). 성능 critical 한 곳은 — initial-exec 또는 local-exec 명시.

코드 예 — LE vs GD

// [user-space - C 예제]
__thread int x;        // 메인 실행 파일

int get_x(void) {
    return x;
}
; Local Exec (메인 실행 파일 빌드 시):
get_x:
    movl %fs:x@tpoff, %eax    ; 매우 빠름!
    ret

; General Dynamic (-fPIC, .so 빌드 시 기본):
get_x:
    leaq x@tlsgd(%rip), %rdi
    call __tls_get_addr@PLT   ; 함수 호출!
    movl (%rax), %eax
    ret

같은 코드 — 컴파일 옵션 따라 매우 다른 성능.

[thread 의 TLS 메모리 layout — x86_64]

  높은 주소
  ┌────────────────────────────────┐
  │ thread stack (8 MB)              │
  │                                  │
  ├────────────────────────────────┤
  │ struct pthread (TCB)             │ ← %fs:0 (의 한 부분)

TLS 사용 가이드

[언제 TLS 사용]

  ✓ 변수가 매 thread 마다 독립적 (mutex 보다 자연스러움)
  ✓ thread 별 buffer, cache, scratch space
  ✓ thread 별 counter, statistic
  ✓ thread 별 context (예: graphics, DB connection)
  
  ✗ thread 간 _공유_ 가 필요한 데이터 — TLS 부적합

TLS 의 함정

[TLS 함정]

  ✗ initializer 의 제약:
     __thread int x = 42;   // OK
     __thread int *p = &y;  // .so 의 GD 모델에선 어려움 (relocation)
     
  ✗ dlopen 후 dlclose:
     DTV 의 entry 가 freed memory 가리킬 수 있음
     glibc 가 이를 처리하나 미묘한 race 가능
     
  ✗ thread 안에서 stack 의 TLS 주소 다른 thread 전달:
     다른 thread 에서 그 주소 무효
     
  ✗ signal handler 의 TLS:
     async-signal-safe 측면에서 위험
     handler 가 같은 thread 의 TLS 만 안전하게 접근
     
  ✗ 매우 많은 thread:
     TLS 영역 mmap 메모리 누적
     
  ✗ pthread_key 의 race:
     pthread_key_create 와 동시 setspecific
     → 보통 pthread_once 로 보호

// [user-space - C 예제, library 의 thread-safe 패턴]
__thread struct my_context ctx = {0};

void mylib_init(void) {
    if (!ctx.initialized) {
        ctx.buffer = malloc(8192);
        ctx.fd = -1;
        ctx.initialized = 1;
    }
}

void mylib_cleanup(void) {
    if (ctx.initialized) {
        free(ctx.buffer);
        if (ctx.fd >= 0) close(ctx.fd);
        ctx.initialized = 0;
    }
}

/* 모든 thread 가 자기만의 ctx — race 없음 */

pthread 에 — 다른 thread 를 강제 종료시키는 기능이 있다. pthread_cancel. 매우 위험한 기능 — 거의 모든 production 코드가 사용을 회피. 이 장은 — 왜 위험한지의 깊이.

cancellation 의 의미

[cancellation 의 동작]

  pthread_cancel(t):
    t 에게 "종료해" 신호 보냄
    실제 종료 시점은 — t 가 cancellation point 도달할 때
    
  cancellation point 의 의미:
    syscall 또는 라이브러리 함수의 _특정 지점_
    POSIX 가 명시한 함수들 (예: read, write, sleep, pthread_cond_wait)
    
  종료 방식:
    내부적으로 — 특수한 exception 발생
    cleanup handlers 실행
    TSD destructors 실행
    pthread_exit(PTHREAD_CANCELED) 와 같은 효과

cancellation 의 두 type

[cancellation type]

  PTHREAD_CANCEL_DEFERRED (기본):
    cancellation point 에서만 종료
    예: read 호출 중 → cancel 도착 → read 가 종료점에서 멈춤
    → 약간 더 안전
    
  PTHREAD_CANCEL_ASYNCHRONOUS:
    ★ 즉시 종료
    어느 시점이든 (memory access, register update 중)
    → 매우 위험 (race, 자원 누수)
    → 거의 안 씀
    
  PTHREAD_CANCEL_DISABLE / ENABLE:
    cancellation 자체를 잠시 끔

대부분 — DEFERRED 만 사용 권장. ASYNCHRONOUS 는 정말 특수한 경우만.

자원 누수의 함정

가장 큰 위험 — lock 잡은 채로 cancel.

// [user-space - C 위험 예제]
pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;

void *worker(void *arg) {
    pthread_mutex_lock(&mtx);
    
    /* 여기서 cancel 도달 가능! */
    /* (mutex_lock 자체가 cancellation point) */
    
    /* 또는: */
    read(fd, buf, n);    /* ← 여기서 cancel */
    
    pthread_mutex_unlock(&mtx);    /* 실행 안 됨! */
    return NULL;
}

/* mtx 영원히 잠긴 채로 thread 종료
   → 다른 thread 영원히 대기 → deadlock */
[cancellation 의 자원 누수]

  ✗ mutex 잡은 채 종료 → 다른 thread 영원히 대기
  ✗ malloc 한 메모리 free 안 함 → 누수
  ✗ open 한 fd 안 닫힘 → 누수
  ✗ malloc 의 중간 상태 (arena lock) → heap 손상 가능
  
  → 모든 cancellation point 마다 _자원 정리_ 필요
  → 매우 복잡

pthread_cleanup_push / pop — 자동 정리

cancellation 시 자동 실행되는 cleanup handler.

// [user-space - C 예제]
void cleanup(void *arg) {
    pthread_mutex_t *mtx = arg;
    pthread_mutex_unlock(mtx);    /* lock 풀어줌 */
}

void *worker(void *arg) {
    pthread_mutex_lock(&mtx);
    
    pthread_cleanup_push(cleanup, &mtx);     /* cancel 시 호출됨 */
    
    /* critical section — cancel 가능 */
    read(fd, buf, n);
    
    pthread_cleanup_pop(1);     /* 1: cleanup 실행, 0: skip */
    /* pop 의 1 이 unlock 함 */
    
    return NULL;
}
[pthread_cleanup_push/pop 의 동작]

  push: stack 에 cleanup 등록
  pop: stack 에서 제거
    인자 0: 등록만 했었음 (cleanup 안 실행)
    인자 1: cleanup 실행 후 제거
    
  cancellation 시:
    push 한 모든 handler 가 LIFO 순서로 호출
    그 다음 thread 종료
    
  ★ push/pop 은 같은 scope (같은 { }) 내에서만!
    매크로로 구현되어 있어 (setjmp/longjmp 비슷)
    if/while 같은 statement 안에서 시작/끝 안 됨

pthread_cleanup_push 의 매크로 트릭

// [user-space - glibc pthread.h master 단순화]
#define pthread_cleanup_push(routine, arg) \
    do { \
        __pthread_cleanup_buffer __clb; \
        _pthread_cleanup_push(&__clb, (routine), (arg));

#define pthread_cleanup_pop(execute) \
        _pthread_cleanup_pop(&__clb, (execute)); \
    } while (0)

영리한 trick — **do { ... } while(0)** 으로 scope 묶음. 같은 scope 내 push/pop 강제.

가장 안전한 패턴 — disable

cancel 받기 싫은 critical section 은 — 잠시 disable.

// [user-space - C 예제]
int oldstate;
pthread_setcancelstate(PTHREAD_CANCEL_DISABLE, &oldstate);

/* critical section — cancel 받지 않음 */
pthread_mutex_lock(&mtx);
do_critical_work();
pthread_mutex_unlock(&mtx);

pthread_setcancelstate(oldstate, NULL);    /* 원래 상태로 */

이 패턴이 — 가장 안전. 단 critical section 길면 — cancel 의 응답성 떨어짐.

거의 모든 라이브러리가 cancellation 거부

대부분의 C library (특히 glibc 자체) — cancellation safety 보장 안 함.

[cancellation 의 현실]

  많은 라이브러리:
    cancellation safe 가 _아님_
    → cancel 시 내부 자료구조 깨질 수 있음
    
  malloc 의 위험:
    arena lock 잡은 채로 cancel → 다른 malloc 영원히 대기
    
  glibc 의 stdio:
    FILE 의 lock 잡은 채로 cancel → 다른 thread 의 printf 영원히 block
    → glibc 의 fopen 에 "c" mode 가 있는 이유 (cancellation point 안 됨)
    
  C++ STL:
    exception safety 와 cancellation 의 호환성 미보장
    
  → 결과적으로: cancellation 거의 안 쓰는 게 권장

권장 대안 — flag + check

// [user-space - C 권장 패턴]
volatile sig_atomic_t should_exit = 0;

void *worker(void *arg) {
    while (!should_exit) {
        do_some_work();
        
        /* 가끔 검사 */
        if (should_exit) break;
        
        do_more_work();
    }
    
    /* 정상 종료 — 모든 자원 정리 */
    cleanup_resources();
    return NULL;
}

int main(void) {
    pthread_t t;
    pthread_create(&t, NULL, worker, NULL);
    
    sleep(5);
    should_exit = 1;            /* cancel 대신 flag */
    
    pthread_join(t, NULL);
    return 0;
}
[flag 패턴의 장점]

  ✓ 종료 시점 명확 (사용자가 제어)
  ✓ 자원 정리 보장
  ✓ atomic 한 사용 (sig_atomic_t)
  ✓ signal handler 에서도 set 가능
  ✓ 디버그 쉬움

blocking I/O 의 처리

flag 패턴의 어려움 — read 같은 blocking syscall 안에서 멈춰있을 때.

[blocking I/O 처리 방법]

  방법 1: timeout 사용
    select/poll/epoll 의 timeout
    또는 SO_RCVTIMEO socket option
    
  방법 2: pipe trick
    pipe 만들고 fd 도 select 에 추가
    종료 시 pipe 에 byte 보냄 → select 깨어남
    
  방법 3: pthread_kill 로 signal
    blocking syscall 이 EINTR 로 반환
    signal handler 가 should_exit 설정
    
  방법 4: eventfd
    pipe 와 비슷하지만 더 가벼움

        {.fd = data_fd, .events = POLLIN},
        {.fd = exit_fd, .events = POLLIN},

내부적으로 — POSIX exception 으로 구현.

[glibc 의 cancellation 구현]

  pthread_cancel:
    target thread 에 signal (SIGCANCEL = 32)

cancellation safety 의 분류

POSIX 가 — 함수들을 cancellation safety 로 분류:

[cancellation safety 분류]

  Async-Cancel-Safe:
    어떤 cancellation 도 안전
    매우 적음 (pthread_cancel, pthread_setcancelstate, ...)
    
  Cancel-Safe (synchronous):
    deferred cancel 만 안전
    예: 대부분의 syscall wrapper
    
  Cancel-Unsafe:
    cancel 시 자원 손상 가능
    예: malloc, free, printf, ...
    매우 많음

대부분의 함수가 — cancel-unsafe. 그래서 cancellation 사용 시 매우 조심해야.

51NPTL — glibc 의 thread 구현

지금까지 본 모든 pthread 메커니즘 — pthread_create, mutex, cond variable, TLS 등 — 의 구현 이름은 NPTL (Native POSIX Thread Library). 2003 년 Red Hat 이 LinuxThreads 를 대체. 이 장은 — NPTL 의 디자인과 그 진화의 역사.

Linux 의 thread 역사

[Linux thread 구현의 진화]

  1996: LinuxThreads (Xavier Leroy)
    Linux 2.0 의 첫 thread 구현
    clone() 기반
    그러나 POSIX 호환 불완전
    
  2003: NPTL (Ulrich Drepper, Ingo Molnar 등)
    Red Hat 주도
    kernel 의 새 primitives 활용 (futex, CLONE_SETTLS 등)
    완전한 POSIX 호환
    엄청난 성능 향상
    
  2003 ~ 현재: glibc 2.4+ 의 기본
    NGPT (IBM 의 대안) 와 경쟁 후 NPTL 승리

LinuxThreads 의 문제점

NPTL 등장 전 LinuxThreads 의 한계:

[LinuxThreads 의 문제]

  1. manager thread:
     pthread_create 마다 — 별도 manager thread 가 처리
     thread 생성 시 manager 와 통신 (pipe)
     → 매우 느림 (20+ μs)
     → 단일 thread bottleneck
     
  2. getpid 가 다름:
     각 thread 가 _다른 PID_ 반환
     POSIX 의 "한 process = 한 PID" 위반
     /proc 에서 thread 가 별도 process 처럼
     
  3. signal 처리:
     signal 이 _하나의 thread_ 에만 (manager 가 받음)
     SIGUSR 같은 process-wide signal 의 잘못된 처리
     
  4. 자원 한도:
     8192 thread 제한 (LDT entries)
     
  5. ptrace, gdb:
     debug 시 thread 들이 _다른 process_ 처럼
     매우 혼란
     
  6. TLS 의 제한:
     LDT 기반 (느림, 한계)

이 모든 문제가 — NPTL 에서 해결.

NPTL 의 설계 원칙

[NPTL 의 설계 결정]

  1. M:N → 1:1 매핑:
     user thread = kernel task (CLONE_THREAD)
     LinuxThreads 도 1:1 였으나 — POSIX 의미 부족
     NPTL: 진정한 POSIX 의 thread 의미
     
  2. kernel 협조:
     kernel 의 새 syscall 과 flag 활용
     - futex (FUTEX_WAIT, FUTEX_WAKE)
     - CLONE_SETTLS, CLONE_CHILD_CLEARTID
     - tgkill, gettid
     - exit_group (전체 thread group 종료)
     
  3. 효율적 동기화:
     uncontended fast path (atomic CAS, syscall 없음)
     contended slow path (futex)
     
  4. 빠른 TLS:
     GDT 기반 (LDT 의 한계 없음)
     %fs register 직접
     
  5. manager thread 제거:
     모든 thread 가 자기 자신 관리
     bottleneck 없음

kernel 의 협조 — 새 primitives

NPTL 이 도입되며 — kernel 도 변경 필요. 새 syscall 들:

[NPTL 을 위한 새 kernel 기능]

  futex (Linux 2.6.0, 2003):
    효율적 사용자 공간 lock
    Fast Userspace muTEX
    uncontended 면 syscall 없음
    
  CLONE_SETTLS:
    clone 의 새 flag
    child 의 TLS 영역 자동 설정
    
  CLONE_CHILD_CLEARTID:
    child 종료 시 tid 위치 0 으로
    futex wake 자동
    → pthread_join 의 효율적 메커니즘
    
  CLONE_PARENT_SETTID:
    parent 에 child tid 자동 전달
    
  CLONE_THREAD:
    같은 thread group (tgid 공유)
    → 같은 PID
    
  tgkill, tkill (2.5.4):
    특정 thread 에 signal
    pthread_kill 의 기반
    
  gettid:
    kernel level tid 얻기
    
  exit_group:
    전체 thread group 종료
    main thread 의 exit() 가 사용
    
  set_robust_list:
    robust mutex 의 cleanup
    thread 죽으면 자동 EOWNERDEAD

이 모든 게 — NPTL 을 위해 kernel 측이 함께 추가한 기능.

NPTL 의 thread 구조

[NPTL 의 thread layout]

  각 thread:
    ─────────────────
    main thread (PID = TID = TGID):

    독립: stack, TLS, errno, kernel task

__pthread_initialize_minimal — 초기화

main thread 가 NPTL 사용 시작 시.

// [user-space - glibc nptl/nptl-init.c master 단순화]
void
__pthread_initialize_minimal_internal(void)
{
    /* main thread 의 struct pthread 초기화 */
    static struct pthread initial_pthread = {.tid = -1};
    
    /* TLS 설정 */
    INIT_THREAD_TCB(&initial_pthread);
    
    /* main thread 의 tid 얻기 */
    initial_pthread.tid = INTERNAL_SYSCALL_CALL(set_tid_address,
                                                  &initial_pthread.tid);
    
    /* SIGCANCEL signal handler 등록 */
    struct sigaction sa;
    sa.sa_handler = sigcancel_handler;
    sigaction(SIGCANCEL, &sa, NULL);
    
    /* stack limit 확인 */
    struct rlimit limit;
    getrlimit(RLIMIT_STACK, &limit);
    __default_pthread_attr.stacksize = limit.rlim_cur;
}
[NPTL 초기화 단계]

  process 시작 시:
    1. main thread 의 struct pthread 설정
    2. TLS 의 %fs base 설정 (arch_prctl)
    3. set_tid_address syscall — main thread 의 tid 저장 위치
    4. SIGCANCEL handler 등록
    5. stack limit 확인
    
  → 이후 pthread API 사용 가능

NPTL 의 stack 캐싱

매우 영리한 최적화 — stack 의 재사용.

// [user-space - glibc nptl/allocatestack.c master 단순화]
static int stack_cache_actsize;
static list_t stack_cache = LIST_HEAD_INIT(stack_cache);
static size_t stack_cache_maxsize = 40 * 1024 * 1024;    /* 40 MB */

static int
allocate_stack(const struct pthread_attr *attr, struct pthread **pdp, ...)
{
    /* 1. cache 에서 적절한 크기의 stack 찾기 */
    pd = get_cached_stack(&size, &mem);
    if (pd != NULL) {
        /* cache 히트! mmap 안 함 */
        goto got_stack;
    }
    
    /* 2. 없으면 mmap */
    mem = mmap(NULL, size, prot, MAP_PRIVATE | MAP_ANONYMOUS | MAP_STACK,
               -1, 0);
    /* guard page 추가 */
    mprotect(mem, guardsize, PROT_NONE);
    
    pd = (struct pthread *)((char *)mem + size - sizeof(struct pthread));
    
got_stack:
    *pdp = pd;
    return 0;
}
[stack 캐싱의 효과]

  pthread_create 호출:
    cache 히트 → mmap 없음, 매우 빠름 (~5 μs)
    cache 미스 → mmap + setup (~20 μs)
    
  thread 종료 시:
    cache 에 저장 (40 MB 한도)
    → 다음 pthread_create 가 재사용
    
  결과:
    thread pool 패턴이 매우 효율적
    short-lived thread 들의 반복 생성도 빠름

NPTL 의 mutex 성능 — futex 의 마법

futex 이전과 이후의 비교:

[mutex 성능 비교 (uncontended lock)]

  pre-NPTL LinuxThreads:
    매번 syscall (kernel mutex)
    ~500 ns
    
  NPTL (futex):
    atomic CAS 만 (kernel 안 감)
    ~20 ns
    
  → 25 배 빠름
  → 짧은 critical section 의 응용에 결정적

futex 의 디자인 천재성

futex 의 핵심 idea — uncontended case 는 kernel 가지 말자.

[전통적 mutex 의 문제]

  mutex_lock:
    무조건 syscall → kernel mode
    kernel 의 wait queue 검사
    경합 없으면 즉시 반환
    user mode 복귀
    
  비용: ~500 ns (syscall 만)
  
  contention 99% 의 경우 — 낭비!

[futex 의 해법]

  user-space 에서 atomic 검사:
    경합 없으면 → 그냥 진행 (syscall 0)
    경합 있으면 → syscall (kernel 의 wait queue)
    
  user-space 의 결정:
    "이 변수 값이 X 일 때 대기" 라는 의미를 atomic 하게 보장
    
  → 99% 의 경우 syscall 없음 → 25x 빠름

futex 의 atomic 보장

futex_wait(addr, val) — "addr 의 값이 val 이면 대기".

[futex 의 race-free 메커니즘]

  thread A:                     thread B:
  ─────────────                  ─────────────
  CAS mutex (0→1)               
  실패 (이미 1)                 CAS mutex (1→2)  /* waiter 표시 */
                                
                                set mutex = 0  /* unlock */
                                futex_wake     /* 깨움 — 그러나 A 아직 안 잠*/
                                
  futex_wait(mutex, 2)          
  kernel: mutex 값이 2 인가?    
  → 아님 (이미 0) → EAGAIN 즉시 반환  /* missed wakeup 없음 */
  
  다시 CAS 시도 → 성공

이 atomic 검사가 — futex 의 race-free 성. user-space + kernel 의 협조의 정수.

52futex — 동기화의 kernel primitives

Part VI 의 마지막 장. 지금까지 본 모든 동기화 primitive — mutex, cond variable, rwlock, semaphore — 의 _밑바닥_ 에 있는 것은 futex (Fast Userspace muTEX). 이 장은 — futex 의 정체와 그 디자인의 천재성.

futex 의 본질

[futex 의 한 줄 정의]

  "user-space 에서 검사하고, 필요할 때만 kernel 로 가는 sleep/wakeup primitive"
  
  핵심 idea:
    user-space: atomic 변수로 빠른 결정
    kernel: wait queue 만 관리
    
  결과:
    uncontended case → 0 syscall (atomic 만)
    contended case → 1 syscall (futex_wait/wake)

futex syscall API

// [user-space - Linux syscall]
#include <linux/futex.h>
#include <sys/syscall.h>

long syscall(SYS_futex, uint32_t *uaddr, int futex_op, uint32_t val,
             const struct timespec *timeout, uint32_t *uaddr2,
             uint32_t val3);
[인자]

  uaddr   : futex 의 user-space 주소 (32-bit aligned)
  futex_op: 동작 (WAIT, WAKE, REQUEUE 등)
  val     : 동작에 따른 값
  timeout : WAIT 의 시간 제한
  uaddr2, val3: 일부 동작의 추가 인자

FUTEX_WAIT — atomic 한 sleep

가장 중요한 동작.

// [user-space - C 의사 코드]
FUTEX_WAIT(addr, val, timeout):
    atomic {
        if (*addr == val) {
            sleep on addr (until WAKE or timeout)
        } else {
            return EAGAIN
        }
    }

핵심 — ***addr == val 검사와 sleep 이 atomic**. race-free 의 마법.

FUTEX_WAKE — sleep 깨움

// [user-space - C 의사 코드]
FUTEX_WAKE(addr, n):
    /* addr 에서 자고 있는 thread 중 _최대 n 개_ 깨움 */
    n == 1: 한 명 깨움 (signal 같은 동작)
    n == INT_MAX: 모두 깨움 (broadcast)

매우 단순한 인터페이스 — WAIT + WAKE. 이걸로 모든 동기화 가능.

가장 단순한 사용 예

// [user-space - C 예제, mutex 구현]
#include <linux/futex.h>
#include <sys/syscall.h>
#include <stdatomic.h>

atomic_int futex = 0;

void simple_lock(void) {
    while (1) {
        int expected = 0;
        if (atomic_compare_exchange_weak(&futex, &expected, 1))
            return;     /* 성공 — lock 획득 */
        
        /* contended — futex 가 0 이 될 때까지 대기 */
        syscall(SYS_futex, &futex, FUTEX_WAIT, 1, NULL, NULL, 0);
        /* "*futex == 1 이면 대기" */
    }
}

void simple_unlock(void) {
    atomic_store(&futex, 0);
    syscall(SYS_futex, &futex, FUTEX_WAKE, 1, NULL, NULL, 0);
    /* 한 명 깨움 */
}
[이 단순한 lock 의 동작]

  uncontended lock:
    CAS 0→1 성공 → 즉시 반환
    syscall 0
    
  contended lock:
    CAS 실패 → futex_wait(futex, 1)
    kernel 에서 sleep
    unlock 시 깨어남
    다시 CAS 시도
    
  uncontended unlock:
    store 0 → futex_wake
    waiter 없으면 wake 는 0 반환 (no-op)
    
  ★ 실제 mutex 는 더 정교 (3 state: 0/1/2 로 wake 도 skip)

futex 의 atomic 한 race-free 보장

매우 중요한 시나리오:

[lost wakeup 방지 시나리오]

  thread A (lock holder):           thread B (waiter):
  ─────────────────────────         ─────────────────────
  현재: futex == 1                  CAS 0→1 실패
  
                                    futex_wait(addr, 1)
                                    
  store 0                           
  futex_wake(addr, 1)               kernel: addr 의 값 == 1 ?
                                            아님 (B 가 검사 직전 A 가 변경)
                                    → EAGAIN 즉시 반환 (안 자고)
                                    → CAS 다시 시도 → 성공
  
  → lost wakeup 없음 ★

이 atomic 검사가 — futex 의 핵심 정확성. kernel 안에서 — addr 의 값 확인 + sleep 이 atomic.

futex 의 kernel 구현 — hash table

kernel 이 — 각 futex 주소 별 wait queue 를 어떻게 관리?

[kernel 의 futex 구현 — global hash table]

  global hash table:
    크기: 2^k 개 bucket (CPU 수 의존)
    각 bucket: spinlock + linked list of waiters
    
  hash 키:
    private futex: mm_struct + address
    shared futex: inode + offset (페이지 단위)
    
  WAIT:
    hash → bucket → lock bucket → insert waiter
    
  WAKE:
    hash → bucket → lock bucket → find waiters

이 디자인이 — kernel 의 메모리 효율. _각 futex 마다 자료구조 만들지 않음_. 충돌 시 같은 bucket 에 — spinlock 으로 직렬화.

FUTEX_PRIVATE_FLAG — 최적화

// [user-space - C]
#define FUTEX_PRIVATE_FLAG  0x80
#define FUTEX_WAIT_PRIVATE  (FUTEX_WAIT | FUTEX_PRIVATE_FLAG)
[private futex 의 최적화]

  process-private futex:
    같은 process 안의 thread 만 사용
    hash 키: mm_struct + address (간단)
    → 빠름
    
  process-shared futex (default):
    여러 process 간 (mmap MAP_SHARED)
    hash 키: inode + offset (복잡)
    → 약간 느림
    
  pthread_mutex 의 기본: PROCESS_PRIVATE → FUTEX_PRIVATE_FLAG
  pthread_mutex 의 PROCESS_SHARED: 일반 futex

대부분의 동기화가 — private 으로 충분. private 의 효율.

futex 의 모든 operation

[futex operations]

  FUTEX_WAIT:
    atomic 검사 + sleep
    
  FUTEX_WAKE:
    n 개 waiter 깨움
    
  FUTEX_REQUEUE:
    한 futex 의 waiter 들을 다른 futex 로 _이동_
    cond_broadcast 의 최적화
    (thundering herd 방지)
    
  FUTEX_CMP_REQUEUE:
    REQUEUE + 값 비교
    
  FUTEX_WAKE_OP:
    WAKE + atomic 연산 (한 syscall 로)
    
  FUTEX_WAIT_BITSET / WAKE_BITSET:
    bit 단위 매칭 (특정 bit set 인 waiter 만)
    
  FUTEX_LOCK_PI, FUTEX_UNLOCK_PI:
    priority inheritance
    real-time 응용
    
  FUTEX_TRYLOCK_PI:
    PI mutex trylock
    
  FUTEX_WAIT_REQUEUE_PI, FUTEX_CMP_REQUEUE_PI:
    PI 의 cond variable 지원

매우 풍부한 API — 다양한 동기화 패턴 효율적 지원.

FUTEX_REQUEUE — broadcast 의 효율

pthread_cond_broadcast 의 단순한 구현:

// 단순한 방법 (느림):
broadcast:
    foreach waiter:
        wake (한 명씩)
    /* 모두 깨움 → 모두 mutex 잡으려 contention → thundering herd */
// FUTEX_REQUEUE 사용 (효율):
broadcast:
    wake 한 명 (cond futex 에서)
    requeue 나머지 → mutex 의 wait queue 로
    /* 한 명만 진짜 깨움, 나머지는 mutex 잡으려 차례로 */
[REQUEUE 의 효과]

  옛 방식:
    100 waiter 모두 깨움
    모두 mutex 잡으려 시도 (99 명 실패)
    99 번의 sleep → 다시 wakeup 사이클
    
  REQUEUE:
    1 명만 깨움
    99 명은 _이미_ mutex 의 wait queue 에
    mutex unlock 시 1 명씩 깨움
    
  → 거대한 성능 차이 (특히 contention 많을 때)

FUTEX_LOCK_PI — priority inheritance

real-time 응용의 매우 중요한 기능.

[FUTEX_LOCK_PI 의 메커니즘]

  보통 futex_lock:
    user-space 의 atomic + futex_wait
    
  FUTEX_LOCK_PI:
    kernel 이 직접 lock 관리 (atomic 이 아닌 kernel level)
    lock holder 의 우선순위를 waiter 의 수준으로 상승
    
  priority inheritance:
    A (낮은 prio) 가 lock 잡음
    B (높은 prio) 가 wait_pi
    → kernel 이 A 의 prio 를 B 수준으로 상승
    → C (중간 prio) 가 A 를 preempt 못 함
    → A 빨리 끝남 → B 가 lock 획득

real-time 시스템 (산업 자동화, 화성 탐사선 등) 에서 필수.

futex 의 성능 측정

[futex 성능 (대략)]

  uncontended (atomic 만):
    ~5 ns (한 atomic CAS)
    
  futex_wait (sleep):
    syscall 자체: ~100 ns
    + context switch: ~1 μs
    + wakeup latency: 가변
    
  futex_wake (waiter 없음):
    syscall: ~100 ns
    no-op 같음
    
  futex_wake (waiter 있음):
    syscall + scheduler wake: ~수 μs

futex 디자인의 천재성

futex 가 진정 천재적인 이유:

[futex 의 디자인 미덕]

  1. user-kernel split 의 영리함:
     자주 일어나는 case (uncontended) 는 user
     드문 case (contention) 만 kernel
     
  2. atomic 검사의 정확성:
     race-free 보장 (lost wakeup 없음)
     
  3. 메모리 효율:
     futex 자체는 4 byte (32-bit int)
     kernel 의 자료구조는 hash table (공유)
     → 백만 개 futex 도 효율적
     
  4. 단순한 인터페이스:
     WAIT, WAKE 만으로 모든 동기화 표현
     mutex, cond var, sem, rwlock, barrier 모두 구현 가능
     
  5. 확장성:
     REQUEUE, BITSET, PI 등 점진적 추가
     모든 케이스에 맞는 변종

[pthread_mutex_lock(&mutex) 한 줄의 깊이]

  user-space:
    atomic CAS: __lock (0 → 1)
       │
       ├─ 성공 → 즉시 반환 (~20 ns, syscall 없음) ★ fast path
       │
       └─ 실패 (contention)
              │
              ▼
    user-space: __lock 을 2 로 표시
              │
              ▼
    kernel:
      futex(SYS_futex, &__lock, FUTEX_WAIT, 2, NULL, NULL, 0)
              │
              ▼
      kernel 의 hash table:
        hash(mm + addr) → bucket
        bucket lock
        atomic check: *addr == 2 ?
          아님 → EAGAIN 반환 (다시 시도)
          맞음 → wait queue 에 추가
                  → schedule_out (sleep)
              │
              ▼
      다른 thread 의 unlock:
        atomic_exchange: 0 → __lock
        이전 값 2 → futex_wake
              │
              ▼
      kernel:
        hash → bucket → waiter 찾기 → wake_up_q
        스케줄러가 우리 깨움
              │
              ▼
    user-space: 다시 atomic CAS 시도
                성공하면 lock 획득

이 모든 게 — pthread_mutex_lock 한 줄.

이제 — thread 의 동작이 완전히 보인다. user-space 의 atomic 부터, futex 의 race-free 보장, kernel 의 hash table, scheduler 의 wakeup 까지.

이 지식이 — multi-threaded 응용의 디버깅, 성능 최적화, 디자인 결정에 직접 도움. 왜 spinlock 이 짧은 critical section 만, 왜 cond variable 이 while loop 필수, 왜 cancellation 안 쓰는지 — 모두 자연스럽게 보임.
