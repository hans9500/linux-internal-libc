Part VIIdynamic linker (ld-linux.so) 의 깊이

Part III 의 14~22 장에서 — _start 부터 main 까지의 program startup 을 봤다. 그때 큰 그림으로 dynamic linking 의 존재를 알았다. Part VII 은 — 그 dynamic linking 의 모든 메커니즘을 본다.

// [user-space - C 단순한 예제]
#include <stdio.h>
int main(void) {
    printf("hello\n");
    return 0;
}

이 단순한 코드 — printf 한 줄 호출에 매우 정교한 기반.

[printf 호출의 dynamic linking 측면]

  컴파일 시:
    printf 는 어디에? — _모름_
    binary 안에 "이 자리에 printf 의 주소가 들어가야 함" 표시만
    
  binary 실행:
    kernel 이 program 과 함께 ld-linux.so 도 mmap
    ld-linux.so 가 entry point
    
  ld-linux.so 가:
    1. /etc/ld.so.cache 에서 libc.so.6 찾음
    2. mmap 으로 libc.so.6 매핑
    3. printf 의 주소 결정 (relocation)
    4. program 의 _start 로 jump
    
  program 의 main:
    printf 호출 시:
      first call: PLT → lazy resolver → libc 의 printf 주소 학습
      subsequent: PLT → 즉시 libc 의 printf

Part VII 의 학습 목표:

[Part VII 의 학습 목표]

  ✓ ld-linux.so 의 정체 (kernel 이 어떻게 로딩)
  ✓ ELF dynamic section 과 link map
  ✓ PLT / GOT 의 구조와 동작
  ✓ lazy binding 의 실제 (_dl_runtime_resolve)
  ✓ symbol resolution 알고리즘
  ✓ relocation types (R_X86_64_*)
  ✓ symbol versioning (GLIBC_2.34)
  ✓ dlopen / dlsym / dlclose
  ✓ LD_PRELOAD / LD_AUDIT 등 hooking
  ✓ ifunc, weak symbol, visibility

Part VII 이 끝나면 — printf("hello\n") 의 호출이 어떻게 libc.so.6 의 printf 주소로 도달하는지의 모든 단계가 보이게 된다.

53ld-linux.so 의 정체 — kernel 의 PT_INTERP 처리

./a.out 실행 시 — 진짜 첫 코드가 실행되는 곳은 a.out 의 _start 가 _아니다_. **ld-linux.so** 가 먼저. 이 장은 — 그 메커니즘을 본다.

file 명령으로 보기

# [user-space - bash]
$ file /bin/ls
/bin/ls: ELF 64-bit LSB pie executable, x86-64, version 1 (SYSV),
         dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2,
         BuildID[sha1]=..., for GNU/Linux 3.2.0, stripped

$ readelf -l /bin/ls | grep INTERP -A 2
INTERP   0x0000000000000318 0x0000000000000318 0x0000000000000318
         0x000000000000001c 0x000000000000001c  R      0x1
         [Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]

ELF binary 안에 — "interpreter" 가 명시되어 있다. 이게 바로 dynamic linker.

PT_INTERP segment

ELF 의 program header 중 하나 — PT_INTERP.

// [user-space - <elf.h>]
#define PT_NULL    0    /* unused */
#define PT_LOAD    1    /* loadable segment */
#define PT_DYNAMIC 2    /* dynamic linking info */
#define PT_INTERP  3    /* program interpreter */
#define PT_NOTE    4    /* auxiliary info */
#define PT_PHDR    6    /* program header itself */
#define PT_TLS     7    /* TLS template */

PT_INTERP 는 — interpreter path 가 들어 있는 segment. 보통 짧은 string.

[PT_INTERP 의 내용]

  64-bit Linux x86: /lib64/ld-linux-x86-64.so.2
  32-bit Linux x86: /lib/ld-linux.so.2
  arm64 Linux:     /lib/ld-linux-aarch64.so.1
  
  static binary: PT_INTERP 없음 (interpreter 안 씀)
  
  특수:
    musl: /lib/ld-musl-x86_64.so.1
    uClibc: /lib/ld-uClibc.so.0

kernel 의 load_elf_binary 처리

execve(2) syscall 의 처리 흐름:

// [kernel - linux fs/binfmt_elf.c master 단순화]
static int load_elf_binary(struct linux_binprm *bprm)
{
    struct elfhdr elf_ex;
    char *elf_interpreter = NULL;
    struct file *interpreter = NULL;
    unsigned long elf_entry;
    unsigned long interp_load_addr = 0;
    
    /* 1. ELF header 읽기 */
    elf_ex = *((struct elfhdr *)bprm->buf);
    
    /* 2. program header 순회 */
    for (i = 0; i < loc->elf_ex.e_phnum; i++) {
        if (elf_ppnt->p_type == PT_INTERP) {
            /* PT_INTERP 발견 — interpreter path 읽기 */
            elf_interpreter = kmalloc(elf_ppnt->p_filesz, GFP_KERNEL);
            kernel_read(bprm->file, elf_interpreter,
                        elf_ppnt->p_filesz, &elf_ppnt->p_offset);
            
            /* 3. interpreter 파일 열기 */
            interpreter = open_exec(elf_interpreter);
            kfree(elf_interpreter);
        }
    }
    
    /* 4. program 의 LOAD segment 모두 mmap */
    for (... PT_LOAD ...) {
        elf_map(...);
    }
    
    /* 5. interpreter 도 mmap */
    if (interpreter) {
        elf_entry = load_elf_interp(...);
        /* elf_entry = ld-linux.so 의 entry point */
        interp_load_addr = ...;
    } else {
        elf_entry = loc->elf_ex.e_entry;   /* program 의 _start */
    }
    
    /* 6. control 을 interpreter 의 entry 로 넘김 (또는 static binary 면 program 의) */
    start_thread(regs, elf_entry, bprm->p);
    return 0;
}

핵심 흐름:

[execve 시의 처리]

  ./a.out 실행
       │
       ▼
  execve syscall
       │
       ▼
  kernel: load_elf_binary
       │
       ▼
  1. ELF magic 검증
  2. program header 순회 — PT_INTERP 찾기
  3. interpreter (ld-linux.so) 파일 열기
  4. program 의 LOAD segment 들 mmap
  5. interpreter 의 LOAD segment 들 mmap
  6. stack 준비 (auxv, argv, envp)
  7. start_thread → CPU 의 IP 를 _interpreter 의 _start_ 로
  
  → 사용자가 보는 첫 user-space 코드:
    program 의 _start 가 _아니라_ ld-linux.so 의 _start

ld-linux.so 의 self-bootstrap

ld-linux.so 의 어려움 — 자기 자신이 dynamic library 인데, 자기를 load 할 dynamic linker 가 없음.

[ld-linux 의 bootstrap 문제]

  일반 library:
    의존성: libc, ld-linux 등
    relocation: ld-linux 가 처리
    
  ld-linux:
    의존성: 없음 (자기 완결적, 정적 링크)
    relocation: ★ 자기 자신을 _스스로_ relocate 해야!
    → "bootstrap" 문제

해결 — ld-linux 의 첫 코드가 self-relocation. 자기 자신의 GOT 와 relocation table 을 만져서 자기 주소 결정.

// [user-space - glibc elf/rtld.c master 단순화]
ElfW(Addr)
_dl_start(void *arg)
{
    /* ★ 매우 조심: 이 시점에 ld-linux 자체가 아직 relocated 안 됨 */
    /* 글로벌 변수, 함수 호출 모두 위험 */
    
    struct dl_start_final_info info;
    ElfW(Addr) start_addr;
    
    /* 1. self-relocation */
    ElfW(Addr) bootstrap_map_base = elf_machine_load_address();
    _dl_relocate_object_no_relro(&bootstrap_map, GL(dl_rtld_map).l_scope, 0, 0);
    
    /* 이제 ld-linux 의 글로벌 변수 등 정상 사용 가능 */
    
    /* 2. _dl_start_final 호출 (main 로직) */
    start_addr = _dl_start_final(arg, &info);
    
    return start_addr;
}

_dl_start 가 — 매우 위험한 코드. 자기 자신을 relocate 하기 전이라 — 글로벌 변수 사용 불가. 모두 stack 기반 + 어셈블리 트릭.

_dl_start_final → dl_main

self-relocation 이 끝나면 — 일반 코드.

[ld-linux 의 main 흐름]

  _dl_start                  ← 어셈블리, self-relocation
       │
       ▼
  _dl_start_final
       │
       ▼
  dl_main                    ← 본격적 로직
       │
       ├─ 1. ELF 의 PT_DYNAMIC 분석
       ├─ 2. DT_NEEDED 의 모든 library 로드
       ├─ 3. symbol resolution (GLOBAL)
       ├─ 4. relocation 수행
       ├─ 5. TLS 초기화
       ├─ 6. init function 호출 (.init_array)
       └─ 7. program 의 _start 로 jump

dl_main 의 마지막에 — program 의 entry point 로 점프.

kernel 이 stack 에 두는 정보 — auxv

ld-linux 는 — kernel 이 전달한 정보를 stack 에서 받는다.

[stack layout (program 시작 시점)]

  argc
  argv[0..N]
  envp[0..N]
  auxv[]:             ← ★ kernel 이 전달
    AT_PHDR    = program 의 program header 주소
    AT_PHENT   = 각 header 크기
    AT_PHNUM   = header 개수
    AT_BASE    = ld-linux 의 base 주소
    AT_ENTRY   = program 의 entry point (_start)
    AT_PAGESZ  = page 크기
    AT_RANDOM  = stack canary 등의 random
    AT_PLATFORM = "x86_64"
    AT_HWCAP   = CPU capability bits
    AT_EXECFN  = "/path/to/a.out"
    AT_SYSINFO_EHDR = vdso 의 위치
    ...

AT_BASE 가 — ld-linux 의 위치. AT_ENTRY 가 — 진짜 program 의 _start. ld-linux 는 — 이걸 받아 동작.

getauxval 로 확인

// [user-space - C 예제]
#include <sys/auxv.h>
#include <stdio.h>

int main(void) {
    printf("ld-linux base: 0x%lx\n", getauxval(AT_BASE));
    printf("program entry: 0x%lx\n", getauxval(AT_ENTRY));
    printf("page size:     %ld\n",   getauxval(AT_PAGESZ));
    printf("execfn:        %s\n",    (char *)getauxval(AT_EXECFN));

}

ld-linux.so 는 — 실행 파일이기도 하다.

# [user-space - bash]

Usage: ld.so [OPTION]... EXECUTABLE-FILE [ARGS-FOR-PROGRAM...]
...

ld-linux 가 — program 을 explicit 하게 로드해 실행. 가끔 사용:

이게 가능한 이유 — ld-linux 의 PT_INTERP 가 자기 자신을 가리킨다 (또는 없다).

static binary — interpreter 없음

-static 으로 컴파일한 binary 는 — PT_INTERP 없음. ld-linux 사용 안 함.

# [user-space - bash]
$ gcc -static hello.c -o hello_static
$ file hello_static
hello_static: ELF 64-bit LSB executable, x86-64, ..., statically linked, ...
$ readelf -l hello_static | grep INTERP
$  # 출력 없음

# kernel 이 직접 program 의 _start 로 jump
[static vs dynamic binary]

  static:
    크기: 큼 (모든 의존성 포함)
    ld-linux 안 거침 → 빠른 시작
    실행 환경의 libc 무관 (이식성)
    dlopen 못 함 (대부분 못 함)
    
  dynamic:
    크기: 작음
    ld-linux 거침 → 약간 느린 시작
    환경의 libc 의존 (호환성)
    dlopen 가능
    여러 program 이 libc 공유 (메모리 절약)

ld-linux 의 entry point

ld-linux 의 진짜 첫 코드 — 어셈블리.

[ld-linux 의 _start]

  glibc/sysdeps/x86_64/dl-machine.h 의 RTLD_START:
  
  asm:
    _start:
      mov   %rsp, %rdi     ; stack pointer → 첫 인자
      call  _dl_start       ; C 함수 호출

      mov   %rax, %rcx     ; entry 저장
      ...
      jmp   *%rcx           ; program 의 _start 로 jump

Part III 의 14 장 (kernel → user) 와 비교

Part III 의 14 장에서 — kernel 이 program 의 _start 로 jump 하는 것처럼 단순화했었다. 사실은:

[정확한 startup 흐름]

  Part III 의 단순화:
    kernel → program 의 _start
    
  정확한 흐름:
    kernel → ld-linux 의 _start
             → _dl_start (self-relocation)
             → dl_main (library 로드, relocation)
             → ld-linux 의 init 호출
             → program 의 _start
             → __libc_start_main
             → program 의 main
    
  (static binary 면 ld-linux 단계 없음)

이 장의 결론

ld-linux.so 의 정체:

[ld-linux.so 정리]

  ✓ ELF 의 PT_INTERP segment 가 path 명시
  ✓ kernel 의 load_elf_binary 가 PT_INTERP 처리
  ✓ kernel 이 program + interpreter 둘 다 mmap
  ✓ control 을 interpreter 의 entry 로 → ld-linux 의 _start
  ✓ ld-linux 가 자기 자신을 먼저 relocate (bootstrap)
  ✓ _dl_start → _dl_start_final → dl_main
  ✓ stack 의 auxv 가 kernel ↔ ld-linux 통신
  ✓ ld-linux 자체가 실행 파일이기도
  ✓ ldd 가 LD_TRACE_LOADED_OBJECTS 의 ld-linux 활용
  ✓ /etc/ld.so.cache 가 library 빠른 lookup
  ✓ LD_DEBUG, LD_PRELOAD, LD_LIBRARY_PATH 등 환경변수
  ✓ static binary 는 ld-linux 안 거침

다음 장에서는 — ld-linux 가 가장 먼저 읽는 정보, ELF 의 dynamic section 과 link map 의 구조를 본다.

54ELF dynamic section 과 link map

ld-linux 의 첫 임무 — program 의 dependency 파악, library 위치 찾기. 이 정보가 모두 — ELF 의 dynamic section 에 있다. 그리고 — ld-linux 가 메모리에 만드는 link map 이 모든 로드된 ELF 의 관리 구조.

_DYNAMIC symbol

// [user-space - C 직접 접근]

#include <stdio.h>

extern Elf64_Dyn _DYNAMIC[];   /* 모든 ELF 에 자동 존재 */

int main(void) {
    Elf64_Dyn *dyn = _DYNAMIC;

이게 — 자기 자신의 dynamic section 을 보는 방법.

Elf64_Dyn 구조

// [user-space - <elf.h> 표준]
typedef struct {
    Elf64_Sxword d_tag;            /* tag (DT_*) */
    union {
        Elf64_Xword d_val;          /* integer value */
        Elf64_Addr  d_ptr;          /* virtual address */
    } d_un;
} Elf64_Dyn;

각 entry — 8 byte tag + 8 byte value (union 으로 다양한 해석).

DT_NEEDED — 의존성 표시

가장 중요한 — DT_NEEDED.

// [user-space - glibc elf/dl-deps.c master 인용]
for (d = l->l_ld; d->d_tag != DT_NULL; ++d)
    if (d->d_tag == DT_NEEDED) {
        /* DT_NEEDED 발견 — library name 추출 */
        name = expand_dst(l, strtab + d->d_un.d_val, 0);
        
        /* 그 library 를 dlopen 같이 로드 */
        dep = openaux(name, ...);
    }

DT_NEEDED 의 d_val 이 — string table 의 offset. 거기에 SO name 이 있음.

$ objdump -p /bin/ls | grep NEEDED

ld-linux 가 — 각 NEEDED 의 library 를 재귀적으로 로드.

DT_STRTAB, DT_SYMTAB — table 들

[ELF 의 두 핵심 table]

  DT_STRTAB:
    string 들의 배열 (null-terminated 들이 연속)
    DT_NEEDED 의 d_val 이 이 table 의 offset
    symbol 의 이름들도 여기에
    
  DT_SYMTAB:
    Elf64_Sym 의 배열
    각 symbol 의 name (strtab offset), value, size, info, ...

Elf64_Sym 의 구조:

// [user-space - <elf.h>]
typedef struct {
    Elf64_Word    st_name;     /* name (strtab offset) */
    unsigned char st_info;      /* type + binding */
    unsigned char st_other;     /* visibility */
    Elf64_Half    st_shndx;     /* section index */
    Elf64_Addr    st_value;     /* value (function 주소 등) */
    Elf64_Xword   st_size;      /* size */

DT_HASH vs DT_GNU_HASH

symbol 검색을 빠르게 — hash table.

[hash table 비교]

  DT_HASH (옛 형식, ELF 표준):
    단순 chained hash
    hash 함수 = ELF hash (간단)
    모든 symbol 검색 가능
    
  DT_GNU_HASH (GCC 5.0+ 기본):
    Bloom filter + hash
    더 빠른 lookup
    매우 많은 symbol 의 library 에 효율적 (libstdc++ 등)
    
  현대 binary: 보통 GNU_HASH 만 있음 (옛 표준 DT_HASH 는 생략)

DT_INIT, DT_INIT_ARRAY — 초기화

library 로드 후 호출할 함수들.

// [user-space - C 예제]
__attribute__((constructor))
void init_lib(void) {
    /* library 로드 시 자동 호출 */
}

__attribute__((destructor))
void fini_lib(void) {
    /* library unload 시 호출 */
}
[init / fini 처리]

  DT_INIT (단수):
    옛 _init 함수 (C runtime 이 호출)
    glibc 는 자체 _init 가짐
    
  DT_INIT_ARRAY:
    초기화 함수 _포인터 배열_
    constructor 들이 여기 들어감
    ld-linux 가 순서대로 호출
    
  DT_FINI / DT_FINI_ARRAY: 종료 시

ld-linux 가 — library 로드 후 INIT_ARRAY 의 모든 함수 호출.

DT_DEBUG — debugger 와의 인터페이스

가장 흥미로운 — DT_DEBUG.

// [user-space - <link.h>]
struct r_debug {
    int r_version;
    struct link_map *r_map;       /* link map list 의 head */
    ElfW(Addr) r_brk;              /* debugger 가 hook 한 함수 */
    enum {
        RT_CONSISTENT,             /* mapping 상태 안정 */
        RT_ADD,                    /* mapping 추가 중 */
        RT_DELETE                  /* mapping 제거 중 */

    ElfW(Addr) r_ldbase;           /* ld-linux 의 base 주소 */

[DT_DEBUG 의 역할]

  static linker 가 DT_DEBUG entry 를 만듬 (d_val = 0)
  ld-linux 가 dl_main 에서 _r_debug 의 주소를 d_val 에 채움
  
  결과:
    DT_DEBUG 의 d_val = &_r_debug
    debugger 가 _DYNAMIC 통해 _r_debug 접근
    _r_debug 의 r_map 통해 모든 loaded library 찾기

이게 — gdb 의 library 발견 메커니즘의 핵심. gdb 가 _DYNAMIC 의 DT_DEBUG 통해 r_debug 접근, link_map 순회.

r_brk — library 로드 알림

ld-linux 가 library 를 로드/언로드할 때 — r_brk 의 함수 호출.

[library 로드의 debugger 통보]

  ld-linux 가 새 library 로드 직전:
    r_debug.r_state = RT_ADD
    r_debug.r_brk() 호출       ← debugger 의 breakpoint
    
  library 로드 후:
    r_debug.r_map 에 새 link_map 추가
    r_debug.r_state = RT_CONSISTENT
    r_debug.r_brk() 호출 다시
    
  debugger 가 _dl_debug_state 에 breakpoint:
    호출 시 link map 다시 읽음
    "Loaded /path/to/libfoo.so" 등 출력

gdb 의 info shared 가 — 이 메커니즘으로 동작.

struct link_map — ld-linux 의 내부 관리

ld-linux 가 — 로드된 모든 ELF 를 link_map 의 linked list 로 관리.

struct link_map {
    /* public part — debugger 도 사용 */
    ElfW(Addr) l_addr;            /* base 주소 (load offset) */

    struct link_map *l_prev;       /* 이전 */
    
    /* private part — glibc 내부 */

    int l_searchlist_loaded;

    ...

각 ELF (program + 모든 library) 가 — 하나의 link_map 객체.

       │

    main program 이 항상 첫 번째
    그 다음 의존 library 들
    마지막 ld-linux 자체

l_addr — load address (slide)

ELF 가 메모리에 매핑된 위치.

[l_addr 의 의미]

  binary 의 ELF 안에 적힌 주소: virtual address (VA)
  실제 메모리에 매핑된 주소: load address + VA
  
  l_addr = 실제_매핑_주소 - VA_in_ELF
  
  PIE (-fPIE) 면:
    VA 가 0 부터 시작 (relative)
    l_addr = 실제_매핑_주소 (slide)
    
  옛 non-PIE 면:
    VA 가 절대 주소 (예: 0x400000)
    l_addr = 0 (slide 없음)
    
  shared library (.so):
    항상 PIE 같이 동작
    l_addr ≠ 0

이 l_addr 가 — 모든 relocation 의 기반. ELF 의 모든 주소가 — l_addr + offset.

l_info[] — DT_* 의 빠른 접근

dynamic section 의 entry 들이 — l_info 의 색인으로 caching.

// [user-space - glibc 사용 예]
struct link_map *map;

/* DT_NEEDED 의 entry */
Elf64_Dyn *needed = map->l_info[DT_NEEDED];

/* DT_STRTAB 의 주소 */
char *strtab = (char *)D_PTR(map, l_info[DT_STRTAB]);
/* D_PTR 매크로가 d_ptr 또는 l_addr + d_val 계산 */

/* DT_SYMTAB 의 주소 */
Elf64_Sym *symtab = (Elf64_Sym *)D_PTR(map, l_info[DT_SYMTAB]);

매 lookup 마다 dynamic section 순회 안 함. 미리 indexing.

_dl_runtime_resolve 의 link map 사용

PLT 의 lazy binding 시:

[lazy binding 의 link map 사용]

  printf 첫 호출:
    PLT slot → _dl_runtime_resolve
    
  _dl_runtime_resolve(link_map, reloc_index):
    1. link_map 의 l_info[DT_SYMTAB] 로 symbol table 접근
    2. reloc_index 통해 symbol 찾기
    3. symbol name 으로 다른 library 에서 정의 검색
    4. 찾은 주소를 GOT 에 기록
    5. 그 주소로 jump
    
  → 다음 호출부터는 직접 GOT 로

55, 56 장에서 자세히.

_dl_runtime_resolve 의 안전성

매우 흥미로운 fact — _dl_runtime_resolve 가 — 재진입 가능 (reentrant) 해야 함.

[reentrant 요구사항]

  multi-thread 환경:
    thread A 가 printf 호출 → resolver 실행 중
    thread B 도 printf 호출 → resolver 실행 중
    
    동시에 같은 GOT 에 쓰면 → race
    
  해결:
    원자적 write (단일 word)
    또는 lock (glibc 의 _dl_load_lock)

_DYNAMIC symbol 의 검출

binary 가 — 자기 자신의 _DYNAMIC 어떻게 찾나?

// [user-space - PIE binary]
/* _DYNAMIC 의 주소는 binary 안에 상대 offset 으로 저장 */
extern Elf64_Dyn _DYNAMIC[];

/* 컴파일러가 자동 처리:
   _DYNAMIC[0] = *(Elf64_Dyn *)(load_base + offset_in_elf) */

ld-linux 의 startup 시 — argv 등에서 자기 load address 알아내고, _DYNAMIC 의 주소 계산.

Practical 사용 — 자기 자신의 library 검사

다음 장에서는 — printf 호출이 어떻게 libc 의 printf 주소로 도달하는지의 핵심 메커니즘, PLT 와 GOT 의 구조를 본다.

55PLT / GOT — lazy binding 의 기반

이제 — printf 호출의 핵심 메커니즘. binary 의 printf call 이 어떻게 libc 의 진짜 printf 주소로 도달하는지. 답은 — PLT (Procedure Linkage Table) 와 GOT (Global Offset Table).

문제의 본질

// [user-space - C 소스]
#include <stdio.h>
int main(void) {
    printf("hello\n");
    return 0;
}
[printf 호출의 문제]

  컴파일 시:
    - main 안에 "printf 호출" 코드 생성
    - printf 의 _주소_ 는? — _모름_
      (libc 가 어디 mapping 될지 컴파일 시점에 결정 불가)
      
  static binary:
    - linker 가 컴파일 시 모든 의존 라이브러리를 합침
    - printf 의 주소 = binary 안에 hardcoded
    - 그러나 binary 크고, ld-linux 못 씀
    
  dynamic binary:
    - libc 의 위치가 runtime 결정
    - 매번 다른 주소 (ASLR)
    - 어떻게 printf 의 주소 알아내나?
    
  답: PLT + GOT 의 indirection

GOT — Global Offset Table

[GOT 의 정체]

  .got 와 .got.plt 두 section:
    
  .got:
    데이터 (글로벌 변수) 의 주소들
    ld-linux 가 startup 시 채움 (eager)
    예: extern int errno → .got 의 entry
    
  .got.plt:
    함수의 주소들
    초기엔 PLT 안의 위치를 가리킴
    첫 호출 시 진짜 주소로 update (lazy)
    예: printf, malloc, free → .got.plt 의 entry

.got.plt 의 처음 3 개 entry 가 특별:

[.got.plt 의 reserved 3 entries]

  GOT[0] = _DYNAMIC 의 link-time 주소 (relocated)
  GOT[1] = link_map 의 주소 (ld-linux 가 채움)
  GOT[2] = _dl_runtime_resolve 의 주소 (ld-linux 가 채움)
  
  GOT[3] 부터: PLT 의 entry 들
  GOT[3]: printf 의 주소 (resolved 후)
  GOT[4]: malloc 의 주소
  ...

이 3 reserved entry 가 — lazy binding 의 기반.

PLT — Procedure Linkage Table

.plt section 의 각 entry — trampoline (도약대).

[PLT 의 구조]

  .plt section:
    PLT[0]:   "resolver 호출용" — 모든 PLT 가 미해결 시 도달
    PLT[1]:   printf@plt
    PLT[2]:   malloc@plt
    PLT[3]:   ...
    
  각 PLT entry 의 크기: 16 byte (x86-64)
  
  PLT[0] 의 동작 (pseudocode):
    push GOT[1]              ; link_map 의 주소를 stack 에
    jmp *GOT[2]              ; _dl_runtime_resolve 호출
    
  PLT[N] (N ≥ 1) 의 동작 (pseudocode):
    jmp *GOT[N+2]            ; .got.plt 의 N+2 번째 entry 로 jump
    push reloc_index_for_N   ; 어떤 symbol 인지 index
    jmp PLT[0]               ; resolver 로

첫 호출의 흐름

[printf 의 첫 호출 흐름]

  1. main 의 어셈블리:
     call printf@plt           ; PLT[1] 호출 (실제로 jmp + push)
                                ;
  2. PLT[1] 에 도달:
     jmp *GOT[3]               ; .got.plt 의 3 번째 entry
                                ;
     ★ GOT[3] 의 초기값:
        binary 컴파일 시 — _PLT[1] 의 두 번째 instruction_ 의 주소
        (즉, jmp 후 push 의 위치)
        → "마치 jmp 가 안 한 것 같음" 효과
                                ;
  3. PLT[1] 의 두 번째 instruction:
     push 0                    ; printf 의 reloc index = 0
                                ;
  4. PLT[1] 의 세 번째 instruction:
     jmp PLT[0]                ;
                                ;
  5. PLT[0]:
     push GOT[1]               ; link_map
     jmp *GOT[2]               ; _dl_runtime_resolve
                                ;
  6. _dl_runtime_resolve:
     - link_map 과 reloc_index 받음
     - reloc table 에서 printf 의 정보 찾기
     - hash table 로 libc.so 의 printf 검색
     - 찾은 주소를 GOT[3] 에 _저장_
     - 그 주소로 jump (→ libc 의 printf)
                                ;
  7. 진짜 printf 실행

이 매우 복잡한 흐름이 — 첫 호출에만 일어난다.

두 번째 호출부터

GOT[3] 가 — 첫 호출 후 진짜 printf 의 주소로 변경됨.

[printf 의 두 번째 호출]

  1. call printf@plt           ; PLT[1] 호출
                                ;
  2. PLT[1]:
     jmp *GOT[3]               ; GOT[3] 의 값으로 jump
                                ;
  ★ GOT[3] 의 현재값: libc 의 printf 의 실제 주소
                                ;
  3. 바로 libc 의 printf 실행 ★ 1 번의 indirect jump
  
  → resolver 안 거침, 매우 빠름

이게 — lazy binding 의 의미. 첫 호출 시만 비용, 이후로는 무료.

.plt.got — non-lazy 변종

gcc 일부 시나리오에서 — .plt.got 라는 다른 section 사용.

[.plt.got 의 의미]

  .plt.got 의 entry (lazy 없음):
    jmp *(GOT 의 적절한 entry)   ; 직접 jump
    
  GOT 의 entry 가 — ld-linux 가 startup 시 미리 resolve
  → lazy 의 indirection 없음
  
  사용처:
    이미 GOT 에 있는 symbol 의 호출 시
    LD_BIND_NOW 환경변수 set 시
    -z now linker flag 시

대부분의 binary 는 .plt 사용. 일부 최적화된 binary 만 .plt.got.

R_X86_64_JUMP_SLOT 가 — PLT 의 GOT entry 들. ld-linux 가 lazy 하게 처리.

R_X86_64_GLOB_DAT vs R_X86_64_JUMP_SLOT

[두 relocation type 의 차이]

  R_X86_64_GLOB_DAT:
    데이터 또는 함수 주소
    ★ eager 처리 (startup 시 즉시)
    .got 에 위치
    GCC -fPIE 의 함수 호출이 사용 (lazy 안 함)
    
  R_X86_64_JUMP_SLOT:
    함수의 PLT entry 용
    ★ lazy 처리 (첫 호출 시)
    .got.plt 에 위치
    R_X86_64_RELATIVE 처럼 빠르지 않음 (resolver 거침)

binary 가 만들어지는 시점 — linker 의 작업

PLT 와 GOT 의 entry 들 — static linker 가 만듬.

[linker (ld) 의 작업]

  컴파일 결과 (.o 파일):
    printf 호출 시 placeholder (relocation entry)
    
  static linker:
    1. 각 외부 symbol 에 PLT entry 생성
    2. .got.plt 의 entry 자리 예약
    3. PLT[N] 가 push N → jmp PLT[0]
    4. .rela.plt 에 R_X86_64_JUMP_SLOT relocation 기록
    5. .dynsym 에 symbol 정보
    
  결과 binary:
    main 의 call printf@plt → PLT[1]
    PLT[1] 의 jmp *GOT[3] → 초기엔 PLT[1]+6 (자기 자신)
    GOT[3] 의 초기값 = PLT[1] + 6 (즉 push 의 주소)

이 디자인이 — 진짜 영리한 trick. 첫 호출 시 — GOT 의 초기값이 _self-reference_ 라 — jmp 가 효과 없는 척 동작, 결국 resolver 로.

ld-linux 가 startup 시 하는 것

[ld-linux 의 PLT 설정]

  binary 로드 직후:
    GOT[0] = _DYNAMIC 의 실제 주소 (link-time 주소 + load offset)
    GOT[1] = link_map 의 주소 (ld-linux 가 만든)
    GOT[2] = _dl_runtime_resolve 의 주소 (ld-linux 안)
    
    GOT[3..N]: 그대로 (linker 의 초기값 — PLT[i]+6)
              → lazy binding 대기
              
  LD_BIND_NOW=1 면:
    startup 시 모든 GOT[3..N] 도 미리 resolve
    → 첫 호출 비용 없음, 그러나 startup 느림

보안 측면 — PLT/GOT 의 공격 표면

.got.plt 가 — 공격자가 노리는 곳.

[GOT overwriting attack]

  공격자 시나리오:
    .got.plt 가 writable (RW)
    overflow 등으로 GOT entry 덮어쓰기
    예: GOT[3] (printf) = 공격자_함수_주소
    
    다음 printf 호출 → 공격자 코드 실행
    
  완화 메커니즘:
    1. RELRO (Read-Only after Relocation):
       startup 후 .got 를 read-only
       gcc -Wl,-z,relro
       
    2. Full RELRO + BIND_NOW:
       .got.plt 도 read-only
       gcc -Wl,-z,relro -Wl,-z,now
       (lazy binding 안 함)
       
    3. PIE + ASLR:
       PLT 의 주소 random
       공격자가 주소 추측 어려움

lazy binding 의 thread safety

_dl_runtime_resolve 가 — multi-thread 동시 호출 시 어떻게?

[lazy binding 의 thread safety]

  thread A, B 가 동시에 같은 printf@plt 첫 호출:
    A 가 resolver 실행, GOT 에 주소 쓰기 중
    B 가 PLT[1] 에서 GOT[3] read
    
    race 가능:
      B 가 옛 값 읽음 → resolver 다시
      또는 A 의 부분 write 읽음 → 잘못된 주소
    
  glibc 의 대응:
    - GOT entry 의 write 가 _원자적_ (64-bit single word)
    - x86-64 의 mov 가 8 byte 원자적
    - 그래도 _dl_runtime_resolve 안에 lock 일부 사용
    
  최악의 경우:
    두 thread 모두 resolver 거쳐 같은 GOT 에 같은 값 write
    → 동일 결과, 약간의 추가 비용만

LD_BIND_NOW 의 의미

환경변수로 lazy binding 끄기:

# [user-space - bash]
$ LD_BIND_NOW=1 ./hello
# startup 시 모든 GOT entry resolve
# 처음 함수 호출 시 추가 비용 없음
# 단 startup 자체가 느림

언제 사용?

  • realtime 응용: 응답시간 일관성
  • 보안 critical: GOT 가 RO 가 되어 안전

binary 의 보호 옵션

[gcc 의 PLT/GOT 관련 옵션]

  -fPIC               : position-independent code
  -fPIE               : position-independent executable
  -pie                : link 시 PIE binary 생성
  
  -Wl,-z,relro        : Partial RELRO (.got read-only)
  -Wl,-z,now          : eager binding (.got.plt 도 RO)
  -Wl,-z,noplt        : PLT 안 만듬 (인라인 PIC 호출)
  
  -fno-plt            : 같은 효과 (gcc 4.8+)

-fno-plt 가 — 매우 흥미로움. PLT 안 거치고 — 직접 GOT 로:

[-fno-plt 의 코드]

  보통:
    call printf@plt    ; PLT 로
    
  -fno-plt:
    call *printf@GOTPCREL(%rip)   ; GOT 의 entry 로 직접 indirect call
    
  장점:
    - PLT 없음 (binary 작음)
    - 분기 1 번 적음 (약간 빠름)
    - 보안 (PLT 의 jmp 가 없어 공격 표면 작음)
    
  단점:
    - GOT 가 항상 채워져 있어야 (lazy 안 됨)
    - LD_BIND_NOW 와 같은 효과

56lazy binding 의 실제 — _dl_runtime_resolve 의 정체

55 장에서 — PLT 가 첫 호출 시 _dl_runtime_resolve 로 점프함을 봤다. 이 장은 — 그 resolver 의 내부.

_dl_runtime_resolve 의 두 부분

resolver 는 — 사실 두 부분.

[_dl_runtime_resolve 의 구조]

  1. _dl_runtime_resolve (어셈블리 stub)
     - 호출자의 모든 register 저장
     - _dl_fixup 의 인자 준비
     - _dl_fixup 호출
     - 반환값 받기
     - 모든 register 복원
     - resolved 함수로 jump
     
  2. _dl_fixup (C 함수, elf/dl-runtime.c)
     - 실제 logic
     - reloc 찾기, symbol lookup, GOT 업데이트
     - resolved 주소 반환

이 두 부분 분리 — 어셈블리는 calling convention 처리만, 복잡한 logic 은 C 로.

왜 어셈블리 stub 가 필요?

[어셈블리 stub 의 이유]

  resolver 호출 직전 상황:
    call printf("hello"):
      rdi = "hello" (첫 인자)
      rsi = 다음 인자
      rax = 변수 인자 수 (varargs)
      ...
      
    PLT 거쳐 _dl_runtime_resolve 도달:
      register 들이 _아직 printf 호출용 상태_
      
  resolver 의 책임:
    1. 이 register 들을 _보존_ 해야 함
       (resolver 가 자기 일 끝나면 printf 호출 _재현_ 해야)
       
    2. resolver 자신의 인자 (link_map, reloc_index) 는
       stack 에 push 된 상태
       
  C 함수로는 이걸 못 함:
    C 함수는 자기 calling convention 따라 register 사용
    원본 인자 register 들 파괴
    
  → 어셈블리 stub 가 manual 한 register 보존 필요

어셈블리 stub 의 흐름 (pseudo)

[_dl_runtime_resolve 의 의사 흐름]

  진입 시 stack:
    [reloc_index]    ← top (PLT[N] 이 push)
    [link_map]       ← (PLT[0] 이 push)
    [return addr]    ← printf 호출자의 return 주소
    ...
    
  stub:
    1. callee-save 가 아닌 register 들 모두 stack 에 save
       (rax, rdi, rsi, rdx, rcx, r8, r9, r10, r11)
       + 부동소수점 register (xmm0~xmm7)
       
    2. C 호출 준비:
       rdi = link_map (stack 의 위치에서)
       rsi = reloc_index
       
    3. call _dl_fixup
    
    4. rax = resolved 주소 (반환)
    
    5. 저장한 register 들 복원
    
    6. stack 정리 (push 한 reloc_index, link_map 제거)
    
    7. jmp rax  → printf 호출 재현
       (이제 register 들이 원래 상태, 인자도 그대로)

이 stub 가 — printf 호출이 resolver 거친 적 없는 것처럼 동작.

_dl_fixup — C 코드의 정체

master 의 _dl_fixup 단순화:

// [user-space - glibc elf/dl-runtime.c master 단순화]
DL_FIXUP_VALUE_TYPE
_dl_fixup(struct link_map *l, ElfW(Word) reloc_arg)
{
    /* 1. dynamic table 에서 필요한 정보 얻기 */
    const ElfW(Sym) *const symtab = 
        (const void *)D_PTR(l, l_info[DT_SYMTAB]);
    const char *strtab = 
        (const void *)D_PTR(l, l_info[DT_STRTAB]);
    
    /* 2. JMPREL (PLT 의 relocation) 의 reloc 가져오기 */
    const PLTREL *const reloc = 
        (const void *)(D_PTR(l, l_info[DT_JMPREL]) 
                       + reloc_offset(reloc_arg, sizeof(PLTREL)));
    
    /* 3. relocation 이 가리키는 symbol */
    const ElfW(Sym) *sym = &symtab[ELFW(R_SYM)(reloc->r_info)];
    
    /* 4. GOT 의 fixup 할 위치 */
    void *const rel_addr = (void *)(l->l_addr + reloc->r_offset);
    
    lookup_t result;
    DL_FIXUP_VALUE_TYPE value;
    
    /* 5. symbol 의 visibility 검사 */
    if (__glibc_likely(ELFW(ST_VISIBILITY)(sym->st_other) == 0)) {
        /* 일반 visibility — 다른 library 에서도 찾기 */
        const struct r_found_version *version = NULL;
        
        /* version 정보 처리 (다음 장에서) */
        if (l->l_info[VERSYMIDX(DT_VERSYM)] != NULL) {
            const ElfW(Half) *vernum = 
                (const void *)D_PTR(l, l_info[VERSYMIDX(DT_VERSYM)]);
            ElfW(Half) ndx = vernum[ELFW(R_SYM)(reloc->r_info)] & 0x7fff;
            version = &l->l_versions[ndx];
            if (version->hash == 0)
                version = NULL;
        }
        
        /* ★ symbol 검색 — 다음 장에서 자세히 */
        result = _dl_lookup_symbol_x(strtab + sym->st_name, l, &sym,
                                     l->l_scope, version,
                                     ELF_RTYPE_CLASS_PLT,
                                     DL_LOOKUP_ADD_DEPENDENCY, NULL);
        
        /* 찾은 symbol 의 실제 주소 */
        value = DL_FIXUP_MAKE_VALUE(result, sym ? LOOKUP_VALUE_ADDRESS(result, true) + sym->st_value : 0);
    } else {
        /* hidden/protected/internal — 같은 ELF 안에서만 */
        value = DL_FIXUP_MAKE_VALUE(l, l->l_addr + sym->st_value);
        result = l;
    }
    
    /* 6. GOT 에 쓰기 + 반환 */
    return elf_machine_fixup_plt(l, result, refsym, sym, reloc, rel_addr, value);
}

이게 — lazy binding 의 모든 핵심.

단계별 분석

[_dl_fixup 의 7 단계]

  1. DT_SYMTAB, DT_STRTAB 추출
     - link_map 의 l_info 가 indexing 해둬서 빠름
     
  2. DT_JMPREL + reloc_arg 로 reloc entry 찾기
     - reloc_arg 는 byte offset (x86-64 의 stub)
     - reloc entry = PLT 의 N 번째 함수의 relocation 정보
     
  3. reloc->r_info 의 상위 부분이 symbol index
     - symtab[N] = 그 symbol
     - Elf64_Sym 구조체
     
  4. reloc->r_offset = GOT 의 entry 의 file offset
     - l_addr + r_offset = 메모리의 GOT entry 주소
     
  5. symbol 의 visibility 분기
     - STV_DEFAULT (0): 정상 lookup
     - STV_PROTECTED/HIDDEN/INTERNAL: 같은 ELF 안만
     
  6. _dl_lookup_symbol_x (다음 장에서 자세히)
     - 의존 library 들을 순회하며 symbol 찾기
     - hash table 활용
     
  7. elf_machine_fixup_plt:
     *rel_addr = resolved_address;     ← GOT 에 write!
     return resolved_address;

elf_machine_fixup_plt — GOT 업데이트

// [user-space - glibc sysdeps/x86_64/dl-machine.h 단순화]
static inline ElfW(Addr)
elf_machine_fixup_plt(struct link_map *map, lookup_t t,
                      const ElfW(Sym) *refsym, const ElfW(Sym) *sym,
                      const ElfW(Rela) *reloc,
                      ElfW(Addr) *reloc_addr, ElfW(Addr) value)
{
    /* GOT 의 entry 에 atomic 하게 쓰기 */
    return *reloc_addr = value;
}

매우 단순 — GOT 에 resolved 주소 쓰기. x86-64 의 64-bit write 가 — 자연스럽게 atomic.

thread safety 의 보장

elf_machine_fixup_plt 가 atomic 한 이유:

[thread-safe lazy binding]

  64-bit aligned 의 64-bit write:
    x86-64 이 자연스럽게 atomic
    multi-thread 가 동시에 같은 GOT 에 쓸 수도 있지만,
    각자 같은 값을 쓰니 무해
    
  GOT[N] 의 read (PLT 의 jmp *):
    한 번에 64-bit read, atomic
    옛 값 또는 새 값 둘 중 하나만 보임 (torn 안 됨)
    
  race 의 worst case:
    thread A, B 동시 첫 호출

resolver 의 cost

_dl_fixup 한 번의 비용:

[lazy binding 비용]

  첫 호출 시:
    - 어셈블리 stub: register 저장/복원 (~50 cycle)
    - _dl_fixup: ~3-10 microsec (symbol lookup 따라)
      ⤷ hash 계산, library 들 순회
    - GOT write
    
  → 첫 호출에 약 5 μs 추가
  → 1000 함수 호출 시: ~5 ms 추가 startup
  
  두 번째 이후:
    - PLT 의 jmp + GOT read: ~3 ns
    - 거의 직접 호출과 동일
    
  LD_BIND_NOW 시:
    - startup 시 모든 PLT resolve
    - startup 이 ~5 ms 느림
    - 첫 호출 시 추가 비용 없음
    - 응답 시간 일관성에 좋음

do_lookup_x — 실제 검색 함수

// [user-space - glibc elf/dl-lookup.c master 단순화]
static int
do_lookup_x(const char *undef_name, uint_fast32_t new_hash,
            unsigned long int *old_hash, const ElfW(Sym) *ref,
            struct sym_val *result, struct r_scope_elem *scope,
            size_t i, const struct r_found_version *const version,
            int flags, struct link_map *skip, int type_class)
{
    size_t n = scope->r_nlist;
    struct link_map *map;
    
    while (n--) {
        map = scope->r_list[i];
        i++;
        
        if (map == skip)
            continue;
        
        /* hash table 활용 */
        Elf64_Sym *symtab = (Elf64_Sym *)D_PTR(map, l_info[DT_SYMTAB]);
        const char *strtab = (const char *)D_PTR(map, l_info[DT_STRTAB]);
        
        /* GNU_HASH 또는 옛 HASH 으로 검색 */
        const Elf64_Sym *sym;
        if (map->l_info[ADDRIDX(DT_GNU_HASH)] != NULL) {
            sym = gnu_hash_lookup(map, undef_name, new_hash);
        } else {
            sym = old_hash_lookup(map, undef_name, *old_hash);
        }
        
        if (sym != NULL) {
            /* symbol 발견 — binding 검사 */
            switch (ELFW(ST_BIND)(sym->st_info)) {
            case STB_WEAK:
                /* weak — 일단 저장, 더 강한 게 있으면 교체 */
                if (! result->m) {
                    result->s = sym;
                    result->m = map;
                }
                break;
            case STB_GLOBAL:
                /* strong — 즉시 반환 */
                result->s = sym;
                result->m = map;
                return 1;
            case STB_GNU_UNIQUE:
                /* unique — process 전체에서 하나만 */
                ...
                break;
            }
        }
    }
    
    return 0;
}

흐름:

[do_lookup_x 의 핵심]

  scope 의 모든 link_map 순회:
    각 link_map 의 hash table 검색
    찾으면 binding 검사:
      STB_GLOBAL: 즉시 채택, 반환
      STB_WEAK: 더 강한 게 안 보이면 사용
      STB_LOCAL: 다른 library 에선 안 보임
    
  결과:
    가장 먼저 발견된 STB_GLOBAL, 또는
    어떤 STB_WEAK

hash table 의 활용

각 ELF 의 .gnu.hash (또는 .hash) 가 — 빠른 lookup.

[GNU hash 의 동작]

  symbol name 의 hash 계산:
    예: hash("printf") = 0x102B0...
    
  bloom filter 검사:
    1. bloom_table 의 bucket = hash % bloom_size
    2. bit 검사 → 없으면 _확실히 없음_
    3. → 다음 library 로 빠른 전환
    
  bucket lookup:
    hash 가 bloom 통과 → bucket 의 entry 검사
    
  chain 순회:
    bucket 의 entry 부터 string 비교
    매치 시 symbol 위치 반환
    
  → 평균 O(1) lookup
  → bloom filter 가 대부분의 "없음" 케이스 빠르게 처리

이 효율적 검색 덕에 — 수많은 symbol 의 library 도 빠른 lookup.

STB_GLOBAL vs STB_WEAK 의 의미

[symbol binding]

  STB_LOCAL:
    그 ELF 안에서만 보임 (다른 library 에서 검색 안 됨)
    static 함수 등
    
  STB_GLOBAL:
    "강한" symbol
    모든 곳에서 검색 가능
    같은 이름의 GLOBAL 가 _여러 개_ 면 error (옛 ld)
    또는 _첫_ 번째가 win (현재 ld)
    
  STB_WEAK:
    "약한" symbol
    STB_GLOBAL 가 있으면 그게 우선
    없으면 weak 가 사용
    여러 weak 가 있으면 첫 번째
    
  STB_GNU_UNIQUE:
    process 전체에서 _하나의 instance_ 만 보장
    C++ 의 template instantiation 등에 사용
    여러 library 가 같은 STB_GNU_UNIQUE 의 symbol 을 export 해도
    runtime 에 같은 주소

weak symbol 의 활용 예

// [user-space - C 예제]

int main(void) {

visibility — symbol 의 외부 노출

[symbol visibility (st_other)]

  STV_DEFAULT:
    일반. dynamic linker 가 검색에 포함
    
  STV_PROTECTED:
    define 한 ELF 안에선 그 자체 사용
    다른 ELF 에선 보이지만, _override 안 됨_
    
  STV_HIDDEN:
    같은 ELF 안에서만 사용 가능
    .so 의 외부에는 안 노출
    static linker 가 빈 entry 안 만듬
    
  STV_INTERNAL:
    STV_HIDDEN 비슷, ABI 측면에서 약간 다름
    
  컴파일러 flag:
    gcc -fvisibility=hidden : default 가 hidden
    gcc -fvisibility=default : default 가 default
    
  __attribute__((visibility("hidden"))) : 함수별 설정

STV_HIDDEN 이 — library 의 내부 함수에 매우 유용. ABI 작아지고, 보안 좋아지고, 성능 약간 좋아짐.

58relocation types — R_X86_64_*

ELF 의 모든 dynamic linking 의 최종 동작 — relocation. 특정 메모리 위치에 적절한 값을 쓰는 작업. 이 장은 — 그 종류들과 정확한 의미.

relocation 의 본질

[relocation 의 본질]

  컴파일 시:
    함수/변수의 _실제 주소_ 모름
    binary 에 placeholder + 정보 만 남김
    
  ld-linux (런타임):
    실제 주소 결정 후
    placeholder 의 위치에 _진짜 값_ 을 씀
    
  이 "기록할 곳 + 무엇을 기록할지" 정보 = relocation entry

Elf64_Rela 구조

// [user-space - <elf.h>]
typedef struct {
    Elf64_Addr  r_offset;       /* 어디에 쓸지 (메모리 주소) */
    Elf64_Xword r_info;          /* type + symbol index */
    Elf64_Sxword r_addend;       /* 더할 값 */
} Elf64_Rela;

/* r_info 의 분해 */
#define ELF64_R_SYM(info)   ((info) >> 32)         /* symbol index */
#define ELF64_R_TYPE(info)  ((info) & 0xffffffff)  /* relocation type */

각 relocation 의 4 가지 정보:

  • **r_offset**: 메모리의 어디에 (binary 의 load address + 이 값)
  • **r_info**: type + 어떤 symbol 인지
  • **r_addend**: 보정값 (덧셈)

Elf64_Rel vs Elf64_Rela

[Rel vs Rela]

  Elf64_Rel  : r_offset, r_info  만
  Elf64_Rela : r_offset, r_info, r_addend
  
  x86-64: 항상 Rela 사용 (DT_RELA)
  옛 i386: Rel 사용 (DT_REL)
  
  addend 가 별도 필드 vs target 위치의 _현재값_ 으로:
    Rel: target 위치의 값을 _read_ 해서 사용 (CPU memory access)
    Rela: addend 가 별도 → 더 효율적, more parallel

x86-64 의 Rela 형식이 — read 없이 처리 가능, 약간 더 빠름.

각각의 정확한 동작이 — 매우 중요.

R_X86_64_RELATIVE — 가장 흔한 type

[R_X86_64_RELATIVE 의 동작]

  공식:
    *(load_base + r_offset) = load_base + r_addend
    
  의미:
    "이 위치에 load_base + 상수 를 써라"
    PIE binary 의 모든 글로벌 pointer 가 이 type
    
  예:
    char *msg = "hello";       /* msg 가 .data 의 변수 */
    "hello" 는 .rodata 의 어느 offset
    
    relocation:
      r_offset = .data 의 msg 위치
      r_addend = .rodata 의 "hello" 의 offset
      
    ld-linux:
      msg = load_base + .rodata 의 hello offset

PIE 의 비밀 — symbol lookup 없이 빠른 처리. symbol index = 0 (없음).

R_X86_64_GLOB_DAT — eager data

[R_X86_64_GLOB_DAT 의 동작]

  공식:
    *(load_base + r_offset) = symbol_address (이미 resolved)
    
  의미:
    외부 symbol 의 GOT entry 채우기
    eager (startup 시)
    
  사용처:
    extern 의 변수 ref (다른 library 의)
    함수 pointer (-fno-plt 시)
    __libc_start_main 같은 startup 필수 symbol

R_X86_64_JUMP_SLOT — lazy PLT

[R_X86_64_JUMP_SLOT 의 동작]

  공식:
    *(load_base + r_offset) = resolved_function_address
    
  의미:
    PLT 의 GOT entry 채우기
    lazy (첫 호출 시)
    
  사용처:
    모든 외부 함수 호출
    R_X86_64_GLOB_DAT 와 비슷, 그러나 PLT 전용

R_X86_64_64 — absolute 64-bit

[R_X86_64_64 의 동작]

  공식:
    *(load_base + r_offset) = symbol_address + r_addend
    
  의미:
    절대 주소 64-bit 직접 쓰기
    PIE 가 아닌 binary 에서 사용
    
  PIE 와의 차이:
    PIE: 모두 RELATIVE 또는 indirect
    non-PIE: 직접 R_X86_64_64

PIE 가 아닌 binary (정적 위치) — 약간 빠르나, ASLR 의 효과 적음.

R_X86_64_PC32 — PC-relative

[R_X86_64_PC32 의 동작]

  공식:
    *(load_base + r_offset) = symbol_address + r_addend - (load_base + r_offset)
    = (symbol - target_location) + r_addend
    
  의미:
    PC 기준의 상대 주소 (32-bit)
    instruction 의 immediate 자리에 들어감
    
  사용처:
    함수 호출의 call 명령어
    jmp 명령어의 target
    가까운 데이터의 참조 (LEA 등)

이게 — PIE 의 코드 안의 모든 분기.

R_X86_64_IRELATIVE — ifunc

GNU 의 indirect function 메커니즘 (62 장 자세히).

[R_X86_64_IRELATIVE 의 동작]

  공식:
    resolver = load_base + r_addend
    *(load_base + r_offset) = resolver()    /* 함수 호출! */
    
  의미:
    resolver 라는 함수를 호출
    그 반환값을 GOT 에 저장
    이후 GOT 의 값으로 함수 호출
    
  사용처:
    memcpy 가 CPU 의 SSE2 / AVX / AVX-512 따라 다른 구현
    resolver 가 CPU detect 후 적절한 구현 반환

memcpy, memset, strlen 같은 — 매우 자주 호출되는 함수들이 ifunc 사용. 한 번의 resolution 으로 CPU 별 최적 구현.

[relocation 처리 순서]

  1. R_X86_64_RELATIVE 먼저 처리
     - symbol lookup 불필요

     - 매우 빠름

// [user-space - glibc elf/dynamic-link.h master 단순화]
static inline void
elf_machine_rela(struct link_map *map, const ElfW(Rela) *reloc,
                 const ElfW(Sym) *sym, const struct r_found_version *version,
                 void *const reloc_addr, int skip_ifunc)
{
    const unsigned long int r_type = ELFW(R_TYPE)(reloc->r_info);
    
    switch (r_type) {
    case R_X86_64_NONE:
        break;
        
    case R_X86_64_RELATIVE:
        *(ElfW(Addr) *)reloc_addr = map->l_addr + reloc->r_addend;
        break;
        
    case R_X86_64_64:
        *(ElfW(Addr) *)reloc_addr = sym_addr + reloc->r_addend;
        break;
        
    case R_X86_64_GLOB_DAT:
    case R_X86_64_JUMP_SLOT:
        *(ElfW(Addr) *)reloc_addr = sym_addr;
        break;
        
    case R_X86_64_PC32:
        *(uint32_t *)reloc_addr = (sym_addr + reloc->r_addend
                                   - (ElfW(Addr))reloc_addr);
        break;
        
    case R_X86_64_COPY:
        /* COPY: source 의 데이터를 target 으로 memcpy */
        memcpy(reloc_addr, (void *)sym_addr, sym->st_size);
        break;
        
    case R_X86_64_IRELATIVE:
        /* resolver 호출 */
        value = ((ElfW(Addr) (*)(void))(map->l_addr + reloc->r_addend))();
        *(ElfW(Addr) *)reloc_addr = value;
        break;
        
    case R_X86_64_TPOFF64:
    case R_X86_64_DTPOFF64:
    case R_X86_64_DTPMOD64:
        /* TLS 처리 */
        ...
        break;
    }
}

static int
do_lookup_x(...)
{
    /* hash 로 symbol 후보 찾기 */
    for (each candidate sym) {
        const char *strtab = (const char *)D_PTR(map, l_info[DT_STRTAB]);
        
        /* 이름 일치 검사 */
        if (strcmp(strtab + sym->st_name, undef_name) != 0)
            continue;
        
        /* version 일치 검사 */
        if (version != NULL && map->l_versions != NULL) {
            const ElfW(Half) *versym = 
                (const ElfW(Half) *)D_PTR(map, l_info[VERSYMIDX(DT_VERSYM)]);
            ElfW(Half) ndx = versym[sym - symtab] & 0x7fff;
            
            if (map->l_versions[ndx].hash != version->hash)
                continue;     /* version mismatch */
            if (strcmp(map->l_versions[ndx].name, version->name) != 0)
                continue;
        }
        
        /* 일치! 사용 */
        return 1;
    }
    return 0;
}

dlopen 의 flags

[dlopen 의 mode flags]

  Resolution timing:
    RTLD_LAZY    : function 의 lazy binding (PLT 통해)
    RTLD_NOW     : 즉시 모든 symbol resolve
                   ★ 또는 LD_BIND_NOW=1 환경변수
    
  Scope:
    RTLD_LOCAL   : symbol 이 다른 library 에 안 보임 (기본)
    RTLD_GLOBAL  : symbol 이 global scope 에 추가
    
  추가 control:
    RTLD_NODELETE: dlclose 시 unmap 안 함 (이후 dlopen 빠름)
    RTLD_NOLOAD  : 로드 안 함, 이미 로드면 handle 반환만
    RTLD_DEEPBIND: 자기 안의 symbol 을 _다른 곳보다 우선_

각 flag 의 의미

**RTLD_LAZY vs RTLD_NOW**:

[lazy vs now]

  RTLD_LAZY (기본):
    function 만 lazy (PLT 통해)
    data symbol 은 즉시
    dlopen 빠름, 첫 호출 시 추가 cost
    
  RTLD_NOW:
    모든 symbol 즉시 resolve
    dlopen 느림, 호출 시 cost 없음
    
    장점: 누락 symbol 즉시 발견 (디버깅 좋음)
    단점: startup 느림

**RTLD_LOCAL vs RTLD_GLOBAL**:

[local vs global]

  RTLD_LOCAL (기본):
    plugin.so 의 symbol 들이 main 의 global scope 에 _안 들어감_
    main 안에서 자유 호출 위해 dlsym 필요
    
  RTLD_GLOBAL:
    plugin.so 의 symbol 들이 global scope 추가
    이후 dlopen 되는 library 들이 이 symbol 사용 가능
    
  예 — plugin chain:
    libplugin_base.so (RTLD_GLOBAL) 로드
       → 그 안의 base_api 함수가 global
    libplugin_addon.so 로드
       → base_api 를 자동으로 사용 (link 시 unresolved)

**RTLD_DEEPBIND** — 위험한 옵션:

[RTLD_DEEPBIND 의 동작]

  보통:
    plugin.so 의 함수 안에서 malloc 호출
    → libc 의 malloc (main 의 LD_PRELOAD 적용)
    
  RTLD_DEEPBIND 시:
    plugin.so 의 함수 안에서 malloc 호출
    → plugin.so 가 정의한 malloc 가 _우선_
    (자기 안의 symbol → main 의 symbol → libc)
    
  사용처:
    plugin 의 self-contained 보장
    
  위험:
    LD_PRELOAD 의 hook 효과 무효화 (debugging 어려움)
    버전 충돌 가능

// [user-space - glibc elf/dl-open.c master 매우 단순화]
void *
_dl_open(const char *file, int mode, const void *caller_dlopen,
         Lmid_t nsid, int argc, char *argv[], char *env[])
{
    struct link_map *new;
    
    /* lock */
    __rtld_lock_lock_recursive(GL(dl_load_lock));
    
    /* 이미 로드된 library 검색 */
    new = _dl_find_dso_in_namespace(file, nsid);
    if (new == NULL || (mode & RTLD_NOLOAD) == 0) {
        /* 새로 로드 */
        new = _dl_map_object(NULL, file, lt_loaded, 0,
                            __RTLD_DLOPEN | mode, nsid);
    } else {
        /* 이미 있음 — refcount 증가 */
        new->l_direct_opencount++;
    }
    
    /* 의존성 로드 (DT_NEEDED 순회) */
    _dl_map_object_deps(new, NULL, 0, 0, 0);
    
    /* lookup scope 결정 */
    if (mode & RTLD_GLOBAL) {
        /* main scope 에 추가 */
        add_to_global_resize(new);
    }
    
    /* relocation 처리 */
    int reloc_mode = (mode & RTLD_LAZY) ? RTLD_LAZY : 0;
    _dl_relocate_object(new, new->l_scope, reloc_mode, 0);
    
    /* init function 호출 (.init_array 등) */
    _dl_init(new, argc, argv, env);
    
    __rtld_lock_unlock_recursive(GL(dl_load_lock));
    return new;
}
[_dl_open 의 5 단계]

  1. _dl_load_lock 획득 (다른 dlopen 과 race 방지)
  2. 이미 로드된 library 인지 검사 (cache)
     - 같으면 refcount 만 증가
  3. 새 library 면 _dl_map_object:
     - 파일 열고 ELF 검증
     - PT_LOAD segment mmap
     - link_map 만들고 link list 에 추가
  4. 의존성 재귀 로드 (_dl_map_object_deps)
     - DT_NEEDED 순회
     - 각각 재귀적으로 _dl_map_object
  5. relocation + init:
     - _dl_relocate_object: GOT 채우기
     - _dl_init: .init_array 의 모든 함수 호출

의존성 로드 — _dl_map_object_deps

dlopen 한 library 의 모든 DT_NEEDED:

[의존성 재귀 로드]

  plugin.so 가 libcrypto.so 의존:
    
    dlopen("plugin.so", ...):
      1. plugin.so mmap
      2. PT_DYNAMIC 분석 → DT_NEEDED: libcrypto.so
      3. _dl_map_object("libcrypto.so", ...)
         3a. libcrypto.so mmap
         3b. libcrypto 의 DT_NEEDED 도 재귀
      4. 모든 의존성 로드 완료
      5. 함께 relocation
      6. 함께 init 호출
      
  → BFS 순서로 의존성 그래프 전체 로드

init function 호출 순서

[init function 호출 순서]

  의존성 그래프의 _topological order_:
    의존되는 것 _먼저_, 의존하는 것 나중
    
  예: plugin.so → libcrypto.so → libc.so
    호출 순서:
      libc.so 의 init      ← 의존되는 것 먼저
      libcrypto.so 의 init
      plugin.so 의 init    ← 마지막
      
  각 library 의 init:
    DT_INIT 의 _init 함수 (옛)
    DT_INIT_ARRAY 의 모든 함수 (순서대로)

// [user-space - glibc elf/dl-sym.c master 단순화]
void *
__dlsym(void *handle, const char *name, void *dl_caller)
{
    void *result;
    
    __rtld_lock_lock_recursive(GL(dl_load_lock));
    
    if (handle == RTLD_DEFAULT) {
        /* main scope 검색 */
        result = _dl_sym(handle, name, dl_caller);
    } else if (handle == RTLD_NEXT) {
        /* 호출자 다음의 scope */
        result = _dl_sym(handle, name, dl_caller);
    } else {
        /* 특정 handle 의 local scope */
        result = _dl_vsym(handle, name, NULL, dl_caller);
    }
    
    __rtld_lock_unlock_recursive(GL(dl_load_lock));
    return result;
}

dlsym 의 검색 scope

[dlsym 의 handle 별 검색]

  dlsym(handle, "name"):
    handle 이 RTLD_DEFAULT (or NULL):
      → main 의 global scope 부터 검색
      → 일반 symbol resolution 과 동일 순서
      
    handle 이 RTLD_NEXT:
      → 호출자 _다음_ 의 scope 부터 검색
      → LD_PRELOAD 의 hook 에서 real 함수 찾는 용도
      
    handle 이 dlopen 의 반환값:
      → 그 handle 의 local scope (RTLD_LOCAL 시)
      → 또는 global scope (RTLD_GLOBAL 시)
      → DT_NEEDED 의존성도 함께 검색

dlerror — thread-local

// [user-space - glibc dl-error.c master 단순화]
struct dl_action_result {
    int errcode;
    const char *objname;
    const char *errstring;
    bool malloced;
};

static __thread struct dl_action_result *last_result;

char *
__dlerror(void)
{
    if (last_result == NULL)
        return NULL;
    
    char *buf = NULL;
    /* error message 조합 (실제는 더 복잡) */
    if (last_result->errstring != NULL) {
        if (last_result->objname != NULL) {
            asprintf(&buf, "%s: %s", last_result->objname, last_result->errstring);
        } else {
            buf = strdup(last_result->errstring);
        }
    }
    
    /* error 를 _소비_ (다시 호출 시 NULL) */
    last_result = NULL;
    return buf;
}

dlclose — reference counting

// [user-space - glibc elf/dl-close.c master 매우 단순화]
int
__dlclose(void *handle)
{
    struct link_map *map = handle;
    
    __rtld_lock_lock_recursive(GL(dl_load_lock));
    
    /* refcount 감소 */
    if (map->l_direct_opencount == 0) {
        _dl_signal_error(EINVAL, NULL, NULL, "shared object not open");
        goto out;
    }
    --map->l_direct_opencount;
    
    /* 아직 ref 가 있으면 unmap 안 함 */
    if (map->l_direct_opencount > 0)
        goto out;
    
    /* 정말 마지막 ref — 진짜 unload */
    _dl_close_worker(map, false);
    
out:
    __rtld_lock_unlock_recursive(GL(dl_load_lock));
    return 0;
}
[dlclose 의 reference counting]

  같은 library 의 dlopen 여러 번:
    각 dlopen 이 refcount 증가
    
  dlclose:
    refcount 감소
    0 이 되면 진짜 unload
    
  의존성 graph 의 처리:
    plugin 이 libfoo 의존
    dlclose(plugin) → libfoo 의 refcount 도 감소
    libfoo 의 ref 0 면 같이 unload

진짜 unload — _dl_close_worker

// [user-space - glibc elf/dl-close.c master 매우 단순화]
void
_dl_close_worker(struct link_map *map, bool force)
{
    /* 1. fini function 호출 (역순) */
    if (map->l_info[DT_FINI_ARRAY] != NULL) {
        ElfW(Addr) *array = (ElfW(Addr) *)
            (map->l_addr + map->l_info[DT_FINI_ARRAY]->d_un.d_ptr);
        unsigned int sz = map->l_info[DT_FINI_ARRAYSZ]->d_un.d_val / sizeof(ElfW(Addr));
        
        while (sz-- > 0)
            ((fini_t)array[sz])();   /* 역순으로 호출 */
    }
    if (map->l_info[DT_FINI] != NULL)
        ((fini_t)(map->l_addr + map->l_info[DT_FINI]->d_un.d_ptr))();
    
    /* 2. global scope 에서 제거 */
    if (map->l_global) {
        remove_from_global_scope(map);
    }
    
    /* 3. TLS 처리 */
    if (map->l_tls_blocksize > 0)
        _dl_tls_module_free(map);
    
    /* 4. memory 해제 */
    __munmap((void *)map->l_map_start, map->l_map_end - map->l_map_start);
    
    /* 5. link_map 자체 해제 */
    _dl_close_free(map);
}
[unload 의 5 단계]

  1. fini function 호출 (역순):
     - DT_FINI_ARRAY 의 함수들 (뒤부터)
     - DT_FINI (옛 _fini)
     - C++ destructor 가 여기 들어감
     
  2. global scope 에서 제거
     - RTLD_GLOBAL 였으면 main scope 에서 제외
     
  3. TLS 의 영역 해제
     - dynamic TLS 의 module ID 등
     
  4. munmap:
     - .text, .rodata, .data, .bss 모두 해제
     - 메모리 OS 로 반환
     
  5. link_map 의 free
     - kernel 의 입장에선 영원히 없는 것처럼

unmap 의 위험

dlclose 의 가장 큰 함정:

[dlclose 의 위험]

  plugin 의 함수 포인터를 main 이 저장:

    → plugin 의 메모리 munmap

61LD_PRELOAD / LD_AUDIT — hooking 메커니즘

dlopen 없이도 — process 의 모든 dynamic linking 에 개입할 방법. LD_PRELOAD 와 LD_AUDIT 두 메커니즘. profiler, tracer, security tool, malware 가 — 모두 이걸 활용.

LD_PRELOAD 의 정확한 동작

[LD_PRELOAD 의 처리]

  ld-linux 가 startup 시:
    1. LD_PRELOAD 환경변수 검사
    2. 거기 명시된 library 들 _가장 먼저_ 로드
       (DT_NEEDED 처리 전에)
    3. scope 의 _앞_에 추가 (main 다음, 다른 의존성 전)
    
  결과:
    scope 순서:
      1. main program
      2. LD_PRELOAD libraries (1, 2, 3, ...)
      3. main 의 DT_NEEDED 들
      ...
      
  symbol resolution 시:
    LD_PRELOAD 의 library 가 _먼저_ 검색됨
    → 거기에 있으면 그게 사용 (override 효과)

기본 hook 예제 — malloc 가로채기

// [user-space - C, malloc_hook.c]
#define _GNU_SOURCE
#include <dlfcn.h>
#include <stdio.h>
#include <stdlib.h>

/* real malloc 의 포인터 */
static void *(*real_malloc)(size_t) = NULL;

/* 우리의 malloc — process 의 모든 malloc 호출이 이 곳으로 */
void *malloc(size_t size) {
    if (!real_malloc) {
        /* 첫 호출 시 진짜 malloc 찾기 */
        real_malloc = dlsym(RTLD_NEXT, "malloc");
        if (!real_malloc) {
            fprintf(stderr, "dlsym: %s\n", dlerror());
            return NULL;
        }
    }
    
    void *ptr = real_malloc(size);
    fprintf(stderr, "[hook] malloc(%zu) = %p\n", size, ptr);
    return ptr;
}

RTLD_NEXT 의 중요성

위 예제의 dlsym(RTLD_NEXT, "malloc") 가 — 핵심.

[RTLD_NEXT 가 없으면]

  dlsym(RTLD_DEFAULT, "malloc"):
    → 첫 번째 malloc 검색
    → 자기 자신 (hook 의 malloc)!
    → 무한 재귀
    
  dlsym(RTLD_NEXT, "malloc"):
    → 호출자 (hook 자체) _다음_ 의 scope 검색
    → libc 의 진짜 malloc
    → 올바름

이게 — hook 의 정형 패턴.

__libc_malloc 직접 호출 — 더 안전한 방법

glibc 내부에서 — __libc_malloc 이라는 internal name 도 export.

  • 단점: 특정 glibc 의 internal name 의존, 비호환

#include <stdarg.h>
#include <fcntl.h>
#include <stdio.h>

        va_end(args);

setuid binary 와 LD_PRELOAD

[보안 — setuid 의 LD_PRELOAD 무시]

  보통:
    /usr/bin/sudo (setuid root)
    LD_PRELOAD=hook.so /usr/bin/sudo
    
  hook.so 의 모든 함수가 _root 권한으로_ 실행 가능
  → 매우 위험
  
  glibc 의 대응:
    setuid binary 면 ld-linux 가 LD_PRELOAD 무시
    (또는 secure 한 path 의 library 만 허용)
    
  관련 환경변수도 모두 무시:
    LD_LIBRARY_PATH, LD_AUDIT, LD_DEBUG 등
    
  __libc_enable_secure flag 가 1

이게 — Linux 보안의 기본 메커니즘.

LD_PRELOAD 의 한계

[LD_PRELOAD 의 한계]

  ✓ override 가능: function symbol (PLT 통한 호출)
  ✗ override 불가:
    - static 함수 (외부 안 보임)
    - inline 된 함수 (compile 시 elided)
    - direct system call (syscall 어셈블리)
    - 같은 library 안의 호출 (자체 호출은 PLT 안 거침)
    
  예 — fprintf 가 vfprintf 호출:
    이는 libc.so.6 _안의_ 호출
    PLT 안 거침 → hook 안 됨
    
  해결:
    LD_AUDIT 또는 ptrace 등 더 강력한 메커니즘

LD_AUDIT — 더 강력한 hooking

LD_AUDIT 가 — 단순한 override 가 아닌, dynamic linker 의 모든 동작에 hook.

[LD_AUDIT 의 동작]

  LD_AUDIT=audit.so ./app
    
  ld-linux 가:
    1. audit.so 로드 (LD_PRELOAD 보다 먼저)
    2. la_version 호출 — handshake
    3. 모든 library load 시 la_objopen 호출
    4. 모든 symbol binding 시 la_symbind64 호출
    5. 모든 PLT 호출 시 la_pltenter/exit (선택)

ifunc — GNU indirect function

[ifunc 의 motivation]

  problem:
    memcpy 가 자주 호출되는 hot function
    CPU 마다 최적 구현 다름:
      - SSE2 (모든 x86-64 CPU)
      - SSSE3
      - AVX
      - AVX2
      - AVX-512
    
  옛 해결:
    binary 가 IFUNC 검사 후 함수 포인터 사용
    매 호출마다 if/else
    → 분기 비용
    
  ifunc 의 해결:
    함수 _이름_ 은 하나 (memcpy)
    runtime 에 한 번 resolver 가 적절한 구현 선택
    이후 모든 호출이 _직접 그 구현_ 으로
    → 분기 없음

ifunc 의 메커니즘

// [user-space - C 예제]
/* 여러 구현 */
static void memcpy_generic(void *dst, const void *src, size_t n) { /* 기본 */ }
static void memcpy_sse2(void *dst, const void *src, size_t n) { /* SSE2 */ }
static void memcpy_avx(void *dst, const void *src, size_t n) { /* AVX */ }
static void memcpy_avx512(void *dst, const void *src, size_t n) { /* AVX-512 */ }

/* resolver — CPU 검사 후 가장 좋은 것 반환 */
static void (*resolve_memcpy(void))(void *, const void *, size_t) {
    if (cpu_has_avx512())
        return memcpy_avx512;
    if (cpu_has_avx())
        return memcpy_avx;
    if (cpu_has_sse2())
        return memcpy_sse2;
    return memcpy_generic;
}

/* ifunc 선언 — 컴파일러가 resolver 호출 처리 */
void *my_memcpy(void *, const void *, size_t)
    __attribute__((ifunc("resolve_memcpy")));

R_X86_64_IRELATIVE 의 동작

[ifunc 의 runtime 처리]

  컴파일 결과:
    binary 의 .rela.dyn 또는 .rela.plt 의 entry:
      type = R_X86_64_IRELATIVE
      r_offset = .got.plt 의 my_memcpy 위치
      r_addend = resolve_memcpy 함수의 offset
      
  ld-linux 의 처리 (startup 시):
    1. R_X86_64_RELATIVE 다 처리
    2. R_X86_64_GLOB_DAT 처리
    3. R_X86_64_IRELATIVE:
       resolver = (function *)(load_base + r_addend)
       result = resolver()           ← resolver 호출
       *(GOT_addr) = result          ← GOT 에 저장
    4. R_X86_64_JUMP_SLOT (lazy)
    
  → resolver 가 startup 시 _한 번_ 호출
  → GOT 가 그 결과로 영구 설정
  
  이후 my_memcpy 호출:
    PLT 통한 jmp *GOT[N]
    GOT 가 이미 resolved → 즉시 직접 호출
    분기 없음

weak symbol — link-time vs runtime

57 장에서 본 — STB_WEAK. 두 종류로 분류:

[weak symbol 의 두 종류]

  1. weak definition (정의):
     이 정의가 _약함_
     같은 이름의 strong 가 있으면 그게 우선
     
  2. weak reference (참조):
     이 _참조_ 가 약함
     정의가 없어도 link OK (NULL 로 resolve)

weak definition 예

// [user-space - C 예제, library 안]
__attribute__((weak))
void hook(void) {
    /* default 동작 (아무것도 안 함) */
}

void main_logic(void) {
    /* hook 호출 */
    hook();
}
// [user-space - C, 사용자 코드]
/* 사용자가 override 가능 */
void hook(void) {
    printf("custom hook called\n");
}

int main(void) {
    main_logic();    /* → "custom hook called" */
    return 0;
}
[weak definition 의 처리]

  link 시:
    libfoo.a 의 hook (weak)
    user.o 의 hook (strong, default)
    
    → user.o 의 hook 가 우선
    
  user 가 안 정의:
    → libfoo.a 의 weak hook 사용 (default)

weak reference 예

// [user-space - C 예제]
/* 이 symbol 의 _존재_ 가 unknown 인 ref */
extern void optional_feature(void) __attribute__((weak));

int main(void) {
    if (optional_feature) {     /* weak ref 면 NULL 가능 */
        optional_feature();
    } else {
        /* fallback */
    }
    return 0;
}
[weak reference 의 처리]

  link 시:
    optional_feature 의 정의 못 찾음
    weak ref 라 — link OK
    binary 안에서 optional_feature 의 값 = NULL
    
  runtime:
    if 가 NULL 검사 → fallback

visibility — 4 종류

[symbol visibility]

  STV_DEFAULT:
    - 일반 (외부에 보임)
    - linker 가 default behavior
    - 다른 library 가 override 가능
    
  STV_PROTECTED:
    - 외부에 보임 (export 됨)
    - 같은 .so 안의 ref 는 _자기 정의_ 사용 (preempt 안 됨)
    - LD_PRELOAD 로 override 가능하지만, 자기 안 호출은 영향 X
    
  STV_HIDDEN:
    - 외부에 _안 보임_
    - 다른 .so 가 dlsym 불가
    - 같은 .so 안에서만 사용
    - static 함수와 비슷, 그러나 다른 .o 끼리 link 가능
    
  STV_INTERNAL:
    - STV_HIDDEN 비슷
    - ABI 측면에서 약간 다름 (덜 strict)
    - 사실 거의 안 씀

printf 호출의 완전한 그림

이제 — Part VII 의 시작에 본 질문에 완전한 답.

[printf("hello\n") 의 전체 흐름]

  컴파일 시:
    gcc 가 binary 에:
    - main 의 코드 (call printf@plt)
    - PLT[printf] = 3 instruction
    - GOT[printf] = PLT[printf]+6 (self-ref)
    - .rela.plt 의 R_X86_64_JUMP_SLOT 의 reloc
    - .dynsym 의 printf symbol
    - .dynstr 의 "printf" 문자열
    - .gnu.version_r 의 "GLIBC_2.2.5"
    - DT_NEEDED: libc.so.6
    - PT_INTERP: /lib64/ld-linux-x86-64.so.2
    
  execve:
    kernel 의 load_elf_binary:
      - hello 의 PT_LOAD 들 mmap
      - PT_INTERP 의 ld-linux mmap
      - stack 에 argv, envp, auxv 준비
      - control 을 ld-linux 의 _start 로
    
  ld-linux 의 startup:
    _dl_start: self-relocation
    dl_main:
      - hello 의 dynamic 분석
      - LD_PRELOAD 검사 (없으면 skip)
      - DT_NEEDED 의 libc.so.6 로드
      - libc 의 의존성 (없음) 로드
      - relocation:
        - R_X86_64_RELATIVE 들 처리
        - R_X86_64_GLOB_DAT 처리 (eager)
        - R_X86_64_IRELATIVE 처리 (memcpy 등의 resolver 호출)
        - R_X86_64_JUMP_SLOT 의 lazy 처리 (printf 는 아직 X)
      - .init_array 호출 (libc 의 init, hello 의 init)
      - hello 의 _start 로 jump
    
  hello 의 startup:
    _start → __libc_start_main → main
    
  printf("hello\n") 첫 호출:
    main 의 call printf@plt
    → PLT[printf]: jmp *GOT[printf]
    → GOT 가 self-ref (PLT+6) → push 0 + jmp PLT[0]
    → PLT[0]: push GOT[1] + jmp *GOT[2]
    → _dl_runtime_resolve (어셈블리)
    → 모든 register 저장
    → _dl_fixup(link_map, 0)
    → DT_SYMTAB 에서 symbol 0 = printf
    → version "GLIBC_2.2.5"
    → _dl_lookup_symbol_x:
      scope 의 BFS:
        hello: 안 함
        libc.so.6: hash → printf 찾음 → version match → 발견
    → printf 의 실제 주소 계산
    → elf_machine_fixup_plt: GOT 에 atomic write
    → return printf 주소
    → register 복원
    → jmp printf
    → printf 실행, "hello\n" stdout 출력
    
  printf("world\n") 두 번째 호출:
    main 의 call printf@plt
    → PLT[printf]: jmp *GOT[printf]
    → GOT 가 이미 resolved
    → 직접 printf 로 jump
    → printf 실행
