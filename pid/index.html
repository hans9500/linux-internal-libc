1가장 흔한 의문

Linux 커널 소스 트리를 처음 열어보면 누구나 비슷한 의문을 품는다. 보통 C 프로그램은 main.c 파일에 int main() 함수가 있고 거기서 실행이 시작된다고 배웠는데, Linux 커널 소스에는 분명 init/main.c 라는 파일이 있다. 그런데 그 파일을 열어봐도 main() 함수는 보이지 않는다. 대신 start_kernel() 이라는 낯선 함수가 있다. "이건 뭐지? 그럼 커널은 어디서 시작하는 거지?" 하는 의문이 생긴다.

이 의문은 사실 두 가지 별개의 사실이 머릿속에서 섞여 있어서 생긴다.

첫째 사실은, userspace 프로그램의 main() 도 사실 CPU 가 직접 호출하는 게 아니다. 우리가 작성한 main 이 호출되기 전에 _start 라는 어셈블리 함수가 먼저 실행되고, 그게 __libc_start_main 이라는 glibc 함수를 호출하고, 그 안에서 비로소 우리의 main 이 호출된다. 즉 "main 이 entry" 는 C 언어와 libc 가 만든 약속일 뿐, 실제 실행 메커니즘은 그렇지 않다.

둘째 사실은, Linux 커널은 freestanding C 환경이라는 것이다. C11 표준은 두 가지 실행 환경을 정의하는데, 그중 freestanding 환경에서는 시작 함수의 이름과 타입을 구현이 마음대로 정할 수 있다. 즉 main 이라는 이름을 강제하는 것은 hosted 환경 (= libc 위에서 동작하는 일반 사용자 프로그램) 에만 적용되는 규칙이다. 커널은 그 규칙을 만드는 쪽이지, 따르는 쪽이 아니다.

이 두 사실을 종합하면 결론은 명료하다. init/main.c 파일은 그저 "커널 시작 로직 모음" 이라는 의미의 역사적 naming 이고, 그 안에는 main() 함수가 없다. 대신 start_kernel() 이 사실상 그 역할을 하며, 이는 C 표준상 완벽히 합법이다.

이 문서는 이 결론에 도달하기까지의 모든 디테일을 단계적으로 풀어간다.

3미리 보는 한 줄 결론

긴 문서를 읽기 전에, 결론을 미리 보면 학습 중에 길을 잃지 않는다.

Linux 커널은 "프로그램" 이 아니라 "실행 환경 그 자체" 다.
일반 프로그램의 main() 은 OS 가 만들어준 환경 위에서 동작하는 약속이고,
커널은 그 환경을 만드는 쪽이라 그 약속에 묶이지 않는다.

start_kernel() 이 사실상 커널의 main 역할을 하며,
PID 0 (swapper) 컨텍스트에서 시작해 PID 1 (systemd) 와 PID 2 (kthreadd)
를 fork 한 뒤 자신은 idle task 로 전환된다.

이후 PID 1 이 모든 userspace 를 관리하고,
PID 2 가 모든 kernel thread 를 관리하며,
PID 0 은 할 일이 없을 때 CPU 절전을 담당한다.

이 한 단락을 머릿속에 두고 다음 Stage 부터 차근차근 디테일을 채워나가자.

Stage 1"main 함수는 어디 있지?" — userspace 와 kernel 의 entry 비교

이 Stage 의 목표는 "Linux 의 main.c 에는 왜 main() 함수가 없는가?" 라는 의문을 완전히 해소하는 것이다. 단순히 "커널은 main 을 안 쓴다" 로 끝나지 않고, 일반 프로그램의 main() 이 어떻게 호출되는지부터 차근차근 짚어나간다. userspace 와 kernel 의 entry point 메커니즘을 나란히 비교하면, 둘이 사실 같은 ELF 메커니즘 위에서 동작한다는 것이 보이고, 그제야 "커널은 왜 다른가" 가 진짜로 이해된다.

4일반 프로그램의 진짜 entry — _start

우리가 보통 C 프로그램을 작성할 때, 실행은 main 함수에서 시작한다고 배운다. 이건 어떤 의미에서는 맞고 어떤 의미에서는 틀린 말이다. 정확히 말하면, C 언어 차원에서는 main 이 시작점이지만, 실제 CPU 실행 차원에서는 main 이 시작점이 아니다. 우리의 main 이 호출되기 전에 이미 여러 일이 일어난다.

가장 단순한 hello world 프로그램으로 직접 확인해보자.

/* hello.c */
#include <stdio.h>
int main(void) {
    printf("hi\n");
    return 0;
}
$ gcc -o hello hello.c
$ readelf -h hello | grep Entry
  Entry point address:               0x1050

readelf -h 가 보여주는 "Entry point address" 가 ELF 헤더의 e_entry 필드 값이다. 이게 OS 가 이 프로그램을 실행할 때 처음으로 점프할 주소다. 그런데 이 주소는 main 이 아니다. objdump 로 확인해보면 이 주소에 _start 라는 함수가 있다.

$ objdump -d hello | grep -A 12 "<_start>:"
0000000000001050 <_start>:
    1050: f3 0f 1e fa           endbr64
    1054: 31 ed                 xor    %ebp,%ebp
    1056: 49 89 d1              mov    %rdx,%r9
    1059: 5e                    pop    %rsi
    105a: 48 89 e2              mov    %rsp,%rdx
    105d: 48 83 e4 f0           and    $0xfffffffffffffff0,%rsp
    1061: 50                    push   %rax
    1062: 54                    push   %rsp
    ...
    106a: 48 8d 3d 6f 01 ...    lea    0x16f(%rip),%rdi
    1071: ff 15 ...             call   *...(%rip)

이 _start 함수가 무엇을 하는지는 glibc 소스에서 확인할 수 있다. glibc 의 sysdeps/x86_64/start.S 파일이 x86_64 의 _start 를 정의하는데, 그 안의 주석이 핵심을 그대로 알려준다.

/*
 * stack 레이아웃 (execve 직후, _start 진입 시):
 *
 *   (%rsp)              argc                ← 정수 (포인터 아님)
 *   8(%rsp)             argv[0]             ← 첫 인자 문자열 포인터
 *   16(%rsp)            argv[1]
 *   ...
 *                       NULL                ← argv 끝 표시
 *                       envp[0]
 *   ...
 *                       NULL                ← envp 끝 표시
 *                       auxv                ← ELF auxiliary vector
 *                       ...
 */

ENTRY (_start)
    cfi_undefined (rip)
    /* frame pointer 클리어 — outermost frame 표시 (ABI 권장) */
    xorl %ebp, %ebp

    /* __libc_start_main 인자 준비 (System V AMD64 ABI):
         main:      %rdi
         argc:      %rsi
         argv:      %rdx
         init:      %rcx
         fini:      %r8
         rtld_fini: %r9
         stack_end: stack
    */
    mov %RDX_LP, %R9_LP     /* rtld_fini → r9 (kernel 이 rdx 에 넣어줌) */
    popq %rsi               /* argc → rsi (stack top 에서 pop) */
    mov %RSP_LP, %RDX_LP    /* argv → rdx (현재 rsp 가 argv 시작점) */

    /* main, init, fini 는 linker 가 채움 */
    /* ... */
    call __libc_start_main
    hlt                     /* 절대 도달하지 않아야 함 */

요약하면 _start 는 두 가지 일을 한다. 하나, kernel 이 stack 에 미리 배치해둔 argc, argv, envp 를 register 로 옮긴다. 둘, __libc_start_main 이라는 glibc 함수를 호출한다. 우리가 작성한 main 의 주소는 linker 가 컴파일 시점에 미리 채워서 register 에 들어간다.

5__libc_start_main 이 진짜로 main 을 호출

_start 가 호출하는 __libc_start_main 은 단순한 wrapper 가 아니다. 이 함수가 main 을 호출하기 전에 여러 가지 setup 을 수행한다. 정확한 시그니처는 glibc 의 csu/libc-start.c 에 있다.

int __libc_start_main(
    int (*main)(int, char **, char **),    /* 우리의 main 함수 포인터 */
    int argc,                              /* 인자 개수 */
    char **argv,                           /* 인자 배열 (envp 도 여기서 계산) */
    void (*init)(void),                    /* __libc_csu_init = 전역 ctor */
    void (*fini)(void),                    /* __libc_csu_fini = 전역 dtor */
    void (*rtld_fini)(void),               /* dynamic linker cleanup */
    void *stack_end                        /* stack 시작 위치 */
);

인자가 무려 7개다. GPT 가 흔히 단순화해서 __libc_start_main(main, ...) 라고 쓰는데, 실제로는 main 함수 포인터뿐 아니라 ctor / dtor 함수 포인터, dynamic linker cleanup 콜백, stack 정보까지 모두 받는다.

내부 동작은 다음과 같이 단순화할 수 있다.

int __libc_start_main(...)
{
    /* 1. 보안 검사 (effective uid vs real uid 비교 등) */
    /* 2. TLS (thread-local storage) 초기화 */
    /* 3. dynamic linker cleanup 등록 (atexit 비슷) */

    /* 4. 전역 ctor 호출 */
    init();   /* = __libc_csu_init → __init_array 안 ctor 들 호출 */

    /* 5. ★ 우리의 main 호출 ★ */
    int result = main(argc, argv, __environ);

    /* 6. exit 호출 (전역 dtor 도 그 안에서) */
    exit(result);
    /* exit 안에서:
         atexit 등록된 함수들 호출
         fini() 호출 (= __libc_csu_fini → __fini_array)
         exit_group syscall → process 종료
       즉 main 이 return 해도 _start 로 돌아가지 않음 */
}

이 흐름은 GDB 로 직접 확인할 수 있다. main 에 breakpoint 를 걸고 backtrace 를 찍어보면 다음과 같은 스택 프레임이 보인다.

$ gcc -g -o hello hello.c
$ gdb hello
(gdb) break main
(gdb) run
(gdb) bt
#0  main () at hello.c:3
#1  0x00007ffff7a05b97 in __libc_start_main (
        main=0x55..., argc=1, argv=0x7fff...,
        init=<optimized out>, fini=<optimized out>,
        rtld_fini=<optimized out>, stack_end=0x7fff...)
        at ../csu/libc-start.c:310
#2  0x000055555555455a in _start ()

_start → __libc_start_main → main 의 호출 체인이 그대로 보인다. 즉 우리가 보통 "프로그램의 시작점" 이라고 부르는 main 은, 사실 두 단계의 어셈블리 + glibc 코드를 거친 뒤에야 호출되는 함수다.

한 가지 master glibc 의 변화도 짚어둔다. 위 도식은 2.34 미만 glibc 의 동작이고, __libc_csu_init / __libc_csu_fini 는 정적 링크 binary 에서만 그대로 사용된다. glibc 2.34 부터 동적 링크 binary 의 ctor 처리는 변경되었다 — __libc_start_main 의 init 인자가 NULL 이면 dynamic linker 가 main map 의 DT_INIT_ARRAY 를 직접 처리하는 방식이다. 이 변경의 동기는 ROP gadget 제거였다 (glibc bugzilla BZ #23323 참조). 새 binary 와 옛 libc 사이의 호환을 위해 __libc_start_main@@GLIBC_2.34 라는 새 symbol version 도 도입되었다. 동작의 본질 — "main 호출 전에 ctor 가 실행되고 후에 dtor 가 실행된다" — 는 동일하다.

6crt0, crt1, crti, crtn — startup 파일 6 종

_start 는 어디서 왔을까? 우리가 직접 작성한 적이 없는데도 gcc hello.c 만으로 실행 파일에 포함된다. 답은 gcc 가 자동으로 startup object 파일들을 링크하기 때문이다. 이 startup 파일들은 6 종류가 있고, 각자 역할이 다르다.

링크 시 gcc 가 자동으로 끼워 넣는 startup 파일들:

  Scrt1.o  /  crt1.o    ← _start 정의 (PIE 면 Scrt1.o)
  crti.o                ← .init / .fini 섹션의 시작 prologue
  crtbegin.o            ← C++ 전역 객체 ctor 등록 시작 — gcc 제공
  ──────────────────────────
  사용자 .o 파일들
  ──────────────────────────
  crtend.o              ← 전역 ctor 등록 마무리 — gcc 제공
  crtn.o                ← .init / .fini 섹션의 끝 epilogue

gcc -v 옵션을 주면 gcc 가 실제로 호출하는 ld 명령을 볼 수 있고, 거기에 이 파일들이 모두 등장한다.

$ gcc -v hello.c 2>&1 | grep -E "collect2|crt" | head -20
 .../collect2 ...
   /usr/lib/x86_64-linux-gnu/Scrt1.o
   /usr/lib/x86_64-linux-gnu/crti.o
   .../crtbeginS.o
   /tmp/cc......o       ← 우리 hello.o
   -lgcc --as-needed -lgcc_s --no-as-needed
   -lc
   .../crtendS.o
   /usr/lib/x86_64-linux-gnu/crtn.o

각 파일의 역할을 정리하면 다음과 같다. crt1.o (또는 PIE 의 경우 Scrt1.o) 가 바로 위에서 본 _start 함수를 정의한다. crti.o 와 crtn.o 는 .init 와 .fini 섹션의 시작과 끝 부분을 차지하며, _init 과 _fini 함수의 prologue / epilogue 역할을 한다. crtbegin.o 와 crtend.o 는 C++ 의 전역 객체 ctor / dtor 등록을 처리하지만, 순수 C 프로그램에서도 __init_array / __fini_array 처리를 위해 필요하다.

이 파일들의 역할이 중요한 이유는, 이것들을 모두 빼면 _start 도 없고 libc 도 안 거치는 minimal 프로그램을 만들 수 있다는 점이다. 다음 예제는 _start 를 직접 작성하고 syscall 로 곧장 종료하는 프로그램이다.

/* tiny.c */
void _start(void) {
    /* libc 거치지 않고 직접 종료 */
    __asm__("mov $60, %rax\n"   /* sys_exit syscall 번호 */
            "xor %rdi, %rdi\n"  /* exit code 0 */
            "syscall");
}
$ gcc -nostdlib -static -o tiny tiny.c
$ ./tiny ; echo $?
0

이게 동작한다는 사실은 중요한 의미가 있다. _start 만 있으면 프로그램은 실행된다. main 도 libc 도 절대 필수가 아니다. 이 사실이 다음 절에서 다룰 "커널은 왜 main 이 없는가" 의 직접적인 답으로 이어진다.

7Linux 커널의 ELF entry — phys_startup_64

이제 커널 차례다. 놀라운 사실 하나를 먼저 짚자: Linux 커널 binary 도 그냥 ELF 파일이다. 일반 프로그램과 정확히 같은 포맷을 쓴다. 차이는 e_entry 가 가리키는 함수의 이름과, 그 함수를 호출해주는 주체뿐이다.

$ file /usr/lib/debug/boot/vmlinux-$(uname -r)
... ELF 64-bit LSB executable, x86-64, ...

$ readelf -h /usr/lib/debug/boot/vmlinux-$(uname -r) | grep -E "Type|Entry"
  Type:                              EXEC (Executable file)
  Entry point address:               0xffffffff81000000

ELF 64-bit, EXEC 타입, entry 주소가 있다. 일반 프로그램과 동일한 메커니즘이다. 이 entry 주소가 어디서 오는지 확인하려면 커널의 linker script 를 봐야 한다. arch/x86/kernel/vmlinux.lds.S 가 그 답을 직접 알려준다.

/* arch/x86/kernel/vmlinux.lds.S — master 기준 그대로 */
OUTPUT_FORMAT(CONFIG_OUTPUT_FORMAT)

#ifdef CONFIG_X86_32
OUTPUT_ARCH(i386)
ENTRY(phys_startup_32)
#else
OUTPUT_ARCH(i386:x86-64)
ENTRY(phys_startup_64)        ← ★ 64-bit kernel 의 entry symbol
#endif

ENTRY() 지시자는 GNU ld linker 에게 "이 symbol 의 주소를 ELF 헤더의 e_entry 에 써라" 라고 명령한다. userspace ELF 가 _start 를 entry 로 등록하는 것과 정확히 동일한 메커니즘이다.

phys_startup_64 의 정의는 같은 linker script 안에 있다.

SECTIONS {
    . = __START_KERNEL;
    phys_startup_64 = ABSOLUTE(startup_64 - LOAD_OFFSET);
    /* ... */
}

즉 phys_startup_64 는 결국 startup_64 의 물리 주소 (가상 주소에서 LOAD_OFFSET 을 뺀 값) 이다. 부트로더는 가상 주소를 모르고 물리 주소만 다루므로, 이 별도 symbol 을 entry 로 등록한다.

8startup_64 어셈블리 — 커널의 _start

startup_64 는 arch/x86/kernel/head_64.S 에 어셈블리로 정의되어 있다. master 기준 시작 부분은 다음과 같다 (include/linux/init.h 의 __HEAD 매크로는 .section ".head.text","ax" 로 확장된다).

    __HEAD
    .code64
SYM_CODE_START_NOALIGN(startup_64)
    UNWIND_HINT_END_OF_STACK
    /*
     * 진입 시 상태:
     * - 64bit mode (CS.L=1, CS.D=0)
     * - identity-mapped page table 이 미리 적재됨
     * - %RSI 에 boot_params 의 물리 주소
     *
     * boot_params 를 %r15 에 보존한다 — C 함수 호출이
     * %rsi 를 덮어쓰더라도 살아남도록.
     */
    mov	%rsi, %r15

    /* (이하 page table fixup, CPU 검증, common_startup_64 로 분기) */
SYM_CODE_END(startup_64)

여기서 두 가지 master 와 옛 자료의 차이를 짚어두자. 옛 글에서는 __INIT 섹션과 pushq %rsi 로 stack 에 boot_params 를 보존하는 코드를 자주 본다. master 에서는 __HEAD 섹션과 mov %rsi, %r15 로 register 에 보존한다 (master 의 주석: "Preserve it in %R15 so C function calls will not clobber it"). 의미는 같지만 master 코드와 일치하도록 표기한다.

또한 한 가지 흔한 오해를 정정하자. 많은 글이 head_64.S 에서 start_kernel 을 직접 호출하는 것처럼 설명하는데, 실제로는 그렇지 않다. startup_64 는 먼저 page table fixup 등 C 함수들을 호출하고, 결국 common_startup_64 로 분기한 뒤 initial_code 변수에 저장된 함수 포인터를 통해 indirect call 로 x86_64_start_kernel 을 호출한다. 그 이후 흐름은 다음과 같다.

startup_64                    (arch/x86/kernel/head_64.S)
   ↓ page table fixup 등 C 함수들 호출
   ↓ jmp common_startup_64
common_startup_64
   ↓ call *initial_code(%rip)   ← initial_code = x86_64_start_kernel
x86_64_start_kernel           (arch/x86/kernel/head64.c)
   ↓ call x86_64_start_reservations
x86_64_start_reservations
   ↓ call start_kernel
start_kernel                  ← ★ 여기가 우리가 찾던 "kernel main" ★

즉 ELF entry (startup_64) 와 우리가 흔히 "커널 메인" 이라 부르는 start_kernel 사이에는 실제로 여러 함수가 끼어 있다. 이 다단계 흐름은 페이지 테이블 fixup 과 가상 주소 전환을 위해 필요한 단계다.

userspace 의 _start 가 __libc_start_main 을 거쳐 main 을 호출하는 것과 비교하면, 커널의 startup_64 가 여러 단계를 거쳐 start_kernel 을 호출하는 구조는 본질적으로 같은 패턴이다. 단지 그 사이에 끼는 함수의 이름과 역할이 다를 뿐이다.

9두 세계의 entry 비교 — 동일한 메커니즘

이제 userspace 와 kernel 을 한 화면에 놓고 비교하면, 두 세계가 사실 같은 ELF 메커니즘 위에서 동작한다는 것이 명확히 보인다.

┌──────────────────────────┬────────────────────────────┐
│  userspace 프로그램        │  Linux kernel              │
├──────────────────────────┼────────────────────────────┤
│ ELF 포맷                   │ ELF 포맷 (동일)             │
│                          │                            │
│ e_entry = _start         │ e_entry = phys_startup_64  │
│   (linker 가             │   (linker 가               │
│    sysdeps/.../start.S    │    vmlinux.lds.S 의        │
│    의 _start 주소 기록)    │    ENTRY() 로 기록)        │
│                          │                            │
│ entry 함수 호출자:         │ entry 함수 호출자:          │
│   kernel (binfmt_elf)    │   bootloader (GRUB)        │
│                          │                            │
│ 호출 형식:                 │ 호출 형식:                  │
│   regs->ip = e_entry,    │   jmp e_entry              │
│   iretq                  │                            │
│                          │                            │
│ entry 다음:               │ entry 다음:                 │
│   _start                 │   startup_64               │
│   → __libc_start_main    │   → __startup_64           │
│   → main()               │   → common_startup_64      │
│                          │   → x86_64_start_kernel    │
│                          │   → x86_64_start_          │
│                          │     reservations           │
│                          │   → start_kernel()         │
│                          │                            │
│ 어떤 runtime 사용?         │ 어떤 runtime 사용?          │
│   glibc + ld-linux.so    │   없음 (freestanding)       │
└──────────────────────────┴────────────────────────────┘

두 entry 함수의 첫 어셈블리 명령을 비교해보면 얼마나 비슷한지 더 명확히 보인다.

# userspace
$ objdump -d /bin/ls | grep -A 3 "<_start>:"
00000000000067f0 <_start>:
    67f0: f3 0f 1e fa           endbr64
    67f4: 31 ed                 xor    %ebp,%ebp        ← frame pointer 클리어

# kernel
$ objdump -d vmlinux | grep -A 3 "<startup_64>:"
ffffffff81000000 <startup_64>:
ffffffff81000000: fc                cld
ffffffff81000001: fa                cli
ffffffff81000002: 31 ed             xor    %ebp,%ebp    ← frame pointer 클리어

둘 다 frame pointer 를 클리어 (xor %ebp, %ebp) 하는 명령으로 시작한다. 이건 System V AMD64 ABI 가 권장하는 outermost frame 표시이며, 두 환경 모두 이 ABI 를 따르기 때문에 동일한 패턴이 나타난다.

핵심 통찰은 다음과 같다.

"커널도 ELF 다."
"e_entry 가 가리키는 이름이 main 이 아닐 뿐이고,"
"그 entry 를 호출해주는 runtime (libc) 이 없을 뿐이다."

따라서:
  - ELF / e_entry 메커니즘 = 같음
  - 호출 주체 (kernel vs bootloader) = 다름
  - runtime 유무 = 다름
  - 함수 이름 (_start vs startup_64) = 다름

main() 이라는 이름은 그저 C 표준 + libc 약속이고,
kernel 은 그 약속을 안 따르는 freestanding 환경.

10kernel 이 userspace _start 로 점프하는 메커니즘

지금까지 본 그림에 한 가지 빠진 조각이 있다. 커널은 어떻게 사용자 프로그램의 _start 주소를 알고 거기로 점프하는가? 이 질문의 답은 ELF 의 e_entry 와 커널의 execve 시스템 콜이 어떻게 협업하는지를 보여준다.

전체 흐름은 다음과 같다.

shell 이 ./hello 입력
   │
   ▼
fork() + execve("./hello", argv, envp)
   │
   ▼ (syscall)
do_execveat_common (fs/exec.c)
   │
   ├─ ELF 헤더 파싱
   ├─ binfmt 매칭 → elf_format
   │
   ▼
load_elf_binary (fs/binfmt_elf.c)
   │
   ├─ PT_LOAD segments mmap (.text, .data, .bss)
   ├─ PT_INTERP 있으면 ld-linux.so.2 도 mmap
   ├─ stack 에 argc/argv/envp/auxv 배치
   │     (사용자 stack — execve 이후 process 가 볼 stack)
   │
   ├─ elf_entry = elf_ex->e_entry
   │              (또는 dynamic 이면 ld-linux 의 entry)
   │
   ├─ start_thread(regs, elf_entry, sp)        ← ★ 핵심
   │     │
   │     └─ regs->ip = elf_entry  (= _start 주소)
   │        regs->sp = sp         (= argc 가 있는 stack top)
   │        segment registers reset
   │
   └─ return 0
   │
   ▼ (syscall return path)
iretq                            ← regs->ip 로 점프
   │
   ▼
_start  (사용자 stack 위에서 실행 시작)

핵심은 start_thread 라는 함수다. arch/x86/kernel/process_64.c 에 정의되어 있고, 단순한 register 설정 함수다.

static void
start_thread_common(struct pt_regs *regs, unsigned long new_ip,
                    unsigned long new_sp,
                    u16 _cs, u16 _ss, u16 _ds)
{
    WARN_ON_ONCE(regs != current_pt_regs());
    /* ... segment register 들 reset ... */

    regs->ip    = new_ip;     /* ← _start 주소 */
    regs->sp    = new_sp;     /* ← stack top */
    regs->cs    = _cs;
    regs->ss    = _ss;
    regs->flags = X86_EFLAGS_IF;
}

이 함수가 하는 일은 syscall 이 return 할 때 사용할 register 값들을 미리 세팅하는 것이다. execve 도 syscall 이므로 끝나면 iretq 로 userspace 로 돌아가는데, 그 때 사용되는 pt_regs 의 ip 가 이미 _start 의 주소로 바뀌어 있다. 그래서 execve 가 return 하는 곳이 호출 직전 위치가 아니라 새 프로그램의 _start 가 된다.

이게 "execve 가 성공하면 return 하지 않는다" 의 정확한 의미다. 실제로는 return 하지만, 다른 process 의 다른 코드로 return 한다. 메모리 매핑도 모두 새것으로 바뀌어 있어서 호출 직전의 코드는 더 이상 존재하지 않는다.

11C 표준의 hosted vs freestanding — 왜 커널은 main 이 필요 없는가

지금까지 본 내용을 한 단계 더 일반화하면 C 언어 표준 차원의 답에 도달한다. C11 표준 (ISO/IEC 9899:2011) 의 §5.1.2 는 "Two execution environments" 라는 제목 아래 두 종류의 실행 환경을 정의한다.

§5.1.2.1 의 freestanding environment 정의는 다음과 같다.

"In a freestanding environment (in which C program execution may take
 place without any benefit of an operating system), the name and type
 of the function called at program startup are implementation-defined.
 Any library facilities available to a freestanding program, other
 than the minimal set required by clause 4, are implementation-defined."

요약하면 OS 없이도 동작 가능한 환경이며, 시작 함수의 이름과 타입은 구현이 결정한다. 즉 main 이라는 이름이 강제되지 않는다. 사용 가능한 표준 라이브러리도 최소 집합 (<float.h>, <iso646.h>, <limits.h>, <stdarg.h>, <stdbool.h>, <stddef.h>, <stdint.h> — 모두 매크로/타입 정의 위주) 만 보장된다.

§5.1.2.2 의 hosted environment 정의는 다음과 같다.

"The function called at program startup is named main. ...
 It shall be defined with a return type of int and with no parameters
 ... or with two parameters ... or equivalent ..."

이쪽은 시작 함수가 반드시 main 이라는 이름이어야 하고, 반드시 int return 이며, 시그니처는 int main(void) 또는 int main(int argc, char *argv[]) 두 가지만 허용된다. 표준 라이브러리 전체 사용 가능.

이 두 정의를 분류표로 정리하면 다음과 같다.

┌───────────────────────────┬──────────────────────────────────┐
│ hosted implementation     │ freestanding implementation      │
├───────────────────────────┼──────────────────────────────────┤
│ 일반 사용자 프로그램         │ OS kernel                         │
│ (Linux 위 ./a.out)        │ (Linux kernel 그 자체)             │
│                           │                                  │
│ shell                     │ bootloader (GRUB, U-Boot)         │
│ daemon (sshd, nginx)      │ embedded firmware                 │
│ DB (postgres, mysql)      │ baremetal program                 │
│ language runtime (CPython)│ device driver (어떤 의미에선)       │
└───────────────────────────┴──────────────────────────────────┘

여기서 결정적인 답이 나온다. Linux 커널은 freestanding implementation 이므로, "시작 함수의 이름이 main 이어야 한다" 는 hosted 규칙이 적용되지 않는다. 그냥 start_kernel 이라는 이름을 썼을 뿐이고, 이는 C 표준상 완벽히 합법이다.

12커널 빌드 옵션 — -ffreestanding 의 효과

C 표준이 freestanding 환경을 허용한다는 것과, 컴파일러가 그 환경으로 동작한다는 것은 별개의 문제다. gcc 는 기본적으로 hosted 환경을 가정하고 동작하므로, 커널을 빌드하려면 명시적으로 freestanding 으로 전환해야 한다. 이를 위한 옵션들이 커널 Makefile 에 들어 있다.

# Linux 커널 빌드 옵션 (master 기준 일부)

-ffreestanding              ← freestanding 환경 명시
-fno-builtin                ← 표준 라이브러리 함수 가정 안 함 (printf 최적화 막음)
-fno-strict-aliasing
-nostdinc                   ← 표준 헤더 (/usr/include) 안 씀, 커널 자체 헤더만
-mno-sse -mno-mmx           ← SSE/MMX 비활성 (커널은 보통 FPU/SIMD 안 씀)
-mno-sse2 -mno-3dnow
-fno-asynchronous-unwind-tables
-fno-jump-tables
-mcmodel=kernel             ← 커널은 가상주소 상위 2GB 영역 사용
-fno-pie -no-pie            ← position-independent 비활성 (커널 자체는 고정 주소)
-fcf-protection=none

링킹 옵션은 더 단호하다.

-nostdlib                   ← libc, crt0/crt1 등 startup 파일 안 씀
-nostartfiles
-z noexecstack
-T arch/x86/kernel/vmlinux.lds.S    ← 자체 linker script 사용

이 옵션들의 효과를 일반 빌드와 비교하면 차이가 한눈에 보인다.

[일반 빌드]                          [kernel 빌드]
ㆍ crt1.o (= _start) 자동 링크        ㆍ 없음 — 자체 head_64.S
ㆍ crti.o / crtn.o 자동 링크          ㆍ 없음
ㆍ libc 링크                          ㆍ 없음 — 자체 lib/ 함수
ㆍ ld-linux.so.2 의존                 ㆍ 없음 — static linking
ㆍ ELF interpreter (PT_INTERP)       ㆍ 없음
ㆍ malloc / printf / 모든 표준 함수    ㆍ 없음 — kmalloc / printk 등

즉 커널은 자기 자신이 runtime 이므로 외부 runtime 을 안 쓴다. 자급자족 환경이다. 이것이 "커널은 freestanding 이다" 라는 말의 실질적 의미다.

13start_kernel 도 사실 return 하지 않는다

userspace 의 main 은 return result 가 가능하다. main 이 return 하면 __libc_start_main 이 그 값을 받아서 exit(result) 를 호출하고, 거기서 atexit 핸들러들이 실행되고 결국 exit_group syscall 로 process 가 종료된다.

그런데 start_kernel 은 다르다. 시그니처에 __noreturn 어트리뷰트가 붙어 있다.

/* init/main.c */
asmlinkage __visible __init __no_sanitize_address __noreturn __no_stack_protector
void start_kernel(void)
{
    /* ... 초기화 ... */
    rest_init();
    /* 여기는 reach 안 함 */
}

__noreturn 은 컴파일러에게 "이 함수는 절대 return 하지 않는다" 를 알리는 어트리뷰트다. 컴파일러는 이 정보로 caller 의 stack frame 에 return 영역을 잡지 않는 등의 최적화를 할 수 있다.

왜 return 하지 않는가? rest_init 안에서 실행 흐름이 분기되기 때문이다.

static noinline void __ref __noreturn rest_init(void)
{
    /* PID 1 생성 */
    pid = user_mode_thread(kernel_init, NULL, CLONE_FS);
    /* PID 2 생성 */
    pid = kernel_thread(kthreadd, NULL, NULL, CLONE_FS | CLONE_FILES);
    /* ... */

    /* boot CPU 의 task → idle task 로 전환 */
    cpu_startup_entry(CPUHP_ONLINE);
    /* cpu_startup_entry 도 __noreturn — 영원히 do_idle 루프 */
}

cpu_startup_entry 가 do_idle 무한 루프로 들어가서 영원히 안 돌아온다. 즉 start_kernel 의 return 시점은 영원히 도래하지 않는다. caller 가 stack frame 을 정리하지 않아도 된다.

이게 "커널은 종료 개념이 없다" 의 정확한 구현이다. 사용자 프로그램은 main 이 return 하면 끝나지만, 커널의 start_kernel 은 끝나는 시점 자체가 정의되지 않는다 — 시스템이 켜져 있는 한.

14Stage 1 정리

Stage 1 의 핵심을 한 번 더 정리하자. "Linux 의 init/main.c 안에 왜 main() 이 없는가?" 라는 의문은 다음과 같이 분해된다.

첫째, userspace 의 main() 도 사실 CPU 가 직접 호출하지 않는다. 진짜 entry 는 _start 이고, 그게 __libc_start_main 을 거쳐 main 을 호출한다. ELF 헤더의 e_entry 필드가 _start 의 주소를 담고 있으며, 커널의 execve 가 start_thread 를 통해 regs->ip 를 그 주소로 세팅한 뒤 iretq 로 점프한다.

둘째, 커널도 ELF 파일이고 똑같은 메커니즘을 쓴다. 다만 e_entry 가 _start 가 아니라 phys_startup_64 를 가리킨다. 그리고 그 entry 를 호출해주는 주체가 OS 가 아니라 부트로더 (GRUB) 이다. 사이에 libc 같은 runtime 도 없다.

셋째, C11 표준이 freestanding 환경에서는 시작 함수 이름을 자유롭게 정할 수 있도록 허용한다. 그래서 커널은 start_kernel 이라는 이름을 쓴다. -ffreestanding 컴파일 옵션이 이 환경을 명시한다.

넷째, start_kernel 도 main 처럼 return 하지 않는다. __noreturn 어트리뷰트가 붙어 있고, 내부의 rest_init 가 boot CPU 를 idle 무한 루프로 전환시키기 때문에 영원히 끝나지 않는다.

이 네 가지가 머릿속에 자리잡으면 더 이상 "main.c 인데 왜 main 이 없지?" 가 헷갈리지 않는다. 다음 Stage 에서는 start_kernel 이 구체적으로 어떤 일을 하는지 — 어떻게 PID 0 이 만들어지고, PID 1 과 PID 2 가 fork 되는지 — 를 따라간다.

Stage 2Linux 커널 부팅 흐름 큰 그림

Stage 1 에서 startup_64 가 ELF entry 이고, 거기서 여러 함수를 거쳐 start_kernel 에 도달한다는 것까지 봤다. Stage 2 의 목표는 이 큰 그림을 시간 순서로 정리하는 것이다. CPU 가 reset 된 직후부터 start_kernel 까지, 그리고 start_kernel 안에서 rest_init 까지의 흐름을 따라가면, "PID 0 이 어디서 등장하고, PID 1 과 PID 2 가 어디서 만들어지는가" 가 자연스럽게 드러난다.

17start_kernel() 안에서 일어나는 일

start_kernel 은 init/main.c 에 정의된 함수이며, 그 안에서 커널의 모든 서브시스템이 순서대로 초기화된다. 메모리 관리, 스케줄러, 인터럽트, VFS (가상 파일시스템), 타이머, 콘솔 등이 차례로 켜진다. 호출 순서는 의외로 엄격하게 정해져 있는데, 이는 한 서브시스템이 다른 서브시스템에 의존하기 때문이다. 예를 들어 printk (커널 메시지 출력) 가 동작하려면 콘솔이 초기화되어 있어야 하고, 콘솔 초기화는 또 메모리 관리에 의존한다.

master 기준 실제 호출 순서는 다음과 같다. 함수 시그니처도 master 그대로 인용한다.

asmlinkage __visible void __init start_kernel(void)
{
    char *command_line;
    char *after_dashes;

    set_task_stack_end_magic(&init_task);    /* ★ 첫 호출 — 의미는 본문 참조 */
    smp_setup_processor_id();
    debug_objects_early_init();
    /* ... */
    cgroup_init_early();
    local_irq_disable();
    early_boot_irqs_disabled = true;
    boot_cpu_init();
    page_address_init();
    pr_notice("%s", linux_banner);
    setup_arch(&command_line);
    setup_command_line(command_line);
    /* ... 수십 개 함수 호출 ... */
    rest_init();                              /* ★ 마지막 — 첫 task 들을 만들고 idle 로 */
}

이 호출 목록의 마지막에 rest_init() 이 있다. 이름이 "나머지 초기화" 인 이유는, 이 함수가 단순한 초기화가 아니라 첫 두 개의 process (PID 1, PID 2) 를 fork 하고, boot CPU 를 idle task 로 전환하기 때문이다. 즉 진짜 system 의 시작은 rest_init 안에서 일어난다.

여기서 한 가지 중요한 사실. 이 모든 초기화가 어떤 task 의 컨텍스트에서 실행되는가? 답은 PID 0 — 즉 swapper / idle task 의 컨텍스트다. 더 정확히는 init/init_task.c 에 정적으로 선언된 init_task 라는 task_struct 의 컨텍스트에서 실행된다. 부팅 시점에는 아직 다른 task 가 만들어지지 않았으므로, 커널 자신이 이 정적 task 의 컨텍스트로 동작하면서 다른 task 들을 만들어내는 셈이다.

이는 위 master 코드의 첫 줄 set_task_stack_end_magic(&init_task); 에서 명시적으로 드러난다. 이 함수는 인자로 받은 task 의 stack 끝에 magic 값 (STACK_END_MAGIC) 을 박아 stack overflow 검출용 sentinel 로 만든다. 부팅 직후 가장 먼저 손대는 task 가 init_task 라는 사실 자체가, 현재 실행 컨텍스트가 PID 0 임을 코드 차원에서 증명한다. 외부에서 부팅 흐름을 추측하지 않아도, 첫 줄 한 줄로 "지금 누가 실행 중인지" 가 드러나는 것이다.

CPU boot
   |
start_kernel()
   |
현재 task == PID 0  (= init_task, statically declared)
   |
PID 0 컨텍스트에서 커널 초기화 수행
   |
rest_init() 에서 PID 1, PID 2 fork, boot CPU 는 idle 로 전환

이 사실은 PID 0 의 정체를 이해하는 핵심이며, Stage 3 에서 자세히 다룬다.

18rest_init() 의 3 단계

rest_init 는 매우 짧지만 매우 중요한 함수다. 이 안에서 PID 1, PID 2 가 차례로 만들어지고, boot CPU 자신은 idle 로 전환된다. master init/main.c 의 본체를 그대로 인용하고 (주석은 한국어로 풀어서) 단계를 따라가자.

static __initdata DECLARE_COMPLETION(kthreadd_done);

static noinline void __ref __noreturn rest_init(void)
{
    struct task_struct *tsk;
    int pid;

    rcu_scheduler_starting();

    /* === Step 1 : PID 1 생성 === */
    /* user_mode_thread() 는 "이 thread 는 결국 userspace 로 가게 될
       thread" 라는 의도를 이름에 담은 5.10 이후의 명칭이다.  */
    pid = user_mode_thread(kernel_init, NULL, CLONE_FS);

    /* PID 1 task 를 가져와서 boot CPU 에 못 박는다.
       sched_init_smp() 가 끝나기 전에는 task migration 이
       제대로 동작하지 않기 때문이다. */
    rcu_read_lock();
    tsk = find_task_by_pid_ns(pid, &init_pid_ns);
    tsk->flags |= PF_NO_SETAFFINITY;
    set_cpus_allowed_ptr(tsk, cpumask_of(smp_processor_id()));
    rcu_read_unlock();

    numa_default_policy();

    /* === Step 2 : PID 2 (kthreadd) 생성 === */
    pid = kernel_thread(kthreadd, NULL, NULL, CLONE_FS | CLONE_FILES);
    rcu_read_lock();
    kthreadd_task = find_task_by_pid_ns(pid, &init_pid_ns);
    rcu_read_unlock();

    /* might_sleep() 검사를 이제 켠다.  PID 1 은 이미 만들어졌지만
       kthreadd_done completion 을 기다리느라 멈춰 있는 상태이므로
       지금 켜도 안전하다. */
    system_state = SYSTEM_SCHEDULING;
    complete(&kthreadd_done);          /* ← PID 1 을 풀어준다 */

    /* === Step 3 : boot CPU 가 idle 로 전환 === */
    /* 처음 한 번은 명시적으로 schedule 해서 흐름을 시동한다. */
    schedule_preempt_disabled();
    /* 그리고 idle 진입 — 이 함수는 절대 return 하지 않는다. */
    cpu_startup_entry(CPUHP_ONLINE);
}

이 함수의 흐름을 단계별로 풀어보자.

Step 1 — PID 1 생성과 boot CPU pinning. user_mode_thread(kernel_init, ...) 가 PID 1 을 만든다. 처음에는 kernel_init 이라는 커널 함수를 실행하지만, 그 함수 안에서 결국 execve("/sbin/init") 를 호출해서 userspace 의 /sbin/init (= systemd) 으로 변신한다. 즉 PID 1 은 처음에는 커널 thread 였다가 도중에 userspace process 로 바뀐다. 이 변신의 메커니즘은 Stage 3 에서 자세히 본다.

만들어진 PID 1 은 곧바로 find_task_by_pid_ns 로 task_struct 를 가져와서 PF_NO_SETAFFINITY flag 를 켜고 set_cpus_allowed_ptr 로 boot CPU 에 못 박는다. 이는 SMP 스케줄러가 아직 완전히 켜지지 않은 단계라 task migration 이 동작하지 않기 때문이다. 나중에 sched_init_smp 가 호출되면 isolated CPU 가 아닌 모든 CPU 로 풀려난다.

Step 2 — PID 2 (kthreadd) 생성과 동기화 약속. kernel_thread(kthreadd, ...) 가 PID 2 를 만든다. 만들어진 task 는 전역 변수 kthreadd_task 에 저장된다. 이후 다른 코드가 kthread_create 같은 API 를 호출하면 그 요청이 kthreadd_task 에게 전달되고, kthreadd 가 실제 task 생성을 대신해 준다.

여기서 kthreadd_done 이라는 completion 변수가 핵심 역할을 한다. PID 1 (kernel_init) 은 kthread API 에 의존하므로 PID 2 (kthreadd) 가 ready 된 다음에야 자기 일을 시작할 수 있다. 그래서 master 코드는 다음과 같이 약속을 만든다.

  • rest_init 시작 직전 static DECLARE_COMPLETION(kthreadd_done); 으로 신호 변수 선언
  • PID 1 의 진입 함수인 kernel_init 의 첫 줄에서 wait_for_completion(&kthreadd_done); 으로 대기 (master init/main.c)
  • rest_init 이 PID 2 를 만든 직후 complete(&kthreadd_done); 으로 신호 — 그제야 PID 1 이 풀려난다

이 명시적 동기화 덕분에 PID 1 은 자기가 의존하는 kthreadd 가 살아 있다는 것을 보장받은 상태에서 작업을 시작한다.

Step 3 — boot CPU 가 idle task 가 된다. 가장 흥미로운 부분이다. 부팅 동안 커널 초기화 코드를 실행해온 task 는 PID 0 이었는데, 모든 일을 마치고 PID 1 과 PID 2 를 만든 뒤에는 자기 자신이 영원히 idle 루프를 도는 것이다. master 는 먼저 schedule_preempt_disabled() 로 한 번 명시적으로 schedule 을 호출해 흐름을 시동한 뒤 cpu_startup_entry(CPUHP_ONLINE) 로 idle 에 진입한다. 이 함수는 __noreturn 이며 안에서 while (1) do_idle(); 무한 루프다.

다른 runnable task 가 있으면 스케줄러가 그쪽으로 switch 하지만, 다른 task 가 없으면 다시 idle 로 돌아온다. do_idle 의 실제 동작은 §22 에서 자세히 본다.

kernel_thread vs user_mode_thread 의 차이. Linux 5.10 이전에는 PID 1 도 kernel_thread(kernel_init, ...) 로 만들었지만, 이후 commit 343f4c49f243 에서 user_mode_thread 로 교체되었다. 시그니처는 같지만 의미가 더 명확해졌다 — "이 thread 는 결국 user mode 로 갈 thread" 라는 의도가 함수 이름에 드러난다.

do_idle() idle 루프 kernel/sched/idle.c
arch_cpu_idle() 아키텍처별 idle (mwait/hlt) arch/x86/kernel/process.c
free_initmem() __init 메모리 회수 init/main.c
fork_idle(cpu) per-CPU idle task 생성 kernel/fork.c
setup_arch() 아키텍처별 초기화 arch/x86/kernel/setup.c
mm_init() 메모리 관리 초기화 init/main.c
sched_init() 스케줄러 초기화 kernel/sched/core.c

20Stage 2 정리

Stage 2 까지 오면 큰 그림이 거의 완성된다. CPU reset 부터 시작해서 부트로더, 어셈블리 entry, C 코드 진입, 그리고 start_kernel 안의 긴 초기화 끝에 rest_init 가 PID 1 과 PID 2 를 만들고 자신은 idle 로 전환된다는 흐름이다.

다음 Stage 에서는 이 세 PID 의 정체를 더 자세히 들여다본다. PID 0 은 정확히 무엇이고 왜 ps 명령에 안 보이는지, PID 1 (systemd) 은 어떻게 kernel 함수에서 userspace process 로 변신하는지, PID 2 (kthreadd) 가 만들어내는 모든 [kworker], [ksoftirqd] 같은 thread 들의 정체는 무엇인지를 풀어간다.

Stage 3PID 0 / 1 / 2 의 정체와 생성 과정

Stage 2 에서 rest_init 가 PID 1 과 PID 2 를 만들고 boot CPU 가 idle 로 전환된다는 것까지 봤다. 이 세 PID 는 Linux 시스템의 가장 기초가 되는 task 들이며, 각자 매우 특별한 역할을 한다. Stage 3 는 이 세 PID 를 하나씩 자세히 들여다보고, 각각이 어떻게 만들어지고 무엇을 하는지를 정확히 정리한다.

21PID 0 — swapper / idle task

PID 0 은 흔히 swapper 또는 idle task 라고 부른다. 일반적인 의미의 process 가 아니라 매우 특수한 커널 내부 entity 다. PID 0 의 가장 두드러진 특징은 다음과 같다.

항목 설명
이름 swapper / idle
역할 CPU idle
kernel thread 인가 맞음
userspace? 아님
종료 가능? 불가능

가장 중요한 사실은 PID 0 은 동적으로 fork 된 게 아니라 정적으로 생성된다는 점이다. init/init_task.c 파일에 init_task 라는 task_struct 가 정적으로 선언되어 있고, 부팅 시 이 정적 인스턴스가 그대로 PID 0 으로 사용된다.

init_task (정적 선언, init/init_task.c)
 ├─ pid = 0
 ├─ comm = "swapper/0"
 ├─ sched_class = idle_sched_class
 └─ context: boot CPU's idle (do_idle loop)

여기서 한 가지 흥미로운 사실이 있다. SMP (멀티 CPU) 시스템에서는 CPU 마다 별도의 idle task 가 필요하다. boot CPU 의 idle task 는 위에서 본 init_task 정적 인스턴스이지만, 다른 CPU 들은 SMP boot 단계에서 fork_idle(cpu) 함수로 각자의 idle task 를 동적으로 생성한다. 이 모든 task 들의 PID 는 0 이다. 즉 PID 0 은 task_struct 인스턴스 차원에서는 CPU 수만큼 존재하지만, PID 번호 차원에서는 모두 동일하게 0 으로 표시된다 — Linux 의 특수 처리다.

이들을 구분하는 방법은 comm 필드다. boot CPU 의 idle 은 swapper/0, 두 번째 CPU 는 swapper/1, 세 번째는 swapper/2, ... 식으로 N 번 CPU 의 idle task 는 swapper/N 이라는 comm 을 갖는다. TASK_COMM_LEN = 16 제약이 있어서 이름은 짧게 표현된다.

fork_idle(cpu)  (kernel/fork.c, SMP 다른 CPU 시작 시)
 └─ creates task_struct for each CPU
     ├─ pid = 0  (special, all share)
     ├─ comm = "swapper/N"  (TASK_COMM_LEN = 16 제한)
     └─ context: CPU N's idle

22idle task 가 실제로 하는 일

idle task 는 단순히 무한 루프를 도는 것이 아니다. 단순 spin 이라면 CPU 가 100% 사용률을 보일 것이고 전력도 엄청나게 소비할 것이다. 실제로는 kernel/sched/idle.c 의 do_idle() 루프 안에서 아키텍처별 idle 명령을 호출해 CPU 를 절전 상태로 진입시킨다. x86 의 경우 mwait 또는 hlt 명령을 사용한다.

cpu_startup_entry 의 master 본체는 단순하다.

void cpu_startup_entry(enum cpuhp_state state)
{
    current->flags |= PF_IDLE;       /* 이제부터 나는 idle task */
    arch_cpu_idle_prepare();
    cpuhp_online_idle(state);
    while (1)
        do_idle();
}

do_idle() 의 master 본체는 더 길지만 핵심을 간추리면 다음과 같다. 옛 글에서 흔히 인용되는 rcu_idle_enter/rcu_idle_exit 는 더 이상 master 의 코드 흐름이 아니다. master 는 tick_nohz_idle_enter/tick_nohz_idle_exit 를 사용하고, RCU 동기화는 arch_cpu_idle_enter/arch_cpu_idle_exit 와 cpuidle_idle_call 안의 ct_cpuidle_enter/ct_cpuidle_exit 에서 처리한다.

static void do_idle(void)
{
    int cpu = smp_processor_id();
    bool got_tick = false;

    nohz_run_idle_balance(cpu);
    __current_set_polling();
    tick_nohz_idle_enter();           /* tick 정지 (NO_HZ) */

    while (!need_resched()) {
        local_irq_disable();
        if (cpu_is_offline(cpu)) {
            cpuhp_report_idle_dead();
            arch_cpu_idle_dead();
        }
        arch_cpu_idle_enter();
        rcu_nocb_flush_deferred_wakeup();

        if (cpu_idle_force_poll || tick_check_broadcast_expired()) {
            tick_nohz_idle_restart_tick();
            cpu_idle_poll();
        } else {
            cpuidle_idle_call(got_tick);  /* x86: mwait / hlt */
        }
        got_tick = tick_nohz_idle_got_tick();
        arch_cpu_idle_exit();
    }

    preempt_set_need_resched();
    tick_nohz_idle_exit();
    __current_clr_polling();
    smp_mb__after_atomic();
    flush_smp_call_function_queue();
    schedule_idle();                  /* runnable 있으면 context switch */
    /* (livepatch 처리 생략) */
}

이 흐름의 요점은 다음과 같다. CPU 가 할 일이 없으면 idle task 가 선택되고, idle task 는 cpuidle_idle_call 을 통해 CPU 를 잠재운다 (x86 에서는 결국 mwait 또는 hlt). CPU 는 인터럽트가 들어올 때까지 거의 전력을 소비하지 않는다. 인터럽트가 들어오면 깨어나서 schedule_idle 을 호출하는데, 그 안에서 runnable 한 다른 task 가 있는지 확인한다. 있으면 그쪽으로 context switch 하고, 없으면 다시 루프의 시작으로 돌아가 또 잠든다.

스케줄러 관점에서 idle task 는 가장 낮은 우선순위 클래스 (DEFINE_SCHED_CLASS(idle)) 에 속한다. stop_sched_class > deadline > rt > fair (CFS) > idle 의 순서이고, 다른 모든 클래스에 runnable task 가 없을 때만 idle 이 선택된다.

여기서 master 의 흥미로운 디테일 하나. idle task 는 sleep 하면 안 된다는 규칙이 코드에 강제되어 있다. kernel/sched/idle.c 의 idle scheduling class 정의 끝부분에 다음 함수가 있다.

static bool dequeue_task_idle(struct rq *rq, struct task_struct *p, int flags)
{
    raw_spin_rq_unlock_irq(rq);
    printk(KERN_ERR "bad: scheduling from the idle thread!\n");
    dump_stack();
    raw_spin_rq_lock_irq(rq);
    return true;
}

이는 idle scheduling class 의 .dequeue_task 콜백으로 등록되어 있다. 만약 누군가 idle task 를 runqueue 에서 dequeue 하려 하면 (= idle task 가 sleep 또는 block 하려 하면), 이 함수가 호출되어 KERN_ERR 메시지와 함께 stack dump 를 남긴다. idle 은 무조건 깨어 있어야 하고 sleep 해서는 안 된다는 약속이 코드 레벨에서 enforce 되는 것이다. 만약 누군가 실수로 idle 컨텍스트에서 sleep 가능한 함수를 호출하면 이 경고가 dmesg 에 찍히게 된다.

23PID 1 — kernel_init 에서 systemd 로의 변신

PID 1 의 생성은 두 단계로 이루어진다. 첫 단계는 커널 thread 로서의 탄생, 두 번째 단계는 userspace process 로의 변신이다.

PID 0 (swapper)
   |
   +--> kernel_thread() / user_mode_thread()
              |
              +--> kernel_init()
                        |
                        +--> execve("/sbin/init")

rest_init 안에서 다음 호출이 PID 1 을 만들어낸다.

pid = user_mode_thread(kernel_init, NULL, CLONE_FS);

이 호출은 새 task 를 만들고, 그 task 가 처음 실행할 함수로 kernel_init 을 지정한다. 새로 만들어진 task 는 PID 1 을 받고 kernel_init 함수를 실행하기 시작한다. 이 시점에는 아직 커널 모드이고, 커널 함수만 실행하고 있다.

kernel_init 함수는 init/main.c 에 정의되어 있고, master 의 본체는 다음과 같다 (주요 흐름만 발췌, 주석은 master 그대로).

static int __ref kernel_init(void *unused)
{
    int ret;

    /*
     * Wait until kthreadd is all set-up.
     */
    wait_for_completion(&kthreadd_done);

    kernel_init_freeable();
    /* need to finish all async __init code before freeing the memory */
    async_synchronize_full();

    system_state = SYSTEM_FREEING_INITMEM;
    kprobe_free_init_mem();
    ftrace_free_init_mem();
    kgdb_free_init_mem();
    exit_boot_config();
    free_initmem();           /* __init 메모리 회수 */
    mark_readonly();

    /*
     * Kernel mappings are now finalized - update the userspace page-table
     * to finalize PTI.
     */
    pti_finalize();

    system_state = SYSTEM_RUNNING;
    numa_default_policy();

    rcu_end_inkernel_boot();
    do_sysctl_args();

    /* 1순위: rdinit= 으로 지정된 ramdisk init */
    if (ramdisk_execute_command) {
        ret = run_init_process(ramdisk_execute_command);
        if (!ret) return 0;
        pr_err("Failed to execute %s (error %d)\n",
               ramdisk_execute_command, ret);
    }

    /* 2순위: init= 으로 지정된 init */
    if (execute_command) {
        ret = run_init_process(execute_command);
        if (!ret) return 0;
        panic("Requested init %s failed (error %d).",
              execute_command, ret);
    }

    /* 3순위: 기본 init 경로 */
    if (CONFIG_DEFAULT_INIT[0] != '\0') {
        ret = run_init_process(CONFIG_DEFAULT_INIT);
        if (ret)
            pr_err("Default init %s failed (error %d)\n",
                   CONFIG_DEFAULT_INIT, ret);
        else
            return 0;
    }

    if (!try_to_run_init_process("/sbin/init") ||
        !try_to_run_init_process("/etc/init")  ||
        !try_to_run_init_process("/bin/init")  ||
        !try_to_run_init_process("/bin/sh"))
        return 0;

    panic("No working init found.  Try passing init= option to kernel. "
          "See Linux Documentation/admin-guide/init.rst for guidance.");
}

execve 가 성공하면 이 task 는 더 이상 kernel_init 함수가 아니라 /sbin/init (= systemd 의 본 binary) 를 실행하게 된다. task_struct 자체는 같지만, 실행하는 코드와 메모리 매핑이 통째로 바뀌는 것이다. 그래서 PID 1 의 task 는 처음에는 kernel_init 이라는 커널 함수였지만 어느 순간 systemd 라는 userspace process 로 변신한다.

run_init_process 의 우선순위는 kernel command line 의 init= 또는 rdinit= 으로 override 할 수 있다.

kernel_init() → run_init_process() 호출 순서:

1. ramdisk_execute_command           ← rdinit= (kernel cmdline)
   기본값: "/init"                     ← initramfs 의 /init
   |
   |  실행 실패 또는 부재 시 ↓
   v
2. execute_command                    ← init= (kernel cmdline)
   예: init=/bin/bash (rescue 모드)
   |  실행 실패 → panic (override 했으므로)
   v
3. CONFIG_DEFAULT_INIT                ← .config 의 default
   |
   |  실행 실패 또는 부재 시 ↓
   v
4. /sbin/init                         ← Ubuntu 기본 (systemd symlink)
5. /etc/init
6. /bin/init
7. /bin/sh
   |
   v
   모두 실패 → panic("No working init found.")

그래서 init=/bin/bash 로 부팅하면 systemd 대신 bash 가 PID 1 으로 실행되며, 이는 시스템 복구 (rescue 모드) 에서 자주 쓰는 기법이다. 단 master 의 동작에는 미묘한 차이가 있다 — init= 으로 override 한 경우 (= execute_command 분기) 실행 실패 시 fallback 없이 곧장 panic("Requested init ... failed") 한다. "내가 지정한 init 이 안 떴으니 더 이상 추측하지 말라" 는 의도다.

24PID 2 — kthreadd 와 커널 thread 가족

PID 2 는 kthreadd 라는 커널 thread 매니저다. 이름의 d 는 daemon 의 d 다. 이 task 는 영원히 커널 모드로 남으며, 다른 커널 thread 를 on-demand 로 만들어내는 역할을 한다.

PID 1 과 마찬가지로 rest_init 안에서 만들어진다.

pid = kernel_thread(kthreadd, NULL, NULL, CLONE_FS | CLONE_FILES);

이 호출 후 새 task 는 PID 2 를 받고 kthreadd 함수를 실행한다. kernel/kthread.c 의 kthreadd 는 본질적으로 다음과 같은 무한 루프다.

int kthreadd(void *unused)
{
    /* ... */
    for (;;) {
        set_current_state(TASK_INTERRUPTIBLE);
        if (list_empty(&kthread_create_list))
            schedule();
        __set_current_state(TASK_RUNNING);
        
        /* kthread_create_list 큐를 polling 하며 thread 생성 요청 처리 */
        while (!list_empty(&kthread_create_list)) {
            struct kthread_create_info *create = ...;
            create_kthread(create);
        }
    }
    return 0;
}

요점은 kthreadd 가 kthread_create_list 라는 큐를 polling 하며 누군가 thread 생성을 요청하면 그것을 만들어 낸다는 것이다. 이 큐에 요청을 넣는 것은 다른 커널 코드에서 호출하는 kthread_create 또는 kthread_run API 다.

이렇게 만들어지는 모든 kernel thread 는 부모가 PID 2 이다. ps -ef 에서 대괄호 [...] 로 둘러싸여 표시되는 모든 thread 가 그 결과물이다.

PID 2 (kthreadd)
  │
  ├─ [kworker/N:M]   workqueue worker (per-CPU N, idx M)
  ├─ [kworker/u8:0]  unbound workqueue worker
  ├─ [ksoftirqd/N]   softirq handler (per-CPU)
  ├─ [rcu_sched]     RCU callbacks
  ├─ [rcu_tasks]
  ├─ [migration/N]   CPU migration (per-CPU)
  ├─ [watchdog/N]    soft lockup detection
  ├─ [cpuhp/N]       CPU hotplug
  ├─ [kcompactd0]    memory compaction (per-NUMA node)
  ├─ [kswapd0]       page swap (per-NUMA node)
  ├─ [oom_reaper]    OOM victim memory reclaim
  ├─ [writeback]     dirty page writeback
  ├─ [jbd2/sda2-8]   ext4 journaling (per-fs)
  ├─ [scsi_eh_N]     SCSI error handler
  └─ ...

확인: ps -e -T | grep '\['

이 thread 들은 각자 매우 중요한 역할을 한다. [kworker] 들은 workqueue 위에서 실행되는 일반 work item 들을 처리하고, [ksoftirqd] 는 softirq 가 너무 많이 쌓이면 process context 에서 처리해주며, [rcu_*] 들은 RCU (Read-Copy-Update) 의 grace period 처리를 담당한다. 모두 PID 2 의 자식이라는 사실이 핵심이다.

25세 PID 의 관계 — 종합 도식

지금까지 본 PID 0, 1, 2 의 관계를 한 화면에 정리하면 다음과 같다.

                    PID 0
          +----------------------+
          | swapper / idle task  |
          +----------+-----------+
                     |
          +----------+----------+
          |                     |
          v                     v
       PID 1                 PID 2
    kernel_init            kthreadd
         |                     |

         |

부모-자식 관계로 보면 PID 1 과 PID 2 는 모두 PID 0 이 만들어낸 자식이다. 이후 PID 1 은 모든 userspace 의 root 가 되고, PID 2 는 모든 커널 thread 의 root 가 된다. PID 0 은 자식들을 다 만든 뒤 boot CPU 의 idle task 로 영원히 남는다.

26ps 에서 PID 0 이 안 보이는 이유

ps -ef 명령을 실행해보면 PID 1, PID 2 는 보이지만 PID 0 은 보이지 않는다.

$ ps -ef
UID        PID  PPID  CMD
root         1     0  /sbin/init splash       ← PID 1: systemd
root         2     0  [kthreadd]               ← PID 2: kernel thread mgr
root         3     2  [rcu_gp]                 ← kthreadd 자식
root         4     2  [rcu_par_gp]
root         5     2  [slub_flushwq]
root         8     2  [kworker/0:0H]
...
root       100     1  /lib/systemd/systemd-journald  ← systemd 자식
root       200     1  /usr/sbin/sshd
...

PID 0 안 보이는 이유:
  ├─ /proc/0 디렉토리 없음
  ├─ procfs 가 PID 0 entry 안 만듦
  └─ ps 는 /proc 만 스캔 → PID 0 invisible

PID 0 확인 방법:
  cat /proc/sched_debug | grep swapper
  perf top          (kernel mode 시 swapper 표시)
  cat /proc/stat    (CPU idle time = swapper 활동량)

이유는 단순하다. ps 는 /proc/<pid>/ 디렉토리를 스캔해서 process 목록을 만드는데, procfs 가 일부러 PID 0 에 대해 디렉토리를 만들지 않기 때문이다. 즉 task_struct 차원에서는 PID 0 이 분명히 존재하지만, userspace 에서 보이는 /proc 인터페이스에서는 의도적으로 노출하지 않는다. 이것은 "PID 0 은 일반 process 가 아니다" 라는 정책적 표현이다.

PID 0 의 활동을 보려면 다른 우회 경로가 필요하다. /proc/sched_debug 에는 swapper/0, swapper/1 같은 항목이 보인다. /proc/stat 의 첫 줄에 있는 idle time 이 사실 swapper (PID 0) 의 활동 시간이다. perf top 도 CPU 가 kernel mode 의 idle 안에 있을 때 swapper 로 표시한다.

27PID 0 / 1 / 2 비교표

세 PID 의 특성을 한 표로 종합하면 다음과 같다.

항목 PID 0 PID 1 PID 2
이름 swapper / idle task (Ubuntu) systemd kthreadd
생성 방식 정적 (init_task) + per-CPU fork_idle() kernel_thread/user_mode_thread(kernel_init) → execve kernel_thread(kthreadd, ...)

여기서 한 가지 흥미로운 디테일이 보인다. PID 1 은 처음에 kernel_init 이라는 이름으로 만들어졌지만, execve("/sbin/init") 후에는 systemd 가 prctl(PR_SET_NAME, ...) 를 호출해서 자기 이름을 systemd 로 바꾼다. 즉 comm 필드는 동적으로 변경 가능하다.

28실제 Ubuntu 시스템에서 확인하기

지금까지 설명한 내용을 실제로 확인해보면 학습이 단단해진다. Ubuntu 시스템에서 다음 명령들을 실행해보자.

PID 1 의 정체 확인:

$ ps -p 1 -f
UID   PID  PPID CMD
root    1     0 /sbin/init

$ ls -la /sbin/init
/sbin/init -> /lib/systemd/systemd

/sbin/init 가 실제로는 systemd 로의 symlink 임이 확인된다. kernel_init 함수가 run_init_process("/sbin/init") 를 호출했고, 그것이 결국 systemd binary 를 실행한 것이다.

PID 2 확인:

$ ps -p 2 -f
UID   PID  PPID CMD
root    2     0 [kthreadd]

대괄호로 표시된 것이 커널 thread 의 표식이다. 일반 userspace process 와 구분된다.

모든 커널 thread 보기:

$ ps -e -T | grep '\['

이 명령은 시스템에 있는 모든 kernel thread 를 나열한다. 출력의 거의 모든 항목이 PPID 가 2 (kthreadd) 일 것이다.

29Stage 3 정리

Stage 3 까지 오면 PID 0, 1, 2 의 정체가 명확해진다.

PID 0 은 부팅 시점에 정적으로 생성되는 특수 task 이며, CPU 마다 별도 인스턴스가 있고 모두 PID 0 으로 표시된다. 역할은 CPU idle — 다른 할 일이 없을 때 CPU 를 절전 상태로 만드는 것이다. /proc/0 이 없어서 ps 에 안 보이지만 /proc/sched_debug 등으로 확인할 수 있다. PID 0 도 PID 1 처럼 보호받는다 — master kernel/exit.c 의 make_task_dead 함수 시작 부분에 다음 검사가 있다.

void __noreturn make_task_dead(int signr)
{
    struct task_struct *tsk = current;
    int oops_dumping_count = ...;
    /* ... */

    if (unlikely(!tsk->pid))
        panic("Attempted to kill the idle task!");
    /* ... */
}

tsk->pid == 0 이면 즉시 panic("Attempted to kill the idle task!") 가 발동한다. 즉 PID 0 을 죽이려는 시도는 곧 시스템 정지로 이어진다. PID 0 이 사라지면 idle 동작 자체가 없어져 OS 의 안정성이 무너지기 때문에, 코드 차원에서 명시적으로 막혀 있는 것이다. PID 1 도 비슷한 보호를 받지만 검사 위치와 조건이 다르다 — 자세한 내용은 Stage 5 §43 에서 본다.

PID 1 은 처음 kernel_init 이라는 커널 함수로 만들어졌다가 execve("/sbin/init") 로 systemd 로 변신한다. 이후 모든 userspace service 의 root 가 된다. PID 1 의 죽음은 kernel panic 을 일으키는데, 그 정확한 메커니즘은 Stage 5 에서 본다.

PID 2 는 끝까지 커널 모드로 남는 kthreadd 다. kthread_create_list 큐를 polling 하며 다른 코드가 요청한 커널 thread 들을 만들어낸다. [kworker], [ksoftirqd], [rcu_*] 등 모든 대괄호 표시된 thread 들의 부모다.

여기까지 봤다면 시스템의 가장 기초가 되는 task 들을 모두 이해한 셈이다. 다음 Stage 에서는 한 단계 더 깊이 들어가서 task_struct 와 struct pid 의 관계, 그리고 thread 와 process 의 진짜 차이 (TGID vs TID) 를 다룬다.

Stage 4PID 자료구조 한 단계 깊이

Stage 3 에서 PID 0, 1, 2 의 정체와 역할을 봤다. 이제 한 단계 더 깊이 들어가서, 커널 내부에서 PID 가 어떻게 자료구조 차원에서 표현되는지를 본다. 단순히 정수 하나가 아니라 struct pid 라는 별도 자료구조가 있고, 이 자료구조와 task_struct 의 관계가 thread / process / process group / session 같은 개념을 모두 표현한다. 또한 우리가 흔히 헷갈리는 "thread 의 PID" 의 정확한 정체 — kernel 의 PID 는 사실 TID 라는 사실 — 도 여기서 정리한다.

30모든 실행 단위는 task_struct 다

커널 내부의 가장 기본적인 사실 하나부터 짚자. Linux 는 process 와 thread 와 kernel thread 와 idle task 를 구분하지 않는다. 모두 같은 자료구조 — task_struct — 로 관리한다.

모든 실행 단위를 task_struct 로 관리:

* userspace process
* userspace thread (pthread)
* kernel thread
* idle task (PID 0)

→ 모두 task_struct

task_struct 는 include/linux/sched.h 에 정의된 거대한 구조체이며, 약 1 천 줄 가까운 필드를 갖고 있다 (CONFIG 옵션에 따라 달라진다). 이 구조체에는 process 가 가져야 할 모든 상태 — 메모리 매핑, 열린 파일, signal handler, scheduler 정보, 자식 리스트 등 — 이 들어 있다.

PID 는 이 구조체 안에 단순히 정수로 저장될 수도 있었다. 실제로 cached value 로는 그렇게 들어 있다. 하지만 커널은 PID 를 별도의 struct pid 라는 자료구조로 분리해서 관리한다. 이 분리에는 명확한 이유가 있다.

31왜 struct pid 를 분리했는가

include/linux/pid.h 의 코멘트는 분리 이유를 그대로 설명한다.

* Storing pid_t values in the kernel and referring to them later has a
* problem.  The process originally with that pid may have exited and the
* pid allocator wrapped, and another process could have come along
* and been assigned that pid.
*
* Referring to user space processes by holding a reference to struct
* task_struct has a problem. ... A task_struct plus a stack consumes
* around 10K of low kernel memory. ... By comparison
* a struct pid is about 64 bytes.
*
* Holding a reference to struct pid solves both of these problems.

요약하면 두 가지 문제를 동시에 해결하기 위해서다. 하나는 PID 정수만 저장했을 때의 문제다. PID 가 wrap 되면 (예: pid_max 까지 갔다가 다시 작은 번호부터 할당) 같은 정수가 다른 task 를 가리키게 된다. 그래서 내가 저장해둔 PID 12345 가 의도한 task 가 아닌 새로 만들어진 다른 task 를 가리키는 사고가 날 수 있다.

다른 하나는 task_struct 포인터를 저장했을 때의 문제다. task_struct 는 매우 크다 — 자기 자신만 약 10KB 인데 stack 까지 합치면 더 커진다. 누군가가 단지 "이 task 를 식별하기 위해" 포인터를 잡고 있으면, 그 task 가 죽어도 메모리가 회수되지 못한다. 메모리 낭비가 심하다.

struct pid 는 이 두 문제를 동시에 해결한다. 크기가 약 64 바이트로 작고, refcount 로 안전하게 식별 가능하며, 새 task 가 같은 PID 정수를 받으면 새 struct pid 객체가 만들어지므로 wrap 후에도 잘못된 참조가 안 일어난다.

32struct pid 와 struct upid 자료구조

master 기준 struct pid 의 정의는 다음과 같다.

struct upid {
    int nr;                       /* namespace 내부 PID 번호 */
    struct pid_namespace *ns;     /* 어느 namespace 의 번호인가 */
};

struct pid {
    refcount_t count;
    unsigned int level;           /* 이 pid 가 속한 namespace 의 깊이 */
    spinlock_t lock;
    /* pidfs 관련 필드들 (생략) */

    /* 이 pid 를 사용하는 task 들의 리스트 (PID/PGID/SID/TGID 별) */
    struct hlist_head tasks[PIDTYPE_MAX];

    struct hlist_head inodes;
    wait_queue_head_t wait_pidfd; /* pidfd 대기열 */
    struct rcu_head rcu;

    /* 가변 길이 배열 — level+1 개의 upid */
    struct upid numbers[];
};

enum pid_type {
    PIDTYPE_PID,    /* 0: 개별 task (= TID) */
    PIDTYPE_TGID,   /* 1: thread group leader (= getpid 결과) */
    PIDTYPE_PGID,   /* 2: process group */
    PIDTYPE_SID,    /* 3: session */
    PIDTYPE_MAX
};

가장 흥미로운 부분은 numbers[] 라는 가변 길이 배열이다. 하나의 struct pid 는 여러 namespace 에서 다른 정수 값으로 보일 수 있다. 예를 들어 container 안의 task 가 host 에서는 PID 12345 이지만 container 내부에서는 PID 1 인 경우가 있는데, 이 경우 numbers[] 에 두 개의 upid 가 들어 있다 — 하나는 host namespace 에서 12345, 하나는 container namespace 에서 1.

이 다중 PID 메커니즘이 PID namespace 의 핵심이며, Stage 5 에서 자세히 다룬다.

tasks[PIDTYPE_MAX] 는 이 struct pid 를 사용하는 task 들의 리스트다. 한 struct pid 에 여러 task 가 연결될 수 있는 이유는, process group 이나 session 이 여러 task 를 묶을 수 있기 때문이다. 각 PID type 별로 별도 hlist 를 유지한다.

33task_struct 안의 PID 관련 필드

6.x 커널 기준 task_struct 안의 PID 관련 필드는 다음과 같다.

struct task_struct {
    /* ... */
    pid_t           pid;          /* cached: thread ID (TID) */
    pid_t           tgid;         /* cached: thread group ID — getpid() 가 반환 */

    struct pid     *thread_pid;   /* PIDTYPE_PID 대응 struct pid 포인터 */

    /* PIDTYPE_TGID/PGID/SID 는 signal_struct 에 위치 */
    struct signal_struct *signal;
    /* signal->pids[PIDTYPE_TGID/PGID/SID] */
    /* ... */
};

여기서 두 가지 사실이 중요하다. 첫째, pid 와 tgid 는 cached scalar 값이다. 빠른 접근을 위해 정수로 따로 저장되어 있고, 실제 안전한 식별은 thread_pid (struct pid 포인터) 를 통해 한다. 둘째, TGID/PGID/SID 는 task_struct 에 직접 들어 있지 않고 signal_struct 에 위치한다. 같은 thread group 의 task 들은 하나의 signal_struct 를 공유하므로, 거기에 있는 pids[PIDTYPE_TGID] 등이 자연스럽게 thread group 차원의 식별자가 된다.

이 구조는 Linux 5.x 중반에 변경되었다. 이전에는 task_struct 가 struct pid_link pids[PIDTYPE_MAX] 를 직접 갖고 있었지만, thread group 이나 process group, session 은 thread group leader 만 가지면 충분하다는 점을 활용해서 그것들을 signal_struct 로 이동시켰다. 이제 task_struct 자체에는 thread_pid 만 남는다.

전체 관계를 도식화하면 다음과 같다.

┌──────────────────────┐
│   task_struct        │
│  ──────────────      │
│  pid (cached TID)    │──┐
│  tgid (cached TGID)  │  │ cached scalar
│                      │  │
│  thread_pid ─────────┼──┼──▶ struct pid
│                      │  │      ├─ count: refcount
│  signal ─────────────┼─▶│      ├─ level: 0 (init_pid_ns) or N (nested)
│                      │  │      ├─ tasks[PIDTYPE_PID]: hlist of tasks
│                      │  │      └─ numbers[]:
└──────────────────────┘  │           ├─ [0] {nr=12345, ns=&init_pid_ns}
                          │           └─ [1] {nr=42,    ns=&container_ns}
        ┌────────────────┘           (level=1 인 경우 2개)
        ▼
   signal_struct
     ├─ pids[PIDTYPE_TGID] ──▶ struct pid (TGID 용)
     ├─ pids[PIDTYPE_PGID] ──▶ struct pid (process group leader 의 pid)
     └─ pids[PIDTYPE_SID]  ──▶ struct pid (session leader 의 pid)

34TGID vs TID — thread 와 process 의 진짜 차이

이제 가장 헷갈리는 주제로 들어간다. getpid() 와 gettid() 는 무엇이 다른가? thread 의 PID 는 무엇인가?

먼저 핵심 사실부터 못박자. 커널 내부에서 PID 는 사실 TID (Thread ID) 다. task_struct 한 개에 PID 하나가 1:1 로 대응한다. 그런데 userspace 에서 getpid() 가 반환하는 값은 task_struct 의 PID 가 아니라 TGID (Thread Group ID) 다. 이름이 정반대로 쓰이는 셈이다.

커널 내부 용어:
  PID  = task_struct 1 개의 식별자 = userspace 의 TID
  TGID = thread group 의 식별자 = userspace 의 PID

userspace 용어:
  getpid()  → 커널의 TGID 반환
  gettid()  → 커널의 PID 반환

이 명명 혼선 때문에 처음 보는 사람은 매우 헷갈린다. 하지만 한 번 정리하고 나면 명확하다.

multi-thread process 를 시각화하면 다음과 같다.

┌─────────────────────────────────────────────────────┐
│ userspace 의 "process" (예: pid=1000)               │
│                                                     │
│  ┌──────────────────────────────────────────────┐  │
│  │ thread group (TGID = 1000)                   │  │
│  │                                              │  │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  │  │
│  │  │ task A   │  │ task B   │  │ task C   │  │  │
│  │  │ pid=1000 │  │ pid=1001 │  │ pid=1002 │  │  │
│  │  │ tgid=1000│  │ tgid=1000│  │ tgid=1000│  │  │
│  │  │ leader   │  │          │  │          │  │  │
│  │  └──────────┘  └──────────┘  └──────────┘  │  │
│  └──────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘

userspace getpid()       → 1000 (모든 thread 동일, = TGID)
userspace gettid()       → 1000 / 1001 / 1002 (thread 별 = kernel pid)
ps -e (default)          → 1000 만 보임 (TGID 만 나열)
ps -eLf 또는 ps -eT      → 1000/1001/1002 모두 보임 (LWP/TID 표시)
/proc/1000/              → process 단위 (TGID 디렉토리)
/proc/1000/task/1001/    → 안의 thread 별 디렉토리

/proc 의 구조도 이 사실을 그대로 반영한다. /proc/1000/ 는 process 단위 디렉토리이고, 그 아래의 /proc/1000/task/1001/, /proc/1000/task/1002/ 같은 서브디렉토리가 thread 단위 정보를 제공한다.

35clone() flag 가 결정하는 thread 와 process 의 차이

C 표준에는 thread 라는 개념이 직접 들어 있지 않다. POSIX threads (pthreads) 가 thread 를 정의하는데, Linux 에서는 이것이 모두 clone() 시스템 콜로 구현된다. fork() 와 pthread_create() 모두 내부적으로는 clone() 을 호출하고, 차이는 어떤 flag 를 주느냐일 뿐이다.

┌───────────────────┬──────────────┬──────────────────────────────┐
│ 호출              │ flag         │ 결과                          │
├───────────────────┼──────────────┼──────────────────────────────┤
│ fork()            │ (없음)       │ 새 PID + 새 TGID (별도 process)│
│ vfork()           │ CLONE_VFORK  │ 새 PID + 새 TGID + parent 정지│
│                   │ |CLONE_VM    │                               │
│ pthread_create()  │ CLONE_THREAD │ 새 PID + 같은 TGID (thread)   │
│                   │ |CLONE_VM    │                               │
│                   │ |CLONE_FS    │                               │
│                   │ |CLONE_FILES │                               │
│                   │ |CLONE_SIGHAND│                              │
│                   │ |CLONE_SYSVSEM│                              │
│                   │ |CLONE_SETTLS │                              │
└───────────────────┴──────────────┴──────────────────────────────┘

핵심은 CLONE_THREAD flag 다. 이 flag 가 켜져 있으면 새 task 는 부모와 같은 TGID 를 공유한다. CLONE_VM 은 메모리 공간 공유, CLONE_FS 는 파일시스템 정보 (working directory 등) 공유, CLONE_FILES 는 file descriptor table 공유, CLONE_SIGHAND 는 signal handler 공유다. 이 모든 flag 가 함께 켜져야 진짜 "같은 process 안의 thread" 가 된다.

copy_process 안의 단순화한 흐름은 다음과 같다.

/* kernel/fork.c — 단순화한 흐름 */
if (clone_flags & CLONE_THREAD) {
    p->tgid = current->tgid;        /* 부모와 같은 TGID */
    p->group_leader = current->group_leader;
    /* 같은 signal_struct 공유 */
    p->signal = current->signal;
} else {
    p->tgid = p->pid;               /* 새 process: TGID = PID */
    p->group_leader = p;            /* 자기 자신이 leader */
    /* 새 signal_struct 할당 */
}

즉 TGID 는 thread group leader 의 PID 값과 같다. task_struct 의 tgid 필드는 단순히 group_leader->pid 의 cached value 라고 봐도 된다.

copy_process 가 실제로 하는 일 — 한 단계 더 깊이

fork(), vfork(), pthread_create(), clone3(), kthread_create() — 이 모든 호출의 종착지는 master kernel/fork.c 의 단 하나의 함수, copy_process 다. 이 함수가 task_struct 를 한 개 만들어내고, clone flag 에 따라 어떤 자원은 공유하고 어떤 자원은 복제한다. 함수의 master 서명을 그대로 본다.

__latent_entropy struct task_struct *copy_process(
                    struct pid *pid,
                    int trace,
                    int node,
                    struct kernel_clone_args *args)
{
    int pidfd = -1, retval;
    struct task_struct *p;
    struct multiprocess_signals delayed;
    struct file *pidfile = NULL;
    const u64 clone_flags = args->flags;
    struct nsproxy *nsp = current->nsproxy;
    /* ... */
}

인자가 struct kernel_clone_args * 하나로 통일된 점이 흥미롭다. 옛 자료에서는 _do_fork(clone_flags, stack_start, stack_size, parent_tidptr, child_tidptr, tls) 처럼 인자가 줄줄이 나열되었는데, master 는 모두 한 구조체에 담는다. 이는 clone3(2) syscall 의 등장 (Linux 5.3) 과 함께 도입된 변경이다.

kernel_clone_args 의 master 정의 (include/linux/sched/task.h) 는 다음과 같다.

struct kernel_clone_args {
    u64 flags;              /* CLONE_VM, CLONE_FS, CLONE_FILES, ... */
    int __user *pidfd;      /* CLONE_PIDFD: 새 task 의 pidfd 받을 곳 */
    int __user *child_tid;  /* CLONE_CHILD_SETTID: child 의 TID 쓸 곳 */
    int __user *parent_tid; /* CLONE_PARENT_SETTID: parent 에 child TID 쓸 곳 */
    const char *name;       /* kthread name */
    int exit_signal;        /* child 종료 시 parent 에 보낼 signal */
    u32 kthread:1;          /* kernel thread 인가? */
    u32 io_thread:1;        /* io_uring worker 인가? */
    u32 user_worker:1;      /* user worker 인가? (vhost 등) */
    u32 no_files:1;         /* file table 공유/복제 모두 skip */
    unsigned long stack;
    unsigned long stack_size;
    unsigned long tls;
    pid_t *set_tid;         /* clone3 의 set_tid 확장 */
    size_t set_tid_size;
    int cgroup;             /* CLONE_INTO_CGROUP 용 cgroup fd */
    int idle;               /* idle task 생성 (fork_idle) */
    int (*fn)(void *);      /* kernel thread 시작 함수 */
    void *fn_arg;
    struct cgroup *cgrp;
    struct css_set *cset;
    unsigned int kill_seq;
};

여러 분기 (kthread, io_thread, user_worker, idle) 가 비트 필드로 들어 있어, 같은 copy_process 가 일반 fork, kernel thread, idle task, io_uring worker 등 다양한 task 종류를 모두 만들어낼 수 있다.

copy_process 의 흐름을 단계별로 따라가자. master 의 실제 호출 순서이며, 각 단계는 master kernel/fork.c 안의 별도 헬퍼 함수다.

각 copy_* 헬퍼의 핵심 분기 패턴은 같다. 예를 들어 copy_files 의 master 본체는 다음과 같다.

static int copy_files(unsigned long clone_flags, struct task_struct *tsk,
                      int no_files)
{
    struct files_struct *oldf, *newf;

    oldf = current->files;
    if (!oldf)
        return 0;

    if (clone_flags & CLONE_FILES) {
        atomic_inc(&oldf->count);     /* 공유: refcount++ */
        tsk->files = oldf;
        return 0;
    }

    newf = dup_fd(oldf, NULL);        /* 복제: 새 fd table */
    if (IS_ERR(newf))
        return PTR_ERR(newf);
    tsk->files = newf;
    return 0;
}

copy_fs, copy_sighand, copy_signal, copy_mm 모두 동일한 패턴이다. flag 가 켜져 있으면 refcount 증가로 공유, 꺼져 있으면 새로 복제. 이게 "thread 와 process 의 차이는 무엇을 공유하느냐" 의 정확한 코드 차원 답이다.

특히 copy_mm 의 CoW 처리가 눈에 띈다.

static int copy_mm(unsigned long clone_flags, struct task_struct *tsk)
{
    struct mm_struct *mm, *oldmm = current->mm;

    if (!oldmm)
        return 0;

    if (clone_flags & CLONE_VM) {
        mmget(oldmm);                 /* mm_users++ (공유) */
        mm = oldmm;
        goto good_mm;
    }

    mm = dup_mm(tsk, current->mm);    /* 새 mm + CoW page table */
    if (!mm)
        return -ENOMEM;

good_mm:
    tsk->mm = mm;
    tsk->active_mm = mm;
    return 0;
}

dup_mm 안에서 dup_mmap 이 호출되어 모든 VMA 를 새 mm 에 복제하는데, page table entry 의 write 권한을 떨어뜨려 Copy-on-Write 를 활성화한다. 즉 fork 직후 부모와 자식의 모든 page 가 read-only 로 공유되고, 한쪽이 write 를 시도하는 순간 page fault → page 복제가 일어난다. 이것이 fork 가 빠른 이유다 — 실제 메모리 복제는 lazy 하게 일어난다.

마지막으로 한 가지 흥미로운 master 의 디테일. copy_process 는 새 task 를 만들기만 하고 schedule 하지 않는다. 함수 주석에 "The actual kick-off is left to the caller." 라고 적혀 있다. 호출자인 kernel_clone (_do_fork 의 master 후신) 이 wake_up_new_task(p) 로 명시적으로 깨워야 비로소 새 task 가 runnable 상태가 된다. 이 분리가 LSM hook 이나 cgroup attach 같은 사후 처리를 안전하게 할 수 있게 한다.

pid_t kernel_clone(struct kernel_clone_args *args)
{
    /* ... */
    struct pid *pid;
    struct task_struct *p;

    p = copy_process(NULL, trace, NUMA_NO_NODE, args);
    /* ... 사후 처리 (pidfd 설치 등) ... */

    pid = get_task_pid(p, PIDTYPE_PID);
    nr = pid_vnr(pid);

    /* CLONE_PARENT_SETTID 처리 */
    if (clone_flags & CLONE_PARENT_SETTID)
        put_user(nr, args->parent_tid);

    /* CLONE_VFORK: parent 일시 정지 setup */
    if (clone_flags & CLONE_VFORK) {
        p->vfork_done = &vfork;
        init_completion(&vfork);
        get_task_struct(p);
    }

    wake_up_new_task(p);                /* ← 여기서 비로소 schedule 가능 */

    /* CLONE_VFORK: child 가 exec/exit 할 때까지 대기 */
    if (clone_flags & CLONE_VFORK) {
        if (!wait_for_vfork_done(p, &vfork))
            ptrace_event_pid(PTRACE_EVENT_VFORK_DONE, pid);
    }

    put_pid(pid);
    return nr;
}

vfork() 가 어떻게 동작하는지도 이 코드에서 명료해진다. CLONE_VFORK | CLONE_VM 으로 호출되면 parent 가 wait_for_vfork_done 에서 멈추고, child 가 exec 또는 exit 할 때까지 기다린다. child 가 부모의 mm 을 잠시 빌려 쓰는 동안 부모는 정지 상태인 셈이다.

fork(), vfork(), clone(), clone3() — master 의 통일된 진입점

지금까지 clone() flag 로 thread/process 의 차이가 정해진다는 점, 그리고 copy_process 가 모든 task 생성의 종착지라는 점을 봤다. 그렇다면 fork(), vfork(), clone(), clone3() 는 master 안에서 정확히 어떻게 구분되는가? 답은 — 이들은 모두 kernel_clone(&args) 라는 단 하나의 함수를 호출하며, 차이는 어떤 kernel_clone_args 를 넘기느냐일 뿐이다.

master kernel/fork.c 의 syscall entry 들을 순서대로 본다.

#ifdef __ARCH_WANT_SYS_FORK
SYSCALL_DEFINE0(fork)
{
#ifdef CONFIG_MMU
    struct kernel_clone_args args = {
        .exit_signal = SIGCHLD,
    };

    return kernel_clone(&args);
#else
    /* can not support in nommu mode */
    return -EINVAL;
#endif
}
#endif

#ifdef __ARCH_WANT_SYS_VFORK
SYSCALL_DEFINE0(vfork)
{
    struct kernel_clone_args args = {
        .flags       = CLONE_VFORK | CLONE_VM,
        .exit_signal = SIGCHLD,
    };

    return kernel_clone(&args);
}
#endif

#ifdef __ARCH_WANT_SYS_CLONE
SYSCALL_DEFINE5(clone, unsigned long, clone_flags, unsigned long, newsp,
                int __user *, parent_tidptr,
                int __user *, child_tidptr,
                unsigned long, tls)
{
    struct kernel_clone_args args = {
        .flags       = (clone_flags & ~CSIGNAL),
        .pidfd       = parent_tidptr,
        .child_tid   = child_tidptr,
        .parent_tid  = parent_tidptr,
        .exit_signal = (clone_flags & CSIGNAL),
        .stack       = newsp,
        .tls         = tls,
    };

}
#endif

SYSCALL_DEFINE2(clone3, struct clone_args __user *, uargs, size_t, size)
{
    int err;
    struct kernel_clone_args kargs;
    pid_t set_tid[MAX_PID_NS_LEVEL];

    kargs.set_tid = set_tid;

    err = copy_clone_args_from_user(&kargs, uargs, size);
    if (err)
        return err;

    if (!clone3_args_valid(&kargs))
        return -EINVAL;

    return kernel_clone(&kargs);
}

네 entry 의 본체가 매우 짧다 — 각자가 하는 일은 단지 kernel_clone_args 를 채워 넣고 kernel_clone(&args) 로 위임하는 것뿐이다. 어떻게 채워 넣느냐가 곧 syscall 의 의미를 결정한다.

이를 표로 압축하면 다음과 같다.

syscall flags exit_signal 기타 인자 의미
fork() 0 (없음) SIGCHLD 없음 완전 분리된 새 process
vfork() `CLONE_VFORK \ CLONE_VM` SIGCHLD 없음 mm 공유, parent 정지 대기
clone(flags, ...) 호출자 지정 flags & CSIGNAL newsp, tls, tid ptr 등 5 개 호출자가 모두 제어
clone3(args) args->flags args->exit_signal struct 안에 모두 확장 가능한 인자

fork() 가 master 안에서 어떻게 정의되는지가 흥미롭다. flags 가 0 인 것이 본질이다 — 어떤 자원도 공유하지 않는다 (CLONE_VM, CLONE_FS, CLONE_FILES, CLONE_SIGHAND 모두 꺼짐). 그래서 child 는 부모의 mm/fs/files/sighand 를 모두 복제해서 자기 것을 갖게 된다. 자원 격리의 극단.

vfork() 는 정반대로 CLONE_VM 만 켜져 있다. mm 을 공유하므로 child 의 메모리 변경이 부모에게도 보인다 (위험!). CLONE_VFORK 가 추가되어 parent 가 child 의 exec/exit 를 기다리도록 한다. 이 두 flag 의 조합이 vfork 의 BSD 시절 의미 — "잠깐 부모의 mm 을 빌려 쓰는 fork" — 를 정확히 재현한다.

exit_signal = SIGCHLD 가 fork/vfork 의 또 다른 본질이다. child 가 종료되면 parent 에게 SIGCHLD 가 전달된다 는 약속을 이 한 줄이 정한다. 그래서 부모가 wait() 로 회수할 수 있다. clone() 에서는 exit_signal 을 호출자가 지정하므로 thread (CLONE_THREAD 사용 시) 처럼 종료 signal 이 전달되지 않게 만들 수도 있다.

glibc 의 fork 래퍼는 단순한 syscall 호출이 아니다

fork() 라는 C 함수를 호출하면 단순히 SYS_fork syscall 로 직진할 것 같지만, 실제로는 glibc 가 그 사이에서 상당한 일을 한다. 이는 fork 의 의미상 (= "memory 가 통째로 복제됨") 필연적이다 — 복제 직전에 정리해야 할 상태들이 있다.

사용자: pid_t pid = fork();
              │
              ▼
       glibc __libc_fork (sysdeps/nptl/fork.c)
              │
              ├─ atfork prepare handlers 실행 (LIFO)
              │    : pthread_atfork(prepare, ...) 으로 등록된 핸들러
              │      malloc lock, stdio lock 등을 child 안전 상태로
              │
              ├─ malloc 의 내부 lock 들 잡기 (__malloc_fork_lock_parent)
              │
              ├─ INTERNAL_SYSCALL(clone) 또는 (fork)
              │    : 실제 kernel 진입 — 여기가 위의 SYSCALL_DEFINE0(fork)
              │
              ├─ [child] atfork child handlers 실행 (FIFO)
              │    : malloc/stdio lock 을 free 상태로 reset
              │    : single-threaded 가정 복원
              │
              └─ [parent] atfork parent handlers 실행 (FIFO)
                   : lock 들을 unlock
                   : 정상 상태로 복원

pthread_atfork(3) man page 가 이 메커니즘이 필요한 이유를 직접 말한다.

The purpose of pthread_atfork() is to recall that fork(2) duplicates the whole memory space, including mutexes in their current locking state, but only the calling thread: other threads are not running in the child process. The mutexes are not usable after the fork and must be initialized with pthread_mutex_init() in the child process. This is a limitation of the current implementation and may or may not be present in future versions.

multi-thread 프로세스에서 한 thread 가 fork 하면, 다른 thread 가 잡고 있던 mutex 가 child 에서는 "잠긴 상태" 그대로 남는데, 그 mutex 를 풀어줄 thread 가 child 에 존재하지 않는다. child 는 영원히 그 lock 을 기다리게 된다 (= deadlock). atfork 메커니즘은 이 위험을 다음과 같이 해결한다.

  • prepare: fork 직전에 부모가 호출. 모든 중요한 lock 을 잡아 둔다 (다른 thread 가 lock 안에 있지 않도록 유도).
  • parent: fork 직후에 부모에서 호출. 잡았던 lock 을 푼다.
  • child: fork 직후에 child 에서 호출. lock 을 "초기 상태" 로 reset (잡혔던 흔적을 지운다).

glibc 자체도 자기 안의 lock 들 (malloc arena lock, stdio lock, locale lock 등) 에 대해 내부적으로 atfork 핸들러를 등록해 둔다. 그래서 사용자가 별 신경 안 써도 printf 직후의 fork 가 잘 동작한다. 단 사용자가 만든 mutex 는 사용자가 직접 pthread_atfork 로 등록해야 한다.

async-signal-safety 와 vfork → posix_spawn 의 등장

fork 의 또 다른 함정은 signal handler 안에서 호출하면 deadlock 가능 이라는 점이다. POSIX 는 fork 를 async-signal-safe 로 정의하지만, atfork handler 가 호출된다는 사실 자체가 deadlock 위험을 만든다. signal handler 가 mutex 를 잡고 있는 와중에 fork 가 prepare handler 를 호출하면, 이미 잡힌 lock 을 다시 잡으려는 시도가 발생할 수 있다.

vfork 는 이 문제에서 자유롭다 — CLONE_VM 으로 mm 을 공유하므로 atfork handler 가 호출될 필요가 없다 (어차피 child 가 곧 exec 할 것이므로 mm 의 상태가 의미없음). 그러나 vfork 는 그 자체로 위험하다. child 의 사소한 stack 사용이 부모의 stack 을 망가뜨릴 수 있다 (같은 stack 을 공유). 그래서 POSIX 는 vfork 를 obsolescent (구식) 로 표시했다.

POSIX 가 제시한 대안이 posix_spawn() 이다. 이 함수는 fork + exec 를 한 번의 호출로 묶고, libc 가 그 사이의 위험한 영역 (atfork handler, stack 공유) 을 신중히 다룬다. Linux glibc 의 구현은 내부적으로 clone(CLONE_VM | CLONE_VFORK) 를 호출해서 vfork 의 장점은 유지하면서 단점은 libc 가 흡수한다. 단 posix_spawn 은 syscall 이 아니라 libc 의 함수이므로 이 문서가 다루는 kernel 범위를 벗어난다. (자세한 내용은 별도의 libc 문서에서 다룬다.)

여기서 핵심을 다시 정리하면 — master 의 process 생성 syscall 은 단 네 개 (fork/vfork/clone/clone3) 이고, 모두 kernel_clone(&args) 라는 단일 함수로 수렴한다. 그 위의 모든 다양한 함수들 (pthread_create, system, popen, posix_spawn, ...) 은 libc 영역에서 이 네 syscall 중 하나를 적절히 호출하는 wrapper 일 뿐이다. kernel 의 관점에서는 흐름이 매우 단순하다.

$ ls /proc/1234/task/
1234/  1240/  1241/  1242/  1243/

PID 컬럼은 모든 thread 가 동일한 1234 를 갖고 (= TGID), TID 컬럼만 다르다. /proc/1234/task/ 안에는 각 thread 별 디렉토리가 있고, 그 디렉토리 이름이 TID 다. 첫 번째 thread (group leader) 의 TID 는 TGID 와 같다.

C 코드에서 두 값을 직접 확인하면 다음과 같다.

#include <sys/syscall.h>
#include <unistd.h>

pid_t pid = getpid();           /* = TGID, kernel 의 task->tgid */
pid_t tid = syscall(SYS_gettid);/* = kernel PID, task->pid */

/* 또는 glibc 2.30+ */
pid_t tid = gettid();

multi-thread 환경에서 getpid() 는 모든 thread 가 같은 값이지만 gettid() 는 thread 마다 다르다. 이를 활용하면 thread 마다 다른 동작을 하도록 코드를 작성할 수 있다.

37PID 할당 메커니즘 — alloc_pid 와 pid_max

PID 가 어떻게 할당되는지도 짚어보자. 옛날 (2.x ~ 3.x) 에는 bitmap 기반이었지만, 현대 커널은 IDR (radix tree) 기반으로 교체되었다. kernel/pid.c 의 alloc_pid 함수가 그 핵심이다.

/* kernel/pid.c */
struct pid *alloc_pid(struct pid_namespace *ns, pid_t *set_tid,
                      size_t set_tid_size)
{
    /* level 0..ns->level 의 모든 namespace 에서 nr 할당 */
    for (i = ns->level; i >= 0; i--) {
        int pid_max = READ_ONCE(tmp->pid_max);
        nr = idr_alloc_cyclic(&tmp->idr, ..., pid_min, pid_max, ...);
        /* ... */
    }
    /* ... */
}

idr_alloc_cyclic 함수가 핵심이다. 이름 그대로 cyclic — 마지막 할당 PID 다음부터 검색한다. 0 부터 다시 시작하지 않으므로 wrap 후에도 충돌이 줄어든다. nested namespace 라면 for i = level..0 루프가 모든 level 에서 각자의 IDR 로부터 nr 을 할당한다.

pid_max 는 PID 의 최대값으로, /proc/sys/kernel/pid_max 로 확인하고 변경할 수 있다.

$ cat /proc/sys/kernel/pid_max
4194304     # 4M (현대 64-bit Ubuntu 기본)
# 과거 32-bit 시절: 32768 (PID_MAX_DEFAULT)

PID_MAX_DEFAULT 는 0x8000 (32768) 이고 32-bit 시절의 한계였다. PID_MAX_LIMIT 은 4M 으로 64-bit 시 자동 증가한다. namespace 별로 별도의 pid_max 가 있어 container 마다 PID 를 제한할 수 있다.

master include/linux/threads.h 의 정확한 매크로 정의는 다음과 같다.

/* This controls the default maximum pid allocated to a process */
#define PID_MAX_DEFAULT (IS_ENABLED(CONFIG_BASE_SMALL) ? 0x1000 : 0x8000)

/*
 * A maximum of 4 million PIDs should be enough for a while.
 * [NOTE: PID/TIDs are limited to 2^30 ~= 1 billion, see FUTEX_TID_MASK.]
 */
#define PID_MAX_LIMIT (IS_ENABLED(CONFIG_BASE_SMALL) ? PAGE_SIZE * 8 : \
                       (sizeof(long) > 4 ? 4 * 1024 * 1024 : PID_MAX_DEFAULT))

PID_MAX_DEFAULT = 0x8000 = 32768 은 평시 부팅 시 pid_max 의 초기값이고, 작은 임베디드 시스템 (CONFIG_BASE_SMALL) 에서는 0x1000 = 4096 으로 떨어진다. 운영 중에는 /proc/sys/kernel/pid_max 로 4M 까지 자유롭게 늘릴 수 있다.

흥미로운 주석은 4M 이라는 상한이 임의의 숫자가 아니라는 점이다. PID 와 TID 는 사실 30 비트 (≈ 10억) 까지가 한계이며 — 그 이유는 FUTEX_TID_MASK = 0x3fffffff (30 비트 mask) 라서 futex (fast userspace mutex) API 가 PID 를 30 비트 안에 인코딩해서 처리하기 때문이다. 그래서 master 는 4M 정도면 "한동안 충분" 이라는 안전 margin 으로 설계한 것이다.

PID wrap 의 흐름을 시각화하면 다음과 같다.

시스템 가동
   ↓
PID 1, 2, 3, ... 순차 할당
   ↓
프로세스 죽으면 PID 회수
   ↓
pid_max 도달 (예: 4194303)
   ↓
다시 처음부터 검색 (300 이상의 빈 자리 찾음)
   ↓
PID 재사용

RESERVED_PIDS = 300 매크로가 있어서 일부 영역은 hint 로 다루어진다. 그래서 wrap 후에도 작은 PID (예: 1, 2) 는 영원히 systemd, kthreadd 가 차지하고 다른 task 가 가져갈 수 없다.

이제 §31 에서 봤던 "PID 정수만 저장하면 wrap 후 다른 task 를 가리킨다" 의 의미가 더 구체적으로 이해된다. wrap 이 일어나면 같은 정수가 다른 task 를 가리키지만, struct pid reference 는 wrap 시 새로 할당된 객체이므로 안전하다.

38PID 로 task 찾기 — find_task_by_* 함수 가족

PID 로 task 를 찾는 헬퍼 함수들도 정리해두자. 어느 namespace 기준인지가 중요하다.

┌────────────────────────────┬─────────────────────────────────┐
│ 함수                        │ 동작                              │
├────────────────────────────┼─────────────────────────────────┤
│ find_pid_ns(nr, ns)        │ 명시한 ns 안에서 nr → struct pid  │
│ find_vpid(nr)              │ current 의 ns 에서 (= virtual)    │
│ find_get_pid(nr)           │ find_vpid + refcount 증가         │
│                            │                                  │
│ find_task_by_pid_ns(nr,ns) │ 명시한 ns 안에서 nr → task_struct │
│ find_task_by_vpid(nr)      │ current 의 ns 에서 task_struct    │
│                            │                                  │
│ pid_task(pid, type)        │ struct pid → task_struct (RCU)   │
│ get_pid_task(pid, type)    │ pid_task + refcount 증가          │
└────────────────────────────┴─────────────────────────────────┘

이름의 패턴은 다음과 같다. _vpid 는 "virtual PID" 의 약자로, 호출 task 의 PID namespace 기준이라는 뜻이다. _pid_ns 는 명시한 namespace 기준. 그냥 pid_task 류는 이미 struct pid 를 갖고 있을 때 task_struct 를 얻는 용도다.

흔한 사용 패턴은 다음과 같다.

/* 어떤 namespace 기준 PID 12345 의 task 가져오기 */
struct task_struct *find_some_task(pid_t nr, struct pid_namespace *ns)
{
    struct pid *pid;
    struct task_struct *task;

    rcu_read_lock();
    pid = find_pid_ns(nr, ns);
    task = pid ? pid_task(pid, PIDTYPE_PID) : NULL;
    if (task)
        get_task_struct(task);  /* 사용 중 사라지지 않게 */
    rcu_read_unlock();

    return task;
}

system call 에서의 함정도 알아두자. kill(pid, sig), waitpid(pid, ...) 등 PID 를 받는 syscall 은 모두 호출자 task 의 PID namespace 기준 으로 해석된다. 즉 container 안에서 kill(1, SIGTERM) 은 container 의 PID 1 (그 안 systemd / init 류) 을 가리키지 host 의 systemd 가 아니다. find_task_by_vpid(pid) 가 그 변환을 담당한다.

39Stage 4 정리

Stage 4 까지 오면 PID 의 자료구조 차원이 잡힌다. 핵심 사실들을 한 번 더 정리하면 다음과 같다.

PID 는 단순한 정수가 아니라 struct pid 라는 별도 자료구조로 관리된다. 이는 PID wrap 시 잘못된 참조를 방지하고, task_struct 포인터 보관의 메모리 낭비도 피하기 위한 의도적 설계다. 64 바이트의 작은 구조체가 refcount 와 함께 안전한 식별자 역할을 한다.

task_struct 안에는 빠른 접근을 위한 cached 값으로 pid 와 tgid 가 정수로 들어 있고, 안전한 식별을 위한 thread_pid 포인터가 별도로 있다. TGID/PGID/SID 는 signal_struct 로 옮겨졌는데, 이는 thread group 차원에서 공유되는 값이기 때문이다.

userspace 와 kernel 의 PID 명명이 정반대다. kernel 의 PID = userspace 의 TID, kernel 의 TGID = userspace 의 PID. getpid() 는 TGID 를, gettid() 는 kernel 의 PID 를 반환한다. multi-thread process 에서 모든 thread 가 같은 TGID 를 공유하지만 각자 다른 TID 를 갖는다.

clone() 의 flag 들이 thread 와 process 의 차이를 결정한다. CLONE_THREAD 가 켜져 있으면 같은 TGID 를 공유하고, 함께 켜지는 CLONE_VM, CLONE_FS, CLONE_FILES, CLONE_SIGHAND 가 진짜 thread 의 자원 공유를 만든다.

PID 할당은 IDR 기반의 idr_alloc_cyclic 으로 이루어지며, pid_max 까지 갔다가 wrap 한다. namespace 별로 별도 pid_max 가 있다.

다음 Stage 에서는 이 자료구조의 가장 강력한 활용처 — PID namespace 와 container — 로 들어간다. struct pid::numbers[] 가 가변 길이 배열인 이유, host PID 1 과 container PID 1 의 차이, 그리고 container 의 PID 1 이 죽었을 때 무엇이 일어나는지가 모두 거기서 풀린다.

Stage 5PID namespace 와 container 의 PID 1

Stage 4 의 마지막에 struct pid::numbers[] 가 가변 길이 배열이라는 사실을 봤다. 이 배열의 존재 이유가 바로 PID namespace 다. Stage 5 의 목표는 이 namespace 메커니즘을 통해 container 가 어떻게 자기만의 PID 공간을 가지게 되는지, container 의 PID 1 이 host 의 PID 1 과 어떻게 다른지, 그리고 PID 1 이 죽었을 때 두 경우에 무엇이 일어나는지를 명확히 정리하는 것이다.

40pid_namespace 자료구조

PID namespace 의 자료구조는 include/linux/pid_namespace.h 에 정의되어 있다. 핵심 필드만 추리면 다음과 같다.

struct pid_namespace {
    struct idr              idr;            /* PID allocator */
    struct rcu_head         rcu;
    unsigned int            pid_allocated;
    struct task_struct      *child_reaper;  /* 이 namespace 의 PID 1 */
    struct kmem_cache       *pid_cachep;
    unsigned int            level;          /* 0 = init_pid_ns, 1+ = nested */
    int                     pid_max;        /* per-ns pid_max (CONFIG) */
    struct pid_namespace    *parent;        /* 부모 namespace */
    struct user_namespace   *user_ns;
    struct ucounts          *ucounts;
    int                     reboot;         /* group exit code on pidns reboot */
    struct ns_common        ns;
    /* ... */
};

extern struct pid_namespace init_pid_ns;

가장 중요한 필드는 child_reaper 다. 이 namespace 안에서 고아가 된 자식들을 입양할 task — 즉 이 namespace 의 PID 1 — 을 가리킨다. 또한 level 은 이 namespace 가 얼마나 nested 되어 있는지를 나타낸다 (host 는 0, 첫 container 는 1, 그 안의 nested container 는 2, ...).

가장 root 가 되는 init_pid_ns 는 kernel/pid.c 에 정적으로 선언되어 있다.

struct pid_namespace init_pid_ns = {
    .ns           = NS_COMMON_INIT(init_pid_ns),
    .idr          = IDR_INIT(init_pid_ns.idr),
    .pid_allocated = PIDNS_ADDING,
    .level        = 0,
    .child_reaper = &init_task,        /* ← boot 시점, init_task (PID 0) 가 임시 reaper */
    .user_ns      = &init_user_ns,
    .pid_max      = PID_MAX_DEFAULT,
    /* ... */
};

여기 흥미로운 디테일이 있다. init_pid_ns.child_reaper 가 boot 단계에서 &init_task (= PID 0, swapper) 로 설정되어 있다. 즉 부팅 시점에는 PID 0 이 임시로 reaper 자리를 차지한다. 이후 kernel_init 이 PID 1 으로 fork 되면 PID 1 의 task 가 reaper 자리를 인계받는다. "PID 1 = init namespace 의 child_reaper" 는 부팅 후의 결과이지 정적 정의가 아니다.

41같은 task 가 여러 PID 를 갖는 의미

struct pid::numbers[] 가 가변 길이 배열인 이유는, 하나의 task 가 여러 namespace 에서 다른 정수 값으로 보일 수 있기 때문이다.

[level 0]  init_pid_ns          host PID = 12345
[level 1]  container_pid_ns_A   container PID = 42
[level 2]  container_pid_ns_B   nested container PID = 1

위 예에서 같은 task 가 host 에서는 12345, 첫 번째 container 에서는 42, 그 안의 두 번째 nested container 에서는 1 (= reaper) 로 보인다. struct pid 의 numbers[] 에 각 level 의 upid 가 들어 있고, level 필드가 가장 안쪽 namespace 의 깊이를 가리킨다.

이를 다루기 위한 헬퍼 함수들이 include/linux/pid.h 에 있다.

/* global PID (init namespace 기준) — pid->numbers[0].nr */
static inline pid_t pid_nr(struct pid *pid);

/* virtual PID (호출 task 의 pid namespace 기준) */
pid_t pid_vnr(struct pid *pid);

/* 명시한 namespace 기준 */
pid_t pid_nr_ns(struct pid *pid, struct pid_namespace *ns);

/* task_struct 헬퍼들 */
static inline pid_t task_pid_nr(struct task_struct *tsk);    /* tsk->pid (cached, init ns) */
static inline pid_t task_pid_vnr(struct task_struct *tsk);   /* current ns 기준 */
static inline pid_t task_tgid_nr(struct task_struct *tsk);   /* tsk->tgid (cached, init ns) */
static inline pid_t task_tgid_vnr(struct task_struct *tsk);  /* current ns 기준 TGID */

_nr 은 init namespace 기준의 글로벌 정수, _vnr 은 호출자가 속한 namespace 기준의 virtual 정수, _nr_ns 는 명시한 namespace 기준이다. 이 분리가 중요한 이유는, 같은 task 라도 어느 namespace 에서 보느냐에 따라 다른 PID 정수가 보이기 때문이다.

PID 1 을 판별하는 두 함수도 있는데, 이들이 host 와 container 의 차이를 정확히 표현한다.

/* include/linux/pid.h */
static inline int is_global_init(struct task_struct *tsk)
{
    return task_tgid_nr(tsk) == 1;
}

static inline bool is_child_reaper(struct pid *pid)
{
    return pid->numbers[pid->level].nr == 1;
}

is_global_init 은 "init namespace 기준 TGID 가 1" 인지 검사한다. host 의 진짜 systemd 만 true 를 반환한다. container 안의 PID 1 은 host 입장에서 큰 PID 를 갖고 있으므로 false 다.

is_child_reaper 는 "자신의 namespace 안에서 PID 가 1" 인지 검사한다. host 의 systemd 도 true 이고, container 안의 PID 1 도 true 다.

이 두 함수의 분리가 핵심이다. host 의 PID 1 (is_global_init 만 통과) ≠ container 의 PID 1 (is_child_reaper 만 통과). 두 곳의 PID 1 은 같은 PID 정수를 가지지만 권한과 책임이 다르다.

42PID 1 의 특수 책임 — child reaper

PID 1 은 단순히 첫 번째 process 가 아니라 매우 특수한 책임을 갖는다. 가장 중요한 책임이 reaper — 고아 프로세스 입양과 zombie 회수 — 이다.

Zombie 의 정확한 정체

"reap" 이라는 동사가 자주 쓰이는데, 그 대상인 zombie 가 코드 차원에서 무엇인지부터 못박자. master include/linux/sched.h 에 exit_state 필드는 두 값만 가질 수 있다.

/* Used in tsk->exit_state: */
#define EXIT_DEAD       0x0010
#define EXIT_ZOMBIE     0x0020
#define EXIT_TRACE      (EXIT_ZOMBIE | EXIT_DEAD)

task 가 do_exit 를 호출하면 master kernel/exit.c::exit_notify 안에서 tsk->exit_state = EXIT_ZOMBIE; 가 명시적으로 설정된다. 이 상태의 task 가 바로 zombie 다. ps 에서 Z 또는 defunct 로 표시되는 그것이다.

/* kernel/exit.c master */
static void exit_notify(struct task_struct *tsk, int group_dead)
{
    bool autoreap;
    struct task_struct *p, *n;
    LIST_HEAD(dead);

    write_lock_irq(&tasklist_lock);
    forget_original_parent(tsk, &dead);   /* orphan 재부모 */

    if (group_dead)
        kill_orphaned_pgrp(tsk->group_leader, NULL);

    tsk->exit_state = EXIT_ZOMBIE;        /* ← 여기서 zombie 화 */

    if (unlikely(tsk->ptrace)) {
        /* ... ptrace 케이스 ... */
        autoreap = do_notify_parent(tsk, sig);
    } else if (thread_group_leader(tsk)) {
        autoreap = thread_group_empty(tsk) &&
                   do_notify_parent(tsk, tsk->exit_signal);
    } else {
        autoreap = true;
        do_notify_pidfd(tsk);             /* pidfd waiter 깨우기 */
    }

    if (autoreap) {
        tsk->exit_state = EXIT_DEAD;      /* ← reap 결정됨 */
        list_add(&tsk->ptrace_entry, &dead);
    }
    /* ... */
}

zombie 상태에서는 task_struct 와 PID 만 남아 있고, mm/files/fs 등 대부분의 자원은 이미 해제된 상태다. 그래서 "이미 죽었는데 process table 의 자리만 잡고 있는" 흉상이라는 비유가 정확하다. kill -9 는 무의미하다 — 이미 죽었기 때문이다.

wait() 가 zombie 를 회수하는 흐름

부모가 wait() / waitpid() / waitid() 를 호출하면 zombie 가 다음 흐름으로 회수된다.

부모: wait4() syscall
   │
   ▼
kernel/exit.c::wait_task_zombie(wo, p)
   │
   ├─ 자식의 exit_code, rusage 등을 부모에게 복사
   ├─ p->exit_state 를 EXIT_ZOMBIE → EXIT_DEAD 로 전환
   │
   └─ release_task(p)
        │
        ├─ __exit_signal(p): signal_struct 정리
        ├─ proc_flush_pid(p->thread_pid): /proc/<pid>/ 정리
        ├─ detach_pid: struct pid 의 tasks hash 에서 제거
        ├─ leader 처리 (CLONE_THREAD 인 경우)
        ├─ release_thread(p): arch 별 정리
        │
        └─ put_task_struct(p)
             └─ refcount 0 이 되면 free_task(p)
                  └─ task_struct + kernel stack 메모리 해제

이게 "PID 가 회수된다" 의 정확한 의미다. release_task 가 detach_pid 로 struct pid::tasks[PIDTYPE_PID] hash list 에서 task 를 분리하고, 더 이상 그 PID 를 가리키는 reference 가 없으면 struct pid 도 free 된다. 이때 비로소 그 PID 정수가 alloc_pid 의 IDR 에서 풀려나 재사용 가능해진다.

orphan 재부모와 reaper 의 등장

부모가 자식보다 먼저 죽으면 자식은 orphan 이 된다. 이 orphan 의 zombie 를 누가 회수할 것인가? master kernel/exit.c::find_new_reaper 가 새 부모를 결정한다.

/* kernel/exit.c master, 단순화 */
static struct task_struct *find_new_reaper(struct task_struct *father,
                                            struct task_struct *child_reaper)
{
    struct task_struct *thread, *reaper;

    /* 1순위: 같은 thread group 의 살아있는 thread */
    thread = find_alive_thread(father);
    if (thread)
        return thread;

    /* 2순위: PR_SET_CHILD_SUBREAPER 설정한 nearest ancestor */
    if (father->signal->has_child_subreaper) {
        unsigned int ns_level = task_pid(father)->level;
        for (reaper = father->real_parent;
             reaper->pid_ns_for_children->level == ns_level;
             reaper = reaper->real_parent) {
            if (reaper == &init_struct_pid->...->child_reaper)
                break;
            if (!reaper->signal->is_child_subreaper)
                continue;
            thread = find_alive_thread(reaper);
            if (thread)
                return thread;
        }
    }

    /* 3순위: 해당 PID namespace 의 child_reaper (= PID 1) */
    return child_reaper;
}

자식 입양은 forget_original_parent → find_new_reaper 로 일어나고, 새 부모가 정해지면 reparent_leader 가 자식의 real_parent 와 parent 를 새 부모로 갱신한다. 그 후 자식이 종료되면 SIGCHLD 가 새 부모에게 가고, 새 부모가 wait() 하면 zombie 가 회수된다.

이 흐름이 끝나면 비로소 "PID 1 의 가장 중요한 책임은 orphan 의 reaper 다" 라는 명제가 코드 차원에서 닫힌다.

[부모 P 가 죽으면 자식 C 는 어디로?]
              │
              ▼
   nearest ancestor 중 PR_SET_CHILD_SUBREAPER 설정한 task 검색
              │
   ┌──────────┴──────────┐
   │ subreaper 있음       │ subreaper 없음
   ▼                     ▼
   해당 subreaper        현재 PID namespace 의 child_reaper
                              (host 면 systemd, container 면 그 안의 PID 1)

reparent 된 자식이 종료 (do_exit) 하면 새 부모 (= reaper) 에게 SIGCHLD 가 전달된다. reaper 는 wait4() 또는 waitpid() 로 zombie 의 task_struct 를 회수해야 한다. systemd 는 PID 1 으로서 끊임없이 SIGCHLD 를 처리하며 zombie 를 reap 한다. 이를 안 하면 zombie 가 누적되어 PID 가 고갈된다.

PR_SET_CHILD_SUBREAPER 의 정확한 효과

subreaper 메커니즘도 중요하다. prctl(PR_SET_CHILD_SUBREAPER, 1) 호출 시 해당 task 는 자기 후손 중 orphan 이 생기면 PID 1 대신 자신이 받는다. 호출 즉시 task->signal->is_child_subreaper = 1 이 설정되고, 자식 task 가 만들어질 때마다 task->signal->has_child_subreaper 비트가 전파된다. find_new_reaper 가 위의 master 코드처럼 이 비트를 검사하는 식이다.

prctl(2) 의 정의를 그대로 인용하면 다음과 같다.

PR_SET_CHILD_SUBREAPER (since Linux 3.4): If arg2 is nonzero, set the "child subreaper" attribute of the calling process; if arg2 is zero, unset the attribute. When a process is marked as a child subreaper, all of the children that it creates, and their descendants, will be marked as having a subreaper. In effect, a subreaper fulfills the role of init(1) for its descendant processes.

용도는 다양하다 — systemd user-session manager (systemd --user) 가 자기 후손을 모두 받기 위해, container runtime 이 container 안 PID 1 죽음을 감지하기 위해, session manager 와 daemon supervisor 일반.

PID 1 (systemd, host)
  └─ user@1000.service (systemd --user)   ← PR_SET_CHILD_SUBREAPER
       └─ shell
            └─ daemon (double-fork)
                 └─ orphan 발생
                      ↓
            shell 죽으면 → daemon 의 부모는 user@1000.service
            (PID 1 으로 reparent 안 됨)

이 메커니즘 덕분에 process tree 의 중간에서 "내 후손은 내가 책임진다" 라는 경계를 설정할 수 있다. Docker daemon, podman, tmux, supervisord 같은 daemon supervisor 들이 이 flag 를 활용한다.

43PID 1 죽음 = panic — 정확한 호출 경로

PID 1 의 또 다른 특수 책임은 자기 자신은 죽지 말아야 한다 는 것이다. host 의 PID 1 (systemd) 이 종료되면 시스템 전체가 panic 한다.

호출 경로는 kernel/exit.c 안에 있다. master 의 do_exit 함수 시그니처와 panic 검사 부분을 그대로 인용한다.

void __noreturn do_exit(long code)
{
    struct task_struct *tsk = current;
    int group_dead;

    synchronize_group_exit(tsk, code);
    WARN_ON(tsk->plug);

    /* ... 일부 사전 처리 ... */

    group_dead = atomic_dec_and_test(&tsk->signal->live);

    /*
     * If the last thread of global init has exited,
     * panic immediately to get a useable coredump.
     */
    if (unlikely(is_global_init(tsk) &&
                 (group_dead || (tsk->signal->flags & SIGNAL_GROUP_EXIT)))) {
        panic("Attempted to kill init! exitcode=0x%08x\n",
              tsk->signal->group_exit_code ?: (int)code);
    }

    /* ... 평소 흐름: 자원 회수, reparent, ... */
}

옛 자료가 흔히 단순화해서 if (tsk->pid == 1) panic(...) 라고 적는데, master 의 정확한 검사는 다음 두 조건의 AND 다.

  • is_global_init(tsk) — 이 task 가 init namespace 의 PID 1 인지. 함수 본체는 task_pid_nr(tsk) == 1 (master include/linux/sched/signal.h).
  • group_dead || (tsk->signal->flags & SIGNAL_GROUP_EXIT) — thread group 의 마지막 thread 가 죽거나, group 전체 exit 가 트리거된 상태인지.

후자의 조건이 핵심이다. PID 1 process 가 multi-threaded 인 경우 (예: systemd) 한 thread 가 종료되어도 다른 thread 가 살아 있다면 process 자체는 죽지 않은 것이다. 그래서 master 는 "마지막 thread 가 죽었거나 group exit 가 트리거됐을 때" 에만 panic 한다. 옛 자료의 tsk->pid == 1 한 줄짜리 검사는 single-threaded init 에 대해서만 정확했고, 현재 master 와는 차이가 있다.

또 한 가지 — master 의 do_exit 시그니처는 void __noreturn do_exit(long code) 다. __noreturn 이 시그니처에 박혀 있어 컴파일러도 이 함수에서 돌아오지 않음을 알고 있다. panic 으로 가는 길이든, 평소 종료 흐름이든 어느 쪽도 caller 로 return 하지 않는다.

실제 panic 메시지의 스택 trace 는 다음처럼 보인다 (Red Hat 의 예).

container 안의 PID 1 은 어떨까? host 입장에서는 is_global_init(tsk) == 0 이므로 panic 조건에 안 걸린다. 대신 다른 메커니즘이 발동한다.

container 의 PID 1 종료
   │
   ▼
zap_pid_ns_processes()         (kernel/pid_namespace.c)
   │
   ├─ namespace 안 모든 process 에게 SIGKILL
   │  (read_lock(&tasklist_lock); send_sig(SIGKILL, ...))
   │
   └─ namespace 자체 정리

이게 "container 안 PID 1 죽으면 container 안 모든 process 가 죽고 container 가 끝난다" 의 정확한 구현이다. pid_namespaces(7) man page 가 같은 사실을 다음과 같이 표현한다.

If the "init" process of a PID namespace terminates, the kernel terminates all of the processes in the namespace via a SIGKILL signal.

44PID 1 signal 보호

PID 1 의 또 한 가지 특수 처리가 있다. PID 1 은 자신이 핸들러를 등록하지 않은 signal 을 받지 않는다. pid_namespaces(7) :

Only signals for which the "init" process has established a signal handler can be sent to the "init" process by other members of the PID namespace.

이게 무슨 뜻인가? 일반 task 라면 SIGTERM 을 받으면 SIG_DFL (= 종료) 동작이 적용된다. 하지만 PID 1 이 SIGTERM 핸들러를 등록하지 않은 상태에서 누가 SIGTERM 을 보내면 — 그 signal 은 무시된다 (drop). 이게 "실수로 PID 1 죽이는 것" 을 막는 추가 방어선이다.

이 동작의 정확한 코드는 master kernel/signal.c 의 sig_task_ignored 함수에 있다.

static bool sig_task_ignored(struct task_struct *t, int sig, bool force)
{
    void __user *handler;

    handler = sig_handler(t, sig);

    /* SIGKILL and SIGSTOP may not be sent to the global init */
    if (unlikely(is_global_init(t) && sig_kernel_only(sig)))
        return true;

    if (unlikely(t->signal->flags & SIGNAL_UNKILLABLE) &&
        handler == SIG_DFL && !(force && sig_kernel_only(sig)))
        return true;

    /* Only allow kernel generated signals to this kthread */
    if (unlikely((t->flags & PF_KTHREAD) &&
                 (handler == SIG_KTHREAD_KERNEL) && !force))
        return true;

    return sig_handler_ignored(handler, sig);
}

이 함수에는 PID 1 보호와 관련된 별개의 두 메커니즘이 들어 있다.

메커니즘 1 — is_global_init && sig_kernel_only (host PID 1 의 SIGKILL/SIGSTOP 차단). sig_kernel_only(sig) 는 SIGKILL 또는 SIGSTOP 을 의미한다 (master include/linux/signal.h). master 주석이 명료하다 — "SIGKILL and SIGSTOP may not be sent to the global init". 즉 host 의 PID 1 (init namespace 의 PID 1) 은 SIGKILL 과 SIGSTOP 마저도 무시된다. 이는 "정의상 catch 불가능한 signal" 인 SIGKILL/SIGSTOP 의 일반 규칙에 대한 host PID 1 한정 예외다.

단 한 가지 주의 — 이 차단은 같은 namespace 안에서 보낼 때의 동작이다. ancestor namespace 에서 보내면 (host PID 1 의 경우 ancestor 가 없으므로 해당 없음, container PID 1 의 경우 host 에서 보낼 때) pid_namespaces(7) 에 따르면 강제 전달된다.

메커니즘 2 — SIGNAL_UNKILLABLE flag. 두 번째 if 문은 t->signal->flags & SIGNAL_UNKILLABLE 이 켜진 task 가 핸들러 없이 (SIG_DFL) signal 을 받으면 모두 무시한다. 이 flag 는 host PID 1 과 container PID 1 모두에 켜져 있다 (fork.c 의 copy_signal 에서 is_child_reaper 인 경우 자동으로 켜짐). 그래서 SIGTERM 처럼 catch 가능한 일반 signal 도 핸들러가 등록되지 않은 PID 1 에 보내면 그냥 drop 된다.

두 메커니즘을 표로 정리하면 다음과 같다.

보낸 signal host PID 1 (init ns) container PID 1 (자기 ns 내부 → 자기) container PID 1 (host → 자기)
SIGTERM (handler 없음) drop (UNKILLABLE) drop (UNKILLABLE) drop (UNKILLABLE)
SIGTERM (handler 있음) handler 호출 handler 호출 handler 호출
SIGKILL drop (is_global_init) drop (UNKILLABLE) 강제 전달 → 종료
SIGSTOP drop (is_global_init) drop (UNKILLABLE) 강제 전달 → stop

이 보호 덕분에 kill -TERM 1 을 잘못 입력해도 host 시스템이 즉시 죽지는 않는다. 단 PID 1 자체에 버그가 있어 segfault 등으로 죽으면 그건 막을 수 없다 — 그 경우 §43 의 panic 검사가 발동한다.

45CLONE_NEWPID — container 의 PID 1 만들기

container 가 자기만의 PID 공간을 갖는 메커니즘은 CLONE_NEWPID flag 다. clone() 시스템 콜에 이 flag 를 주면 새 task 는 새 PID namespace 안에서 만들어지고, 그 namespace 에서 PID 1 을 받는다.

# host 에서
$ ps -ef | grep nginx
root  12345  12340  ...  nginx: master process

# container 안에서 (같은 nginx 프로세스를 봄)
$ docker exec mycontainer ps -ef
PID  PPID  CMD
  1     0  nginx: master process
  ...

같은 task_struct 가 host 에서는 PID 12345, container 안에서는 PID 1 으로 보인다.

shell 에서 직접 만들어볼 수도 있다.

$ sudo unshare --pid --fork --mount-proc /bin/bash
# (새 namespace 안에서)
$ echo $$
1
$ ps
  PID TTY          TIME CMD
    1 pts/0    00:00:00 bash
   ...

--pid 가 새 PID namespace 를, --fork 가 새 task 를 만들어 그 안에 진입시킨다. --mount-proc 는 새 procfs 를 마운트해서 ps 가 새 namespace 의 PID 들만 보도록 한다.

C 코드로는 다음과 같다.

#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>

static int child_main(void *arg) {
    /* 이 함수는 새 PID namespace 안에서 PID 1 으로 실행 */
    printf("child: my pid = %d\n", getpid()); /* prints 1 */
    /* 이 시점에 /proc 도 새로 mount 해야 ps 가 정상 동작 */
    execlp("/bin/sh", "sh", NULL);
    return 0;
}

int main(void) {
    char stack[1024*1024];
    pid_t pid = clone(child_main,
                      stack + sizeof(stack),
                      CLONE_NEWPID | SIGCHLD,
                      NULL);
    printf("parent: child host pid = %d\n", pid); /* host PID, 보통 1 아님 */
    waitpid(pid, NULL, 0);
    return 0;
}

커널 내부에서는 kernel/fork.c 의 copy_process 안에서 CLONE_NEWPID flag 가 처리된다. copy_pid_ns 가 호출되어 부모의 nsproxy->pid_ns_for_children 기반으로 새 pid_namespace 를 생성하고, level = parent->level + 1 로 깊이를 한 단계 늘린다. alloc_pid 가 모든 level 에서 PID 를 할당한다 — level 0 (init_pid_ns) 에서 host 가 볼 PID, level 1 에서 container 가 볼 PID (= 1, 첫 task 이므로). 첫 task 가 그 namespace 의 child_reaper 로 등록된다.

46/proc/<pid>/status 의 NSpid

같은 task 의 모든 namespace 별 PID 를 한 번에 보고 싶다면 /proc/<pid>/status 를 보면 된다.

# host 에서, container 안 nginx (host PID 12345) 의 namespace 정보 보기
$ cat /proc/12345/status | grep -E "^(Pid|NSpid)"
Pid:    12345
NSpid:  12345   1

NSpid: 는 모든 level 의 PID 를 표시한다. 첫 항목이 init_pid_ns 의 PID, 마지막이 가장 안쪽 namespace 의 PID 다. 이 값은 정확히 struct pid::numbers[] 의 내용이다.

47namespace 계층의 제약 사항

PID namespace 에 관한 몇 가지 중요한 제약이 있다. pid_namespaces(7) 인용:

A process's PID namespace membership is determined when the process is created and cannot be changed thereafter.

즉 task 는 한 번 만들어지면 자기 PID namespace 를 영원히 가진다. setns(2) 로 다른 process 의 PID namespace 안에 들어가도, 그 task 자체의 namespace 는 안 바뀐다. 단지 이후 fork 한 자식만 새 namespace 에 속한다 (pid_ns_for_children 메커니즘).

또한 CLONE_NEWPID 와 CLONE_THREAD 는 결합 불가다. thread 는 같은 PID namespace 여야 한다 — 같은 process 의 thread 들이 서로를 PID 로 식별할 수 있어야 하므로.

namespace 의 깊이도 제한된다. master include/linux/ns_common.h 의 다음 매크로가 한계를 정의한다.

/* MAX_NS_LEVEL is needed for limiting size of 'struct pid' */
#define MAX_NS_LEVEL 32

pid_namespaces(7) 의 설명: "Since Linux 3.7, the kernel limits the maximum nesting depth for PID namespaces to 32." create_pid_namespace 가 새 namespace 생성을 요청받으면 if (level > MAX_NS_LEVEL) goto out 으로 거부한다. 이 제한이 있는 이유는 두 가지다.

첫째, struct pid 의 numbers[] 가 namespace 깊이만큼 자라는 flexible array 이므로, 깊이가 무제한이면 한 struct pid 가 page 크기를 초과할 수 있다. 32 깊이로 제한해야 한 page 안에 들어간다.

둘째, namespace 가 깊을수록 find_vpid 같은 함수의 비용이 높아진다. namespace fork bomb 으로 깊이를 늘려가는 공격이 시스템 응답성을 망가뜨릴 수 있어 차단된 것이다.

            → container 내부 모든 task 에 SIGKILL
            → container 종료 (host 는 영향 없음)

이 한 도식이 Linux 의 PID 시스템을 부팅부터 container 까지 일관된 그림으로 보여준다. struct pid 의 다중 numbers, pid_namespace 의 계층 구조, child_reaper 의 책임 — 이 모든 것이 하나의 일관된 모델 안에서 동작한다.

50Stage 5 정리

Stage 5 까지 오면 container 의 비밀이 풀린다. container 안의 PID 1 은 host 의 PID 1 과 완전히 다른 task 이지만, 자료구조 차원에서는 같은 메커니즘 (struct pid::numbers[], pid_namespace::child_reaper) 위에서 동작한다.

PID namespace 는 nested 구조이며, 각 level 마다 독립적인 PID 공간을 갖는다. 같은 task 가 여러 namespace 에서 다른 정수로 보이며, 이는 struct pid::numbers[] 가 가변 길이 배열인 직접적 이유다.

PID 1 의 책임은 두 가지로 구분된다. 첫째, 고아 프로세스의 reaper — 이건 host 와 container 모두 공통이다. 둘째, 자기 자신의 죽음 처리 — host PID 1 의 죽음은 panic 이고, container PID 1 의 죽음은 zap_pid_ns_processes 발동으로 namespace 전체가 SIGKILL 받는다.

is_global_init 과 is_child_reaper 의 분리가 이 두 책임의 차이를 정확히 표현한다. host PID 1 은 양쪽 모두 true 이지만, container PID 1 은 is_child_reaper 만 true 다.

다음 Stage 에서는 지금까지 다루지 않은 깊은 주제들 — initramfs 와 switch_root, /proc 의 동적 dentry, __init 메모리 회수 — 을 정리한다.

Stage 6깊은 주제 — initramfs, /proc 동작, __init 메모리 회수

지금까지의 Stage 들이 부팅 흐름과 PID 시스템의 큰 줄기를 다뤘다면, Stage 6 은 그 사이사이에 있는 디테일들 — initramfs 가 왜 필요한지, /proc/<pid>/ 디렉토리는 정확히 언제 만들어지는지, __init 으로 표시된 메모리는 부팅 후 어떻게 회수되는지 — 을 정리한다. 이 디테일들은 핵심 흐름을 따라가는 데 필수는 아니지만, 시스템의 동작을 더 깊이 이해하는 데 큰 도움이 된다.

51initramfs 의 역할 — switch_root 메커니즘

Stage 2 와 Stage 3 에서 kernel_init 이 /sbin/init 를 execve 한다고 했다. 그런데 한 가지 질문이 남는다. /sbin/init 가 들어 있는 root 파일시스템은 어떻게 mount 되어 있는 상태인가? 부팅 직후의 커널은 root 파일시스템이 어디에 있는지 모르는 상태인데?

답은 initramfs 라는 임시 root 파일시스템에 있다. 부트로더가 커널과 함께 로드해주는 작은 ramdisk 이미지로, 그 안에 root device 를 찾고 mount 하는 데 필요한 도구들 (블록 드라이버, udev, mount 명령 등) 이 들어 있다. 부팅 흐름은 다음과 같다.

[Kernel boot]
    ↓
[start_kernel → rest_init → kernel_init]
    ↓
ramdisk_execute_command = /init        ← initramfs 안의 /init 스크립트
    ↓
PID 1 = /init (임시)
    ↓
/init 작업:
  ├─ 필수 모듈 로드 (블록/파일시스템 드라이버)
  ├─ udev 시작
  ├─ root device 탐색 (UUID, LABEL)
  ├─ root 파일시스템 mount → /sysroot or /new_root
  └─ exec switch_root /new_root /sbin/init
                              ↓
                        switch_root 동작:
                          ├─ initramfs 메모리 free
                          ├─ pivot_root(2) /new_root
                          └─ exec /sbin/init  ← 진짜 PID 1
                                  ↓
                              systemd (Ubuntu)

이 흐름의 핵심은 PID 1 이 두 번 바뀐다는 것이다. 처음에는 initramfs 의 /init 스크립트가 PID 1 으로 실행되어 root 파일시스템을 찾고 mount 한다. 그 다음 switch_root 라는 명령 (또는 systemd 의 경우 그에 상응하는 동작) 이 진짜 root 파일시스템으로 root 를 전환하고, 거기에 있는 /sbin/init 를 execve 한다. 이 순간 task_struct 자체는 같지만 (PID 1 그대로) 실행되는 코드가 통째로 바뀐다.

switch_root 의 동작은 다음 세 단계다. 먼저 initramfs 가 차지하던 RAM 을 free 한다 (이미 root 가 다른 파일시스템으로 옮겨졌으므로 더 이상 필요 없다). 다음 pivot_root(2) 시스템 콜로 새 root 를 마운트한다. 마지막으로 exec /sbin/init 으로 진짜 init 을 실행한다.

이렇게 두 단계로 부팅하는 이유는 모듈러리티 때문이다. 커널 자체에 모든 블록 드라이버와 파일시스템 드라이버를 정적으로 빌드해 넣으면 커널이 비대해지고, 새 하드웨어를 지원할 때마다 커널을 다시 빌드해야 한다. initramfs 를 쓰면 커널은 작게 유지하고, 필요한 모듈은 ramdisk 안에서 동적으로 로드할 수 있다.

52/proc/<pid>/ 디렉토리는 동적으로 만들어진다

Linux 에서 process 정보를 조회할 때 /proc/<pid>/ 디렉토리를 본다. 그런데 이 디렉토리는 task 가 만들어질 때 미리 생성되는 게 아니다. 흔한 오해 중 하나다.

procfs 는 가상 파일시스템이고, lookup 시점에 동적으로 dentry 를 만든다. 즉 ls /proc 또는 cat /proc/12345/status 를 실행하는 그 순간에 procfs 가 task list 를 보고 적절한 dentry/inode 를 만들어낸다.

$ ls /proc/12345
   │
   ▼
VFS lookup("12345" in /proc)
   │
   ▼
proc_root_lookup()            (fs/proc/root.c)
   │  첫 글자가 '0'~'9' 이면 → proc_pid_lookup
   ▼
proc_pid_lookup()             (fs/proc/base.c)
   ├─ 문자열 → pid_t 변환 (name_to_int)
   ├─ find_pid_ns(tgid, ns)   — IDR 에서 struct pid 찾기
   │   └─ 없으면 → -ENOENT
   ├─ pid_task(pid, PIDTYPE_PID) — task_struct 찾기
   └─ proc_pid_instantiate()  — dentry + inode 동적 생성

/proc 자체는 PID 디렉토리 목록을 미리 갖고 있지 않다. readdir(/proc) 호출 시에만 proc_pid_readdir 가 next_tgid → find_ge_pid(iter.tgid, ns) 로 IDR 을 순회하며 PID 들을 동적으로 enumerate 한다.

이 설계가 갖는 결과적 효과는 다음과 같다. 새 task 가 fork 되면 별도 등록 작업 없이 즉시 /proc/<new_pid>/ 접근이 가능하다 (alloc_pid 가 IDR 에 새 entry 를 만들었으므로 find_pid_ns 가 찾을 수 있다). task 가 죽으면 별도 삭제 작업 없이 다음 lookup 부터 ENOENT 가 반환된다.

PID 0 이 /proc 에 안 보이는 진짜 이유. 흔히 "정책적 거부" 라고 설명하지만, master 코드를 따라가 보면 더 정확한 답이 나온다. init_task 는 init/init_task.c 에서 정적으로 선언된 struct task_struct 인스턴스이며, alloc_pid 의 IDR allocator 를 거치지 않는다. master kernel/pid.c 에서 init namespace 의 IDR 은 다음과 같이 초기화된다.

struct pid_namespace init_pid_ns = {
    .idr             = IDR_INIT(init_pid_ns.idr),    /* ← 빈 IDR */
    .pid_allocated   = PIDNS_ADDING,
    .level           = 0,
    .child_reaper    = &init_task,
    /* ... */
    .pid_max         = PID_MAX_DEFAULT,
};

그리고 PID 0 자신을 표현하는 init_struct_pid 는 별도 정적 인스턴스로 존재한다.

struct pid init_struct_pid = {
    .count   = REFCOUNT_INIT(1),
    .level   = 0,
    .numbers = { { .nr = 0, .ns = &init_pid_ns, } },
};

이 init_struct_pid 는 IDR 에 등록되지 않았다. 따라서 /proc/0 lookup 시 proc_pid_lookup 이 find_pid_ns(0, &init_pid_ns) 를 호출해도 IDR 검색 결과가 NULL → -ENOENT 가 반환된다. 마찬가지로 proc_pid_readdir 가 find_ge_pid(0, ns) 로 enumerate 를 시작해도 IDR 에 PID 0 entry 가 없으니 자연스럽게 첫 번째 결과가 PID 1 부터 시작한다.

즉 PID 0 거부 코드가 따로 있는 게 아니라, init_task 가 IDR 에 등록된 적이 없다는 정적 사실의 자연스러운 귀결이다. SMP 시스템에서 다른 CPU 의 idle task 들도 같은 이유로 보이지 않는다 — fork_idle 로 만들어지지만 IDR 에는 들어가지 않는다.

namespace 별 procfs 동작도 이 메커니즘 위에서 자연스럽게 구현된다. pid_namespaces(7) :

A /proc filesystem shows (in the /proc/[pid] directories) only processes visible in the PID namespace of the process that performed the mount.

container 시작 시 mount -t proc proc /proc 를 새로 해야 container 의 PID 들만 보인다. 안 하면 host /proc 가 그대로 보여 일관성이 깨진다. 그래서 unshare --pid --mount-proc 옵션이 필요한 것이다.

53__init 메모리 회수

커널 코드를 보면 함수 정의 앞에 __init 어트리뷰트가 붙어 있는 경우가 많다. start_kernel 자체도 그렇다.

asmlinkage __visible __init __no_sanitize_address __noreturn __no_stack_protector
void start_kernel(void)

이 __init 어트리뷰트는 무엇인가? include/linux/init.h 에 정의되어 있고, 의미는 다음과 같다. 이 함수의 코드가 .init.text 라는 별도 섹션에 배치되며, 부팅 완료 후 회수될 메모리 영역에 들어간다. 마찬가지로 __initdata 는 데이터를 .init.data 섹션에 배치한다.

[Kernel image layout]
  .text          (일반 코드 — 부팅 후에도 살아있음)
  .data
  .rodata
  .init.text     ← __init 함수들 (start_kernel 등)
  .init.data     ← __initdata 변수들
  .bss

부팅이 완료되면 kernel_init 함수 안에서 free_initmem() 가 호출되어 .init.* 섹션의 메모리를 모두 회수한다.

[Boot 진행]
  start_kernel ────────► .init.text 사용
        │
        ▼
  rest_init ────────► PID 1 / PID 2 / idle 전환
        │
        ▼
  kernel_init() {
    ...
    free_initmem();   ← .init.* 섹션 메모리 회수
    ...
  }
        │
        ▼
  printk: "Freeing unused kernel memory: NK"
        │
        ▼
  RAM 절약 (수 MB)

[Boot 완료 후]
  start_kernel 등 호출 시도 → SEGV / panic
  (메모리 회수됨, 더 이상 호출 불가)

부팅 시 콘솔에 출력되는 "Freeing unused kernel memory: 2048K" 같은 메시지가 바로 이 회수의 결과다. 보통 수 MB 정도가 절약된다.

이 메커니즘 덕분에 부팅 시점에만 필요한 코드 (CPU 검출, 페이지 테이블 셋업, 드라이버 초기화, 명령행 parsing 등) 를 마음껏 작성해도 런타임 메모리에 부담을 주지 않는다. 단, 부팅 후 이 함수를 호출하려고 시도하면 segfault 또는 panic 이 발생한다 — 메모리가 사라졌으니 당연하다.

asmlinkage 와 __visible 같은 다른 어트리뷰트도 이 헤더에 정의되어 있다. asmlinkage 는 ABI 명시 (x86 32-bit 에서 stack 으로만 인자 전달, x86_64 에서는 no-op), __visible 은 LTO 시 symbol 가시화다.

54PID 0 의 task_struct 와 SMP 처리

Stage 3 에서 PID 0 의 task_struct 가 정적으로 선언되어 있다고 했다. 이를 더 정확히 들여다보자.

init/init_task.c 에 단 하나의 init_task 가 정적으로 선언되어 있다.

init_task (정적 선언, init/init_task.c)
 ├─ pid = 0
 ├─ comm = "swapper/0"
 ├─ sched_class = idle_sched_class
 └─ context: boot CPU's idle (do_idle loop)

fork_idle(cpu)  (kernel/fork.c, SMP 다른 CPU 시작 시)
 └─ creates task_struct for each CPU
     ├─ pid = 0  (special, all share)
     ├─ comm = "swapper/N"  (TASK_COMM_LEN = 16 제한)
     └─ context: CPU N's idle

kernel_thread(kernel_init, ...) → PID 1
 ├─ comm = "kernel_init" → execve 후 systemd 가 prctl PR_SET_NAME 로 변경
 └─ execve("/sbin/init") → userspace transition

kernel_thread(kthreadd, ...) → PID 2
 ├─ comm = "kthreadd"
 └─ kernel mode forever (kernel-only thread, never enters userspace)

부팅 초기에는 boot CPU 만 동작하고 다른 CPU 들은 꺼져 있다. SMP 초기화 (smp_init) 단계에서 다른 CPU 들이 하나씩 켜지는데, 그때마다 fork_idle(cpu) 가 호출되어 각 CPU 의 idle task 가 생성된다. 모두 PID 0 으로 표시되며 comm 만 swapper/N 으로 다르게 설정된다.

master 의 정확한 흐름은 kernel/smpboot.c 의 idle_threads_init 가 모든 가능한 CPU 에 대해 idle_init(cpu) 을 호출하고, idle_init 안에서 per-CPU 변수 idle_threads 에 task 가 없으면 fork_idle(cpu) 로 새로 만드는 식이다.

/* kernel/smpboot.c master */
static DEFINE_PER_CPU(struct task_struct *, idle_threads);

void __init idle_thread_set_boot_cpu(void)
{
    per_cpu(idle_threads, smp_processor_id()) = current;
    /* boot CPU 의 idle 은 새로 만들지 않고 init_task 를 그대로 등록 */
}

static __always_inline void idle_init(unsigned int cpu)
{
    struct task_struct *tsk = per_cpu(idle_threads, cpu);
    if (!tsk) {
        tsk = fork_idle(cpu);
        if (IS_ERR(tsk))
            pr_err("SMP: fork_idle() failed for CPU %u\n", cpu);
        else
            per_cpu(idle_threads, cpu) = tsk;
    }
}

void __init idle_threads_init(void)
{
    unsigned int cpu, boot_cpu = smp_processor_id();
    for_each_possible_cpu(cpu) {
        if (cpu != boot_cpu)
            idle_init(cpu);
    }
}

이 코드의 흐름이 명료하다. boot CPU 의 idle 은 init_task 그대로 — 정적 선언이므로 다시 fork 할 필요가 없다. 다른 CPU 들은 fork_idle(cpu) 로 새 task_struct 를 만들고 per-CPU 변수에 저장한다. CPU hotplug 의 경우에도 같은 task 를 재사용할 수 있도록 per-CPU 캐시 구조다.

이 디자인 덕분에 어떤 CPU 에서 current 매크로를 호출해도 (= 현재 task_struct 를 얻어도) 항상 유효한 task 가 반환된다. 비록 그 task 가 PID 0 (= idle) 이라도.

56자주 받는 질문

Q1. main 함수가 있는가?

A: 없다. 커널은 int main() 진입점이 아닌 start_kernel() 진입점을 사용한다. C 표준의 main 은 libc + crt0 가 필요하나 커널은 둘 다 없는 freestanding 환경이다. 부트로더가 직접 phys_startup_64 (= ELF entry symbol) 로 jump 한다.

Q2. PID 0 은 process 인가?

A: 커널의 task_struct 로는 존재하지만, userspace 관점에서는 process 가 아니다. /proc/0 도 없다. CPU idle 만 담당하는 특수 kernel thread 로 보면 된다.

Q3. PID 0 이 여러 개인가?

A: task_struct 인스턴스로는 CPU 수만큼 존재한다. boot CPU = init_task (정적 선언), 나머지 CPU = fork_idle() 로 생성. 모두 PID 0 으로 표시되며 — Linux 의 특수 처리 — comm 으로 구분한다 (swapper/0, swapper/1, ...).

Q4. systemd 없이 부팅 가능한가?

A: 가능하다. kernel command line 에 init=/bin/bash 를 추가하면 bash 가 PID 1 으로 실행된다 (rescue / single-user 모드). 단 service 관리 등 systemd 의 기능은 없다.

Q5. PID 1 이 죽으면?

A: kernel panic. PID 1 의 종료는 do_exit() 안의 panic("Attempted to kill init!") 호출로 트리거되며 (Stage 5 §43 참조), 운영 시스템 전체가 정지한다.

Q6. ps -p 0 출력은?

A: 출력 없음. /proc/0 이 없으므로. 단 cat /proc/sched_debug | grep swapper 또는 perf top 으로 idle task 활동을 확인할 수 있다.

Q7. kernel 에 _start 함수가 있나?

A: 없다. 일반 프로그램의 _start 는 glibc 의 crt1.o 가 제공하는데, 커널은 -nostartfiles 옵션으로 빌드되어 그것을 안 쓴다. 대신 arch/x86/kernel/head_64.S 의 startup_64 가 그 역할을 하며, linker script (vmlinux.lds.S) 의 ENTRY(phys_startup_64) 지시자가 이를 ELF entry 로 등록한다.

Q8. main 안 쓰면 컴파일러가 에러 안 내나?

A: hosted 환경에서는 main 이 없으면 보통 linker error 가 난다 (crt1.o 가 main 을 참조하므로). freestanding 환경에서는 그런 참조 자체가 없다. -ffreestanding -nostdlib -nostartfiles 옵션이 이를 구현한다.

Q9. container 안 PID 1 이 죽으면 host 도 영향받나?

A: 아니다. is_global_init 가 false 이므로 host panic 조건에 안 걸린다. 대신 zap_pid_ns_processes 가 발동되어 그 namespace 안의 모든 task 가 SIGKILL 을 받고 namespace 가 정리된다. host 의 systemd 와 다른 container 들은 영향 없다.

Q10. getpid() 와 gettid() 의 차이는?

A: getpid() 는 커널 내부의 TGID (= thread group leader 의 PID) 를 반환하고, gettid() 는 커널 내부의 PID (= 개별 task 의 식별자) 를 반환한다. 명명이 정반대로 된 이유는 역사적 — POSIX 가 "process" 와 "thread" 라는 추상화로 정의했고, Linux 가 "task" 라는 더 일반적인 개념으로 구현했기 때문이다. multi-thread process 에서 모든 thread 의 getpid() 는 같지만 gettid() 는 다르다.

57pidfd — PID 재사용 race 의 master 해결책

이 문서 전반에 걸쳐 "PID 는 정수일 뿐이며, wrap 후에는 다른 task 를 가리킨다" 는 사실을 여러 번 언급했다 (§31, §37). 이게 실무에서 일으키는 문제가 있다.

타임라인 — 고전적인 race condition:

t0: A 가 자식 B (PID 12345) 를 fork
t1: A 가 다른 일을 잠깐 처리
t2: B 가 종료 → zombie → reap → PID 12345 해제
t3: 무관한 task X 가 fork → PID 12345 재할당 (wrap)
t4: A 가 "내 자식 B 에게 신호를 보내자" → kill(12345, SIGTERM)
                                            ↓
                                        X 에게 전달됨!

이 race 는 PID 가 wrap 될 수 있다는 사실에서 본질적으로 비롯된다. PID_MAX 를 4M 까지 늘려도 (§37) 줄어들 뿐 사라지지 않는다. master 가 도입한 정답은 pidfd (PID file descriptor) 다.

pidfd 의 기본 개념

pidfd 는 단순히 file descriptor 다. int 정수지만 fd table 의 한 슬롯을 차지하고, 그 fd 가 가리키는 struct file 이 struct pid * 를 안에 들고 있다. 핵심은 pidfd 가 살아 있는 한 struct pid 의 refcount 가 유지된다 는 점이다. task 가 죽고 PID 정수가 재사용되어도, pidfd 가 가리키는 struct pid 객체는 그대로 살아 있다. 그래서 pidfd 는 task 에 대한 안정된 reference 가 된다.

                  pidfd (fd 5)
                    │
                    ▼
              struct file
                    │
                    ▼
              struct pid {
                count: 2     ← pidfd 가 잡고 있음
                tasks[...]
                numbers[]
              }
                    │
              ┌─────┴─────┐
              ▼ (task 살아있음) ▼ (task 죽음)
          struct task     NULL
          (alive)         (zombie 또는 reaped)

task 가 죽은 뒤에도 struct pid 는 살아 있다. pidfd 를 통해 task 가 죽었는지 검사할 수 있고, 죽었으면 정확히 ESRCH 를 받는다 — 다른 task 에 시그널이 잘못 가는 일이 없다.

세 가지 획득 경로

pidfd 를 얻는 방법은 master 에 세 가지 있다.

/* 1. 직접 open (5.3+) */
int pidfd = syscall(SYS_pidfd_open, pid, 0);

/* 2. clone3 의 CLONE_PIDFD (5.2+) — race-free */
struct clone_args args = {
    .flags = CLONE_PIDFD,
    .pidfd = (uint64_t)(uintptr_t)&pidfd,
};
pid_t child = syscall(SYS_clone3, &args, sizeof(args));

/* 3. /proc/<pid> 디렉토리 open (5.1+) */
int pidfd = open("/proc/1234", O_DIRECTORY | O_CLOEXEC);

이 중 CLONE_PIDFD 가 진정한 race-free 다. clone3 가 atomic 하게 새 task 를 만들고 pidfd 를 반환하므로, PID 정수가 노출되기 전에 이미 pidfd 가 fd table 에 있다. pidfd_open 은 PID 를 입력으로 받으므로 미세한 race 가 있을 수 있다 — 그 PID 가 이미 wrap 된 상태로 호출될 가능성. 그래서 자식을 만들 때는 가능하면 CLONE_PIDFD 를 쓰는 게 권장된다.

pidfd 의 세 가지 용도

#### (1) pidfd_send_signal — race-free signal

int pidfd_send_signal(int pidfd, int sig, siginfo_t *info, unsigned int flags);

pidfd_send_signal(2) 의 man page 가 다음 사실을 직접 말한다.

The pidfd_send_signal() system call allows the avoidance of race conditions that occur when using traditional interfaces (such as kill(2)) to signal a process. The problem is that the traditional interfaces specify the target process via a process ID (PID), with the result that the sender may accidentally send a signal to the wrong process if the originally intended target process has terminated and its PID has been recycled for another process.

target 이 이미 죽었다면 ESRCH 를 받는다. kill(pid, ...) 은 이걸 보장하지 못한다.

#### (2) poll(pidfd) — task 종료 알림

struct pollfd pfd = { .fd = pidfd, .events = POLLIN };
poll(&pfd, 1, -1);   /* task 가 종료되면 POLLIN 이 켜짐 */

부모-자식 관계가 아니어도 임의의 task 의 종료를 감지할 수 있다. 이건 옛 SIGCHLD 만으로는 불가능했다 — SIGCHLD 는 자식에 한정되고 시그널 기반이라 다루기 어렵다. pidfd 는 일반적인 fd 라 epoll, select, io_uring 어디에든 끼울 수 있다.

내부적으로 task 가 종료되면 master kernel/signal.c::do_notify_pidfd 가 호출되어 그 task 의 struct pid::wait_pidfd waitqueue 를 깨운다 — §32 에서 봤던 wait_queue_head_t wait_pidfd 필드가 정확히 이 용도다.

/* kernel/signal.c master */
void do_notify_pidfd(struct task_struct *task)
{
    struct pid *pid = task_pid(task);

    WARN_ON(task->exit_state == 0);

    __wake_up(&pid->wait_pidfd, TASK_NORMAL, 0,
              poll_to_key(EPOLLIN | EPOLLRDNORM));
}

poll_to_key 로 EPOLLIN | EPOLLRDNORM event 를 함께 전달하므로 poll/epoll waiter 가 올바른 event mask 로 깨어난다. 추가로 release_task 가 호출하는 __unhash_process (kernel/exit.c) 안에서도 wake_up_all(&pid->wait_pidfd) 가 호출되어, task 가 완전히 reap 된 시점에서도 waiter 가 깨어난다.

#### (3) waitid(P_PIDFD, ...) — race-free wait

siginfo_t info;
waitid(P_PIDFD, pidfd, &info, WEXITED);

waitid 의 P_PIDFD 옵션이 pidfd 기반 wait 를 가능하게 한다. 부모-자식 관계인 경우에 한정되지만, race 없이 깔끔하게 자식의 종료 정보를 회수할 수 있다.

실무 사용처

container runtime (runc, crun), service manager (systemd), pod manager (podman) 들이 모두 master pidfd 를 활용한다. 특히 systemd 는 cgroup 와 pidfd 를 결합해서 service 의 main process 를 race-free 로 추적한다 — "내가 시작한 process 가 정확히 이것이며, 그것이 죽었는지" 를 PID 정수에 의존하지 않고 안정적으로 알 수 있다.

이 모든 메커니즘이 §32 에서 짧게 지나간 wait_queue_head_t wait_pidfd; 한 줄에 응축돼 있다. master 의 단순한 한 필드가 PID 시대의 race condition 을 정리한 셈이다.

58cgroup 과 task 의 관계 — namespace 와는 별개 축

이 문서가 다룬 PID 와 namespace 는 task 의 "정체성" 차원이다. 그러나 task 에는 또 하나의 차원이 있다 — 자원 분류 (resource accounting and limiting). 이것이 cgroup (control group) 이다. cgroup 은 namespace 와 비슷한 시기에 도입되었고 container 의 다른 한 축을 담당하지만, 메커니즘은 완전히 별개다.

두 차원의 분리

task 는 두 개의 직교 차원을 가진다:

차원 1: namespace — "무엇이 보이는가" (visibility)
  ├─ pid namespace      : PID 격리
  ├─ mnt namespace      : mount 격리
  ├─ net namespace      : network 격리
  ├─ uts namespace      : hostname 격리
  └─ ...

차원 2: cgroup — "얼마나 쓸 수 있는가" (resource accounting / limiting)
  ├─ cpu      : CPU time 제한
  ├─ memory   : RAM 사용 제한
  ├─ io       : block I/O 제한
  ├─ pids     : 자식 task 수 제한
  └─ ...

container = 두 차원의 조합
  • namespace 가 격리를 제공
  • cgroup 이 자원 제한을 제공

container runtime (Docker, runc, podman) 가 컨테이너를 만들 때 두 차원을 모두 설정하지만, 커널 차원에서는 완전히 별개의 자료구조와 hook 으로 처리된다.

task_struct 의 cgroup 필드

master include/linux/sched.h 의 task_struct 안에 다음 필드가 있다.

struct task_struct {
    /* ... */

#ifdef CONFIG_CGROUPS
    /* Control Group info protected by css_set_lock: */
    struct css_set __rcu        *cgroups;
    /* cg_list protected by css_set_lock and tsk->alloc_lock: */
    struct list_head            cg_list;
#endif

    /* ... */
};

task->cgroups 는 struct css_set * 로 직접 cgroup 을 가리키지 않는다. 이 우회 구조가 master cgroup 설계의 핵심이다.

css_set — cgroup subsystem 의 묶음

master include/linux/cgroup-defs.h 가 다음 구조를 정의한다 (필드 순서 master 그대로).

struct css_set {
    /*
     * Set of subsystem states, one for each subsystem. This array is
     * immutable after creation apart from the init_css_set during
     * subsystem registration (at boot time).
     */
    struct cgroup_subsys_state *subsys[CGROUP_SUBSYS_COUNT];

    /* Reference count */
    refcount_t refcount;

    /* the default cgroup associated with this css_set */
    struct cgroup *dfl_cgrp;

    /* internal task count, protected by css_set_lock */
    int nr_tasks;

    /* tasks 와 mg_tasks: 이 css_set 을 쓰는 task 들의 list */
    struct list_head tasks;
    struct list_head mg_tasks;

    /* css_task_iters currently walking this cset */
    struct list_head task_iters;

    /* 같은 hash slot 안의 css_set 들 */
    struct hlist_node hlist;

    /* 이 css_set 이 참조하는 cgroup 들과의 link */
    struct list_head cgrp_links;

    /* ... migration 관련 필드, RCU 필드 등 ... */
};

master Documentation/admin-guide/cgroup-v1/cgroups.rst 의 정확한 표현:

A css_set contains a set of reference-counted pointers to cgroup_subsys_state objects, one for each cgroup subsystem registered in the system. There is no direct link from a task to the cgroup of which it's a member in each hierarchy, but this can be determined by following pointers through the cgroup_subsys_state objects.

즉 task → cgroup 직접 링크는 없다. 대신 task → css_set → cgroup_subsys_state[ss_id] → cgroup 의 2 단계 우회 경로다.

왜 css_set 이라는 중간층이 있는가

같은 cgroup 조합을 사용하는 모든 task 가 하나의 css_set 을 공유한다. 시스템에 1000 개의 process 가 모두 (memory=/system.slice/foo, cpu=/system.slice/foo, io=/) 라는 같은 cgroup 조합을 쓴다면, css_set 객체는 단 1 개만 존재하고 1000 개 task 가 그 css_set 을 가리킨다. 메모리 절약이 극적이다.

task A ──┐
task B ──┼──▶ css_set #42 ──▶ subsys[memory] ──▶ /system.slice/foo (memory cgroup)
task C ──┘                ──▶ subsys[cpu]    ──▶ /system.slice/foo (cpu cgroup)
                          ──▶ subsys[io]     ──▶ /                  (io cgroup)
                          ──▶ subsys[pids]   ──▶ /system.slice/foo (pids cgroup)

cgroup 가 다른 새 조합을 쓰는 task 가 등장하면 새 css_set 이 만들어지고, 그 task 의 cgroups 포인터가 새 css_set 을 가리키도록 갱신된다.

task_struct::cg_list 가 그 task 가 속한 css_set 의 tasks 리스트에 연결된다. css_set 입장에서는 "나를 가리키는 task 들의 리스트" 가 되고, task 입장에서는 "내가 어느 css_set 에 속하는지" 를 알 수 있다.

struct css_set #42
    .tasks ───┬─▶ task A.cg_list
              ├─▶ task B.cg_list
              └─▶ task C.cg_list

init_css_set — boot 시점의 css_set

PID 0 (= init_task) 의 cgroup 은 무엇인가? master init/init_task.c 와 kernel/cgroup/cgroup.c 가 정의하는 init_css_set 이 그 답이다.

struct css_set init_css_set = {
    .refcount       = REFCOUNT_INIT(1),
    .dom_cset       = &init_css_set,
    .tasks          = LIST_HEAD_INIT(init_css_set.tasks),
    .mg_tasks       = LIST_HEAD_INIT(init_css_set.mg_tasks),
    /* subsys[] 는 모두 init_cgroup 의 css 를 가리킴 (= root cgroup) */
    /* ... */
};

init_task.cgroups 가 정적으로 &init_css_set 으로 초기화되어 있다. 즉 boot 시점부터 모든 task 는 어떤 cgroup 에 속해 있다 — 처음에는 root cgroup. 이후 systemd 등이 service 를 시작하면서 새 cgroup 으로 옮긴다.

fork 시점의 cgroup 처리

§35 에서 copy_process 의 단계별 흐름을 보면서 cgroup_can_fork 라는 hook 이 있었다. master kernel/fork.c 에서:

/* copy_process 안 */
retval = cgroup_can_fork(p, args);
if (retval)
    goto bad_fork_cancel_cgroup;

/* ... */

/* 새 task 를 task list 에 add 한 뒤 */
cgroup_post_fork(p, args);

cgroup_can_fork 가 "이 task 가 새 자식을 만들어도 되는가" 를 검사한다 — pids.max cgroup 제한이 여기서 적용된다. 통과하면 새 task 가 부모와 같은 css_set 을 공유하고 (= 부모와 같은 cgroup), cgroup_post_fork 가 그 css_set 의 tasks list 에 새 task 의 cg_list 를 연결한다.

clone3(CLONE_INTO_CGROUP) flag (Linux 5.7+) 가 있으면 자식이 부모와 다른 cgroup 으로 바로 들어갈 수 있다. systemd 같은 service manager 가 활용한다.

실무 확인

/proc/<pid>/cgroup 가 한 task 가 어떤 cgroup 에 속해 있는지 보여준다.

$ cat /proc/self/cgroup
0::/user.slice/user-1000.slice/session-3.scope

# 모든 task 의 cgroup 보기
$ cat /proc/1/cgroup
0::/init.scope

# cgroup 안의 task 들 보기 (cgroup v2)
$ cat /sys/fs/cgroup/user.slice/user-1000.slice/session-3.scope/cgroup.procs
1234
1235
1236
...

cgroup v2 통합 hierarchy 에서는 모든 subsystem 이 같은 hierarchy 를 공유하므로 출력이 한 줄이다. v1 에서는 subsystem 별로 여러 줄로 보인다.

namespace 와의 비교

마지막으로 두 차원의 관계를 한 표로 정리한다.

┌──────────────────┬────────────────────┬────────────────────┐
│ 항목              │ namespace          │ cgroup             │
├──────────────────┼────────────────────┼────────────────────┤
│ 목적              │ 격리 (visibility)   │ 제한 (limiting)    │
│ task_struct 필드 │ nsproxy            │ cgroups            │
│ 가리키는 타입     │ struct nsproxy *   │ struct css_set *   │
│ 우회 단계         │ 1단계 (직접)        │ 2단계 (css_set →   │
│                  │                    │  subsys_state →    │
│                  │                    │  cgroup)           │
│ 종류              │ pid/mnt/net/uts/   │ cpu/memory/io/     │
│                  │ ipc/user/cgroup/   │ pids/devices/...   │
│                  │ time               │                    │
│ fork hook        │ copy_namespaces    │ cgroup_can_fork +  │
│                  │                    │ cgroup_post_fork   │
│ 생성 syscall     │ clone(CLONE_NEW*)  │ mkdir on cgroupfs  │
│                  │ unshare(2)         │ + write to         │
│                  │ setns(2)           │ cgroup.procs       │
│ 보는 도구         │ lsns               │ systemctl,         │
│                  │ ip netns           │ cgroup tools       │
└──────────────────┴────────────────────┴────────────────────┘

PID namespace 가 "container 안에서 어떤 PID 가 보이는가" 를 결정한다면, cgroup 은 "container 가 CPU 와 memory 를 얼마나 쓸 수 있는가" 를 결정한다. 두 메커니즘이 완전히 별개의 자료구조 위에서 동작하지만, container runtime 이 두 차원을 함께 설정해서 우리가 아는 "container" 라는 추상을 만들어낸다.

cgroup 자체는 매우 큰 주제이고 (특히 cgroup v2 의 통합 hierarchy 와 unified delegation 등), 이 문서가 다루는 PID 와 namespace 의 학습 흐름을 벗어난다. 다만 task 의 두 직교 차원을 이해하는 데는 이 정도면 충분하다 — task 는 "어디에 보이는가 (namespace)" 와 "얼마나 쓸 수 있는가 (cgroup)" 라는 두 축으로 분류되며, 둘 다 task_struct 의 다른 필드에 연결되어 있다.

59한 줄 정리

지금까지의 모든 내용을 한 단락으로 압축하면 다음과 같다.

Linux 는 freestanding C 환경이라 main() 이라는 이름을 안 쓴다.
커널의 ELF entry 는 startup_64 이고, 부트로더가 직접 그곳으로 jump 한다.
거기서 여러 단계를 거쳐 init/main.c 의 start_kernel() 에 도달한다.
start_kernel 은 PID 0 (swapper) 컨텍스트에서 실행되며,
모든 커널 서브시스템을 초기화한 뒤 rest_init() 를 호출한다.
rest_init 는 PID 1 (kernel_init → systemd) 와 PID 2 (kthreadd) 를 fork 하고,
boot CPU 자신은 idle task 로 영원히 do_idle 루프를 돈다.
PID 1 은 모든 userspace 의 root 가 되어 service 를 관리하고,
PID 2 는 모든 kernel thread 의 root 가 되어 work 를 처리하며,
PID 0 은 다른 할 일이 없을 때 CPU 를 절전 상태로 만든다.
container 의 PID 1 은 새 PID namespace 안에서의 PID 1 일 뿐이며,
host 의 PID 1 과는 task_struct 가 다른 별개의 task 다.
같은 task 도 여러 namespace 에서 다른 PID 정수로 보일 수 있다 —
struct pid::numbers[] 가 그 모든 PID 를 함께 보관한다.
이 모든 메커니즘이 한 일관된 모델 위에서 동작한다.
